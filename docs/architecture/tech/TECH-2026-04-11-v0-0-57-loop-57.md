# v0.0.57 - Loop 57 - 2026-04-11

## 本轮目标
- 把 `shared-channel sync` 从“pending backlog 会在 auto-retry / repair 入口被持续重试”推进到“重复失败请求进入 dead-letter 并退出自动重试范围”的最小可审阅边界

## 本轮实现
- `services/control-plane-api/src/lib.rs`
  - `SocialControlState` 当前新增 `dead_letter_shared_channel_sync_requests`
  - request 连续失败达到固定阈值 `3` 后，会从 `pending_shared_channel_sync_requests` 转入 `dead_letter_shared_channel_sync_requests`
  - dispatch 队列会显式跳过 dead-lettered request
  - `repair-shared-channel-sync` 当前只消费 pending backlog，并回传 `deadLetterBefore / deadLettered / deadLetterAfter`
  - `SocialAggregateCountsResponse` 与 `repair-social-runtime-dir` 当前会显式报告 `deadLetterSharedChannelSyncRequests`
  - snapshot/journal repair 当前会保留 dead-letter backlog
- `services/control-plane-api/tests/social_external_collaboration_test.rs`
  - 新增 dual-service e2e，证明 repeated failure 后 request 会进入 dead-letter；健康 repair 返回 `noop`；后续新的健康 ready-pair 写入只会 materialize 新 actor，而 dead-lettered actor 仍不能读取 shared history

## 风险与未闭环
- 当前 dead-letter 只有隔离语义，没有 requeue / replay route
- 当前没有后台 retry scheduler
- 当前没有 remote republish ownership / lease / SLA，也没有 release-ready exactly-once semantics
- `https://` public runtime target 仍未进入本轮 truth

## 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop57 已把 `shared-channel sync` 补齐到 `durable pending backlog + operator repair + next healthy ready-pair write auto-retry + repeated-failure dead-letter`
- 下一主缺口收敛为 `automatic retry scheduler / dead-letter requeue / remote republish / delivery ownership semantics`
