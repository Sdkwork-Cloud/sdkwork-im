### 2. 已把 PowerShell `open-chat-test` 升级为同入口双模式工
- 更新：`bin/open-chat-test.ps1`
- 当前新增能力
  - `-ScriptedValidation`
  - `-ValidationMessage`
  - `-Json`
  - `Invoke-ScriptedValidation(...)`
- scripted mode 当前会执行：
  - owner `create-conversation`
  - owner `add-member`
  - guest `watch --event-type message.posted --exit-after-events 1`
  - owner `send-message`
  - guest `timeline`
  - 输出 machine-readable JSON 摘要
- 当前 JSON 摘要明确暴露
  - `mode`
  - `conversationId`
  - `ownerUserId`
  - `guestUserId`
  - `validationMessage`
  - `watchFrameTypes`
  - `watchDelivered`
  - `timelineContainsValidationMessage`

### 3. 已把 Bash `open-chat-test` 追平到相scripted surface
- 更新：`bin/open-chat-test.sh`
- 当前 Bash 入口已具备：
  - `--scripted-validation`
  - `--validation-message`
  - `--json`
- 当前脚本逻辑PowerShell 对齐为同一 operator 语义
  - 先建 conversation
  - 再加成员
  - 再跑 guest `watch`
  - 再发 owner 消息
  - 再用 guest `timeline` 做落地确
- 这避免出PowerShell Bash 各自维护不同 smoke 语义

### 4. 已把 Step 12 operator 文档追平scripted validation
- 更新：`docs/部署/CLI聊天验证与兼容矩阵md`

  - scripted mode 当前必须观察的关键帧
    - `realtime.connected`
    - `event.window`
- 这让发布前验证不再依赖隐性知识

### 5. 已修复PowerShell watch 成功判定的真实坑
- 更新：`bin/open-chat-test.ps1`
- 当前处理方式
  - 不再PowerShell `Start-Process` 的空`ExitCode` 当作可靠成功信号
  - 转而以实际解析到的 watch frames 作为交付成功依据
- 这消除了 scripted mode Windows 下的一个伪阳/ 伪阴性风险
