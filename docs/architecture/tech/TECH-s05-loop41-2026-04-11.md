# S05 Loop41补充 - 2026-04-11

## 1. 本轮主任务
- 关闭 `repair-runtime-local.*` 尚未统一接入 standalone social repair CLI 的 operator surface 缺口。

## 2. 已兑现事实
- `repair-runtime-local.ps1` 与 `repair-runtime-local.sh` 不再只执行 generic `repair-runtime-dir`；现在会在 `state/social-commit-journal.json` 存在时自动追加 `control-plane-api repair-social-runtime-dir --runtime-dir <path>`。
- wrapper 保持了原有 generic runtime-dir repair 的前置顺序，避免把 social journal repair 混回 `sdkwork-im-server` 的 generic empty-file repair 逻辑。
- 当 social journal 缺失时，wrapper 维持原有成功路径；当 social journal 存在但 social repair 失败时，wrapper 显式失败并传播退出码，避免 operator 误判本地修复已完成。
- `docs/prompts/反复执行Step指令.md` 被压缩为 10 行 concise prompt，满足 `provider_plugin_docs_test` 的 repeated-step 门禁，同时保留 `step / 波次 / 是否闭环`、`docs/review`、`docs/step`、`docs/架构`、`deployment_profile` 与 `下一轮动作` 关键约束。
- 新增并保持通过：
  - `test_repair_runtime_local_ps1_propagates_social_repair_failure_when_social_journal_exists`
  - `test_repair_runtime_local_sh_invokes_social_repair_after_generic_repair_when_social_journal_exists`
- 为了恢复本轮 `sdkwork-im-server` package regression 的可编译状态，`services/sdkwork-im-cloud-gateway/src/node.rs` 同步补齐了 `conversation_runtime::RuntimeError::InvalidInput` 与 `ConversationBindingNotFound` 的 `ApiError` 映射。
- fresh 包级回归：
  - `cargo test -p sdkwork-im-cloud-gateway --offline --test deployment_profile_test -- --nocapture` = `64 passed`
  - `cargo test -p sdkwork-im-cloud-gateway --offline --tests -- --nocapture` = `passed`

## 3. 仍未关闭
- 仍无真正 `atomic multi-file tx`
- `S05` 仍未具备 `step_closure`
- `repair-runtime-local.* --json` 当前继续保留 generic runtime repair 的 stdout 合同；social repair 作为追加 gate 执行，而非合并成新的 wrapper-level machine-readable envelope

## 4. 当前结论
- Loop41 把 `S05` 从“已有 standalone social repair CLI，但 operator 仍需二次手工调用”推进到“已有统一 wrapper/operator 入口”。`S05` 因 `atomic multi-file tx + step_closure` 仍未兑现，只能继续标记为 `not_closed / local_closure`。
