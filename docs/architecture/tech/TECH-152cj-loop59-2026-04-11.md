# 152CJ Loop59 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“dead-letter requeue 会保留旧失败预算”推进到“operator requeue 会显式重置失败预算，使 request 重新获得最小 pending 生命周期”的恢复语义

## 2. As-Built 事实
- `SocialControlState::requeue_dead_letter_shared_channel_sync_requests()` 当前在 `dead_letter -> pending` 迁移时会把 `failureCount` 重置为 `0`
- 同一条迁移当前仍保留：
  - request payload
  - `lastError`
- 因此 requeue 后的 request：
  - 在状态上是“新的 pending item”
  - 在审计上仍保留上一次失败原因上下文
- `repair_shared_channel_sync(...)` 当前不需要改协议
  - rearmed request 的首次失败会把 `failureCount` 从 `0` 增加到 `1`
  - request 会继续停留在 `pending_shared_channel_sync_requests`
  - 不会因历史 dead-letter 前的旧失败次数而立即再次回到 dead-letter

## 3. 明确边界
- 本轮没有：
  - targeted requeue / selection contract
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 单体内的 operator recovery boundary，而不是 release-ready cross-service delivery orchestration

## 4. Fresh Evidence
- red:

- green:

- package regression:
  - `cargo fmt -p control-plane-api` = `passed`
  - `cargo test -p control-plane-api --offline --tests -- --nocapture` = `70 passed`

## 5. 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop59 补齐的是 dead-letter requeue rearm seam，而不是 targeted requeue / remote republish / delivery ownership semantics
