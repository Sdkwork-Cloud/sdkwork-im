# 152CJ Loop71补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“takeover 冲突写路径只返回 `code/message`”推进到“takeover 冲突写路径显式回写 machine-readable `details`”的最小 contract

## 2. As-Built 事实
- `ControlPlaneError` 当前支持可选 `details`
- active foreign takeover conflict 当前显式返回 `details`
- legacy override-required conflict 当前也显式返回同一组 `details`
- `details` 当前至少覆盖：
  - `requestKey`
  - `ownerActorId`
  - `leaseExpiresAt`
  - `leaseStatus`
  - `takeoverEligible`
  - `legacyTakeoverRequired`
- operator 当前在 takeover 被拒绝时，不再必须回头再读 inventory 才能知道 lease / ownership / legacy override gate 的原因
- 本轮没有改变 takeover allow/deny 语义；补的是 write-path conflict surface symmetry，而不是新的 stale policy

## 3. 明确边界
- 本轮没有：
  - dead-letter inventory metadata parity 的 dedicated 断言
  - claim/release conflict `details` parity
  - configurable lease window / SLA
  - timeout reclaim / automatic stale detection
  - stale-aware bulk `repair-shared-channel-sync`
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 的 error surface 对齐增量，而不是完整的 operator policy/governance engine

## 4. Fresh Evidence
- red：
  - `cargo test -p control-plane-api --offline --test social_external_collaboration_test test_control_plane_social_shared_channel_pending_takeover_transfers_claim_to_foreign_operator -- --nocapture`
- red 失败点：
  - `active_takeover_json["details"]["requestKey"] = null`
- green：
  - active conflict 当前显式返回 `details.leaseStatus = active`
  - legacy override-required conflict 当前显式返回 `details.leaseStatus = untracked`
  - 同一条 takeover lifecycle 继续证明 active conflict、legacy override gate、stale takeover success 与 republish ownership guard 并存
  - `cargo fmt -p control-plane-api` = `passed`
  - `cargo test -p control-plane-api --offline --tests -- --nocapture` = `75 passed`

## 5. 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop71 补齐的是 `takeover conflict details symmetry seam`，不是 `dead-letter metadata parity`、`claim/release conflict details parity` 或 `stale-age governance`
