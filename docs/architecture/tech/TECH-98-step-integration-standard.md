# Step 并行车道交付包与集成交接标准

## 1. 文档定位

本文件用于把“并行车道”从概念变成真正可执行的协作机制。

目标是回答：

- 每个并行车道必须交什么
- 交付到什么程度才允许合并
- 阻塞时如何升级
- 怎样保证车道越多，收口越快，而不是返工越多

## 2. 适用范围

适用于所有在同一波次内并行推进的 step，特别是：

- `02-03`
- `04-06`
- `07-09`
- `10-12`

## 3. 并行车道的最小交付包

每个并行车道在任一集成窗口前，必须提交以下内容：

### 3.1 边界冻结单

至少包括：

- 本车道名称
- 主写入目录
- 禁止写入目录
- 共享边界文件
- 本轮是否改动共享协议/权威字段/状态机

### 3.2 本轮输入

- 依赖的 step 文档
- 依赖的 `docs/架构/` 文档
- 依赖的上一车道结论
- 依赖的配置、协议版本、测试资产

### 3.3 本轮输出

- 已完成代码路径
- 已完成测试路径
- 已完成脚本或运行验证
- 对 `docs/review/` 的新增产物

### 3.4 当前阻塞项

- 阻塞描述
- 是否影响其他车道
- 是否需要回写 `docs/架构/` / `docs/step/`
- 推荐升级等级：`P1 / P2 / P3`

### 3.5 合并条件

- 当前 step 检查点通过项
- 本车道 smoke 通过项
- 与其他车道接口一致性确认
- 是否允许进入集成分支

## 4. 车道交接的硬规则

- 没有边界冻结单，不允许并行开工。
- 没有本轮输出，不允许说“已完成一半”。
- 没有阻塞判定，不允许把风险留到波次末尾才暴露。
- 没有合并条件验证，不允许直接合并。
- 没有 `docs/review/` 证据，不允许进入下一 step。

## 5. 集成窗口标准

每次集成窗口至少检查五项：

- 共享协议与字段边界是否变化
- 共享状态机是否变化
- 主写入目录是否冲突
- smoke 与测试是否仍通过
- 是否需要回写 `docs/架构/`

若以上任一项未通过，本轮只能停留在车道内修复，不得强行进入集成。

## 6. 推荐交付频率

### 6.1 高频车道

适用于：

- Link / Route Runtime
- Message 主链路
- Stream / RTC

要求：

- 每天至少一次可集成状态
- 每天至少一次最小 smoke

### 6.2 中频车道

适用于：

- Control Plane
- Storage / Projection
- Deploy / CLI / SDK

要求：

- 每 1-2 天一次可集成状态
- 每次集成前补齐文档与验证证据

## 7. 阻塞升级标准

### 7.1 P1

必须立即停下并升级：

- 权威字段边界冲突
- 协议状态机冲突
- 两个车道主写同一高风险文件
- 架构文档与实现明显偏移

### 7.2 P2

可在当日内修复后继续：

- 测试资产不一致
- smoke 口径不一致
- review 文档缺失

### 7.3 P3

记录并进入下一轮排期：

- 非主链路优化项
- 附加文档美化项
- 不影响当前 gate 的次要重构

## 8. 推荐交接模板

```md
# 车道交付包 - Step XX / Lane YY

## 1. 边界冻结单
- 主写入目录：
- 禁止写入目录：
- 共享边界文件：

## 2. 本轮输入
- ...

## 3. 本轮输出
- 代码：
- 测试：
- 脚本/运行：
- review 文档：

## 4. 当前阻塞项
- ...

## 5. 合并条件检查
- [ ] 检查点通过
- [ ] smoke 通过
- [ ] 接口一致
- [ ] 是否需回写 docs/架构 已确认

## 6. 结论
- 可合并 / 不可合并
```

## 9. 与其他治理文档的关系

- `94`：告诉你怎么拆车道
- `96`：告诉你怎么排节奏
- `97`：告诉你完成后怎么回写架构能力
- `98`：告诉你每个车道到底交什么、怎么交、交到什么程度才能合并

## 10. 结论

真正高效的并行执行，不是“大家同时写”，而是：

- 先冻结车道边界
- 再按统一交付包推进
- 再按统一集成窗口收口

只有这样，step 才能又快又完整地并行跑完。
