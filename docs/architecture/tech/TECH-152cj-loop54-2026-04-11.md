# 152CJ Loop54补充 - 2026-04-11

- `control-plane-api` 当前除了 embedded in-process consumer 之外，还显式提供 standalone public HTTP consumer 形态
  - 该 consumer 的 `PublicSharedChannelLinkedMemberSyncTrigger` 实现
  - trigger 不依赖 `conversation-runtime` runtime 类型，只消费已存在的 public sync HTTP surface
  - trigger 当前通过 public bearer 生成 `actor_kind = system` 的最小身份，将 ready-pair payload 投递到 `/im/v3/api/chat/conversations/shared-channel-links/sync`
- standalone `control-plane-api` 的组合根当前已被更新
  - `services/control-plane-api/src/main.rs` 现在会在检测到 `sdkwork_im_SHARED_CHANNEL_SYNC_TARGET_BASE_URL` 时装配真实 consumer
  - 未配置该环境变量时，standalone 仍保持原有无 trigger 降级，不会伪装成已闭环
- `S07` 的 shared-channel auto-sync truth 现在有两条真实装配路径
  - `sdkwork-im-server`：同进程 consumer，直接调用 `ConversationRuntime::sync_shared_channel_linked_member(...)`
  - `standalone control-plane-api`：跨进程 public HTTP consumer，调用 standalone `conversation-runtime` 的 public sync route
- 当前仍然不能把系统解释为“shared-channel sync 已具备 release-ready cross-service delivery semantics”
  - 还没有 outbox / retry / dead-letter / remote republish
  - 当前 bridge 只支持 `http://` public runtime target，`https://` 仍未进入 as-built truth
  - linked member beyond `shared history read` 的其他 read surfaces 仍未一并宣称
- As-Built 结论现在更新为：
  - 已闭合到 `standalone control-plane public HTTP runtime consumer wiring`
  - 仍待闭合到 `cross-service shared-channel sync outbox / retry / delivery ownership boundary`
