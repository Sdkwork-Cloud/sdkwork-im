# v0.0.54 - Loop 54 - 2026-04-11

## 本轮目标
- 把 `standalone control-plane / cross-service shared-channel sync orchestration` 从“只有 embedded local-minimal 装配面有真实 consumer”推进为 standalone `control-plane-api` 的最小 public HTTP consumer 闭环

## 本轮实现
- `control-plane-api`
  - 新增 `PublicSharedChannelLinkedMemberSyncTrigger`
  - 新增 `build_public_shared_channel_sync_trigger(...)`
  - 新增 `build_public_app_with_shared_channel_sync_trigger(...)`
  - 新增 `configured_public_shared_channel_sync_trigger(...)`
  - 新增 `SHARED_CHANNEL_SYNC_TARGET_BASE_URL_ENV`
- `services/control-plane-api/src/main.rs`
  - 当前会在检测到 `SDKWORK_IM_SHARED_CHANNEL_SYNC_TARGET_BASE_URL` 时装配真实 public HTTP trigger
  - 未配置时继续回落到原有 `build_public_app()`
- 测试
  - 扩展 `services/control-plane-api/tests/social_external_collaboration_test.rs`
  - 启动 standalone `conversation-runtime::build_public_app()`，再通过 public `control-plane-api` 写入 `external_connection / shared_channel_policy / external_member_link`
  - 最终验证 linked actor 能经 public runtime 直接读取 shared history

## 风险与未闭环
- 当前 bridge 只支持 `http://` public runtime target；`https://` 仍未进入本轮 truth
- 当前没有 outbox / retry / dead-letter / remote republish，仍不是跨服务 exactly-once 闭环
- linked member beyond `shared history read` 的其他 read surfaces 当前未一并验证

## 结论
- `S07` 仍是 `not_closed / local_closure`
- standalone `control-plane-api` 现已具备 `ready-pair auto-trigger -> public HTTP sync bridge -> standalone runtime linked history read` 的最小跨进程闭环
- 下一主缺口收敛为 `cross-service shared-channel sync outbox / retry / delivery ownership boundary`
