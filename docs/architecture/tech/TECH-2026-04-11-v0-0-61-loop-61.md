# v0.0.61 - Loop 61 - 2026-04-11

## 本轮目标
- 把 `shared-channel sync` 从“pending backlog 只有全量 repair，缺少 inventory 与定点 republish 契约”推进到“operator 可读取 pending inventory，并按 `requestKey` targeted republish”的最小可审阅边界

## 本轮实现
- `services/control-plane-api/src/lib.rs`
  - 新增 `GET /backend/v3/api/control/social/runtime/pending-shared-channel-sync`
  - pending inventory item 当前显式返回 `requestKey / request / failureCount / lastError`
  - 新增 `POST /backend/v3/api/control/social/runtime/republish-pending-shared-channel-sync-targeted`
  - `republish_pending_shared_channel_sync_targeted(...)` 当前允许按 `requestKey` 只投递被选中的 pending request 到 remote runtime
  - 成功投递的选中 request 会从 pending backlog 清理；未选中的 pending request 保持不动
  - 现有 `repair-shared-channel-sync` 全量 route 保持不变
- `services/control-plane-api/tests/social_external_collaboration_test.rs`
  - 新增 dual-service e2e，证明 pending inventory 可列出两条 pending request 的稳定 `requestKey`
  - targeted republish 后只有被选中的 actor 恢复 shared history 可读；未被选中的 actor 仍然 `403`

## 风险与未闭环
- 当前 pending inventory 还没有 filter / pagination / search contract
- 当前 targeted pending republish 只覆盖 operator 显式投递，不包含 lease owner / SLA / background scheduler
- 当前仍没有 release-ready exactly-once semantics
- `https://` public runtime target 仍未进入本轮 truth

## 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop61 已把 `shared-channel sync` 补齐到 `durable pending backlog + operator repair + pending inventory + targeted pending republish + next healthy ready-pair write auto-retry + repeated-failure dead-letter + dead-letter inventory + dead-letter targeted/all requeue + failure-budget rearm`
- 下一主缺口收敛为 `delivery ownership / lease / SLA semantics`
