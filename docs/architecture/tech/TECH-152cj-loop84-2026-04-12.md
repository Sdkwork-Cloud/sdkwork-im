# 152CJ Loop84 Addendum - 2026-04-12

## 1. As-Built Update
- Targeted pending-republish now has the same stale same-owner lease-renewal normalization that claim gained in Loop83, but only on dispatch-enabled paths.
- A same-owner republish attempt now implies:
  - if the lease is active, republish is owner-idempotent
  - if the lease is stale and dispatch is attempted, the lease is renewed before the trigger fires
- No `deployment_profile`, script entry, or operator route changed in this loop.

## 2. Architectural Meaning
- Before Loop84, a same-owner stale republish failure could erase owner metadata because failure persistence saw a stale lease and reclaimed it.
- After Loop84, owner-initiated republish is lease-consistent on failing dispatch paths:
  - republish first renews the stale same-owner lease
  - failure persistence keeps the request in the pending owner pool
- This remains a path-local normalization seam, not a scheduler or timeout sweeper.

## 3. Deferred Architecture Work
- trigger-unconfigured targeted republish stale normalization
- automatic stale detection / timeout reclaim / scheduler
- release-ready exactly-once delivery semantics for shared-channel sync
