# Step 13 - 发布就绪与持续迭代闭环

## 1. 目标与范围

本 step 用于把整个实施计划最终收口到“可发布、可验收、可持续迭代”的闭环上，避免项目停留在“代码写了很多，但没有最终发布标准”的状态。

本 step 主要覆盖：

- 全量回归与验收
- 发布清单
- 风险收口
- 回滚预案
- 下一轮迭代输入

### 1.1 执行输入

- `00-12` 的复盘文档与质量评分结果
- 当前发布相关脚本、配置模板、SDK / CLI 和 compatibility 资产
- 当前性能、高可用、灾备演练报告
- 当前 `docs/review/` 中的未决问题和遗留风险

### 1.2 本步非目标

- 不在本 step 内启动新一轮架构重构
- 不在本 step 内继续引入新的高风险能力改动
- 不在本 step 内用“即将修复”替代发布门禁

### 1.3 最小输出

- 发布清单、升级说明、回滚说明
- 全量验收结论与阻塞项清单
- 当前版本是否允许交付的正式判断
- 下一轮 backlog 与迭代输入
- 面向 provider/plugin 的最小增量迭代包

## 2. 架构对齐

本 step 是对前面所有架构文档和 step 文档的总收口，特别对齐：

- `docs/架构/README.md`
- `docs/架构/09-实施计划.md`
- `docs/step/README.md`
- `docs/review/`

## 3. 当前现状与问题

在没有最终闭环之前，常见风险包括：

- 功能实现了，但没有明确 release gate
- 压测做过了，但结果没有进入发布标准
- SDK、CLI、脚本、服务端之间没有统一验收
- 问题列表没有沉淀为下一轮计划

因此本 step 的重点不是继续扩功能，而是完成交付收口。

同时必须避免另一类问题：

- provider / plugin 文档已经冻结，但没有进入下一轮可执行 backlog
- 默认 provider、租户覆盖、协议适配、设备接入体系没有进入正式 release gate

## 4. 设计

### 4.1 发布就绪标准

发布前必须同时满足：

- 架构一致性
- 功能正确性
- 安全与权限边界
- 性能与可用性证据
- 部署与恢复能力
- CLI / SDK / compatibility 完整性

### 4.2 发布清单

建议形成以下清单：

- 构建清单
- 配置清单
- 部署清单
- 测试清单
- 兼容矩阵
- 风险清单
- 回滚清单

### 4.3 持续迭代闭环

必须形成如下循环：

1. review 问题
2. 修复与补测
3. 验证与复盘
4. 发布与监控
5. 形成下一轮 backlog

对于 provider/plugin 相关能力，下一轮 backlog 的最小粒度必须至少覆盖：

- 契约
- provider / protocol adapter
- control-plane / profile 配置
- conformance / smoke
- docs/review 回写

### 4.4 验收对象

最终验收对象不是某个单服务，而是整套应用系统：

- 核心服务
- 协议和契约
- 脚本
- 部署
- CLI / SDK
- 运维与恢复

## 5. 实施落地规划

### 5.1 任务拆解

1. 汇总前面所有 step 的完成情况和遗留问题
2. 运行全量构建、测试、smoke、E2E、压测、恢复验证
3. 生成发布清单、版本说明、升级说明、回滚说明
4. 汇总 `docs/review/` 中的未决问题，形成下一轮 backlog
5. 确认 release gate 后再执行正式发布

### 5.2 重点路径

重点涉及：

- `README.md`
- `docs/架构/`
- `docs/step/`
- `docs/review/`
- `bin/`
- `deployments/`
- `sdks/`
- `tools/chat-cli/`

### 5.3 发布前最小必跑集

建议至少跑以下集合：

- `cargo fmt --check`
- `cargo clippy --workspace --all-targets --all-features -- -D warnings`
- `cargo test --workspace`
- 本地安装、启动、停止、恢复脚本 smoke
- CLI 聊天验证
- 协议兼容验证
- 备份恢复验证
- 至少一轮性能或稳定性回归验证

### 5.4 下一轮计划输入

本 step 完成后必须至少输出：

- 发布后遗留问题清单
- 下一轮性能优化清单
- 下一轮能力拓展清单
- 下一轮文档修订清单
- 下一轮 provider / plugin 增量清单

## 6. 测试计划

本 step 本身就是全量测试与验收 step，重点关注：

- 全 workspace 回归
- 多环境部署 smoke
- CLI / SDK 兼容回归
- 备份恢复与升级回滚演练
- 安全边界和权限回归

## 7. 结果验证

本 step 完成后，需要验证：

- 当前应用已经具备清晰发布口径
- 所有关键能力都有证据链支撑
- 交付不依赖口头说明和隐性知识
- 下一轮迭代已经有明确输入，而不是重新从零梳理
- provider / plugin 增量已经被拆成可执行最小闭环，而不是重新回到大而全方案

## 8. 检查点

- `CP13-1`：完成全量回归与验收清单
- `CP13-2`：完成发布、升级、回滚说明
- `CP13-3`：完成遗留问题和下一轮 backlog 整理
- `CP13-4`：形成完整交付闭环并可进入正式发布

### 8.1 推荐 review 产物

- `docs/review/step-13-执行卡-YYYY-MM-DD.md`
- `docs/review/step-13-release-readiness-YYYY-MM-DD.md`
- `docs/review/step-13-go-no-go清单-YYYY-MM-DD.md`
- `docs/review/step-13-next-wave-backlog-YYYY-MM-DD.md`

### 8.2 推荐并行车道

- `13-A`：发布清单、升级/回滚说明、交付资产归档
- `13-B`：最终验收、go/no-go 决策、阻塞项收口
- `13-C`：下一轮 backlog、provider/plugin 增量包、技术债、运营与治理续航计划
- 收口要求：`13-Owner` 必须统一汇总 `00-12` 的证据与阻塞项，任何车道都不能跳过 release gate 直接宣布完成。
- 车道编排参考：[`94-Step并行执行编排与车道拆分建议`](./94-Step并行执行编排与车道拆分建议.md)

### 8.3 架构能力闭环判定

- 必须形成正式发布结论、升级回滚路径和下一轮 backlog，才能支撑“当前版本可交付”。
- 如果只有总结性说明，没有 go/no-go 判断和 release gate，本 step 不算闭环。
- 闭环验收以 [`95-架构能力闭环验收标准`](./95-架构能力闭环验收标准.md) 中 Step 13 条目为准。

### 8.4 快速并行执行建议

- 先冻结 go/no-go 模板、release evidence 索引和阻塞项口径，再并行推进发布清单、最终验收、下一轮 backlog。
- 推荐本步只做收口，不再引入新功能或新协议决策，避免 release gate 被持续打断。
- 任一阻塞项若影响交付结论，必须在 `docs/review/` 中明确记录并回传到下一轮计划。

### 8.5 完成后必须回写的架构文档

- 强制范围：`docs/架构/README.md`、`docs/架构/09-实施计划.md` 以及本步执行中实际被修订过的全部架构文档。
- 回写重点：当前版本最终实现能力、未完成能力、发布边界、运维限制、下一轮 backlog 是否已经反映到架构与实施计划。
- 必备证据：`docs/review/step-13-架构兑现-YYYY-MM-DD.md` 与 `docs/review/step-13-架构回写决议-YYYY-MM-DD.md`。

## 9. 风险与回滚

### 9.1 风险

- 没有正式 release gate，容易在“差不多能用”时提前交付
- 没有回滚说明，升级风险会被放大
- 没有下一轮 backlog，项目会失去迭代连续性

### 9.2 回滚

- 若发布前验证未通过，应回到对应 step 修复，不允许带病发布
- 若发布后验证发现问题，应依据已准备好的回滚脚本和配置快照执行回滚
- 重大问题必须同步回写到 `docs/review/` 和下一轮 step/backlog 中

## 10. 完成定义

满足以下条件时，本 step 完成：

- 所有关键能力已完成闭环验证
- 发布、升级、回滚、恢复说明齐全
- 下一轮迭代输入已准备完毕

## 11. 下一步准入条件

本 step 是当前实施计划的最终收口 step。完成后可进入下一轮审计、优化与扩展循环。
