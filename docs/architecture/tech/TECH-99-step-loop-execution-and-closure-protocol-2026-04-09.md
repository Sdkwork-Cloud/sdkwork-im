# 99 - Step 循环执行与持续迭代闭环协议

## 1. 目标

为 `docs/step` 体系补上“循环执行直到真正闭环”的统一协议，避免以下常见失真：

- 只完成局部代码修改，就误判整个 Step 完成
- 代码、测试、架构文档、review 文档长期漂移
- 本轮只写计划，不关真实缺口
- 发现架构偏移后没有重新回写 Step 状态

本协议定义的不是一次性交付，而是 `loop -> step -> wave -> release` 的连续闭环机制。

## 2. 核心概念

| 概念 | 含义 | 说明 |
| --- | --- | --- |
| `loop` | 一次真实迭代循环 | 每轮只关闭一个最高价值主缺口 |
| `step` | 某个能力阶段 | 一个 step 通常包含多轮 loop |
| `wave` | 一组强相关 step 的阶段性收束 | 用于中期验收 |
| `release closure` | 面向发布的最终闭环 | 需要更高等级证据 |
| `local closure` | 当前改动点局部闭合 | 不等于 step 闭环 |
| `step closure` | 当前 step 的全部目标与文档已对齐 | 需要重新评估 |

## 3. 闭环层级

### 3.1 `local closure`

只表示本轮修改点已经有证据支撑，例如：

- 一个 bug 已修复
- 一个测试已补齐
- 一个文档漂移已纠正

但这不代表：

- 当前 step 已完成
- 当前 wave 已完成
- 架构已无漂移

### 3.2 `step closure`

只有当以下条件同时成立时，才能把 step 标记为闭环：

1. 该 step 的核心缺口已经全部关闭，没有残留高优先级未完成项。
2. 代码、测试、文档、review 与 Step 状态一致。
3. 相关架构文档已经回写，且不再与当前实现冲突。
4. 没有“局部已完成但整体主链路仍断裂”的伪完成。

### 3.3 `wave closure`

`wave` 闭环要求：

- 该波次内各 step 的前置依赖都已满足
- 交叉模块的接口、配置、文档、验证样本已经一致
- 回写文档能清楚说明这一波次的剩余缺口与下一波次入口

### 3.4 `release closure`

只有当：

- 当前发布范围内没有高风险未闭环能力
- 验证证据真实存在
- `docs/step`、`docs/review`、`docs/架构` 与当前仓库状态一致

才允许宣称发布闭环。

## 4. 每轮 loop 的强制动作

每次循环执行都必须按以下顺序进行：

1. 判断当前 `step / wave / closure level`
2. 检查当前仓库的真实状态
3. 识别所有剩余缺口
4. 选择“单个最高价值主缺口”
5. 真正修改代码或文档，而不是只写计划
6. 运行验证
7. 回写 `docs/review`、`docs/step`、`docs/架构`
8. 重新评估当前是不是 `local closure / step closure / wave closure`
9. 明确输出下一轮应该继续做什么

## 5. 每轮必须检查的证据

每轮 loop 至少要检查以下证据源：

- 相关代码文件
- 相关测试文件
- 相关 Step 文档
- 相关架构文档
- 相关 review 文档
- `git status`
- 已执行验证命令及结果

如果这些证据没有查，就不能声称“已经闭环”。

## 6. 硬规则

### 6.1 不允许 plan-only 逃逸

如果当前缺口可以被直接关闭，就必须优先关闭，不允许只输出计划。

### 6.2 一轮只主攻一个核心缺口

每轮 loop 可以顺手修复邻近小漂移，但只能有一个主目标，否则很容易把状态做假。

### 6.3 不允许跳过验证

没有验证就没有完成。验证可以是：

- 自动化测试
- 文档契约测试
- 结构测试
- 本地脚本验证
- 明确记录的人工验证

### 6.4 不允许跳过回写

以下情形必须回写文档：

- 架构边界调整了
- Step 状态变化了
- review 结论变化了
- 剩余缺口优先级变化了

### 6.5 不允许把投影当真值

一旦发现当前实现把可重建投影视为真值，必须在该轮 loop 里显式指出并纠正，不允许带着错误前提继续推进。

## 7. Step 体系中的状态机

推荐统一状态：

- `identified`
- `in_progress`
- `locally_verified`
- `backwritten`
- `step_reassessing`
- `step_closed`
- `wave_reassessing`
- `wave_closed`
- `blocked`

说明：

- `locally_verified` 只能说明本轮改动有证据
- `backwritten` 才表示文档同步完成
- `step_closed` 必须经过重新评估，而不是代码一绿就直接升级

## 8. 架构回写触发器

出现以下情况时，必须回写 `docs/架构`：

- 新增限界上下文
- 实体命名发生变化
- 真值与投影边界被重新定义
- 跨服务职责重新分配
- 数据库主模型新增或删除关键表

如果只改代码不改架构文档，Step 不得判定为真正闭环。

## 9. Review 回写触发器

出现以下情况时，必须回写 `docs/review`：

- 本轮关闭了真实缺口
- 本轮发现旧完成声明不成立
- 本轮把“局部完成”降级为“整体未闭环”
- 本轮调整了下一轮优先级

## 10. 停止条件

循环执行只能在以下三种情形之一停止：

1. 当前范围内已经没有高优先级和中优先级缺口，且 Step / Wave / 架构 / Review 完整一致。
2. 明确被外部依赖阻塞，并且已经写出最小阻塞说明与下一轮入口。
3. 当前轮次已经完成一项真实闭环，需要把控制权交给下一轮继续推进。

“感觉差不多了”不是停止条件。

## 11. 推荐输出结构

每轮输出都建议固定包含：

- 当前 `step`
- 当前 `wave`
- 当前闭环层级
- 本轮主缺口
- 本轮实际完成
- 修改文件
- 验证结果
- 文档回写结果
- 剩余缺口
- 下一轮动作
- 是否允许停止

## 12. 与 Prompt 的关系

本协议是 Step 体系规则层。

`docs/prompts/反复执行Step指令.md` 是给大模型重复喂入的执行层。二者关系如下：

- `99` 负责定义闭环规则
- `反复执行Step指令.md` 负责把规则翻译成可循环执行的提示词

后续如果 prompt 强化了，但违反本协议，视为无效强化。
