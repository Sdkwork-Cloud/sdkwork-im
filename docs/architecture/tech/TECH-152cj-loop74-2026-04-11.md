# 152CJ Loop74补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“dead-letter inventory metadata parity 主要由共享 helper 隐式保证”推进到“dead-letter inventory metadata parity 被 dedicated assertion 显式锁定”的最小 contract

## 2. As-Built 事实
- pending inventory 与 dead-letter inventory 当前继续复用同一套 response helper
- dead-letter inventory item 当前与 pending inventory 一样显式暴露：
  - `leaseStatus`
  - `takeoverEligible`
  - `legacyTakeoverRequired`
  - `ownerActorId / ownerActorKind / claimedAt / leaseExpiresAt`
- 对 dead-letter item 的当前 truth 是：
  - `leaseStatus = unclaimed`
  - `takeoverEligible = false`
  - `legacyTakeoverRequired = false`
  - owner/lease 相关元数据为空
- 本轮没有引入新的 operator route；补的是 read-model parity 的 direct regression lock

## 3. 明确边界
- 本轮没有：
  - 修改 inventory response schema
  - 修改 claim / republish / release / takeover 语义
  - 提供 richer remediation workflow
  - 提供 automatic stale detection / timeout reclaim / scheduler
  - 提供 stale-aware bulk `repair-shared-channel-sync`
  - 提供 release-ready exactly-once delivery semantics

## 4. Fresh Evidence
- green：
  - `cargo test -p control-plane-api --offline --test social_external_collaboration_test test_control_plane_social_shared_channel_dead_letter_inventory_and_targeted_requeue_only_restores_selected_actor -- --nocapture`
- green 事实：
  - dead-letter inventory parity assertions 通过
  - 这证明 shared helper 的读模型对齐当前已被 dedicated test 直接覆盖，而不是仅靠间接推断

## 5. 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop74 补齐的是 `dead-letter metadata parity assertion lock`，不是 `stale-age governance` 或 `batch remediation`
