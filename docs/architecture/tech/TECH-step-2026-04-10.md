# 101-Step执行卡与并行交付模板-2026-04-10

## 1. 文档定位

- `100-*` 负责总顺序、能力挂靠、并行窗口；`101-*` 负责把单个 `Sxx` 变成可直接执行的工作单。
- 任一步真正开工前，都必须先实例化一份执行卡；若 `100-*` 与本模板冲突，以 `100-*` 为准。
- 模板目标不是补格式，而是冻结边界、约束并行、压实回写、阻断假完成。

## 4. 最小使用规则

1. 先写执行卡，再开车道；未冻结目标、写集、owner，不允许并行。
2. 车道只按写集拆分；共享 schema、README、总计划、公共契约、架构索引只能串行收口。
3. `LV` 独立于实现车道，只写测试、脚本、review、evidence，不主写业务聚合。
4. 交付包若未覆盖“变更 + 测试 + 证据 + 集成注意事项”，不得合流。
5. `CPR` 未过、回退不清、回写未做、出口 6 问任一 `no`，统一视为 step 未完成。
6. 若无法回答“本轮兑现了哪些架构结论、哪些仍 deferred、归属到谁”，统一视为未闭环。

## 5. 最快并行建议

- `Wave` 开始前只冻结六项：目标、非目标、写集、共享文件 owner、最小交付包、集成窗口。
- `Wave` 进行中只同步四项：阻塞、共享契约漂移、交付完成度、下一次合流时间。
- `Wave` 收口时只看六项：CPR、测试、review、evidence、架构回写结论、出口 6 问。
- 若发现能力未挂靠、回写结论悬空或 Deferred 无 owner，先回写执行卡或 `100-*`，再继续实现。
