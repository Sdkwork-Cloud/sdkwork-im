# 多入口服务外部认证收敛与控制面保护标准（2026-04-05）

## 1. 背景

在 Review Cycle 3 完成 `sdkwork-im-server` 的外部 Bearer-only 收敛后，工作区里仍然保留多个可单独部署的服务入口：

- `session-gateway`
- `streaming-service`
- `im-call-runtime`
- `control-plane-api`

这些服务一旦被单独部署，就不能继续把测试/内部装配用的 trusted identity headers 当成公网认证来源，也不能把控制面写接口暴露为匿名可调用。

## 2. 本轮发现

### 2.1 高风险：`control-plane-api` 的控制写接口此前没有任何认证保护

受影响接口：

- `POST /backend/v3/api/control/nodes/{nodeId}/drain`
- `POST /backend/v3/api/control/nodes/{nodeId}/activate`
- `POST /backend/v3/api/control/nodes/{nodeId}/routes/migrate`

根因：

- handler 未解析任何认证上下文
- `main.rs` 直接启动 `build_app()`
- 因此 drain / migrate / activate 可以被匿名调用

风险：

- 未授权用户可触发节点排空、路由迁移、节点恢复
- 可直接影响实时路由归属、集群可用性与投递稳定性

修复：

- `control-plane-api` 新增 `build_public_app()`
- `main.rs` 改为启动 `build_public_app()`
- 非健康检查入口统一要求 Bearer
- handler 层补上 `resolve_auth_context(...)`，即使 internal/test builder 也必须至少带可信认证上下文

### 2.2 高风险：`session-gateway` 单独部署时对外仍接受 trusted headers

受影响入口：

- `POST /im/v3/api/presence/heartbeat`
- `GET /im/v3/api/presence/me`
- `POST /im/v3/api/realtime/subscriptions/sync`
- `GET /im/v3/api/realtime/events`
- `POST /im/v3/api/realtime/events/ack`
- `GET /im/v3/api/realtime/ws`

根因：

- 服务只暴露 `build_app()`
- handler 使用 `resolve_auth_context(...)`
- WebSocket 握手同样继承 trusted headers fallback

修复：

- 新增 `build_public_app()`
- public builder 在 `/healthz` 之外统一要求 Bearer
- `main.rs` 改为启动 `build_public_app()`
- 新增 HTTP 与 WebSocket 握手 Bearer-only 契约测试

### 2.3 高风险：`streaming-service` 单独部署时对外仍接受 trusted headers

根因：

- `main.rs` 启动 `build_default_app()`
- 各 stream 路由通过 `resolve_auth_context(...)` 接受 trusted headers fallback

修复：

- 新增 `build_public_app()`
- `main.rs` 改为启动 `build_public_app()`
- `/healthz`、`/readyz` 之外统一要求 Bearer

### 2.4 高风险：`im-call-runtime` 单独部署时对外仍接受 trusted headers

根因：

- `main.rs` 启动 `build_default_app()`
- RTC 会话与信令路由通过 `resolve_auth_context(...)` 接受 trusted headers fallback

修复：

- 新增 `build_public_app()`
- `main.rs` 改为启动 `build_public_app()`
- `/healthz`、`/readyz` 之外统一要求 Bearer

## 3. 冻结标准

本轮之后，以下标准冻结：

1. 所有可对外部署的独立服务 `main.rs` 默认必须启动 `build_public_app()`。
2. `build_public_app()` 只允许 health/ready 这类探针接口匿名访问，其余入口统一要求 `Authorization: Bearer ...`。
3. `trusted identity headers` 只能保留在 internal/test builder 或显式内部链路中，不能作为公网入口默认认证来源。
4. 即使是 internal/test builder，也不允许控制面写接口匿名调用；至少要存在 trusted/bearer 认证上下文。
5. WebSocket 握手属于外部入口的一部分，必须与 HTTP 路由执行相同的 Bearer-only 边界。

## 6. 剩余风险与下一轮计划

本轮收敛了最直接可部署的 session/stream/rtc/control-plane 入口，但以下服务若后续被单独部署，仍需继续 review 是否也要 public/internal builder 拆分：

- `audit-service`
- `automation-service`
- `media-service`
- `ops-service`
- `conversation-runtime`
- `projection-service`
- `notification-service`

下一轮优先级：

1. 继续 review 剩余独立服务的 `main.rs` 与认证边界。
2. 评估是否将 internal/test builder 统一显式重命名为 `build_internal_app()`，进一步减少误用。
3. 在控制面上继续补“认证之后的授权”标准，例如 operator/admin 角色或运维策略校验。
