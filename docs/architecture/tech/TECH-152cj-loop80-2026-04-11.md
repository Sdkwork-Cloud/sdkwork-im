# 152CJ Loop80 Addendum - 2026-04-11

## 1. As-Built Update
- Dead-letter requeue now treats ownership as non-transferable backlog metadata.
- When a dead-letter item is restored into pending backlog, owner metadata is cleared before the item re-enters the pending pool.
- This makes dead-letter requeue consistent with:
  - explicit stale reclaim
  - retry-failure reclaim
  - repair-time reclaim

## 2. Architectural Meaning
- Before Loop80, dead-letter requeue could reintroduce historical stale owner metadata into pending backlog.
- After Loop80, requeue behaves as a true backlog restoration step rather than a claim restoration step.
- The pending pool remains the single source of truth for active ownership, and dead-letter acts only as a failure bucket.

## 3. Deferred Architecture Work
- automatic stale detection / timeout reclaim / scheduler
- release-ready exactly-once delivery semantics for shared-channel sync
- richer stale-age / SLA policy beyond the current manual and path-local reclaim seams
