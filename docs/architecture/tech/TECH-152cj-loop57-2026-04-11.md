# 152CJ Loop57 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“pending backlog 会在 auto-retry / repair 入口被持续重试”推进到“重复失败请求进入 dead-letter 并退出自动重试范围”的最小 poison backlog 边界

## 2. As-Built 事实
- `SocialControlState` 当前新增 durable `dead_letter_shared_channel_sync_requests`
- `control-plane-api` 当前使用固定失败阈值 `3`
  - 第一次失败：request 进入 `pending`
  - 后续失败累计到阈值：request 从 `pending` 转入 `dead_letter`
- `pending_shared_channel_sync_dispatch_queue(...)` 当前会显式跳过 dead-lettered request
  - 因此 `next healthy ready-pair write` 自动重试不会继续消费 poison backlog
- `repair_shared_channel_sync(...)` 当前只消费 `pending`，不会继续重试 dead-lettered request
- `repair-shared-channel-sync` 响应当前显式暴露：
  - `deadLetterBefore`
  - `deadLettered`
  - `deadLetterAfter`
- `SocialAggregateCountsResponse` 与 `repair-social-runtime-dir` operator report 当前显式暴露 `deadLetterSharedChannelSyncRequests`
- `repair-derived-snapshot` 与 `repair-social-runtime-dir` replay journal 时当前都会保留 snapshot 中已有的 dead-letter backlog

## 3. 明确边界
- 本轮 dead-letter 是“隔离边界”，不是“恢复边界”
- 当前没有：
  - dead-letter requeue / replay route
  - 后台 retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once delivery semantics
- 因此 dead-letter 目前只证明 poison backlog 不会被无限自动重试，不证明 release-ready cross-service delivery 已闭环

## 4. Fresh Evidence
- red:
  - `cargo test -p control-plane-api --offline --test social_external_collaboration_test test_control_plane_social_shared_channel_repeated_failure_moves_request_to_dead_letter_and_stops_repair_retry -- --nocapture`
  - 初始失败断言为 repeated failure 后 request 仍留在 pending backlog，而未进入 dead-letter
- green:
  - 同一条 dual-service e2e 通过，证明 request 在第三次失败后进入 dead-letter
  - 切换到健康 trigger 后，repair route 返回 `noop`
  - 后续新的健康 ready-pair 写入只会 materialize 新 actor；dead-lettered actor 仍不能读取 shared history
- package regression:
  - `cargo fmt -p control-plane-api` = `passed`
  - `cargo test -p control-plane-api --offline --tests -- --nocapture` = `68 passed`

## 5. 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop57 补齐的是 repeated-failure dead-letter 边界，而不是 dead-letter 恢复链路
- 下一主缺口继续收敛为 `automatic retry scheduler / dead-letter requeue / remote republish / delivery ownership semantics`
