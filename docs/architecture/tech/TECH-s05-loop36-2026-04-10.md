# S05 Loop36补充 - 2026-04-10

## 1. 本轮主任务
- 关闭 `journal durable` 已成立但客户端收到 `503` 后，same-event retry 仍无法精确确认提交结果的缺口。

## 2. 已兑现事实
- `SocialControlState` 新增 committed-event 回查能力，当前按 `(tenant_id, event_id)` 识别 replay truth。
- `SocialControlRuntime` 新增统一 replay 路径：若同租户同 `eventId` 的 commit 已存在且内容一致，则直接返回已提交结果，并 best-effort 修复 snapshot cache。
- 若同租户复用相同 `eventId` 但提交不同 commit，则显式返回 `social_event_id_conflict`，阻止重复 journal truth。
- 新增并保持通过：
  - `test_control_plane_social_file_runtime_replays_same_event_id_after_snapshot_save_failure`
- fresh 包级回归：`cargo test -p control-plane-api --offline --tests -- --nocapture` = `56 passed`

## 3. 仍未关闭
- 仍无真正 `atomic tx boundary`
- 仍缺 `snapshot save fail` 的 failpoint / retry / repair 证据
- 仍缺 operator 级 replay / roll-forward / repair 闭环
- `S05 = not_closed / local_closure`

## 4. 当前结论
- Loop36 已把 `S05` 的客户端确认语义从“truth 已提交但只能返回 503”收敛到“首次可能 503，但 same-event retry 可精确确认已提交结果”；下一主缺口转为 failpoint 证据与 operator repair 闭环。
