# 08-B - IoT MQTT 协议插件运行时基线

## 本轮目的

把 `iot-mqtt` 从“只有 provider registry 矩阵项”推进到“真实 adapter crate + contract test + 文档回写”的第一条 IoT provider/plugin 实链，避免 IoT 主线继续停留在设计层。

## 发现的问题

- `DeviceAccessProvider / IotProtocolAdapter` 已在 `im-platform-contracts` 冻结。
- `StaticProviderRegistry::platform_default()` 也已经声明：
  - `iot-access-local`
  - `iot-mqtt`
  - `iot-xiaozhi`
- 但仓库里还没有：
  - `adapters/iot-mqtt`
  - 真实 `decode_uplink / encode_downlink / provider_health_snapshot`
- 结果是：IoT 当前仍只有 registry 与 step 文档，没有真实 protocol adapter baseline。

## 本轮决策

- 先做最小且诚实的 `iot-mqtt` baseline，不伪造完整设备 runtime。
- 本轮只交付：
  - `adapters/iot-mqtt`
  - `IotProtocolAdapter` 的真实实现
  - adapter contract test
  - step / 架构 / review 回写
- 暂不声称已经完成：
  - `DeviceAccessProvider` runtime 注入
  - HTTP provider health / protocol surface
  - `iot-xiaozhi` 真实 adapter

## 实际落地

- 新增：
  - `adapters/iot-mqtt/Cargo.toml`
  - `adapters/iot-mqtt/src/lib.rs`
  - `adapters/iot-mqtt/tests/adapter_contract_test.rs`
  - `adapters/iot-mqtt/README.md`
- 当前 `iot-mqtt` 已实现统一 `IotProtocolAdapter` contract：
  - `protocol_key`
  - `decode_uplink`
  - `encode_downlink`
  - `provider_health_snapshot`
- 归一化边界已冻结：
  - uplink -> `device.telemetry`
  - downlink -> `device.command`
  - MQTT 私有 `topic / qos / brokerEndpoint` 只保留在 envelope attributes
