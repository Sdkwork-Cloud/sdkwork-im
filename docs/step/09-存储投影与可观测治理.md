# Step 09 - 存储投影与可观测治理

## 1. 目标与范围

本 step 用于把系统的持久化、投影、可观测、生命周期治理和恢复治理统一起来，支撑前面已经落地的消息、流、RTC、Agent、IoT 能力。

本 step 主要覆盖：

- `MessageLog / StreamStore / RouteStore / BlobStore / MetadataStore` 抽象
- PostgreSQL / Redis / Object Storage 适配
- projection rebuild
- metrics / tracing / logging
- retention / archive / backup / restore

### 1.1 执行输入

- step 05-08 已形成的主链路与扩展能力边界
- 当前 `adapters/local-memory`、`adapters/local-disk`、`projection-service` 与持久化测试资产
- `132`、`140`、`141` 的存储与治理文档
- 当前 backup / restore / repair 脚本和运行时目录工具

### 1.2 本步非目标

- 不在本 step 内直接自建通用数据库或全面替换成熟组件
- 不在本 step 内完成容量压测结论
- 不在本 step 内完成多 region 部署治理

### 1.3 最小输出

- 存储抽象层与本地适配器对齐
- projection rebuild 与恢复链路基线
- plane 级 metrics / tracing / logging 入口
- 备份、恢复、修复、归档的最小可运行闭环

## 3. 当前现状与问题

当前仓库已存在：

- `adapters/local-memory`
- `adapters/local-disk`
- 与本地最小运行时相关的大量持久化、备份、恢复测试
- `projection-service`

但还需要进一步治理：

- 存储抽象尚未系统固化为目标层次
- projection rebuild 需要更统一的模型
- 可观测性指标需要按 plane 收口
- 备份、恢复、归档、冷热分层规则还需要纳入统一架构口径

## 4. 设计

### 4.1 存储分层

建议统一形成：

- `storage-core`
- `storage-postgres`
- `cache-redis`
- `object-s3`
- `adapter-local-memory`
- `adapter-local-disk`

### 4.2 抽象优先

第一阶段坚持：

- 先冻结 `MessageLog / StreamStore / RouteStore / ProjectionRebuild`
- 底层先落成熟组件
- 自研热路径留在后续演进，不在此 step 直接自建数据库

### 4.3 Projection 模型

Projection Plane 至少要覆盖：

- timeline
- inbox
- unread
- summary
- topic / thread
- AI context
- notification view

### 4.4 可观测性

必须按 plane 建立统一指标和追踪：

- Link Plane
- Route Plane
- Messaging Plane
- Stream / AI Plane
- Projection Plane
- Storage Plane
- Control Plane
- Ops Plane

## 9. 风险与回滚

### 9.1 风险

- 如果不先统一抽象，后续自研热路径将无法平滑替换
- 如果 projection 与主写路径耦合过深，会影响扩容
- 如果 observability 只补日志、不补指标和 tracing，运维闭环不成立

### 9.2 回滚

- 允许阶段性保留现有 local-memory / local-disk 实现作为验证路径
- 新适配器与新抽象先并行引入，再逐步切换服务依赖
- rebuild 和恢复路径变更必须保留旧测试作为回归基线

## 10. 完成定义

满足以下条件时，本 step 完成：

- 存储和投影已经进入统一抽象与统一观测框架
- backup / restore / repair 可重复验证
- 各主链路的持久化与恢复语义稳定

## 11. 下一步准入条件

进入 step 10 前必须确认：

- 部署脚本、配置模板和多环境 profile 能够复用当前统一的存储与观测模型

### 阶段 1：工程骨架重构

目标：

重点动作：

- 新建基础 crate

验收标准：

- 新 workspace 拓扑可编译
- 高风险 `lib.rs` 不再承载大块业务逻辑

### 阶段 2：协议与存储抽象冻结

目标：

- 冻结统一的：
  - `CCP protocol / envelope / schema / capability negotiation`
  - `sender / actor`
  - `conversation / member`
  - `message / stream`
  - `MetadataStore / RouteStore / MessageLog / StreamStore / BlobStore`

重点动作：

- 契约 crate 落地
- `ccp-*` 协议基础设施 crate 落地
- 存储 core 端口落地
- 消息和流的 envelope 统一
- 冻结 `ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`、`ccp/mqtt/1`
- 冻结 `hello / auth_bind / session_resume / heartbeat / goaway / error`

验收标准：

- API、事件、持久化模型不再漂移
- 协议版本与 schema 版本的演进规则不再漂移
- 存储层具备可替换性

### 阶段 3：连接层与路由层重构

目标：

- 建立真正的 `Link Plane` 与 `Route Plane`
- 建立 `resume / reconnect / fencing / drain` 统一模型

重点动作：

- 从 `session-gateway` 抽出连接热路径
- 建立连接分片、出站队列、背压控制
- 建立 route ownership 与节点排空模型

验收标准：

- 节点排空与 route 迁移语义稳定
- 连接层可独立扩容

### 阶段 4：消息主链路迁移

目标：

- 完整迁移：
  - conversation
  - member
  - message
  - read-cursor
  - presence
  - device sync

重点动作：

- 建立 `domain-conversation / domain-message / app-conversation`
- 对齐外部 API 与内部命令模型
- 统一 sender、member、read-cursor 和多端补偿逻辑

验收标准：

- 主链路可运行
- 编辑、撤回、已读、多端同步语义稳定

### 阶段 5：流与 RTC 基线

目标：

- 建立标准流能力与 RTC 信令能力

重点动作：

- `stream.open / delta / checkpoint / finalize / abort` 落地
- `rtc.invite / accept / reject / end` 落地
- 流与消息、RTC 与消息的桥接规则落地

验收标准：

- 普通流、AI 流、RTC 信令回流至少一条主路径可验证

### 阶段 6：AI / Agent 基线

目标：

- Agent 作为一等主体进入系统
- 支持 tool call / tool result
- 支持流式 AI 输出并物化

重点动作：

- 冻结 agent 事件模型
- 建立 AI stream materialization 规则

验收标准：

- 人与 Agent 对话可跑通

### 阶段 7：IoT / 智能硬件接入

目标：

- Device 成为一等主体
- MQTT / device stream / command stream 基线落地

重点动作：

- 引入 device session / device presence / device twin
- 统一用户、Agent、设备的实时通信模型

验收标准：

- 至少一种智能硬件接入路径跑通

### 阶段 8：性能与高可用打磨

目标：

- 在不破坏上层协议和产品能力的前提下，下探连接密度、吞吐和恢复能力

- 压测连接数、消息吞吐、流式吞吐

验收标准：

- 单机连接能力和恢复能力有明确量化结果

## 7. 技术路线结论

当前最优实施路线不是：

- 一开始就重度微服务化
- 一开始就引入大量重中间件
- 一开始就自建通用分布式数据库

而是：

`连接优先 + 协议优先 + 抽象优先 + 组件最小依赖 + 热路径后续自研演进`
