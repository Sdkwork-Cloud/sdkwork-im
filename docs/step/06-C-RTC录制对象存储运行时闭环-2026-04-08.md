# Step 06-C: RTC录制对象存储运行时闭环

## 本轮目标

- 把 `RtcRecordingArtifact` 从“RTC provider 直接给最终播放 URL”收口到“RTC provider 只给对象定位信息，运行时统一通过 `ObjectStorageProvider` 重签名”。
- 保持 provider/plugin 的标准顺序不变：
  1. tenant override
  2. deployment_profile
  3. global default

## 本轮为什么做

- 06-B 已经完成对象存储在媒体链路上的真实闭环，但 RTC 录制产物仍由各 RTC adapter 自己拼接播放地址。
- 这会导致：
  - 对象存储 deployment_profile 无法影响 RTC 播放面
  - tenant override 无法在 RTC 录制场景生效
  - 业务层继续依赖 provider 私有 URL 模板

## 实际完成项

- `RtcRecordingArtifact` 新增：
  - `bucket`
  - `storage_provider`
- `rtc-signaling-service` 内置对象存储 provider map，并默认通过 deployment profile 选择 `object-storage-volcengine`。
- `recording_artifact(...)` 改为：
  - 先从 RTC provider 获取 `bucket + object_key`
  - 再从 `ObjectStorageProvider` 生成 `playback_url`
  - 回填标准化后的 `storage_provider`
- `rtc-volcengine / rtc-aliyun / rtc-tencent` adapter 不再拼接最终播放 URL。
- standalone 与 local-minimal 的 `GET /im/v3/api/rtc/sessions/{rtcSessionId}/artifacts/recording` 已走统一运行时路径。

## 当前结论

- RTC 录制产物已和媒体下载链路采用同一套 `ObjectStorageProvider` 重签名标准。
- 默认 deployment profile 为 `object-storage-volcengine`，tenant override 可以覆盖 RTC 录制播放地址签发 provider。
- `RtcRecordingArtifact` 现已具备后续接入控制面 effective binding 展示所需的标准字段。

## 剩余差距

- 控制面尚未暴露 RTC / Object Storage 的 effective binding 与 deployment profile。
- RTC 录制播放 TTL 目前为运行时固定值，后续可纳入 deploy 配置标准。
