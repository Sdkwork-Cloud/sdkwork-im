# 100-面向DDD的IM重构升级总实施计2026-04-09

## 1. 文档定位

- 本文档是当前 `docs/step/` 的推荐入口，面向 `docs/架构/` 最新基线与当前仓库真实状态编制
- 若本文与`00-13` step 文档冲突，以本文为准；旧文档继续作为历史执行资产、细节参考和已落地证据索引
- 本文不替`docs/架构/09-实施计划.md`、`150CJ`、`151CJ`、`152CJ`，而是把它们收敛为可执行的主脊、并行车道、检查点和验收口径

## 2. 当前应用真相

- 当前仓库已具备连接、消息、流式、RTC、provider 治理、部署脚本、step/evidence 体系等基础能力
- 当前仓库尚未`friendship / space / chat_group / chat_channel / invitation / membership_request / ban_record` 收敛为独durable truth
- `conversation-runtime` 仍承担过多业务语义；目标态中 `conversation` 只能是消息运行时容器，不再承载好友、群组、空间真相
- `services/control-plane-api` 是最适合承接 durable truth 写入、治理命令、审计与 outbox 的现有服务；`conversation-runtime` 继续收敛为消息与线程运行时
- 当前 `/docs/架构/130-149`、`150CJ`、`151CJ`、`152CJ` 已把 `Current State / Target State / Deferred` 分层表达；后step 禁止再把现状和目标态混写成一句话

## 3. 规划原则

- 统一语言先于编码：冻`user / actor / member / friendship / direct_chat / space / chat_group / chat_channel / conversation / thread`
- DDD 先于实现：先冻结边界、聚合、所有权，再改数据表、API、runtime
- durable truth runtime projection 分离：关系与治理真相control-plane，消息时序写 conversation-runtime，投影视图可重建
- 大文件与service 先拆边界：继续压`services/*/src/lib.rs`，服务只保留装配和运行入口
- 协议与业务同时收敛：CCP、registry、compatibility matrix 必须和新的业务对象模型同步升级
- 证据先于完成：每step 都必须有设计、实施、测试、验证、检查点、review/backwrite 证据

## 4. 编号、完成定义与并行规则

### 4.1 编号

- step 编号统一`S00-S14`

### 4.3 并行安全规则

- 每个 step 只能有一个收口owner，负`CPR` 结论和主干合流

- 合流顺序固定为：`domain/schema -> app/API -> projection/test/docs -> review/backwrite`
- 任何涉及 `docs/架构/` 主基线、共享协议、共享契约、共享索引的变更，都必须串行收口

## 8. 统一执行卡要

- 本总计划负责排总序；真正开工前，必须为当前 `Sxx` 派生一份执行卡或在对应执行文档中补齐本节字段；优先复用 `docs/step/101-Step执行卡与并行交付模板-2026-04-10.md`
- 每个 step 开工前必须冻结：`本轮目标 / 本轮非目标/ 主写入范围/ 共享文件 owner / 集成窗口 / 验证口径`
- 每个 step 开工前必须同时冻结：`Current State / Target State / Deferred / 迁移策略 / 回退策略 / 最小交付包`
- 每个 step 完成时必须交出：`代码或文档变/ 测试与验证结论/ review 结论 / backwrite 结果 / CPR 结论 / 下一步准入判断`
- 每个 step 都必须显式回答四个问题：
  - step 对应哪几篇架构文档的哪类能力
  - 本轮最小输出是什么，少一项就不能算闭环
  - 哪些文档必须回写，不回写会造成哪类漂移
  - 下一 step 何时允许启动，何时必须阻塞
- 每个 step 都必须显式声明：`本轮已覆盖的架构能力` `本轮刻意延后的架构能力`，禁止把未分配能力留在隐含角落
- 每个 step 的完成判定至少同时满足：
  - 主链路能力已闭环，不是局部补丁
  - 自动化验证、脚本验证或结构验证已落证据
  - `docs/架构/`、`docs/step/`、`docs/review/` 已同步
  - 未完成项被显式降级到下一 step backlog，而不是混入“已完成”
- 每个 step 的并行必须遵守：
  - 只按写入范围拆车道，不按概念口号拆车道
  - 验证车道尽量只写测试、脚本、review、回写文档
  - 高风险共享文件只允许一个车道主写
  - 集成前必须有边界冻结单和最小交付包

### 8.1 Step 出口硬门

- 任一 step 收口前必须连续回写`6` yes/no 问题
  1. `95` 是否判定step 对应架构能力已闭环，而不是局部补丁
  2. `97` 是否已判定本步兑现了哪些架构结论、哪些仍 deferred
  3. `docs/架构/` 是否已完成“无需回写 / 实现更具/ 实现有偏移”三选一结论
  4. 本轮未完成能力是否已明确归属到后`Sxx`，而不是悬空
  5. 自动化测试、脚本验证、运行证据、review 证据是否至少具备 `1` 个主证据`1` 组辅证据
  6. 回退、降级、阻塞升级条件是否已写入执行卡并review 认可
- 以上任一问题若回答为 `no`，该 step 只能标记为“未闭环/阻塞/待回写”，不得进入下一步

## 12. 最快并行执行建

### 12.1 推荐车道

- `L0-架构与审计车道`：主`docs/架构/`、`docs/review/`、总计划与状态回写
- `L1-骨架与持久化车道`：主`crates/im-domain-*`、持久化端口、transaction/outbox/audit
- `L2-关系域车道`：主social/direct-chat/contact projection
- `L3-空间域车道`：主space/group/channel/member/acl
- `L4-runtime 与投影车道`：主`conversation-runtime`、`session-gateway`、`projection-service`
- `L5-协议与交付车道`：主`ccp-*`、SDK/CLI、deploy/profile/release
- `LV-验证车道`：主写测试、脚本、review、evidence，不主写业务聚合

### 12.2 最快波次

1. `Wave-0`：`S00-S02` 串行主脊，只允许 `L0 + L1 + LV` 协同
2. `Wave-1`：`S03-S04` 串行收口，`L1` 主写，`L0/LV` 跟随
3. `Wave-2`：并行执行卡`S05 || S06 || S09`，分别由 `L2 / L3 / L5` 主写，`LV` 独立验证
4. `Wave-3`：串行汇`S07 -> S08`，由 `L4` 主写，`L2/L3` 只提供绑定支持
5. `Wave-4`：先 `S10`，再并行 `S11 || S12`，由 `L4/L5` 收口基础能力，`L2/L3` 分别支撑 actor device/external 模型
6. `Wave-5`：`S13 -> S14` 串行终局，`L0 + L5 + LV` 收口发布级证据

### 12.3 集成节奏

- 每个车道只允许一个收口owner
- 每日一次边界同步，只同步：主写目录、共享契约、阻塞项、下一集成窗口
- `2-3` 天一次集成窗口；未满足最小交付包的车道不得进主干
- 验证车道始终独立，避免“实现者自己证明自己”
- 共享高风险文件先owner，再开车道；若 owner 变更，先回写 step 状态

### 12.4 小团队最优解

- 若只`1-2` 名工程师或单 agent，按宏顺序执行：`S00-S04 -> S05/S06/S09 -> S07/S08 -> S10-S12 -> S13-S14`
- 若有 `3-5` 名工程师，优先保`L1/L2/L3/L4/LV` 五条车道，不要盲目再拆
- 若有 `6-8` 名工程师，固`L0-L5 + LV` 七条车道；多出来的人只放`LV` 或独docs/review 辅助位，不拆共享聚合
- 若要多子 agent 并行，严格按照“单车道、单写入范围、单集成窗口”派工；不要把同一 step 拆成agent 同写一个聚合

### 12.5 最快完工派工法

1. `Step owner` 先写执行卡，冻结：目标、非目标、写集、共享文owner、最小交付包、集成窗口；未冻结前不派工
2. `L1/L2/L3/L4/L5` 只按写集领任务；共享契约、README、总计划、公schema 一律不并写
3. `LV` `Wave-0` 就独立启动，只写测试、脚本、review、evidence，不接业务聚owner
4. 每个 `Wave` 只做一次主干集成，合流顺序固定：`domain/schema -> app/API -> projection/test/docs -> review/backwrite`
5. 任一车道若触发共享契约漂移、CPR 未通过、回退路径不清，立即降级为阻塞项，回写执行卡后再继续

### 12.6 并行排产

| 阶段 | owner | 必做动作 | 未完成时禁止|
| --- | --- | --- | --- |
| `Freeze` | `Step owner + L0` | 冻结执行卡、写集、共享文owner、最小交付包、回退条件 | 禁止派工、禁止并行写共享文件 |
| `Build` | `L1-L5` | 按写集开发，提交车道交付物| 禁止跨车道接管共享聚|
| `Verify` | `LV` | 补测试、脚本、review、evidence、CPR 预判 | 禁止实现车道自判通过 |
| `Integrate` | `Step owner` | 按固定顺序合流，处理共享契约漂移 | 禁止跳过 review/backwrite 直接进下一 step |
| `Backwrite` | `L0 + Step owner` | 回写 `docs/架构/`、`docs/step/`、`docs/review/`，更新准入结论| 禁止以“代码已合并”替代架构兑现|

### 12.7 每波次最短节

1. `T0`：冻结执行卡，确认车owner 与共享文owner
2. `T0-T1`：实现车道开发，`LV` 同步写测试与 review 提纲
3. `T1`：一次主干集成，只合入满足最小交付包的车道
4. `T1-T2`：补 CPR、review、backwrite、Deferred 归属
5. `T2`：通过出口硬门禁后，才允许进入下一 `Wave` 或下一 `Sxx`

## 13. 主风险与应对

- 风险：仍`conversation_member` 充当群成员真相
  - 应对：`S02/S06/S07` 明确禁止，所有相关分支一律标记为迁移缺口
- 风险：并行车道同时修改共享协议、共享契约或同一聚合
  - 应对：按写集拆车道，共享文件只允许单 owner 串行收口
- 风险：文档先声称完成，代码与 evidence 未跟上
  - 应对：所step 完成必须同时满足 `代码 + 测试 + review + backwrite + CPR 结论`
- 风险：历`00-13` 被误当作当前主计划直接执行
  - 应对：README 明确本文为推荐入口，旧文档仅作历史参考
- 风险：行业对标标配能力未被显式挂靠，导致 step 做完后仍有结构空洞
  - 应对：强制使`9.1` 挂靠+ `101-*` 执行卡能力覆盖声明，未挂靠能力不得进入“已完成”

## 14. 结果标准

- `/docs/架构/` 能清楚描述当前真相、目标真相、延后项
- `/docs/step/` 能直接指导串行推进与多子 agent 并行执行
- 行业对标能力 `thread/external/history-visibility/retention/reaction/pin/member-directory` 全部有明step owner，不留未分配能力
- 关系域、空间域、会话域、治理域边界稳定，命名友好且符合 DDD
- 协议、SDK、CLI、部署、性能、HA/DR 与业务模型一致，不再各走一套
