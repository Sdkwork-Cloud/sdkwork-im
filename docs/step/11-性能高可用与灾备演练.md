# Step 11 - 性能、高可用与灾备演练

## 1. 目标与范围

本 step 用于对前面已完成的架构与实现做量化验证，确认系统不仅“功能上能跑”，而且在连接密度、吞吐、恢复、排空、升级和回滚上具备真实可交付能力。

本 step 主要覆盖：

- 单机连接密度验证
- 消息吞吐验证
- 流式吞吐验证
- drain / rebalance / failover 演练
- 备份恢复与 RTO / RPO 演练
- 多 cell / 多 region 升级与兼容演练

### 1.1 执行输入

- step 10 已统一的部署入口和 profile
- step 04-09 已稳定的运行时、主链路、治理和存储基线
- 当前 metrics / tracing / health / diagnostic 能力
- 当前本地测试、脚本和 CLI 可形成的多客户端场景

### 1.2 本步非目标

- 不在本 step 内继续大规模功能开发
- 不在本 step 内变更协议骨架或消息语义
- 不在本 step 内锁定最终基础设施选型为不可替换状态

### 1.3 最小输出

- 一套分层压测与演练场景
- 至少一轮连接、消息、流三类量化结果
- 排空、恢复、升级回滚的演练结果
- 可进入发布评审的性能与可用性证据

## 3. 当前现状与问题

当前仓库已经具备很多本地持久化、恢复和路由排空相关测试，但还缺少更系统的量化演练体系：

- 缺少统一的压测分层目标
- 单机连接能力和吞吐能力的量化结论还未固化
- 多环境故障演练和协议升级演练还未形成标准步骤

## 4. 设计

### 4.1 压测分层

建议把性能验证分为三层：

- `CI Smoke Tier`：小规模、快速、每次改动都可跑
- `Pre-Release Tier`：中等规模、用于版本发布前
- `Capacity Tier`：专用环境、用于验证单机和集群设计目标

### 4.2 指标目标

建议至少量化：

- 握手成功率
- 活跃连接数
- reconnect / resume 成功率
- 消息发送 TPS
- stream 帧吞吐
- route 迁移成功率
- 节点排空完成时间
- 备份恢复时间

### 4.3 高可用与灾备目标

必须明确：

- 连接故障恢复路径
- 节点排空路径
- 路由迁移路径
- 数据恢复路径
- 协议升级回滚路径

### 4.4 单机目标口径

建议采用分层目标：

- CI 不追求高连接，重在回归验证
- 预发布环境验证 `1k-10k` 级别连接与稳定性
- 专用容量环境验证更高连接密度和尾延迟

如果环境允许，应逐步逼近架构文档提出的：

- 第一落地目标：`10 万级长连接 / 单机`
- 设计上限预留：更高密度演进空间

## 5. 实施落地规划

### 5.1 任务拆解

1. 建立统一 benchmark / load / chaos 脚本
2. 建立连接、消息、流、RTC 的性能场景集
3. 建立 drain / rebalance / failover 演练脚本
4. 建立 backup / restore / replay 演练脚本
5. 建立协议升级与兼容回滚演练脚本
6. 输出量化报告并写入 `docs/review/`

### 5.2 重点路径

重点涉及：

- `services/sdkwork-im-gateway/tests/`
- `services/session-gateway/tests/`
- `services/streaming-service/tests/`
- `services/projection-service/tests/`
- `bin/`
- `deployments/`
- `tools/chat-cli/`

### 5.3 演练顺序

推荐顺序：

1. 单点连接与消息压测
2. reconnect / resume 稳定性验证
3. route drain / rebalance 验证
4. backup / restore 验证
5. 协议升级 / 回滚演练

### 5.4 输出要求

每类演练必须输出：

- 环境说明
- 输入规模
- 执行命令
- 指标结果
- 异常点
- 结论和改进建议

## 6. 测试计划

建议重点测试与演练：

- 握手与在线连接压测
- 群聊 / 单聊消息吞吐压测
- AI 流、普通流、设备流吞吐压测
- route drain / rebalance E2E 演练
- runtime backup / restore / replay 演练
- 协议版本切换与兼容矩阵验证

可基于现有测试和脚本扩展：

- `cluster_drain_rebalance_e2e_test.rs`
- `cluster_realtime_routing_e2e_test.rs`
- `stream_runtime_persistence_test.rs`
- `runtime_dir_restore_test.rs`
- `open-chat-test` 与 `chat-cli` 驱动的多客户端场景

## 7. 结果验证

本 step 完成后，需要验证：

- 性能和可用性已有量化证据，而不是主观判断
- 高可用和灾备路径已经过实际演练
- 升级和回滚具有可重复执行的脚本化能力
- 架构的扩容主张与真实结果能够对应上

### 8.2 推荐并行车道

### 8.3 架构能力闭环判定

- 至少要沉淀一轮量化压测结果和一轮真实演练复盘，才能支撑“可扩容、可恢复、可运维”的架构主张。
- 如果只有理论指标或只做压测不做故障演练，本 step 不算闭环。
- 闭环验收以 [`95-架构能力闭环验收标准`](./95-架构能力闭环验收标准.md) 中 Step 11 条目为准。

### 8.4 快速并行执行建议

- 先冻结压测拓扑、容量指标和故障场景，再并行做性能、HA/DR、升级回滚三类演练。
- 推荐让验证 owner 独立保存压测与演练证据，避免实现 owner 同时裁决结果。
- 未形成可重复执行的脚本和结论前，不允许用单次跑分或口头结论代替验收。

### 8.5 完成后必须回写的架构文档

- 强制范围：本文件 `## 2. 架构对齐` 中列出的全部架构文档。
- 回写重点：单机连接与并发目标、容量边界、HA/DR 演练结论、跨 region 升级兼容结论是否已经以量化结果回写架构文档。

## 9. 风险与回滚

### 9.1 风险

- 没有量化验证就进入发布，会把风险推给生产
- 只做压测不做故障演练，会误判高可用能力
- 只做单服务测试不做多服务场景，无法验证真实闭环

### 9.2 回滚

- 压测与演练脚本先在 self-hosted.split-services.development 和受控环境执行
- 先做小规模再逐步扩大规模
- 任何优化调整都必须能回退到上一个稳定配置或实现

满足以下条件时，本 step 完成：

- 已建立分层性能与演练体系
- 已输出至少一轮量化结果
- 已找到并记录需要进入下一轮优化的问题
