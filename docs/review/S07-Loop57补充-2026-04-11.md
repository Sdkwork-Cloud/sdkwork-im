# S07 Loop57补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop57 固定只做 `shared-channel repeated failure -> dead-letter`
- `dead-letter requeue`、`retry scheduler`、`remote republish`、`delivery ownership / lease`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 的 `SocialControlState` 当前新增 `dead_letter_shared_channel_sync_requests`
- 对同一 shared-channel sync request：
  - 初始失败会进入 `pending_shared_channel_sync_requests`
  - 失败累计达到固定阈值 `3` 后，会从 `pending` 转入 `dead_letter`
- 转入 dead-letter 后：
  - `next healthy ready-pair write` 自动重试不会继续消费该 request
  - `POST /api/v1/control/social/runtime/repair-shared-channel-sync` 也不会继续消费该 request
- `repair-shared-channel-sync` 响应与 aggregate/report surface 当前显式暴露 dead-letter 计数
- 新增真实 dual-service e2e：
  - 首个 actor 连续失败三次后进入 dead-letter
  - repair route 在健康 trigger 下返回 `noop`
  - 后续新的健康 ready-pair 写入只 materialize 第二个 actor
  - dead-lettered actor 继续拿不到 shared history，新 actor 可以读取 shared history

## 3. 运行时边界
- 本轮 dead-letter 只是 poison backlog 隔离，不是 operator 恢复链路
- 当前仍没有：
  - dead-letter requeue / replay route
  - retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once semantics
- dead-lettered request 当前仍会在重复命中相同 key 的失败记录时刷新 `failureCount / lastError`，但不会重新进入 pending
