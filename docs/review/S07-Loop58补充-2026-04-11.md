# S07 Loop58补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop58 固定只做 `shared-channel dead-letter requeue`
- `automatic retry scheduler`、`remote republish`、`delivery ownership / lease`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 的 `SocialControlState` 当前新增 dead-letter -> pending 的显式状态迁移函数
- 控制面新增 `POST /api/v1/control/social/runtime/requeue-dead-letter-shared-channel-sync`
- 新 route 当前会：
  - 读取 `dead_letter_shared_channel_sync_requests`
  - 将全部 dead-letter request 转回 `pending_shared_channel_sync_requests`
  - 返回 `status / pendingBefore / deadLetterBefore / requeued / pendingAfter / deadLetterAfter`
  - 记录 `control.social_runtime_shared_channel_sync_dead_letter_requeued` audit 事件
- requeue 后的投递链路当前不新增第二套实现，而是继续复用既有 `POST /api/v1/control/social/runtime/repair-shared-channel-sync`
- 新增真实 dual-service e2e：
  - 首个 actor 连续失败三次后进入 dead-letter
  - healthy trigger 下调用 requeue route，会把 backlog 回灌到 pending
  - 随后调用 repair route，会真实 materialize remote runtime linked member truth
  - 原 dead-letter actor 最终可以读取 shared history，证明 recovery path 已闭合到“operator requeue + existing repair”

## 3. 运行时边界
- 本轮 requeue 是“operator 恢复边界”，不是“后台自愈边界”
- 当前仍没有：
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once semantics
  - dead-letter 单条/筛选 requeue body contract
- requeue 当前会保留 `failureCount / lastError`
  - 这样做的含义是：operator 已经显式重武装请求，但不会伪装成全新 request
  - 若环境仍然不健康，请求在后续 repair 失败后会较快重新进入 dead-letter
