# Step 08 / CP08-3 质量审计与复盘 - 2026-04-08

## 审计范围
- `services/local-minimal-node/src/node/access.rs`
- `services/streaming-service/src/lib.rs`
- `services/local-minimal-node/tests/device_stream_e2e_test.rs`
- `services/streaming-service/tests/device_scope_guard_test.rs`

## 审计结论
- 本轮未发现阻塞 `CP08-3` 交付的剩余缺陷。
- 设备 telemetry / command 已进入统一 stream 主链路，且没有为 IoT 再造独立 runtime/protocol。
- `device` scope 的访问现在必须经过 authorizing gateway，减少了后续 Step 08/09 再返工鉴权边界的风险。

## 正向结果
- `device.telemetry` 的 sender 已真实复用 `DeviceSubject::sender(...)`，不再把 owner principal 错当设备主体。
- `device.command.send` 与 `device.telemetry.read` 已从文档能力名变成真实 capability gate。
- 独立 `streaming-service` 明确拒绝 `device` scope，协议母体继续保持单一入口，不向旁路扩散。
- 既有普通 stream 回归、conversation-bound stream 权限回归仍然通过，说明本轮没有打坏普通 IM/stream 主路径。

## 本轮发现并修正的问题
- `device.telemetry` frame 之前沿用通用 `AuthContext -> Sender` 映射，导致 `sender.id` 错误落成 owner principal。
- `device.command` 之前没有显式 capability gate，缺少 `device.command.send` 仍可写入。
- `device.telemetry` 之前也没有显式读权限，缺少 `device.telemetry.read` 仍可读取。
- 独立 `streaming-service` 之前允许直接打开 `device` scope stream，形成潜在绕过路径。

## 剩余风险
- 当前 `CP08-3` 仍是最小闭环，没有把 MQTT transport、device twin/shadow、批量设备编排一次性做完。
- `device.command` 目前仍以最小 stream 路径落地，还没有补充更复杂的 ACK / retry / timeout 语义。
- `Step 09` 尚未接手更完整的投影、恢复、可观测治理，因此设备链路的长期 durability 仍需后续波次继续加固。

- 设备路径继续复用统一 stream 主链路的收益已经很清楚：补的是边界和 sender，不是再开一条新协议。
