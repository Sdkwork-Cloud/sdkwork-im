# S07 Loop53补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop53 固定只做 `self-hosted.split-services.development embedded real runtime consumer wiring`
- `standalone control-plane build_app* / main.rs`、跨服务 outbox / retry / remote republish、更强 `im_thread_subscription` durable model / per-user unread-notification level 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 抽出了不带 `/healthz` 的 embedded control surface builder，可供组合装配面直接 merge `/backend/v3/api/control/*`
- `control-plane-api` 补齐了 `runtime_dir + governance_sinks + shared_channel_sync_trigger` 组合 builder，使 file-backed social runtime 与 injected trigger 可以在同一装配点同时成立
- `sdkwork-im-server` 当前把 embedded control-plane surface 合并进默认/公开 router，并以生产依赖方式接入 `control-plane-api`
- `sdkwork-im-server` 新增同进程 `SharedChannelLinkedMemberSyncTrigger` implementation，直接把 ready-pair payload 投递到 `ConversationRuntime::sync_shared_channel_linked_member(...)`
- `projection-service` 的 timeline 鉴权当前已接受 `can_read_shared_history()` 的 linked member，因此 self-hosted.split-services.development merged app 上的 linked actor 能直接消费 shared history

## 3. 运行时边界
- 本轮闭环只覆盖 `sdkwork-im-server` 这条 embedded 装配面
- standalone `control-plane-api::build_app*` 与 `services/control-plane-api/src/main.rs` 仍不注入真实 runtime consumer
- 当前没有 outbox / retry / remote republish；shared-channel auto-sync 仍不是跨服务 exactly-once 闭环
- `projection-service` 本轮只放宽了 `timeline_from_auth_context(...)`；其他 read surface 是否也应接受 linked member，当前未宣称
