# S07 Loop59补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop59 固定只做 `dead-letter requeue failure-budget rearm`
- `requeue selection contract`、`remote republish`、`delivery ownership / lease`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 当前将 `requeue-dead-letter-shared-channel-sync` 明确为一次 operator 再授权动作
- `SocialControlState::requeue_dead_letter_shared_channel_sync_requests()` 当前会：
  - 把 dead-letter request 搬回 pending
  - 将 `failureCount` 重置为 `0`
  - 保留原 request payload 与 `lastError`
- 新增真实 dual-service e2e：
  - request 先在连续失败后进入 dead-letter
  - operator 执行 requeue 后，pending item 的 `failureCount` 变为 `0`
  - 下一次 repair 即使仍失败，也只会把 request 留在 pending，并把 `failureCount` 记为 `1`
  - request 不会因为历史失败预算被立即再次送回 dead-letter

## 3. 运行时边界
- 本轮 rearm 只是修正 operator recovery semantics，不是新增后台自愈
- 当前仍没有：
  - targeted requeue / selection contract
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once semantics
- 当前选择保留 `lastError`
  - 含义是保留上一次失败上下文，便于 operator 审计
  - 但这不是新的失败计数，requeue 后的 request 重新从 `failureCount = 0` 起步
