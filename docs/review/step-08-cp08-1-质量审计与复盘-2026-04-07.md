## 正向结果
- `AgentSubject` 与 `DeviceSubject` 都直接复用 `im_domain_core::message::Sender`，没有新造平行 sender 结构。
- `DeviceSubject::sender(...)` 显式回填 `device_id`，与设备主体语义保持一致。
- 平台 facade 同步导出新的主体契约，避免后续平台层测试继续依赖 crate 私有路径。
- 红灯测试先证明缺口存在，再由最小实现转绿，符合当前仓库的持续迭代要求。

## 本轮发现并修正的问题
- `craw-chat-contract-agent` 只有 automation execution 契约，缺少 agent 主体模型。
- `craw-chat-contract-iot` 只有 twin 契约，缺少 device 主体模型。
- `im-platform-contracts` 缺少对新主体契约的统一导出。
- `craw-chat-contract-iot` 原先没有接入 `im-domain-core`，无法直接复用统一 `Sender`。

## 剩余风险
- 当前 sender 物化只解决了主体快照问题，还没有把 tool call、stream frame、telemetry frame 接到统一事件主链路。
- `DeviceSubject` 目前保留 `owner_principal_id`，但本轮尚未把设备主体权限边界真正接入鉴权上下文。
- `AgentSubject` 仅完成主体建模，尚未证明流式输出、handoff、tool result 生命周期与现有 stream/message 主链路完全收敛。
