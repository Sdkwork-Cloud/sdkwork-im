# S07 Loop56补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop56 固定只做 `shared-channel pending backlog auto-retry on next healthy ready-pair write`
- `dead-letter`、`remote republish`、`delivery ownership / lease`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 当前会在 dispatch shared-channel sync 前先合并：
  - `SocialControlState.pending_shared_channel_sync_requests`
  - 本次 ready pair 新生成的 sync request
- 合并后的 dispatch 队列会按 request key 去重，并按“pending backlog 优先、本次新请求随后”的顺序投递
- 当 trigger 已恢复健康且新的 ready pair 写入发生时：
  - 既有 backlog 会先被自动重放
  - 成功的 backlog 项会被即时清理
  - 本次新请求会在同一轮 dispatch 中继续投递
- 新增真实 dual-service e2e：
  - 首个 external-member-link 写入在 failing trigger 下落成 durable backlog
  - 切换到 public HTTP trigger 后，不调用 repair route，而是再写入第二个 ready pair
  - 最终验证 backlog 被自动清空，且 `actor_alice / actor_bob` 都能从 remote runtime 读取 shared history

## 3. 运行时边界
- 本轮自动重试只存在于“下一次健康 ready-pair 写入口”，不是独立后台任务
- `repair-shared-channel-sync` operator surface 继续保留，适用于“没有后续 ready pair 写入”时的显式人工修复
- backlog 清理路径当前仍是 best-effort snapshot save；若清理持久化失败，旧 backlog 仍可能残留并被后续 repair / auto-retry 再次消费，因此继续依赖 runtime sync 幂等语义兜底
- 当前仍没有：
  - retry scheduler
  - dead-letter queue
  - remote republish ownership / SLA / lease
