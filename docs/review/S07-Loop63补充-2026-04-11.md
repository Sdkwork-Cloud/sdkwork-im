# S07 Loop63补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop63 固定只做 `pending targeted release lifecycle`
- `stale-claim / lease / SLA`、`automatic retry scheduler`、`https:// public runtime target` 继续 deferred，不与本轮并行混做

## 2. 本轮交付
- `control-plane-api` 当前新增 `POST /backend/v3/api/control/social/runtime/release-pending-shared-channel-sync-targeted`
- targeted release 当前会：
  - 只释放被选中且归当前 operator 所有的 pending request
  - 把 owner 元数据清空，使 request 返回 unowned pool
  - 保持未选中的 pending request 原样不动
- foreign operator 对已 claim request 的 targeted release 当前会收到 `409 shared_channel_sync_owner_conflict`
- released request 当前可被其他 operator 再次 claim，并复用既有 targeted republish 路径
- 新增真实 dual-service e2e：
  - operator A claim pending request
  - operator B release 被 `409` 拦下
  - operator A release 成功，inventory owner 字段恢复为 `null`
  - operator B 重新 claim 并 republish 成功，actor 恢复 shared history 可读

## 3. 运行时边界
- 本轮补的是 claim lifecycle 的 manual release seam，不是 stale-claim / lease / SLA 实现
- 当前仍没有：
  - `claimedAt / leaseExpiresAt / SLA` 元数据
  - foreign operator 的 force release / takeover contract
  - ownership-aware `repair-shared-channel-sync`
  - release-ready exactly-once semantics
- 现有 `repair-shared-channel-sync` route 仍保留：
  - 含义是当前系统同时拥有 `全量 repair`、`claim + targeted republish`、`owner-only targeted release` 三条 operator 投递/恢复路径
  - 但这还不是完整统一的 stale-claim / lease policy language
