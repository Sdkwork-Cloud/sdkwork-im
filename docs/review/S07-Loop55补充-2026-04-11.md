# S07 Loop55补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop55 固定只做 `shared-channel sync durable pending backlog + operator repair seam`
- `automatic retry scheduler`、`dead-letter`、`remote republish`、`exactly-once delivery semantics` 与 `https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 的 `SocialControlState` 新增 `pending_shared_channel_sync_requests`
  - 当 ready pair 存在但 `shared_channel_sync_trigger` 未配置时，请求会被 durable 记入 pending backlog，而不再静默丢弃
  - 当 trigger dispatch 在中途失败时，失败请求及其后的未投递请求会被 durable 记入 pending backlog；已提交的 social durable truth 不回滚
- 控制面新增 `POST /api/v1/control/social/runtime/repair-shared-channel-sync`
  - operator 可显式重放 pending shared-channel sync backlog
  - 重放成功的请求会被清空；失败的请求会保留并刷新 `failureCount / lastError`
- `repair-derived-snapshot` 与 `repair-social-runtime-dir` 当前 replay `social-commit-journal.json` 时会保留 snapshot 中已有的 pending shared-channel sync backlog，不再把 operator backlog 清空
- `SocialAggregateCountsResponse` 当前显式暴露 `pendingSharedChannelSyncRequests`，使 snapshot repair / CLI repair 的观测面能够直接反映 backlog 规模
- 新增真实 dual-service e2e：
  - control-plane 先用 failing trigger 触发 ready pair 写入
  - 验证 `social-state.json` 已持久化 pending backlog
  - 再切换到真实 public HTTP trigger 并调用 repair route
  - 最终验证远端 standalone `conversation-runtime` materialize linked member 后，linked actor 可读取 shared history

## 3. 运行时边界
- 本轮把“失败持久化 + 人工 repair”补齐到了 operator 可消费的最小闭环，但没有声称自动 delivery 已闭环
- 当前 pending backlog 仍属于 `control-plane-api` 的 snapshot-backed operator state，而不是 social commit journal 的领域事件
- 正常成功路径当前对同 key pending backlog 的清理是 best-effort；若清理持久化失败，可能残留可重复 repair 的 backlog，依赖 runtime sync 的幂等语义兜底
- 当前仍没有：
  - 自动 retry scheduler
  - dead-letter / poison message 隔离
  - remote republish ownership / lease / SLA
  - release-ready cross-service exactly-once semantics
