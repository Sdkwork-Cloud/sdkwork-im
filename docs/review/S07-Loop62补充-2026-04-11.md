# S07 Loop62补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop62 固定只做 `pending targeted claim + republish ownership`
- `claim lifetime / lease / SLA`、`automatic retry scheduler`、`https:// public runtime target` 均继续 deferred，不与本轮并行混做

## 2. 本轮交付
- `control-plane-api` 当前新增 `POST /backend/v3/api/control/social/runtime/claim-pending-shared-channel-sync-targeted`
- pending inventory item 当前显式暴露：
  - `requestKey`
  - 原始 `request` payload
  - `failureCount`
  - `lastError`
  - `ownerActorId`
  - `ownerActorKind`
- `PendingSharedChannelSyncRequest` 当前以 durable 方式保留可选 owner 元数据
- targeted claim 当前会：
  - 只 claim 被选中的 pending request
  - 把 owner 绑定为当前 control operator 的 `actorId / actorKind`
  - 保持未选中的 pending request 原样不动
- targeted republish 当前会：
  - 只允许当前 operator 投递“自己已 claim”的 pending request
  - 对 foreign operator 或未 claim request 返回 `409 shared_channel_sync_owner_conflict`
  - 成功后清理被选中的成功 request，未选中的 pending request 继续留在 backlog
- 新增真实 dual-service e2e：
  - 两条 shared-channel request 先进入 pending backlog
  - operator A 先 claim 其中一条
  - operator B 对 A 已 claim 的 request 做 targeted republish 时收到 `409`
  - operator A republish 成功后，只有被选中的 actor 恢复 shared history 可读
  - 未选中的 actor 继续 `403`

## 3. 运行时边界
- 本轮补的是 operator ownership boundary，不是 claim lifetime / lease / SLA 实现
- 当前仍没有：
  - `claimedAt / leaseExpiresAt / SLA` 元数据
  - targeted unclaim / release / force-takeover operator contract
  - ownership-aware 的 `repair-shared-channel-sync` policy language
  - release-ready exactly-once semantics
- 现有 `repair-shared-channel-sync` route 仍保留：
  - 含义是当前系统同时拥有 `全量 repair` 与 `claim + targeted republish` 两条 operator 投递路径
  - 但这还不是完整、统一的 delivery ownership / lease policy
