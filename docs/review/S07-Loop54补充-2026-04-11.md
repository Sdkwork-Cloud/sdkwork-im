# S07 Loop54补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop54 固定只做 `standalone control-plane public HTTP runtime consumer wiring`
- `cross-service outbox / retry / remote republish`、`https public runtime target support`、更强 `im_thread_subscription` durable model / per-user unread-notification level 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 新增 `PublicSharedChannelLinkedMemberSyncTrigger`
  - 该 trigger 不依赖 `conversation-runtime` runtime 类型
  - 它通过 public bearer 身份把 `SharedChannelLinkedMemberSyncRequest` 投递到 standalone `conversation-runtime` 的 `POST /api/v1/conversations/shared-channel-links/sync`
- `control-plane-api` 当前显式提供：
  - `build_public_shared_channel_sync_trigger(...)`
  - `build_public_app_with_shared_channel_sync_trigger(...)`
  - `configured_public_shared_channel_sync_trigger(...)`
  - `SHARED_CHANNEL_SYNC_TARGET_BASE_URL_ENV`
- `services/control-plane-api/src/main.rs` 现在会在检测到 `CRAW_CHAT_SHARED_CHANNEL_SYNC_TARGET_BASE_URL` 时装配真实 public HTTP trigger；未配置时仍回落到原有 `build_public_app()`
- 新增跨进程 e2e：public control-plane 写入 ready pair 后，远端 standalone runtime 的 linked actor 可立即读取 shared history

## 3. 运行时边界
- 本轮闭环覆盖了 standalone `control-plane-api` 的最小 public HTTP 组合根，而不再局限于 embedded `local-minimal-node`
- 当前 bridge 只支持 `http://` public runtime target；`https://` 仍未进入本轮 truth
- 当前仍没有 outbox / retry / dead-letter / remote republish，因此 shared-channel sync 仍不是跨服务 exactly-once 闭环
- 本轮验证的 linked-member 消费面是 `shared history read`；`projection-service` 其他 read surfaces 是否同步放宽，当前未宣称

## 5. 结论
- `S07` 仍只能标记为 `not_closed / local_closure`
- `standalone control-plane-api` 当前已具备 `ready-pair auto-trigger -> public HTTP sync bridge -> standalone runtime linked history read` 的最小跨进程闭环
- 下一单点主缺口收敛为 `cross-service shared-channel sync outbox / retry / delivery ownership boundary`，而不再是 standalone control-plane 完全没有真实 consumer
