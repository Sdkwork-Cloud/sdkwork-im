# S07 Loop60补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop60 固定只做 `dead-letter inventory visibility + targeted requeue`
- `remote republish`、`delivery ownership / lease`、`automatic retry scheduler`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 当前新增 `GET /api/v1/control/social/runtime/dead-letter-shared-channel-sync`
- dead-letter inventory item 当前会显式暴露：
  - `requestKey`
  - 原始 `request` payload
  - `failureCount`
  - `lastError`
- `control-plane-api` 当前新增 `POST /api/v1/control/social/runtime/requeue-dead-letter-shared-channel-sync-targeted`
- `SocialControlState::requeue_selected_dead_letter_shared_channel_sync_requests(...)` 当前会：
  - 只回灌被选中的 dead-letter request
  - 把被选中 request 的 `failureCount` 重置为 `0`
  - 保留原 request payload 与 `lastError`
  - 让未被选中的 dead-letter request 继续停留在 poison backlog
- 新增真实 dual-service e2e：
  - 两条 shared-channel request 先一起进入 dead-letter
  - operator 先通过 inventory 抓到两条稳定 `requestKey`
  - 只 targeted requeue 其中一条
  - repair 后只有被选中的 actor 恢复 shared history 可读
  - 未被选中的 actor 继续 `403`

## 3. 运行时边界
- 本轮补的是 operator 可读 / 可选恢复边界，不是新增后台自愈或 cross-service delivery orchestration
- 当前仍没有：
  - inventory filter / pagination / search DSL
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once semantics
- 现有“全量 dead-letter requeue” route 仍保留
  - 含义是 operator 现在同时拥有 `全量回灌` 与 `定点回灌` 两条最小恢复路径
  - 但这还不是统一的 delivery policy language

- `S07` 仍只能标记为 `not_closed / local_closure`

- 下一单点主缺口继续收敛为 `remote republish / delivery ownership semantics`
