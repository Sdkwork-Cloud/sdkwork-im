# S07 Loop61补充 - 2026-04-11

## 1. 本轮唯一主缺口
- Loop61 固定只做 `pending inventory visibility + targeted republish`
- `delivery ownership / lease / SLA`、`automatic retry scheduler`、`https:// public runtime target` 均保持 deferred，不与本轮并行混写

## 2. 本轮交付
- `control-plane-api` 当前新增 `GET /backend/v3/api/control/social/runtime/pending-shared-channel-sync`
- pending inventory item 当前会显式暴露：
  - `requestKey`
  - 原始 `request` payload
  - `failureCount`
  - `lastError`
- `control-plane-api` 当前新增 `POST /backend/v3/api/control/social/runtime/republish-pending-shared-channel-sync-targeted`
- targeted republish 当前会：
  - 只投递被选中的 pending request
  - 成功后清理被选中的成功 request
  - 保留未选中的 pending request，不触碰其 backlog 状态
- 新增真实 dual-service e2e：
  - 两条 shared-channel request 先进入 pending backlog
  - operator 先通过 pending inventory 抓到两条稳定 `requestKey`
  - 只 targeted republish 其中一条
  - 只有被选中的 actor 恢复 shared history 可读
  - 未被选中的 actor 继续 `403`

## 3. 运行时边界
- 本轮补的是 operator 显式 remote republish 边界，不是新增后台 delivery owner 或 lease 协议
- 当前仍没有：
  - pending inventory filter / pagination / search DSL
  - delivery lease owner / SLA
  - automatic retry scheduler
  - release-ready exactly-once semantics
- 现有 `repair-shared-channel-sync` route 仍保留
  - 含义是 operator 现在同时拥有 `全量 repair` 与 `定点 republish` 两条最小投递路径
  - 但这还不是统一的 delivery ownership policy language

## 5. 结论
- `S07` 仍只能标记为 `not_closed / local_closure`
- Loop61 已把 shared-channel sync 从“operator 只能全量 repair pending backlog”推进到“operator 可先看 pending inventory，再按 `requestKey` 定点 republish”
- 下一单点主缺口继续收敛为 `delivery ownership / lease / SLA semantics`
