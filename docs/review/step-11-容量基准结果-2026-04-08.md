# Step 11 容量基准结果 - 2026-04-08

## 环境说明
- 波次 / Step：`Wave D / Step 11 / CP11-2`
- 执行基线：
  - `profile = self-hosted.split-services.development`
  - `tier = CI Smoke Tier`
- 执行入口：
  - `tools/perf/step-11-cp11-2-local-baseline.json`
  - `services/sdkwork-im-gateway/tests/performance_quant_baseline_test.rs`
- 执行命令：
  - `cargo test -p sdkwork-im-gateway --offline --test performance_quant_baseline_test -- --nocapture`

## 输入规模
- `connectionCount = 32`
- `messageCount = 64`
- `frameCount = 64`

## 结果

| 场景 | 成功数 | 总时长 | P95 | 吞吐 |
| --- | --- | --- | --- | --- |
| `connection` | `32 / 32` | `17.754 ms` | `connectP95Ms = 15.108 ms` | `1802.431 connections/s` |
| `message` | `64 / 64` | `8.263 ms` | `postP95Ms = 0.152 ms` | `7745.652 message/s` |
| `stream` | `64 / 64` | `6.03 ms` | `appendP95Ms = 0.117 ms` | `10613.071 frames/s` |

## 异常点
- `connection` 的 `P95` 明显高于 `message / stream`，说明在当前本地 smoke 规模下，握手与 websocket 建联仍是主要固定成本。
- `message` 与 `stream` 当前结果极高，原因是测试运行在进程内 `self-hosted.split-services.development` 基线上，不能直接外推到预发布或容量环境。

## 当前结论
- `CP11-2` 已满足“至少一轮连接、消息、流三类量化结果”的最低要求。
- 当前结论只适用于：
  - `self-hosted.split-services.development`
  - `CI Smoke Tier`
- 当前不应得出的结论：
  - 单机 `1k-10k` 或更高连接能力已经验证
  - Step 11 整步已经闭环

## 下一步建议
1. 维持当前 `CI Smoke Tier / self-hosted.split-services.development` 量化基线，作为 `Step 12` 之后回归与更高 tier 对照的固定底座。
2. 在后续波次补齐 `Pre-Release Tier` 与 `Capacity Tier`，避免把本地 smoke 数据误写成容量结论。
