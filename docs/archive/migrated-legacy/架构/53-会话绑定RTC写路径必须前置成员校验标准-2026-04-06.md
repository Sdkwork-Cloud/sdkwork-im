# 53-会话绑定 RTC 写路径必须前置成员校验标准-2026-04-06

## 1. 问题背景

在继续 review `sdkwork-im-server` 与 `im-call-runtime` 的 RTC 能力时，发现会话绑定型 RTC 写路径存在一类越权缺陷：

- 非会话成员此前可以直接调用 `POST /im/v3/api/calls/sessions`，创建绑定任意已知 `conversationId` 的 RTC 会话。
- 非会话成员此前调用 `POST /im/v3/api/calls/sessions/{id}/signals` 时，虽然 IM 投影阶段会因为消息写入权限不足返回 `403`，但 `im-call-runtime` 内部已经先写入了 RTC signal 状态，导致 `signalingStreamId` 等 RTC 会话字段被脏写。
- 同一类风险同样存在于 `invite / accept / reject / end` 这类先写 RTC 状态、后做 IM 投影的路径。

这违反了前面已经冻结的“会话私有读写必须以前置成员校验作为入口门槛”的标准，也会导致“HTTP 返回失败但 RTC 内核状态已被污染”的隐蔽一致性问题。

## 2. 标准冻结

### 2.1 适用范围

只要 RTC 请求满足以下任一条件，就必须视为“会话绑定 RTC 写请求”：

- 创建 RTC 会话时，请求体携带 `conversationId`
- 对既有 RTC 会话执行写操作，且该 RTC 会话已绑定 `conversationId`

### 2.2 强制规则

对于会话绑定 RTC 写请求，服务端必须在任何 RTC 状态写入之前完成 active member 校验。

受此规则约束的最小接口集合为：

- `POST /im/v3/api/calls/sessions`
- `POST /im/v3/api/calls/sessions/{id}/invite`
- `POST /im/v3/api/calls/sessions/{id}/accept`
- `POST /im/v3/api/calls/sessions/{id}/reject`
- `POST /im/v3/api/calls/sessions/{id}/end`
- `POST /im/v3/api/calls/sessions/{id}/signals`

### 2.3 错误语义

- 若当前操作者不是目标会话的 active member，返回 `403 conversation_permission_denied`
- 若目标 RTC 会话不存在，返回现有 `404 rtc_session_not_found`
- 不允许出现“鉴权失败，但 RTC 状态已经写入成功”的部分提交

### 2.4 创建路径的优先级约束

`POST /im/v3/api/calls/sessions` 还必须同时满足 RTC 会话幂等标准：

- 若同一 `rtcSessionId` 已存在，应先基于现有 RTC 会话聚合执行幂等/冲突判定
- 对完全相同的重试请求，继续返回既有 RTC 会话
- 对内容不同的重试请求，继续返回 `409 rtc_session_conflict`
- 只有在确认这是一次“新建 RTC 会话”后，才对请求体中的 `conversationId` 执行 active member 校验

这样可以避免把原本应返回 `409 rtc_session_conflict` 的重复创建场景错误提前打成 `404 conversation_not_found` 或 `403 conversation_permission_denied`

## 3. 落地方式

### 3.1 `sdkwork-im-server`

- 创建 RTC 会话时，先尝试读取同一 `rtcSessionId` 的既有会话
- 若会话已存在，优先保留 RTC 幂等/冲突语义；若该既有会话绑定 `conversationId`，则基于既有会话做成员校验
- 若会话不存在且请求带 `conversationId`，先执行 `ensure_conversation_member(...)`
- 对既有 RTC 会话执行写操作前，先读取 RTC 会话聚合
- 若会话绑定 `conversationId`，先执行 `ensure_conversation_member(...)`
- 只有成员校验通过后，才允许调用 `rtc_runtime` 的实际写方法

### 3.2 `im-call-runtime`

- 提供只读 `session(...)` 查询能力，供接入层在写入前判断 RTC 会话是否绑定会话
- 不改变 RTC 内核现有状态机语义，只补足接入层授权前置判断所需的读能力

  - `test_non_member_cannot_create_conversation_bound_rtc_session`

第二个用例验证的不是简单 `403`，而是更关键的“不允许先被越权请求脏写 `signalingStreamId`，再被后续合法请求继承脏状态”。

## 5. 本轮实现结果

本轮实现完成以下收敛：

- 会话绑定型 RTC 写路径全部前置成员校验
- 阻止非成员创建绑定会话的 RTC 会话
- 阻止非成员通过失败请求污染 RTC session 的 `signalingStreamId`
- 保持纯 RTC 会话（未绑定 `conversationId`）仍可按既有语义独立使用

- 会话绑定 RTC 的 `invite / accept / reject / end` 是否存在更多顺序性副作用
