# 152CJ Loop72补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“只有 takeover conflict 写路径显式回写 machine-readable `details`”推进到“targeted republish / release owner-conflict 也显式回写 machine-readable `details`”的最小 contract

## 2. As-Built 事实
- 新增共享 helper，把 pending item 的 `requestKey / ownerActorId / leaseExpiresAt / leaseStatus / takeoverEligible / legacyTakeoverRequired` 收敛成统一 conflict `details`
- targeted republish 的 `shared_channel_sync_owner_conflict` 当前显式返回 machine-readable `details`
- targeted release 的 `shared_channel_sync_owner_conflict` 当前也显式返回 machine-readable `details`
- takeover / republish / release 三条 owner-conflict 写路径当前复用同一 details 语汇，operator 不再必须回头再读 inventory 才能判断被拒绝时的 lease / owner / legacy takeover 背景
- 本轮没有改变 owner boundary allow/deny 语义；补的是写路径 conflict surface symmetry，而不是新的 stale policy

## 3. 明确边界
- 本轮没有：
  - dead-letter inventory metadata parity 的 dedicated 断言
  - claim targeted path 的逐 item conflict details
  - configurable lease window / SLA
  - timeout reclaim / automatic stale detection
  - stale-aware bulk `repair-shared-channel-sync`
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 的 conflict surface 对齐增量，而不是完整的 operator policy/governance engine
