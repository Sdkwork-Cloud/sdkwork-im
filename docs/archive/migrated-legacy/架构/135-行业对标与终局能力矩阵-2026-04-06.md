# 行业对标与终局能力矩阵

## 1. 文档目标

本文档用于把 `sdkwork-im` 的推荐路线放到行业坐标中进行对比，明确：

- 向哪些开IM 学什。
- 向哪些头部产品学什。
- 哪些能力是现在就要冻结的
- 哪些能力属于后续分阶段演。

## 2. 总体结论

`sdkwork-im` 不应复制任何单一产品，而应采用混合型路线：

`以消费级消息内核为底，以企业协作上下文为中层，以 AI / Agent / IoT 扩展为上层，以连接优先和分层弹性扩容为工程基线`

这条路线兼顾。

- 用户体验
- 工程性能
- 商业可控
- 后续智能化扩。

## 3. 开源项目对。

### 3.1 WuKongIM

应吸收：

- 高性能连接与消息热路径意识
- 连接、路由、消息的内核化思路
- 少依赖、可自主演进的方。

不直接照搬：

- 从第一天就重度自研所有底层存。
- 过早将复杂分布式协议作为落地前提

### 3.2 OpenIM

应吸收：

- 平台化、SDK 化与多端开放能。
- 后台接口、Webhook、管理能。
- 面向业务方交付的产品组织方式

不直接照搬：

- 第一阶段的大量微服务与复杂依赖栈

### 3.3 Matrix

应吸收：

- 事件优先和协议可扩展。
- 长期桥接和联邦潜。

当前不优先：

- 联邦复杂。
- 大规模跨组织协议兼容负担

## 4. 头部产品对标

### 4.1 WhatsApp

重点吸收：

- 多客户端路由一致。
- 私聊隐私边界
- 极简稳定的主聊天体验

### 4.2 Telegram

重点吸收：

- 频道、超大群、Bot 生。
- 云同步与大规模消息体。

### 4.3 WeChat / Weixin

重点吸收：

- 聊天作为平台入口
- 对外扩展生态的能力设计

### 4.4 Lark / Feishu

重点吸收：

- 聊天与文档、任务、会议、工作流的上下文融合
- 协作内容沉淀能力

## 5. 推荐终局能力矩阵

| 能力| 现在必须冻结 | 后续分阶段落|
| --- | --- | --- |
| 消息内核 | 会话内有序、编辑、撤回、已读、多端同| 大群、频道、广播优|
| 连接能力 | Link Plane、Route Plane、resume、drain、fencing | 更高连接密度、QUIC |
| 流能| 流为一等能力、checkpoint、materialize | 更复patch、二进制流优|
| 协作能力 | topic / thread / card / document reference 预留 | 深度文档、任务、知识工作台 |
| AI 能力 | Agent 一等主体、tool call、流式输| Agent 协同、复handoff |
| IoT 能力 | Device 一等主体、MQTT 预留、telemetry / command stream 预留 | device twin、边缘网关、复杂编|
| 安全能力 | 权威认证上下文、双轨安全、tenant 隔离 | 端到端加密增强、合规插件化 |
| 存储能力 | PostgreSQL + Redis + Object Storage + 自有抽象 | 自研 MessageLog / StreamStore |

## 6. 为什么采用混合型路线

如果只做消费IM，会缺少 AI 和协作时代的扩展位。

如果只做企业协作，会丢掉消息内核和连接热路径的极致要求。

如果只做 AI 容器，会把聊天体验、群聊、设备、多端一致性做弱。

如果一开始就全做自研基础设施，会拖慢落地和商业验证。

因此最优路线必须是。

- 内核按消费级标准建设
- 协作AI 扩展从第一天预。
- IoT 作为未来一等入口预。
- 组件依赖保持最小化
- 热路径在合适阶段逐步自研

## 7. 对工程实施的直接要求

- 连接层必须独立出来，不能继续塞进大而全服务
- 消息与流必须并列，不能把流降级为消息字段
- `sender / actor / tenant` 必须统一，不能各接口各自表达
- 代码必须crate 边界治理，不能继续用巨型 `lib.rs`
- 第一阶段存储不得回到“重型默认栈”，必须坚持 `PostgreSQL + Redis + Object Storage`

## 2026-04-09 增补：行业对标、终局能力与当前应用边。

### A. 本文定位

- 本文回答的是“`sdkwork-im` 应向谁对标、终局能力线应如何定义”，不是“当前仓库已经完整具备哪些能力”。
- 5 节“推荐终局能力矩阵”属于目标能力冻结线；其中“现在必须冻结”表示架构口径现在就要冻结，不等于代码今天已全部落地。
- 当前应用信息、真实已实现边界与缺口，应以 `152CJ-current-architecture-as-built-alignment-2026-04-09.md`、`02 / 03 / 05 / 09` 和当workspace 为准。

### B. 对标吸收规则

- 消费级消息体验对`WhatsApp / Telegram / WeChat`：强调送达、已读、多端一致、轻交互心智。
- 企业协作与空间治理对`Slack / Teams / Feishu / Discord`：强`space / channel / thread / member / permission` 分层清晰，`conversation` 不承载群组或关系真相。
- 协议开放性与可扩展性对`Matrix`：吸收事件化、可扩展、桥接能力，但不照搬 federation 复杂度。
- 开源工程路线优先吸`WuKongIM / OpenIM` 的连接内核、平台化与开放接口经验，不复制其全部基础设施栈。

### C. 文档口径规则

- 写“当前应用能力”时，必须显式区`已落/ 已冻结待实施 / 后续阶段`。
- 写“行业终局能力”时，可以保持高标准，但必须显式标记`目标态`、`终局能力` `对标能力`。
- 若当前实现与行业终局不一致，禁止再用“已具备”“已支持”“已落地”做笼统混写。

## 8. 结论

`sdkwork-im` 的目标不是追随某个开源项目，而是吸收 WuKongIM 的连接内核意识、OpenIM 的平台开放性、Telegram / WhatsApp 的消息体验、Feishu 的协作上下文AI / IoT 时代的新需求，最终形成一套自主可控的商业级实时通信平台。
