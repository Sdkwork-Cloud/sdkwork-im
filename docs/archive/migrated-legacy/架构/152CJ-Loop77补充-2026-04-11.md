# 152CJ Loop77 Addendum - 2026-04-11

## 1. As-Built Update
- The system-owned next-write auto-retry path now respects pending claim ownership.
- Auto-retry no longer treats all pending backlog as uniformly dispatchable.
- After Loop77:
  - `Unclaimed` pending work can be auto-retried
  - `Stale` pending work can still be auto-retried
  - `Active` claimed pending work is blocked from auto-retry
  - `Untracked` legacy-claimed pending work is blocked from auto-retry
- Same-key incoming requests are also blocked while an `Active` or `Untracked` pending claim exists.

## 2. Architectural Meaning
- This closes an ownership-consistency hole between:
  - operator claim / republish / release / takeover surfaces
  - system-driven next-write auto-retry
- Before Loop77, the architecture exposed an inconsistency:
  - operator surfaces modeled claim ownership
  - next-write auto-retry ignored that ownership and could flush claimed backlog anyway
- After Loop77, the ownership model is consistent across both surfaces.

## 3. Deferred Architecture Work
- automatic stale detection / timeout reclaim / scheduler
- next-write stale-owner metadata cleanup before dispatch
- release-ready exactly-once delivery semantics for shared-channel sync
