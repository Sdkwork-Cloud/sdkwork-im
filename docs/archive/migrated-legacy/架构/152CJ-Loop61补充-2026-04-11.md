# 152CJ Loop61 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“pending backlog 只有全量 repair，没有可读 inventory 与定点投递”推进到“operator 可读取 pending inventory，并按 `requestKey` targeted republish”的恢复语义

## 2. As-Built 事实
- `control-plane-api` 当前新增 `GET /backend/v3/api/control/social/runtime/pending-shared-channel-sync`
- inventory item 当前显式输出：
  - `requestKey`
  - `request`
  - `failureCount`
  - `lastError`
- `requestKey` 当前与 `shared_channel_sync_request_key(...)` 使用同一稳定 key 规则
- `control-plane-api` 当前新增 `POST /backend/v3/api/control/social/runtime/republish-pending-shared-channel-sync-targeted`
- `SocialControlRuntime::republish_pending_shared_channel_sync_targeted(...)` 当前会：
  - 只尝试投递被选中的 pending request
  - 成功时清理对应 pending item
  - 失败时沿用现有失败累计 / dead-letter 规则
  - 保持未选中的 pending request 原样不动
- 现有 `repair-shared-channel-sync` 全量 operator route 仍保留，因此 operator 当前拥有“全量 repair + 定点 republish”两条最小投递路径

## 3. 明确边界
- 本轮没有：
  - pending inventory filter / pagination / search language
  - delivery lease owner / SLA
  - automatic retry scheduler
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 单体内的 operator delivery boundary，而不是 release-ready cross-service delivery orchestration
