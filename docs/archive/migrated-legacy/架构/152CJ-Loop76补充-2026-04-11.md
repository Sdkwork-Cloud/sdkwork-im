# 152CJ Loop76 Addendum - 2026-04-11

## 1. As-Built Update
- The shared-channel sync repair path is now stale-aware.
- Before `repair-shared-channel-sync` dispatches pending work, it clears expired ownership metadata from stale pending claims and counts how many claims were reclaimed.
- The operator-facing repair response now exposes `reclaimed`.
- The repair audit payload now also records `reclaimed`.

## 2. Architectural Meaning
- This change hardens the operator repair seam, not the background runtime behavior.
- The current architecture after Loop76 is:
  - claim metadata can become stale over time
  - operator inventory can expose stale lease state
  - operator repair can reclaim stale ownership before dispatch
  - reclaimed count is visible in both the HTTP response and the audit stream
- The architecture is still intentionally conservative:
  - no automatic scheduler-driven reclaim exists
  - no background timeout sweep exists
  - no claim reclaim is triggered by unrelated traffic unless a future loop adds that behavior explicitly

## 3. Contract Surface
- Response contract:
  - `SocialSharedChannelSyncRepairResponse.reclaimed`
- State transition contract:
  - expired pending claim ownership is cleared before repair dispatch
- Audit contract:
  - `control.social_runtime_shared_channel_sync_repaired` now includes `reclaimed`
- Regression contract:
  - pre-repair inventory must show `leaseStatus = stale`
  - repair must return `reclaimed = 1` in the stale-claim scenario

## 4. Deferred Architecture Work
- automatic stale detection / timeout reclaim / scheduler
- reclaim semantics before unrelated ready-pair writes
- release-ready exactly-once delivery semantics for shared-channel sync
