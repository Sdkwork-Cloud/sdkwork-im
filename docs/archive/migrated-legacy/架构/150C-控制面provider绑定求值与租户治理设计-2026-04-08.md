# 150C 控制面provider绑定求值与租户治理设计

## 设计目标

- 把 provider/plugin 的“静态矩阵”与“动态求值结果”拆成两个清晰的控制面视图。
- 让 RTC、对象存储、用户模块、设备接入、IoT 协议都遵循同一套 binding 求值语义。
- 不让运行时和控制面各自维护两套 provider 选择逻辑。

## 控制面接口分层

### 1. 静态矩阵

- `GET /backend/v3/api/control/provider-registry`

用途：

- 查看 provider/plugin 支持矩阵
- 查看 descriptor、capability、默认标记
- 查看全局优先级定义

它不负责回答“某个租户当前到底选中了谁”。

### 2. 动态求值

- `GET /backend/v3/api/control/provider_bindings`
- 可选查询参数：`tenantId`

用途：

- 查看当前控制面求值后的 `effectiveBindings`
- 验证 `tenant override / deployment_profile / global default` 的实际结果
- 作为后续 runtime / ops / audit 漂移检测的只读基线

## 求值语义

固定顺序：

1. `tenant_override`
2. `deployment_profile`
3. `global_default`

约束：

- 任何 domain 都必须复用同一套优先级规则
- response 必须返回 `selectionSource`
- domain contract 不得泄漏云厂商 SDK 类型

## 当前冻结示例

- RTC 默认 provider：`rtc-volcengine`
- 对象存储 deployment profile 示例：`object-storage-volcengine`
- tenant override 示例：
  - `rtc-aliyun`
  - `object-storage-aws`

这意味着控制面现在既能看见默认矩阵，也能看见特定租户的真实选择结果。

## 返回体建议

- `interfaceVersion`
- `tenantId`
- `effectiveBindings`
- `precedence`

其中 `effectiveBindings` 直接复用标准 `EffectiveProviderBinding`，避免控制面重新定义一套平行字段。

## 安全边界

- 该接口当前为只读治理接口，必须受 `control.read` 或 `control.write` 保护
- public bearer 不得绕过权限直接读取 provider binding
- 写接口、审计记录、配置版本与回滚能力留在后续波次

## 后续演进

- 将 `effective provider bindings` 下发给 runtime / ops，形成漂移检测与运维展示
- 增加 provider policy 写接口与审计闭环
- 将 provider binding 差异纳入 cell / region rollout 与回滚模型
