# 架构能力闭环验收标准

## 1. 文档定位

本文件用于回答一个比“step 做完了没有”更严格的问题：

`step 做完之后，是否真的把对应架构文档中的能力完整落地了？`

为了避免出现“名义完成、实质未闭环”的情况，本文件为每个主要 step 定义：

- 该 step 应闭环的架构能力
- 判定该能力已被完整实现的最低标准
- 常见的伪完成表现

## 2. 使用方式

每个 step 结束时，除了检查本 step 自身的检查点外，还应使用本文件核对：

- 当前是否已经达到该 step 的能力闭环标准
- 是否只是完成了代码迁移，却没有完成能力闭环
- 是否仍存在会导致下一步返工的伪完成状态

## 3. Step 级能力闭环标准

### 3.1 Step 01 闭环标准

对应能力：

- 当前仓库能力盘点
- 差距矩阵
- 高风险识别

闭环标准：

- 已能准确指出当前仓库与目标架构的差距
- 已能明确给出优先处理路径
- 已能把问题映射到后续 step

伪完成表现：

- 只有概览，没有真实路径和高风险文件
- 没有差距优先级

### 3.2 Step 02 闭环标准

对应能力：

- crate 分层骨架
- `services/*` 装配层收敛
- 超大文件拆分起点

闭环标准：

- 新骨架已能承载后续协议、运行时、主链路改造
- 高风险大文件不再继续增长
- 新代码不会再被迫堆回旧 `lib.rs`

伪完成表现：

- 只是新建空目录，但旧服务边界没有开始收敛
- 仍然继续在大文件中新增逻辑

### 3.3 Step 03 闭环标准

对应能力：

- `CCP` 协议骨架
- `contract-*` 与 `ccp-*` 分层
- 权威字段收口

闭环标准：

- 任一入口都能映射到统一协议骨架
- `tenant`、`sender`、`actor` 权威边界稳定
- transport binding 不再各写一套协议细节

伪完成表现：

- 只建了 crate 名称，没形成统一握手/envelope 规则
- 客户端仍能伪造权威字段

### 3.4 Step 04 闭环标准

对应能力：

- Link Plane
- Route Plane
- resume / reconnect / drain / fencing

闭环标准：

- 连接热路径与业务逻辑已完成边界分离
- route epoch / drain / rebalance 有稳定语义
- Link / Route Plane 已具备独立扩容和独立观测的雏形

伪完成表现：

- 只是把代码从一个文件搬到多个文件
- route 迁移与 reconnect 仍不稳定

### 3.5 Step 05 闭环标准

对应能力：

- conversation / member / message / read-cursor / presence
- sender 结构化建模
- tenant 权威收口

闭环标准：

- 单聊、群聊、频道主路径稳定
- `sender` 已取代老式平铺 `senderId`
- 已读、多端同步、消息编辑撤回的语义稳定

伪完成表现：

- 仅改 DTO，不改应用与领域边界
- `tenantId` 仍在主路径中由客户端控制

### 3.6 Step 06 闭环标准

对应能力：

- stream 一等能力
- rtc 一等能力
- 与消息主链路桥接

闭环标准：

- stream lifecycle 可验证
- rtc signaling 可验证
- 流与消息、rtc 与消息桥接规则稳定

伪完成表现：

- 流仍然只是消息附属字段
- RTC 仍然混入媒体面逻辑

### 3.7 Step 07 闭环标准

对应能力：

- registry
- compatibility
- rollout
- kill switch
- drain 编排

闭环标准：

- 控制面已经成为治理权威入口
- 热路径不再依赖散落硬编码
- admin / ops / audit 形成最小闭环

伪完成表现：

- 只有接口，没有治理状态和运行时消费链
- rollout / compatibility 仍停留在文档中

### 3.8 Step 08 闭环标准

对应能力：

- Agent 一等主体
- Device 一等主体
- AI / IoT 扩展与主链路统一

闭环标准：

- Agent 输出与 Device 数据都接入统一 sender / actor / scope
- 至少一条 Agent 路径和一条 Device 路径可跑通
- 扩展能力不破坏普通 IM 路径

伪完成表现：

- Agent / Device 只是旁路模块
- AI / IoT 单独造了一套实时协议

### 3.9 Step 09 闭环标准

对应能力：

- 存储抽象
- projection rebuild
- observability
- backup / restore / repair

闭环标准：

- 主链路已接到统一存储抽象
- 投影可重建
- 恢复与诊断能力可重复执行
- 关键指标和 tracing 已按 plane 收口

伪完成表现：

- 只有存储接口，没有可运行适配器和恢复链路
- 只有日志，没有 metrics / tracing / health 体系

### 3.10 Step 10 闭环标准

对应能力：

- 安装
- 启动
- 停止
- 状态
- 恢复
- 多环境 profile

闭环标准：

- 通过统一脚本可完成最小交付闭环
- 本地与 Docker 至少一套链路稳定
- 恢复和修复命令具备真实可执行性

伪完成表现：

- 只有脚本文件，没有真实 smoke 验证
- profile 只是名称存在，没有清晰职责边界

### 3.11 Step 11 闭环标准

对应能力：

- 性能量化
- 高可用演练
- 灾备演练
- 升级与回滚演练

闭环标准：

- 至少一轮量化结果沉淀
- 至少一轮排空、恢复或升级演练完成
- 可用性与性能结论进入 `docs/review/`

伪完成表现：

- 只有理论指标，没有实测结果
- 只做压测，不做故障演练

### 3.12 Step 12 闭环标准

对应能力：

- CLI
- SDK
- compatibility matrix

闭环标准：

- CLI 可真实验证主路径
- SDK 契约边界清晰
- 兼容矩阵有文档也有验证支撑

伪完成表现：

- 只有 README，没有真实 E2E 验证
- 兼容矩阵只是列表，没有测试或控制面映射

### 3.13 Step 13 闭环标准

对应能力：

- 发布清单
- 升级与回滚说明
- 最终验收
- 下一轮 backlog

闭环标准：

- 当前是否可交付有正式结论
- 发布、升级、回滚路径完整
- 下一轮输入清晰且已归档

伪完成表现：

- 只是总结，没有 release gate
- 没有阻塞项判断和下一轮 backlog

## 4. 使用建议

建议在每个 step 的复盘文档中增加一节：

- `本 step 架构能力闭环判定`

至少回答：

- 对应能力是否真正闭环
- 证据是什么
- 若未闭环，差的是什么

## 5. 与其他治理文档的关系

- `90`：能力映射
- `91`：质量评分
- `92`：输入输出与阻塞控制
- `93`：波次总验收
- `95`：能力闭环判定

这意味着：

- `90` 解决“该做什么”
- `91` 解决“做得好不好”
- `92` 解决“这一轮怎么控”
- `93` 解决“这一波能不能过”
- `95` 解决“能力到底有没有真正实现”

## 6. 结论

如果没有能力闭环判定，step 很容易停留在“代码迁移已做、能力还没真正稳定”的半完成状态。

本文件的作用，就是把“对应架构能力是否完整实现”也变成一条硬门禁。
