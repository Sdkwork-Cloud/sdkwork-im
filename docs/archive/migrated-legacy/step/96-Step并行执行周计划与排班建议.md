# Step 并行执行周计划与排班建议

## 1. 文档定位

本文件用于把 [`94-Step并行执行编排与车道拆分建议`](./94-Step并行执行编排与车道拆分建议.md) 的“并行原则”进一步落到“如何快速完整执行”的操作层。

目标不是重新定义 step 顺序，而是回答以下问题：

- 多人或多 agent 并行时，按什么节奏推进最快且风险可控
- 每一波次如何分车道、分 owner、分验证窗口
- 怎样保证并行执行后还能稳定收口，而不是把返工放到最后

## 2. 执行前提

- 串行主脊柱不变：`00 -> 01 -> 02 -> 03 -> 04 -> 05 -> 06 -> 07 -> 08 -> 09 -> 10 -> 11 -> 12 -> 13`
- 波次内并行，不等于跨越架构依赖并行。
- 每个车道必须有明确主写入目录和单一 owner。
- 每个 step 完成后必须产出 `docs/review/` 证据，并通过 `91` 与 `95` 的双门禁。
- 验证车道必须独立，不与实现车道长期混写。

## 3. 最快完成原则

- 优先压缩“等待时间”，不要压缩“门禁”。
- 先冻结共享边界，再放开车道并行。
- 一个波次内，先拆分实现车道，再单独保留验证车道，最后由集成 owner 统一收口。
- 不允许两个车道同时主写同一个高风险文件或同一组权威模型。
- 每天必须至少有一次集成检查窗口，避免车道各自漂移到不可合并状态。

## 4. 推荐串行主脊柱

### 4.1 波次 A：基线与骨架

- `00-01`：执行门禁、差距审计、风险清单冻结
- `02-03`：工程骨架与协议骨架同步建立

### 4.2 波次 B：核心实时链路

- `04-06`：先跑通 Link / Route，再稳定消息主链路，再收口流式与 RTC

### 4.3 波次 C：治理与扩展

- `07-09`：控制面治理、AI / Agent / IoT 扩展、存储投影与可观测并行落地

### 4.4 波次 D：交付与发布

- `10-12`：部署交付、性能/HA/DR、CLI / SDK / 兼容矩阵并行收口
- `13`：统一做发布就绪判断和下一轮迭代输入整理

## 5. 推荐周节奏

以下节奏用于“希望尽快完整跑通 step 体系”的团队，不是固定工期承诺，而是建议排程骨架。

### 5.1 第 1 周

- 冻结 `00`
- 完成 `01`
- 启动 `02` 的 workspace / crate 骨架车道
- 启动 `03` 的协议骨架预研，但最终冻结要等 `02` 的目录落点确定

### 5.2 第 2 周

- 完成 `02`
- 完成 `03`
- 启动 `04` 的 Link / Route 双车道
- 预热 `05` 的 message / conversation 领域模型车道

### 5.3 第 3 周

- 完成 `04`
- 完成 `05`
- 完成 `06`
- 同步开始 `07` 的控制面注册表与 rollout 车道

### 5.4 第 4 周

- 完成 `07`
- 完成 `08`
- 完成 `09`
- 启动 `10` 的脚本与交付车道

### 5.5 第 5 周

- 完成 `10`
- 完成 `11`
- 完成 `12`
- 启动 `13` 的 release gate 与最终归档

### 5.6 第 6 周

- 完成 `13`
- 复盘所有遗留阻塞项
- 输出下一波次 backlog

## 6. 小团队并行建议

### 6.1 2-3 人团队

- `Owner A`：协议/契约 + Link / Route Runtime
- `Owner B`：消息主链路 + Stream / RTC
- `Owner C`：控制面 + 存储投影 + 交付脚本 + 验证

建议方式：

- 每个波次先由 `Owner A` 冻结共享边界，再由 `Owner B/C` 并行展开
- 每周至少 2 次集成窗口
- 每个 step 收尾时必须先补 review 文档，再允许进入下一步

### 6.2 4-6 人团队

- `A`：协议 / 契约 / 安全字段
- `B`：Link / Route Runtime
- `C`：消息与会话主链路
- `D`：Stream / RTC
- `E`：控制面 + 存储 / 投影
- `F`：部署 / CLI / SDK / 验证

建议方式：

- 波次内多车道并行
- 每日固定一次集成窗口
- 每个波次末尾统一通过 `93` 做总验收

### 6.3 多 agent 模式

- 主 agent 只负责总协调、边界冻结、最终集成、验收决策
- 子 agent 只负责单一车道、单一主写入范围
- 每个子 agent 完成后必须提交：变更范围、测试结果、阻塞项、下一步影响面

## 7. 推荐每日节奏

### 7.1 上午

- 同步当日 step / 车道目标
- 确认共享边界、接口、协议、权威字段没有变化
- 更新 `docs/review/step-XX-执行卡-YYYY-MM-DD.md`

### 7.2 下午

- 各车道独立推进
- 验证车道开始回归已完成的子任务
- 若共享边界发生变化，必须即时回写 step 文档与 review 文档

### 7.3 晚间收口

- 合并当日可集成变更
- 运行最小 smoke、回归、脚本验证
- 记录剩余阻塞项，决定次日是否继续当前 step 或进入下一 step

## 8. 提交、集成与验证节奏

- 每个车道每天至少一次可合并状态，不允许长时间本地大分叉。
- 集成分支只接收通过当前 step 检查点的改动。
- 验证车道提交优先包含测试、脚本、review 证据，不承担大规模业务重构。
- 高风险目录变更必须在 merge 前进行一次集成 owner review。
- 波次切换前必须先跑一次总 smoke，避免把问题带入下一波次。

## 9. 推荐 review 产物节奏

- step 启动时：`docs/review/step-XX-执行卡-YYYY-MM-DD.md`
- step 中期：`docs/review/step-XX-中期审计-YYYY-MM-DD.md`
- step 收尾时：`docs/review/step-XX-复盘-YYYY-MM-DD.md`
- 波次收尾时：`docs/review/wave-X-总验收-YYYY-MM-DD.md`

命名建议：

- `step-01-基线审计-YYYY-MM-DD.md`
- `step-03-CCP协议冻结决议-YYYY-MM-DD.md`
- `step-05-message-conversation主链路-YYYY-MM-DD.md`
- `step-10-profile-smoke验证-YYYY-MM-DD.md`

## 10. 最快且完整的执行建议

如果目标是“尽快且完整地把 step 执行完毕”，推荐采用以下固定打法：

1. `00-01` 用最短时间冻结规则与差距，不在此阶段陷入实现细节。
2. `02-03` 作为第一批必须优先完成的共享边界，任何后续功能车道不得提前越过。
3. `04-06` 采用“三主车道 + 一验证车道”模型：
   `Link/Route`、`Message`、`Stream/RTC`、`Validation`
4. `07-09` 采用“三主车道 + 一治理车道”模型：
   `Control Plane`、`AI/IoT`、`Storage/Projection`、`Validation`
5. `10-12` 采用“三主车道 + 一交付车道”模型：
   `Deploy`、`Perf/HA/DR`、`CLI/SDK`、`Release Evidence`
6. `13` 只做总收口，不再扩散需求和新设计。

## 11. 风险与止损规则

- 如果同一波次出现 2 个以上共享边界冲突，立即降级并行度，先收口边界再继续。
- 如果验证车道连续两次发现同类问题，停止继续扩写功能，先修正执行规则或协议边界。
- 如果 review 文档落后于代码超过 2 天，视为执行体系失控，必须先补文档证据。
- 如果 `95` 的闭环判定无法通过，不允许用“先进入下一步再补”来掩盖问题。

## 12. 结论

最快的并行执行方式，不是把所有 step 同时展开，而是：

- 先冻结共享边界
- 再按波次拆分车道
- 再保留独立验证与统一集成窗口
- 最后用 release gate 做最终收口

这样才能既快，又不会把返工推迟到最后集中爆发。
