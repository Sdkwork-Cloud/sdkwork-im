# 波次里程碑与阶段总验收矩阵

## 1. 文档定位

`00-13` 定义了逐 step 的顺序，但如果没有波次级里程碑与总验收门禁，执行中容易出现一种问题：

单个 step 看起来都完成了，但整波次没有形成稳定可交付状态。

本文件用于补齐“step 之上”的阶段门禁，即：

- 波次 A 结束时，系统至少应该达到什么状态
- 波次 B、C、D 结束时，分别应该交付什么
- 哪些条件不满足时，不允许进入下一波次

## 2. 波次定义

当前 step 体系分为四个波次：

- 波次 A：`00-03`
- 波次 B：`04-06`
- 波次 C：`07-09`
- 波次 D：`10-13`

## 3. 波次 A 门禁：基线与骨架完成

### 3.1 目标状态

波次 A 完成后，至少应达到：

- step 执行体系稳定
- 当前仓库差距已审计清楚
- workspace 与 crate 骨架可承载目标架构
- `CCP` 与 `contract-*` 的基础边界不再漂移

### 3.2 必须通过的条件

- `00-03` 的检查点全部通过
- 高风险大文件已进入拆分轨道
- `CCP` crate 落点与 `contract-*` 分层已成型
- 权威字段收口方案已确定

### 3.3 不允许进入波次 B 的情况

- `session-gateway` 仍准备继续直接承载协议骨架
- `tenant`、`sender`、`actor` 权威边界仍未冻结
- `services/*` 与 `runtime-*` 边界仍完全不清楚

## 4. 波次 B 门禁：核心实时链路完成

### 4.1 目标状态

波次 B 完成后，至少应达到：

- Link Plane 与 Route Plane 已独立成形
- 消息主链路稳定
- 流与 RTC 已成为一等能力

### 4.2 必须通过的条件

- `04-06` 的检查点全部通过
- reconnect / resume / route drain 有自动化验证
- conversation / message / read-cursor / presence 主链路有自动化验证
- stream lifecycle 与 rtc signal flow 有自动化验证

### 4.3 不允许进入波次 C 的情况

- 消息主链路仍依赖旧大服务文件的核心逻辑
- 流仍只是消息附属字段
- route 迁移、epoch 或 drain 语义仍不稳定

## 5. 波次 C 门禁：治理与扩展完成

### 5.1 目标状态

波次 C 完成后，至少应达到：

- 控制面已接管协议和能力治理
- AI / Agent / IoT 已纳入统一主体和统一实时模型
- 存储、投影、可观测已完成统一治理

### 5.2 必须通过的条件

- `07-09` 的检查点全部通过
- registry / compatibility / rollout / kill switch 已进入控制面
- agent / device 事件与主链路已打通
- projection / rebuild / backup / restore 可重复验证
- metrics / tracing / logging 已按 plane 基本收口

### 5.3 不允许进入波次 D 的情况

- 控制面仍只是“接口集合”而不是治理平面
- AI / IoT 扩展破坏普通 IM 主链路
- 可观测与恢复能力不足以支撑交付验证

## 6. 波次 D 门禁：交付与发布完成

### 6.1 目标状态

波次 D 完成后，至少应达到：

- 脚本、配置、部署与恢复形成交付体系
- 性能、高可用、灾备有量化证据
- CLI / SDK / compatibility 收口
- 发布就绪与下一轮 backlog 形成闭环

### 6.2 必须通过的条件

- `10-13` 的检查点全部通过
- 跨平台脚本与本地 / Docker 启动验证通过
- 至少一轮性能和灾备演练结果沉淀
- CLI 与 SDK 能验证核心主路径
- 发布、升级、回滚说明齐全

### 6.3 不允许宣称“可正式交付”的情况

- 部署脚本无法一键跑通最小闭环
- 性能和恢复没有量化报告
- 兼容矩阵没有测试或 CLI / SDK 验证支撑
- 发布清单和回滚说明缺失

## 7. 阶段总验收清单

建议在每个波次结束后执行总验收：

### 7.1 文档总验收

- 相关 `docs/架构/` 是否需要回写
- 相关 `docs/step/` 是否需要修订
- `docs/review/` 是否已有对应复盘

### 7.2 工程总验收

- 高风险文件是否持续收敛
- 新 crate 边界是否被遵守
- 测试和脚本是否持续可用

### 7.3 能力总验收

- 本波次承诺的能力是否真的形成闭环
- 是否引入了新的结构性债务
- 是否具备进入下一波次的前提

## 8. 波次评分建议

建议对每个波次也做一次总评分，参考 `91` 的思路：

- 架构完成度
- 代码结构完成度
- 测试完成度
- 运维完成度
- 复盘与证据完成度

建议标准：

- `95+`：可以稳定进入下一波次
- `90-94`：基本可进入下一波次，但要带遗留清单
- `< 90`：不建议进入下一波次

## 9. 与 `README`、`90`、`91`、`92` 的关系

- `README`：总入口
- `90`：能力到 step 的映射
- `91`：step 质量评分与复盘模板
- `92`：step 执行卡、输入输出和阻塞升级规则
- `93`：波次级总验收与跨 step 里程碑门禁

这五类文档叠加后，形成：

- 总顺序
- 细粒度能力映射
- 单 step 质量控制
- 单 step 执行控制
- 跨 step 波次控制

## 10. 结论

如果没有波次门禁，step 再完整，也可能在整体上跑偏。

本文件的作用，是把 `docs/step/` 从“逐步计划”进一步提升为“带阶段里程碑和总验收门禁的实施系统”。
