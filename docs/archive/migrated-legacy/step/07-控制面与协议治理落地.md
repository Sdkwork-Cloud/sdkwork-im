# Step 07 - 控制面与协议治理落地

## 1. 目标与范围

本 step 用于把控制面从“辅助接口”升级为真正的治理平面，承接租户、配置、配额、协议注册表、灰度、排空和 kill switch 等职责。

本 step 主要解决：

- 协议升级和能力开关不能靠热路径内硬编码
- 租户策略、配额、节点状态、排空编排需要统一治理入口
- `CCP` 的 registry、compatibility matrix、release channel 需要真正落地

### 1.1 执行输入

- step 03 的协议骨架与 registry 设计
- step 04 的 runtime hook 和 step 05 的主链路边界
- 当前 `control-plane-api`、`ops-service`、`audit-service` 的目录与测试现状
- `139`、`142`、`148`、`149` 的治理与多版本文档

### 1.2 本步非目标

- 不在本 step 内重写消息热路径或连接热路径
- 不在本 step 内执行性能压测和灾备演练
- 不在本 step 内完成 CLI / SDK 对控制面的全部消费

### 1.3 最小输出

- 协议注册表、provider registry / effective bindings 与兼容矩阵的控制面入口
- capability / quota / rollout / kill switch 基线
- runtime 的只读治理消费链
- admin / ops / audit 的治理闭环起点

## 3. 当前现状与问题

当前仓库已存在：

- `services/control-plane-api`
- `services/ops-service`
- `services/audit-service`

但还需要进一步演进为目标控制面：

- 配置与协议治理尚未统一
- drain / rebalance / rollout 还未形成完整控制闭环
- 协议注册表、兼容矩阵和发布策略未固化到控制面
- 能力开关、配额、租户 lane 等治理模型未系统化

## 4. 设计

### 4.1 控制面职责

控制面必须统一承接：

- 租户
- 身份与策略
- 配额
- capability 开关
- 协议注册表
- 兼容矩阵
- 发布通道
- drain / rebalance 控制
- kill switch
- 配置下发

### 4.2 运行态分离原则

必须坚持：

- 控制面不侵入消息热写路径
- 运行时只读取治理结果，不承载治理决策过程
- registry 是控制面的权威来源
- runtime 对 registry 和 config 的读取是缓存化、只读化、可降级的

### 4.3 核心对象

建议建立以下核心模型：

- `tenant_policy`
- `capability_profile`
- `protocol_registry`
- `schema_registry`
- `compatibility_matrix`
- `release_channel`
- `rollout_policy`
- `node_drain_plan`
- `kill_switch_rule`

### 4.4 与协议治理的关系

本 step 必须把 step 03 中的协议冻结升级为可治理资产：

- 哪个 schema 处于 `beta / ga / deprecated`
- 哪个 binding 在哪个环境可用
- 哪个租户开启哪些 capability
- 哪个 cell 或 region 使用哪个 release channel

## 5. 实施落地规划

### 5.1 任务拆解

1. 重构 `control-plane-api` 的模型与目录边界
2. 建立协议注册表和兼容矩阵的持久化与读取接口
3. 建立 capability / quota / rollout / kill switch 模型
4. 打通 `runtime-link`、`runtime-route`、`interface-*` 的只读配置消费链
5. 打通 `ops-service` 与 `audit-service` 的治理审计闭环
6. 为多 cell、多 region 的升级和回滚预留治理字段

### 5.2 重点路径

重点涉及：

- `services/control-plane-api/`
- `services/ops-service/`
- `services/audit-service/`
- `crates/im-auth-context/`
- 新增或重构的 `ccp-registry`、`contract-admin`

### 5.3 治理切入顺序

推荐顺序：

1. 先治理协议注册表和 schema registry
2. 再治理 capability / rollout / kill switch
3. 再治理节点排空和 rebalance 编排
4. 最后补齐多 cell / 多 region 发布模型

### 5.4 安全边界

控制面接口必须使用独立、更严格的认证和审计模型：

- 区分 public auth 与 admin auth
- 所有写操作带审计 actor
- 所有变更具备可回溯配置版本

## 7. 结果验证

本 step 完成后，需要验证：

- 协议发布、配置变更和节点编排有统一控制面入口
- runtime 可以读取治理后的结果，而不是写死在代码中
- 升级、灰度、回滚具备最小可运行治理基础
- control plane 与 ops / audit 形成闭环

## 8. 检查点

- `CP07-1`：协议注册表、provider registry / effective bindings 与兼容矩阵进入控制面治理

### 8.3 架构能力闭环判定

- 控制面必须真正成为治理真源，运行时热路径不得继续依赖散落硬编码。
- 如果只有接口和文档，没有注册表状态、发布链路、运行时消费链路，本 step 不算闭环。
- 闭环验收以 [`95-架构能力闭环验收标准`](./95-架构能力闭环验收标准.md) 中 Step 07 条目为准。

### 8.4 快速并行执行建议

- 先冻结 registry、rollout、kill-switch 的状态模型，再并行做控制面接口、运行时消费、审计链路。
- 推荐把“配置真源”作为单点裁决，不允许运行时和控制面同时维护两套配置语义。
- 本步结束前必须用最小实链路验证 rollout / drain / compatibility，而不是停留在接口层。

## 9. 风险与回滚

### 9.1 风险

- 把控制面做得过重，会反向拖累热路径
- 不做 registry 治理，后续多版本兼容会失控
- 控制面与运维面脱节，会导致排空和升级难以执行

### 9.2 回滚

- 先以最小 registry + capability 模型落地，避免一次性全量治理
- runtime 先保留默认本地配置作为应急 fallback
- 变更需保留配置版本，支持快速回滚到上一个治理快照

## 10. 完成定义

满足以下条件时，本 step 完成：

- 控制面已经成为协议和配置治理的权威入口
- 热路径不再依赖硬编码能力开关和协议判断
- admin / ops / audit 已形成基本闭环

## 11. 下一步准入条件

进入 step 08 前必须确认：

- Agent / IoT 的扩展能力可以直接复用控制面的 capability、registry 和 rollout 机制
