# 技术选型与可插拔策略

## 1. 文档目标

本文档用于冻结 `craw-chat` 的第一阶段技术选型基线，并明确哪些组件是当前推荐依赖，哪些能力要通过抽象层预留，哪些重型中间件不应成为第一阶段前置条件。

## 2. 选型总原则

- 连接密度、正确性、工程可维护性优先于“组件名气”。
- 优先选择成熟基础设施承载通用能力。
- 核心知识产权应沉淀在协议、消息、流、路由与存储抽象，而不是厂商实现细节。
- 所有基础设施都必须通过端口接入，允许逐步替换。
- 第一阶段控制依赖数量，避免把系统做成重中间件拼装工程。

## 3. 第一阶段推荐实现栈

### 3.1 服务端语言与核心框架

- `Rust`
- `Tokio`
- `Axum`
- `Tower`
- `Serde`
- `Tracing`
- `OpenTelemetry`

推荐原因：

- 对高并发长连接、低内存占用和高吞吐友好
- 适合建设连接优先、运行时拆分清晰的架构
- 便于后续对热路径进行细粒度性能治理

### 3.2 长连接、传输与应用协议

- 标准传输：`HTTP / WebSocket / SSE / MQTT`
- 统一应用协议族：`CCP = Craw Chat Protocol`
- 主实时绑定：`ccp/ws/1`
- 命令与补偿绑定：`ccp/http/1`
- 单向流输出绑定：`ccp/sse/1`
- IoT 接入绑定：`ccp/mqtt/1`
- 第一阶段编码：`JSON`，设备侧可选 `CBOR`
- 后续演进：`QUIC`

推荐策略：

- 普通移动端与桌面端优先走 `ccp/ws/1`
- 命令提交、上传初始化、补偿拉取优先走 `ccp/http/1`
- 流式只读或轻交互场景可补充 `ccp/sse/1`
- 智能硬件和弱联网设备优先预留 `ccp/mqtt/1`
- 不自研新的 TCP 级传输协议，自主知识产权沉淀在 `CCP envelope / schema / capability negotiation`

### 3.3 持久化与缓存

- `PostgreSQL`
- `Redis`
- `S3-compatible Object Storage`

职责基线：

- `PostgreSQL`：权威元数据、消息日志、流 checkpoint、投影读模型
- `Redis`：route 热状态、presence snapshot、resume hint、限流窗口
- `Object Storage`：媒体资源、流式大对象、导出与归档

provider/plugin 补充基线：

- RTC 采用 provider plugin 体系，支持 `火山引擎 / 阿里云 / 腾讯云`，默认 provider 为 `火山引擎`
- 对象存储统一走 `S3` 标准接入，覆盖 `阿里云 / 腾讯云 / 火山引擎 / AWS / Google / Microsoft`
- 用户模块采用 plugin 体系，只允许 `本地实现` 与 `外部系统集成` 两种形态，默认 `本地实现`
- IoT 采用“设备接入 + 协议适配”双层插件，首批冻结 `MQTT` 与开源 `小智协议`

### 3.4 可观测与运维

- `Prometheus` 指标抓取
- `OpenTelemetry` 链路与标准埋点
- `Loki-compatible logs` 或兼容日志栈

### 3.5 本地闭环基线

- `local-minimal` profile
- `local-memory` / `local-disk` 适配器
- 单节点可启动、可安装、可验证

## 4. 为什么当前不推荐重型默认栈

以下组件在第一阶段不作为默认前置依赖：

- `Kafka / Redpanda`
- `CockroachDB`
- `ScyllaDB / Cassandra`
- `etcd`
- `NATS JetStream`
- `重型 Service Mesh`

原因如下：

- 连接层和消息层主问题不在于是否引入更多中间件
- 会显著提高部署、运维和私有化门槛
- 会分散对连接热路径、消息模型和架构边界的精力
- 现阶段不符合“最小依赖 + 可替换 + 可商用落地”的目标

## 5. 存储与运行时抽象

第一阶段必须冻结以下抽象：

- `MetadataStore`
- `RouteStore`
- `MessageLog`
- `StreamStore`
- `BlobStore`
- `RtcProviderPort`
- `ObjectStorageProvider`
- `UserModuleProvider`
- `DeviceAccessProvider`
- `IotProtocolAdapter`
- `DeviceRegistry`
- `ProjectionRebuild`

这些抽象才是后续演进自研热路径和替换外部组件的核心边界。

## 6. 能力矩阵

| 能力域 | 第一阶段推荐 | 后续可替换 |
| --- | --- | --- |
| MetadataStore | PostgreSQL | 自定义分库分表、专用元数据库 |
| RouteStore | Redis + PostgreSQL 审计 | 自研 route directory |
| MessageLog | PostgreSQL append model | 自研 segment log |
| StreamStore | PostgreSQL + Redis + Object Storage | 自研 stream segment store |
| BlobStore | S3-compatible Object Storage | 私有对象存储、客户对象存储 |
| Presence / Resume Cache | Redis | 内嵌状态服务或自研热状态层 |
| Observability | OpenTelemetry + Prometheus | 企业内部观测平台适配 |

## 7. 可插拔策略

### 7.1 契约稳定

- 更换后端不得改变 API 契约。
- 更换后端不得改变事件 envelope。
- 更换后端不得改变 `sender / actor / tenant` 的权威字段来源。

### 7.2 Adapter 约束

每个 adapter 必须声明：

- `required capabilities`
- `optional capabilities`
- `degraded behavior`
- `unsupported features`

### 7.3 一致性验证

每个 adapter 必须通过统一 conformance test，至少覆盖：

- 消息有序写入
- 幂等提交
- route fencing
- checkpoint 恢复
- projection rebuild 正确性

### 7.4 领域插件基线

- `rtc` 域只允许通过 `RtcProviderPort` 接入厂商 SDK，禁止在 signaling、消息投影或权限逻辑中写厂商分支。
- `object-storage` 域只允许通过 `ObjectStorageProvider` 暴露 `S3` 兼容能力，禁止业务层直接依赖 `OSS / COS / TOS / GCS / Azure` 私有接口。
- `user-module` 域只允许通过 `UserModuleProvider` 暴露用户查询与维护能力，禁止主链直接绑定某个固定用户中心实现。
- `iot` 域必须拆成 `DeviceAccessProvider + IotProtocolAdapter`，设备管理与协议编解码不能混在一起。

### 7.5 默认 provider 与治理要求

- RTC 全局默认 provider 固定为 `火山引擎`，tenant 或 deployment 级覆盖只能从控制面下发。
- 对象存储默认不绑定单一厂商，但必须满足 `S3` 契约，否则该环境不符合当前标准。
- 用户模块全局默认形态固定为 `本地实现`，外部集成只能通过统一 provider 配置切换。
- IoT 协议启用顺序固定为：先 `MQTT` 主路径，再 `小智协议` 适配，任何新增协议都必须先补插件元数据、配置模板和 conformance 证据。

## 8. 部署 profile 策略

### 8.1 Local Minimal

- 单节点
- 本地内存或本地磁盘适配
- 用于安装验证、契约回归和快速开发

### 8.2 Private Standard

- PostgreSQL
- Redis
- 对象存储
- 标准 HTTP / WebSocket 接入

### 8.3 Private Restricted / Air-Gapped

- 允许客户自有 IAM、KMS、对象存储
- 支持离线镜像与离线升级
- 关闭非必要外部集成

### 8.4 SaaS Shared

- 多租户共享 cell
- 通用基础设施池
- 热租户治理和 lane 隔离

### 8.5 SaaS Dedicated

- 大租户独占 cell
- 独立路由 lane、连接 lane 或存储 lane

## 9. 后续演进策略

### 9.1 优先自研的能力

- `MessageLog`
- `StreamStore`
- route ownership directory
- 连接运行时高密度优化

### 9.2 暂不优先自研的能力

- 通用分布式数据库
- 通用对象存储
- 通用消息队列

## 10. 结论

`craw-chat` 当前最优技术路线不是“先把组件堆满”，而是：

`Rust + CCP + PostgreSQL + Redis + Object Storage + 自有协议抽象 + 后续热路径演进`

这条路线既能快速落地，又能为后续的自主可控和高性能演进保留充分空间。

## 2026-04-09 增补：当前实现基线与技术口径

### A. 当前实现基线

- 当前 workspace 技术基线以 Rust workspace 为中心，核心依赖可见于 `Cargo.toml`：`axum / tokio / serde / serde_json / tower`。
- 当前已存在的 adapter 真实清单包括：`local-memory`、`local-disk`、`object-storage-s3`、`rtc-volcengine`、`rtc-aliyun`、`rtc-tencent`、`iot-access-local`、`iot-mqtt`。
- 当前已存在的服务真实清单包括：`session-gateway`、`conversation-runtime`、`control-plane-api`、`projection-service`、`streaming-service`、`rtc-signaling-service`、`notification-service`、`automation-service`、`media-service`、`audit-service`、`ops-service`、`local-minimal-node`。

### B. 技术选型边界

- 本文中的 PostgreSQL、Redis、S3-compatible Object Storage 是当前推荐标准基线，不等同于每个 profile 都已完整产品化交付。
- provider/plugin 的目标是隔离厂商依赖，不是让业务域直接知道具体云厂商 SDK。
- 在好友、空间、群组、频道 durable truth 稳定前，不引入新的重型中间件来掩盖领域边界缺失。

### C. 文档口径规则

- 写“当前应用信息”时，必须优先引用当前 workspace 中真实存在的服务和 adapter 名称。
- 写“推荐技术路线”时，必须标明它是 `标准基线`、`推荐选型` 或 `后续演进`。
- 技术文档允许抽象，但禁止把候选栈写成当前已交付栈。
