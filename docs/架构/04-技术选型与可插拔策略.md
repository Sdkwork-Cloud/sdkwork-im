# 技术选型与可插拔策略

## 1. 选型原则

- 默认采用当前成熟、先进、可支撑高并发和大规模部署的实现。
- 所有后端通过标准端口接入。
- 不把厂商特性写死进领域模型。
- 支持 `先进默认实现` 与 `替换实现` 共存。

## 2. 第一版推荐实现栈

### 2.1 语言与框架

- Rust
- Tokio
- Axum / Hyper
- Tonic gRPC
- Serde
- Tracing

### 2.2 默认后端

- Commit Journal: `Redpanda`
- Metadata Store: `CockroachDB`
- Timeline Projection Store: `ScyllaDB`
- Cache: `Valkey`
- Lease / Coordination: `etcd`
- Async Integration Bus: `NATS JetStream`
- Object Storage: `S3 API`
- Edge Policy: `Envoy`
- Workload Identity: `SPIRE/SPIFFE`
- Observability: `OpenTelemetry`

### 2.3 本地安装基线

- `local-minimal` profile 采用 `adapters/local-memory`
- 该基线只用于单节点开发、安装验证和契约回归
- 生产默认栈目录保留为 `journal-redpanda`、`meta-cockroach`、`timeline-scylla`

## 3. 替换后端矩阵

| 端口 | 默认实现 | 替换实现 |
| --- | --- | --- |
| CommitJournal | Redpanda | Kafka |
| MetadataStore | CockroachDB | PostgreSQL |
| TimelineProjectionStore | ScyllaDB | Cassandra |
| CacheStore | Valkey | Redis-compatible |
| LeaseStore | etcd | Control-plane embedded lease |
| ObjectStore | S3 | 客户对象存储实现 |

## 4. 端口能力定义

### 4.1 CommitJournal

必需能力：

- ordered append
- durable ack
- replay
- checkpoint
- retention
- partition routing

可选能力：

- tiered storage
- idempotent producer
- rack awareness
- geo replication

### 4.2 MetadataStore

必需能力：

- transaction
- unique constraint
- optimistic concurrency
- secondary index
- tenant scope

### 4.3 TimelineProjectionStore

必需能力：

- append-friendly write
- conversation range query
- pagination by seq
- idempotent upsert
- rebuild from journal

### 4.4 LeaseStore

必需能力：

- lease acquire/renew/release
- fencing token
- watch
- compare-and-set

### 4.5 ObjectStore

必需能力：

- put/get/delete
- multipart upload
- presigned URL
- checksum

## 5. 可插拔策略

### 5.1 接口稳定

所有 adapter 只能实现 `contracts` 暴露的能力，不允许修改领域对象。

### 5.2 Capability 模型

每个适配器必须声明：

- required capabilities
- optional capabilities
- degraded behavior
- unsupported features

### 5.3 Conformance Test

每个适配器必须通过统一一致性测试：

- journal ordering
- idempotent append
- lease fencing
- metadata transaction semantics
- projection replay correctness

## 6. Profile 策略

### 6.1 SaaS Shared

- Redpanda
- ScyllaDB
- CockroachDB
- Valkey
- NATS

### 6.2 SaaS Dedicated

- 同 Shared
- 但允许为头部租户独立 cell 和独立数据 lane

### 6.3 Private Standard

- 可替换为客户环境的元数据库、对象存储、身份系统

### 6.4 Private Air-Gapped

- 完整支持本地依赖装配
- 支持关闭外部集成适配器

### 6.5 Local Minimal

- `local-memory` adapter bundle
- 单节点
- 进程内状态
- 用于最小闭环与安装验证

## 7. 设计要求

- 更换后端不得改变 API 契约。
- 更换后端不得改变事件语义。
- 更换后端不得改变消息体结构。
- 仅允许性能、成本、运维体验变化。
- 认证派生的 `tenant` / `actor` / `sender` 结构不得因后端切换而改变。
