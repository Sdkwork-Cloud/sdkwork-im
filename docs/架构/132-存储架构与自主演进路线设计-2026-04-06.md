# 存储架构与自主演进路线设计

## 1. 文档目标

本文档用于定义 `craw-chat` 的存储分层、职责边界、扩容策略与自主演进路线，解决以下问题：

- 哪些数据应该进入 PostgreSQL
- 哪些状态应该进入 Redis
- 哪些内容必须进入对象存储
- 是否应立即自建分布式数据库
- 哪些存储能力必须掌握自主知识产权
- 如何从成熟组件平滑演进到自研热路径

## 2. 总体判断

结论明确如下：

- 第一阶段不建议自建通用分布式数据库
- 第一阶段建议使用：
  - `PostgreSQL`
  - `Redis`
  - `S3-compatible Object Storage`
- 第一阶段就必须自研并冻结：
  - `MessageLog` 抽象
  - `StreamStore` 抽象
  - `RouteStore` 抽象
  - `ProjectionRebuild` 机制
  - 消息与流的统一事件标准

系统真正的核心知识产权应放在：

- 消息模型
- 流模型
- 路由模型
- 投影模型
- 存储抽象层

而不是一开始就去重造数据库内核。

## 3. 存储分层

### 3.1 MetadataStore

职责：

- 租户
- 用户
- principal
- device 元数据
- conversation 元数据
- member 关系
- topic/thread 元数据
- 权限与策略
- 管理面配置

推荐实现：

- PostgreSQL

### 3.2 RouteStore

职责：

- 在线路由
- session fence
- reconnect hint
- route epoch
- node drain 状态

第一阶段推荐：

- Redis 作为热点状态存储
- PostgreSQL 保留长期元数据与必要审计

### 3.3 MessageLog

职责：

- 消息写入顺序
- 幂等
- 回放
- durable truth
- 事件重建

第一阶段推荐：

- 定义独立抽象
- 底层先落在 PostgreSQL append model
- 后续再演进到段文件 / 自研日志存储

### 3.4 StreamStore

职责：

- stream session
- frame append
- checkpoint
- finalize / abort
- partial materialization

第一阶段推荐：

- 短态进 Redis
- durable state 进 PostgreSQL
- 大 payload / blob 进对象存储

### 3.5 BlobStore

职责：

- 图片
- 音频
- 视频
- 文件
- 流式大对象
- LLM 输出附件
- 设备上传资源

推荐实现：

- S3-compatible Object Storage

### 3.6 ProjectionStore

职责：

- inbox
- timeline
- unread
- summary
- AI 上下文视图
- topic/thread 查询视图

第一阶段推荐：

- PostgreSQL 内部读模型表

后续可选：

- 专用读库
- 搜索引擎
- embedding 索引层

## 4. 数据分类

### 4.1 强事务元数据

特征：

- 需要强约束
- 需要唯一性
- 需要关系模型
- 写入量中等但必须正确

代表对象：

- tenant
- user
- principal
- conversation metadata
- member
- device registration
- policy
- admin config

### 4.2 热状态

特征：

- 高频读写
- 允许短期最终一致
- 丢失后可从权威源恢复

代表对象：

- current route
- session hint
- resume hint
- rate limit counters
- presence hot snapshot

### 4.3 事件与日志型数据

特征：

- 有序
- 需要回放
- durable truth

代表对象：

- message appended
- message edited
- message recalled
- stream lifecycle
- rtc lifecycle
- automation side effect event

### 4.4 大对象

特征：

- 容量大
- 访问模式独立
- 不适合直接放进主事务库

代表对象：

- media asset
- recording
- transcript artifact
- stream blob
- IoT media upload

## 5. 为什么不第一阶段自建分布式数据库

原因包括：

- 正确性成本极高
- 副本与恢复复杂度过高
- 数据修复、压缩整理、迁移成本高
- 会显著拖慢产品落地
- 与当前阶段最重要的“连接内核 + 消息协议 + 扩容能力”主目标不一致

真正符合商业可控的策略是：

`先冻结自己的消息与流存储抽象，再逐步替换底层实现。`

## 6. 自主知识产权边界

### 6.1 必须掌握

- `MessageLog` 协议与抽象
- `StreamStore` 协议与抽象
- `RouteStore` 协议与抽象
- 消息序号与回放规则
- stream frame checkpoint 规则
- materialize 规则
- projection rebuild 规则

### 6.2 可先依赖成熟组件

- PostgreSQL
- Redis
- S3-compatible Object Storage

### 6.3 后续可演进自研

- message log segment storage
- stream segment storage
- route directory
- local embedded event log

## 7. 第一阶段推荐表结构方向

### 7.1 PostgreSQL 权威表

建议至少包括：

- `tenants`
- `principals`
- `devices`
- `conversations`
- `conversation_members`
- `conversation_topics`
- `message_log_entries`
- `stream_sessions`
- `stream_frames_checkpoint`
- `rtc_sessions`
- `media_assets`
- `notification_tasks`
- `automation_executions`
- `projection_inbox`
- `projection_timeline`

### 7.2 Redis 热状态键空间

建议至少包括：

- route ownership
- session fence
- reconnect hint
- presence hot snapshot
- rate limit windows
- hot summary cache

## 8. MessageLog 设计建议

### 8.1 关键字段

- `tenant_id`
- `scope_kind`
- `scope_id`
- `aggregate_id`
- `event_id`
- `event_type`
- `ordering_key`
- `ordering_seq`
- `idempotency_key`
- `actor`
- `occurred_at`
- `payload`

### 8.2 关键约束

- 会话内有序
- ordering_seq 单调递增
- 幂等键可重复提交但不可重复副作用
- 回放必须可重建投影

### 8.3 第一阶段实现建议

- PostgreSQL 表承载 append log
- 按 `tenant + scope` 建索引
- 保持事件 envelope 稳定

### 8.4 第二阶段演进建议

- 按段文件分片
- 增加冷热分层
- 增加压缩与归档
- 增加副本与恢复策略

## 9. StreamStore 设计建议

### 9.1 流式状态拆分

流不应只是一张大表。

建议拆为：

- `stream_session`
- `stream_checkpoint`
- `stream_materialization`
- `stream_blob_reference`

### 9.2 适用负载

必须支持：

- 文本 delta
- JSON patch
- 音频 chunk
- telemetry frame
- agent output
- long task progress

### 9.3 冷热分层

建议：

- 热帧短时放 Redis 或本地缓存
- durable checkpoint 进 PostgreSQL
- 大帧 blob 进对象存储
- materialize 后可按策略清理原始 frame

## 10. Projection 与查询层设计

### 10.1 原则

- 投影不是权威源
- 投影可以重建
- 投影故障不能阻塞 commit

### 10.2 第一阶段

- PostgreSQL 内部读模型表
- 定时修复与重放支持

### 10.3 后续阶段

- 搜索索引
- embedding 索引
- AI 上下文索引
- topic/thread 优化视图

## 11. 扩容与分区策略

### 11.1 PostgreSQL

优先策略：

- 按 tenant 范围索引
- 大表按时间或 tenant 分区
- 读写职责分离

### 11.2 Redis

优先策略：

- route / session / presence 明确 key space
- 有 TTL 的状态与无 TTL 的状态分离
- 允许热点键监测与 tenant 限流

### 11.3 Object Storage

优先策略：

- bucket / prefix 按 tenant 或 resource domain 规划
- presigned URL
- checksum
- lifecycle policy

## 12. 备份与恢复

### 12.1 必须支持

- metadata snapshot
- message log replay
- projection rebuild
- object store backup
- tenant scoped recovery

### 12.2 恢复顺序

- 恢复 metadata
- 恢复 message log / stream checkpoint
- 重建 route 热状态
- 重放 projection
- 恢复大对象引用一致性

## 13. 未来演进路线

### 阶段 1

- PostgreSQL + Redis + Object Storage
- 抽象层冻结

### 阶段 2

- 自研 MessageLog
- 自研 StreamStore
- 继续保留 PostgreSQL 作为权威元数据存储

### 阶段 3

- 自研热点日志存储
- 更强的 segment / replica / compaction

### 阶段 4

- 按需引入更高阶的自有存储内核

## 14. 结论

最佳路线不是“第一天就全自研数据库”，而是：

`先自研协议与热路径抽象，先用成熟基础组件承载数据，再在最有价值的消息与流路径上逐步实现自有存储能力。`
