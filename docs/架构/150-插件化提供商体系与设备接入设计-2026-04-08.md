# 插件化提供商体系与设备接入设计

## 1. 文档目标

本文档用于冻结 `craw-chat` 的插件化提供商体系标准，确保 RTC、对象存储、IoT 不再以内嵌厂商分支或旁路协议的方式演进，而是统一纳入：

- 可插拔 provider/plugin 架构
- 控制面可治理的 provider registry
- 面向租户、环境和发布域的 provider 选择策略
- 可重复迭代的 conformance 与 rollout 机制

本轮冻结的能力域包括：

- RTC provider plugin
- Object Storage provider plugin
- UserModule plugin
- IoT access / protocol plugin
- 设备管理与接入体系

## 2. 设计总原则

- 业务契约优先，厂商实现后置。
- 控制面负责 provider 选择与能力治理，数据面只消费 `effective provider snapshot`。
- 任何 provider 接入都不得把厂商 SDK 类型泄漏到消息、流、RTC、设备领域模型。
- RTC 默认 provider 固定为 `火山引擎`，但必须保留 `阿里云`、`腾讯云` 的标准接入位。
- 对象存储一律通过 `S3` 标准接入，业务层不允许出现厂商私有对象存储分支。
- IoT 必须拆成“设备接入/认证”和“协议适配”两层，避免把 `MQTT`、`小智协议` 直接揉进业务流程。
- 每个插件都必须声明能力边界、降级行为、配置模式和验证证据，支持反复迭代而不破坏总架构。

## 3. 插件体系总览

### 3.1 能力域

统一插件域如下：

- `rtc`
- `object-storage`
- `user-module`
- `iot-access`
- `iot-protocol`

其中：

- `rtc` 负责媒体房间、入会凭证、回调、录制与产物回流。
- `object-storage` 负责媒体、录制、导出、归档和大对象。
- `user-module` 负责用户主数据、用户维护、外部用户目录映射与用户查询。
- `iot-access` 负责设备注册、鉴权、会话、绑定和接入策略。
- `iot-protocol` 负责协议编解码、topic / channel 归一化和上下行映射。

### 3.2 分层标准

每个插件域都必须遵守相同分层：

1. `contract / port`
2. `runtime registry`
3. `provider adapter`
4. `control-plane policy`
5. `conformance / rollout`

禁止直接从 service handler 调用厂商 SDK。所有接入都必须先落到 `port`，再由 runtime 装配具体 plugin。

### 3.3 标准元数据

每个 plugin 至少声明以下元数据：

- `pluginId`
- `domain`
- `providerKind`
- `interfaceVersion`
- `configSchemaRef`
- `requiredCapabilities`
- `optionalCapabilities`
- `degradedBehavior`
- `unsupportedFeatures`
- `healthSignals`

### 3.4 推荐代码组织

推荐的代码组织方向如下：

- `contract-provider-rtc`
- `contract-provider-storage`
- `contract-provider-iot`
- `runtime-provider-registry`
- `adapter-rtc-volcengine`
- `adapter-rtc-aliyun`
- `adapter-rtc-tencent`
- `adapter-storage-s3`
- `adapter-user-module-local`
- `adapter-user-module-external`
- `adapter-iot-mqtt`
- `adapter-iot-xiaozhi`
- `app-device-management`

文档先冻结边界，代码实现可按 step 拆批次推进。

## 4. RTC provider plugin 标准

### 4.1 统一边界

RTC provider 只负责以下厂商相关能力：

- 房间 / 会话创建与销毁
- 参会 token / credential 下发
- 房间成员状态回调
- 服务端录制、转写、混流等可选能力
- 录制产物与回放地址回流
- 区域、机房和套餐能力选择

以下能力不属于 provider 插件：

- `CCP` 信令协议
- `rtc.invite / accept / reject / end`
- 会话成员鉴权
- timeline 物化
- 消息与通知主链路

也就是说，`RTC signaling` 仍由平台统一掌控，provider 插件只承接媒体与房间基础设施差异。

### 4.2 标准契约

RTC 插件至少要暴露：

- `create_session`
- `close_session`
- `issue_participant_credential`
- `refresh_participant_credential`
- `map_provider_callback`
- `export_recording_artifact`
- `provider_health_snapshot`

录制、转写、回放等能力可以是可选 capability，但 `session`、`credential`、`callback` 和 `health snapshot` 必须是必选能力。

### 4.3 provider 矩阵

| Provider | pluginId | 默认状态 | 备注 |
| --- | --- | --- | --- |
| 火山引擎 | `rtc-volcengine` | 默认 | 当前全局默认 RTC provider |
| 阿里云 | `rtc-aliyun` | 受支持 | 与统一 signaling / callback contract 对齐 |
| 腾讯云 | `rtc-tencent` | 受支持 | 与统一 signaling / callback contract 对齐 |

### 4.4 默认与选择策略

- 全局默认 RTC provider 固定为 `火山引擎`。
- tenant 级、profile 级和 region 级可以覆盖默认值，但必须通过控制面下发。
- 不允许在业务代码中按 `if provider == xxx` 选择实现。
- 状态中的 RTC session 一经创建，其 provider 归属必须冻结到会话生命周期结束，避免跨 provider 漂移。

### 4.5 降级与回退

- 若某 provider 不支持录制或转写，允许关闭相关 capability，但不得影响基础通话建立。
- provider 失败后的回退必须由控制面或 operator 明确触发，不允许对活跃会话做静默自动跨厂商迁移。
- RTC 产物必须统一回流到 `BlobStore` / `ObjectStorageProvider`，禁止把录制文件长期锁死在 provider 私有路径中。

## 5. Object Storage provider plugin 标准

### 5.1 统一接口

对象存储统一暴露为：

- `BlobStore`
- `ObjectStorageProvider`

统一能力通过 `S3` 标准完成：

- bucket / prefix 管理
- pre-signed URL
- multipart upload
- object get / put / delete
- lifecycle / retention / archive
- server-side encryption

业务层只认 `S3` 兼容契约，不认厂商 SDK。

### 5.2 provider 矩阵

| Provider | pluginId | 接入要求 |
| --- | --- | --- |
| 阿里云 | `object-storage-aliyun` | 通过 `S3` 兼容端点接入 |
| 腾讯云 | `object-storage-tencent` | 通过 `S3` 兼容端点接入 |
| 火山引擎 | `object-storage-volcengine` | 通过 `S3` 兼容端点接入 |
| Amazon Web Services | `object-storage-aws` | 原生 `S3` |
| Google | `object-storage-google` | 必须通过 `S3` 兼容网关或代理端点接入 |
| Microsoft | `object-storage-microsoft` | 必须通过 `S3` 兼容网关或代理端点接入 |

### 5.3 强约束

- 业务代码不得直接依赖 `OSS / COS / TOS / GCS / Azure Blob` 私有 SDK 类型。
- 若 Google 或 Microsoft 环境未提供可用的 `S3` 兼容入口，则该部署不满足本架构标准。
- RTC 录制、会话附件、导出包、归档文件、设备媒体回执都必须复用同一套对象存储插件体系。

## 6. UserModule plugin 标准

### 6.1 统一定位

“用户中心”在当前架构中统一收敛为 `用户模块 plugin`，而不是单独做一个固定中心服务。

`user-module` 只允许两种形态：

- `local`：平台本地维护用户
- `external`：集成外部系统进行用户维护

全局默认形态固定为 `local`。

### 6.2 标准契约

用户模块至少要暴露：

- `get_user`
- `batch_get_users`
- `search_users`
- `create_or_bind_user`
- `update_user_profile`
- `disable_user`
- `map_external_principal`

说明：

- `local` 形态由平台直接维护用户主数据。
- `external` 形态由平台维护最小映射与缓存，真实用户维护动作落到外部系统。

### 6.3 边界要求

- conversation / member / sender 只能依赖统一 `UserModuleProvider` 或等价 port，不能直接依赖本地用户表。
- `sender / actor / principal` 的解析必须允许同时来自本地用户和外部用户映射。
- `external` 形态下，平台仍需保留最小本地影子标识，用于成员、消息、审计和权限闭环。

### 6.4 形态矩阵

| 形态 | pluginId | 默认状态 | 说明 |
| --- | --- | --- | --- |
| 本地实现 | `user-module-local` | 默认 | 平台本地维护用户与资料 |
| 外部系统集成 | `user-module-external` | 受支持 | 外部系统维护用户，平台维护映射与缓存 |

### 6.5 外部集成要求

- 外部系统可以是企业主数据、组织、人事、CRM、IAM 或其他统一用户源。
- 平台必须支持 `pull / webhook / event` 三类同步策略中的至少一种。
- 外部用户集成不能绕过平台权限、租户、成员和审计主链。

## 7. IoT plugin 标准

### 7.1 双层抽象

IoT 必须拆成两类插件：

- `DeviceAccessProvider`
- `IotProtocolAdapter`

前者负责设备接入与治理，后者负责协议映射与编解码。

### 7.2 统一设备管理与接入体系

设备管理必须至少覆盖：

- product / product model
- device registry
- credential / certificate / token issuance
- tenant binding
- owner binding
- device session
- device presence
- device twin / shadow
- telemetry uplink
- command downlink
- command ack / error
- 审计与封禁

这意味着 IoT 不是“只有 MQTT topic 收发”，而是一整套设备管理与接入体系。

### 7.3 协议矩阵

| Protocol | pluginId | 定位 |
| --- | --- | --- |
| MQTT | `iot-mqtt` | 通用设备接入与上下行主协议 |
| 小智协议 | `iot-xiaozhi` | 开源小智协议适配器，映射到统一设备模型 |

### 7.4 协议适配规则

每个 `IotProtocolAdapter` 必须负责：

- 鉴权握手映射
- uplink -> `device.telemetry` / `stream` 归一化
- downlink -> `device.command` / `command stream` 归一化
- topic / channel / QoS / ack 归一化
- 错误码和设备状态归一化
- 协议版本与 capability 协商

`小智协议` 适配器也必须落在这一层，不得绕开统一 `device session / twin / command / telemetry` 领域模型。

## 8. 控制面与部署治理

### 8.1 provider registry

控制面必须维护统一 provider registry，至少包括：

- domain
- providerKind
- pluginId
- version
- configSchemaRef
- capability matrix
- rollout channel
- kill switch
- health state

### 8.2 生效顺序

provider 选择优先级固定为：

1. 紧急 operator override
2. tenant policy
3. deployment profile default
4. global default

RTC 的 global default 固定为 `火山引擎`。
用户模块的 global default 固定为 `local`。

### 8.3 配置分层

provider 配置必须分层：

- build-time：编译特性与插件装配开关
- deploy-time：endpoint、region、bucket、AK/SK、callback secret
- runtime control：provider profile、quota、kill switch、rollout
- tenant policy：租户级 provider 选择与 capability 限制

### 8.4 交付要求

每个 provider/plugin 交付都必须带齐：

- 插件说明
- 配置模板
- capability matrix
- conformance 结果
- review 文档
- 架构回写

## 9. 一致性验证与反复迭代策略

### 9.1 conformance 套件

各域至少需要如下验证：

- RTC：session / credential / callback / artifact / degrade behavior
- Object Storage：put / get / delete / multipart / presign / retention
- UserModule：local / external 两种形态下的 user lookup / sender mapping / member binding
- IoT：register / auth / connect / telemetry / command / twin / protocol mapping

### 9.2 最小迭代单元

后续反复迭代时，每一轮只做一个“最小可闭环增量”，例如：

- `rtc-volcengine` 基线
- `rtc-aliyun` 补齐
- `object-storage-s3` 公共契约
- `user-module-local` 基线
- `user-module-external` 外部映射
- `iot-mqtt` 主路径
- `iot-xiaozhi` 协议适配

每轮都必须同步交付：

- 架构文档回写
- step 文档回写
- review 证据
- 自动化验证
- 下一轮输入

### 9.3 禁止事项

- 禁止先写厂商分支、后补抽象。
- 禁止为单个 provider 单独造一套消息、RTC、设备事件模型。
- 禁止让 deployment profile 与 provider 选择规则各写一份私有逻辑。

## 10. 冻结结论

当前版本冻结以下结论：

- RTC 采用 provider plugin 体系，支持 `火山引擎 / 阿里云 / 腾讯云`，默认 `火山引擎`。
- 对象存储统一走 `S3` 标准，支持 `阿里云 / 腾讯云 / 火山引擎 / AWS / Google / Microsoft`。
- 用户模块采用 plugin 体系，只允许 `本地实现` 与 `外部系统集成` 两种形态，默认 `本地实现`。
- IoT 采用“设备管理与接入 + 协议适配”双层插件架构，首批支持 `MQTT` 与开源 `小智协议`。
- 设备管理和接入体系是架构主链的一部分，不是协议插件的附属脚本。
- 后续任何 provider 接入都必须先满足本文档，再进入代码实现。
