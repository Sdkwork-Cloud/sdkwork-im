# 23-节点排空与路由迁移控制标准

## 1. 目标

本标准定义 realtime 集群在节点排空、路由迁移与恢复激活时的控制面写路径。

目标不是引入一套新的会话系统，而是在现有 `device route ownership`、`RealtimeDeliveryRuntime`、`ops/cluster` 与 `ops/diagnostics` 基础上，把以下语义冻结下来：

- 节点进入 `draining` 后，不能再接收新的稳定 route 绑定
- 已有 route 可以按控制面命令迁移到新的 `owner node`
- route 迁移不能只迁目录，还必须迁移最小 realtime window 状态
- `ops` 视图必须能够反映 drain 与 rebalance 的当前阶段
- SaaS 与私有化部署共享同一套 drain 协议，不因底层组件替换而重做协议

## 2. 核心原则

### 2.1 drain 采用双阶段

节点排空不是一个瞬时布尔值，而是两个明确阶段：

1. `drain requested`
2. `route migrated`

进入 `draining` 只意味着：

- 新 route bind 被拒绝
- 旧 route 仍然可见
- 当前节点进入待迁移状态

只有在 route 已迁走且本节点持有的 route 数量变成 `0` 时，节点才允许进入 `drained`。

### 2.2 route ownership 与 runtime state 必须一起迁移

只修改 `ownerNodeId` 不足以完成商业级 drain，因为 realtime 能力还依赖以下设备级状态：

- subscriptions
- realtime event window
- latest realtime seq
- acked through seq
- trimmed through seq

因此，控制面执行 route migration 时，最小实现也必须完成 runtime state handoff。

### 2.3 target node 必须是可接收状态

route 只能迁移到：

- `drainStatus = active`
- `rebalanceState = stable`

的目标节点。

控制面不得把 route 迁移到另一个已经进入 `draining` 或 `drained` 的节点。

### 2.4 transport cutover 与 route migration 解耦

本阶段标准把 `route ownership` 与 `realtime window state` 的迁移冻结下来，但不要求最小实现同步完成：

- WebSocket 连接踢迁
- L4/L7 负载均衡摘流
- 连接代理侧的连接搬迁

商业版可以在此标准之上追加 transport cutover 编排，但不得改变 route drain 协议本身。

## 3. 节点生命周期状态机

### 3.1 `drainStatus`

- `active`
  - 节点正常接收新 route bind
- `draining`
  - 节点已进入排空，拒绝新 bind，等待 route 迁移
- `drained`
  - 节点 route 已清空，可安全摘除或升级

### 3.2 `rebalanceState`

- `stable`
  - 当前没有 route 迁移动作
- `moving_routes`
  - 控制面正在执行 route migration，或 drain 后仍存在待迁移 route
- `awaiting_resume`
  - 预留给商用版的 transport cutover / client resume 协调阶段

当前最小实现至少要正确支持：

- `active + stable`
- `draining + moving_routes`
- `drained + stable`

## 4. 控制面写路径

控制面服务对外暴露内部运维 API。当前标准冻结以下最小写路径：

### 4.1 `POST /backend/v3/api/control/nodes/{nodeId}/drain`

作用：

- 将节点标记为 `draining`
- 如果节点仍持有 route，则 `rebalanceState` 进入 `moving_routes`
- 后续所有针对该节点的新 route bind 都必须被拒绝

响应示例：

```json
{
  "nodeId": "node_a",
  "drainStatus": "draining",
  "rebalanceState": "moving_routes",
  "ownedRouteCount": 3
}
```

### 4.2 `POST /backend/v3/api/control/nodes/{nodeId}/activate`

作用：

- 取消 drain
- 节点重新进入 `active + stable`
- 允许后续 route bind 重新落到该节点

响应示例：

```json
{
  "nodeId": "node_a",
  "drainStatus": "active",
  "rebalanceState": "stable",
  "ownedRouteCount": 0
}
```

### 4.3 `POST /backend/v3/api/control/nodes/{nodeId}/routes/migrate`

请求体：

```json
{
  "targetNodeId": "node_b"
}
```

作用：

- 把源节点当前持有的全部 device route 迁移到目标节点
- 同步迁移最小 realtime runtime state
- 源节点 route 清零后进入 `drained + stable`
- 目标节点保持 `active + stable`

响应示例：

```json
{
  "sourceNodeId": "node_a",
  "targetNodeId": "node_b",
  "migratedRouteCount": 3,
  "sourceDrainStatus": "drained",
  "sourceRebalanceState": "stable",
  "targetDrainStatus": "active",
  "targetRebalanceState": "stable"
}
```

## 5. bind 与 publish 语义

### 5.1 bind 语义

`bind_device_route(tenantId, principalId, deviceId, ownerNodeId, connectionKind)` 必须遵守：

- 当 `ownerNodeId` 处于 `active` 时，bind 正常成功
- 当 `ownerNodeId` 处于 `draining` 或 `drained` 时，bind 直接拒绝
- 拒绝 bind 时，不得覆盖已有 route ownership
- 拒绝 bind 时，调用方必须收到显式错误，而不是静默 fallback

### 5.2 publish 语义

route migration 完成后，后续 realtime publish 必须路由到新的 owner node。

如果 route 不存在，仍允许保留当前的 `origin node fallback` 语义；但这只用于缺失 route 的兜底，不得作为 drain/migration 的常规路径。

## 6. 最小 runtime state handoff 标准

迁移单个设备 route 时，源 runtime 必须导出并移除该设备的以下状态：

- subscriptions
- events window
- latest sequence
- acked sequence
- trimmed sequence

目标 runtime 导入后必须满足：

- `list_events` 能继续读到迁移前尚未 trim 的 realtime window
- `ack_events` 能在迁移后的目标节点继续推进 `ackedThroughSeq`
- `publish_scope_event` 能基于迁移后的 subscriptions 继续命中设备
- notifier 在目标节点重新建立，初始序列等于迁移后的 `latestRealtimeSeq`

当前标准不要求迁移：

- 传输层连接对象
- TCP/WebSocket socket 本身
- 设备 presence 的跨节点强一致归属

这些能力可以后续继续演进，但不能破坏已冻结的 runtime handoff 语义。

## 7. 运维可观测标准

### 7.1 `ops/cluster`

运维摘要至少要看到：

- `drainStatus`
- `rebalanceState`
- `deviceRouteCount`

### 7.2 `ops/diagnostics`

运维诊断至少要看到：

- 当前节点持有的 `deviceRoutes`
- route 迁移前后的 owner 变化
- route 清空后 `deviceRouteCount = 0`

### 7.3 状态收敛要求

排空完成的最小收敛条件：

1. 源节点 `deviceRouteCount = 0`
2. 源节点 `drainStatus = drained`
3. 源节点 `rebalanceState = stable`
4. 目标节点 `deviceRouteCount` 增加到预期值
5. 目标节点可以继续接收 realtime publish

## 8. 当前落地边界

当前最小实现允许采用共享进程内的 cluster bridge 完成标准验证，但必须预留后续替换点：

- route directory store
- node lifecycle store
- drain orchestration
- runtime state checkpoint / persistence

这意味着当前实现可以是 `in-memory`，但接口与语义必须已经适合未来替换为：

- shared metadata store
- MQ bridge
- route directory service
- commercial control plane orchestration

## 9. 与后续标准的关系

本标准冻结的是：

- drain 写路径
- route migration 语义
- minimal runtime handoff

后续还需要继续冻结：

- ack checkpoint 持久化标准
- route lease / epoch / fencing 标准
- transport cutover 与 graceful disconnect 标准

但这些后续标准不得推翻本标准定义的控制面接口与状态机。
