# 关键业务链路与跨 Plane 时序设计

## 1. 文档目标

本文档用于把 `craw-chat` 的总体架构进一步具体化为可执行的关键链路时序，回答以下问题：

- 一个请求如何跨越 `Link / Route / Messaging / Stream / Projection / Storage` 各个 plane
- 哪些步骤必须同步完成，哪些步骤必须异步解耦
- 在消息、流、AI、IoT、节点迁移场景下，系统不变量是什么
- 日后代码重构时，哪些链路不能被随意打散

## 2. 总体时序原则

- 所有外部请求先进入 `Link Plane` 完成接入、认证、限流与协议绑定。
- 所有需要在线归属的信息都要经 `Route Plane` 获取或校验，而不是由业务层猜测。
- 所有 durable 业务状态必须先进入 `Messaging Plane / Stream Plane + Storage Plane` 的权威写路径。
- 所有 `Projection / Notification / Automation / Search` 都是提交成功后的异步后续动作。
- 任意链路都必须能清晰区分：
  - `同步主链`
  - `异步副链`
  - `恢复链`

## 3. 核心链路一：连接建立与恢复

### 3.1 首次连接

标准时序如下：

1. 客户端进入 `Link Plane`，完成 TLS、WebSocket 握手与连接注册。
2. `Link Plane` 调用认证管线，解析 JWT / session / trusted identity，生成权威上下文。
3. `Link Plane` 请求 `Route Plane` 检查或创建当前 `device_id + session_id` 的 route 归属。
4. `Route Plane` 返回当前 `owner_node_id`、`route_epoch`、`resume_token` 和必要的 fencing 信息。
5. `Link Plane` 在本机 shard 中绑定连接，并为该连接创建出站队列、心跳状态和背压状态。
6. 客户端再通过 `session.resume / subscriptions.sync` 拉起补偿链和订阅链。

### 3.2 断线恢复

标准时序如下：

1. 客户端重连，带上 `deviceId`、`sessionId`、`resumeToken`、`lastSeenSyncSeq`、`lastCheckpoint`。
2. `Link Plane` 完成认证后，请求 `Route Plane` 校验 route epoch 是否仍匹配。
3. 若 route epoch 已升级，旧连接必须被 fencing，新的归属成为唯一 owner。
4. `Link Plane` 从 `RouteStore` 与 `Projection / Sync Feed` 获取恢复起点。
5. 恢复链先补偿 `device sync feed`，再恢复实时订阅和流式状态。
6. 恢复成功后刷新 `presence snapshot` 和 `resume hint`。

### 3.3 链路不变量

- 一个 `device session` 任一时刻只能有一个权威 route owner。
- 旧 epoch 的连接不能继续写入、继续投递、继续 ACK。
- `resume` 失败时允许退化为全量补拉，但不能直接伪造“已同步”。

## 4. 核心链路二：普通消息发送

### 4.1 同步主链

标准时序如下：

1. 客户端通过 HTTP 或 WebSocket 发起 `message.post`。
2. `Link / Interface` 层只做参数校验、权威上下文绑定和命令 envelope 组装。
3. `Messaging Plane` 校验：
   - tenant
   - conversation membership
   - capability
   - idempotency key
   - 会话当前写入状态
4. `Storage Plane` 先写入 `MessageLog` 或 append model 权威记录。
5. `Messaging Plane` 分配 `message_seq`，产出 `message.posted` 事件。
6. 同步返回 `messageId / messageSeq / committedAt / sender snapshot`。

### 4.2 异步副链

提交成功后异步触发：

- `Projection Plane` 更新 timeline / summary / inbox / unread
- `Delivery Plane` 或运行时向在线连接投递
- `Notification` 生成通知请求
- `Audit` 写入审计锚点
- `Automation / Webhook` 按策略处理

### 4.3 链路不变量

- timeline 不是真相，MessageLog 才是真相。
- 副链失败不能回滚主链提交。
- `tenantId`、`sender`、`actor` 必须来自权威上下文或成员快照，不可由客户端业务体伪造。

## 5. 核心链路三：AI 流式对话

### 5.1 标准时序

1. 用户向会话发送一个普通消息或 `agent.invoke` 命令。
2. `Messaging Plane` 提交用户消息。
3. `Agent Runtime` 订阅到会话上下文和触发条件，创建 `stream session`。
4. `Stream / AI Plane` 写入 `stream.open`，分配 `stream_id`。
5. AI 推理过程持续写入：
   - `stream.delta`
   - `stream.patch`
   - `stream.checkpoint`
6. 在线用户通过 WebSocket / SSE 接收流式增量。
7. 推理结束后，`Stream Plane` 写入 `stream.finalize`。
8. 根据策略物化为一条普通消息、卡片、文件或知识引用。

### 5.2 设计约束

- 流本身是一等能力，AI 只是流的一类使用场景。
- AI 失败可以中止流，但不能破坏会话原有消息顺序。
- 物化消息必须保留对原始 `stream_id` 的引用关系。

## 6. 核心链路四：群聊 fanout 与多端同步

### 6.1 群聊 fanout

1. 一条群消息提交后，`Projection Plane` 先更新群视图。
2. `Route Plane` 查询在线成员设备的 route 分布。
3. 每个目标连接通过本机 `Link Plane` 的出站队列投递。
4. 离线成员依赖 `sync feed / unread projection` 进行后续补偿。

### 6.2 多端同步

1. 对同一主体的多设备，服务端为每台设备构建独立的 `device sync feed`。
2. 当前发送设备可以通过 `actorDeviceId` 做本机去重。
3. 其他设备按 `sync_seq` 顺序补拉，直到对齐。

### 6.3 设计约束

- 在线实时投递和离线补偿必须共享同一套事件来源。
- 任何设备级补偿都不能绕过成员关系与租户边界。

## 7. 核心链路五：IoT 设备上行与指令下行

### 7.1 设备上行

1. 设备通过 MQTT / WebSocket 接入 `Link Plane`。
2. 认证通过后，设备作为 `device principal` 建立 session。
3. 设备的 telemetry 进入 `Stream Plane`，按 `stream.delta` 或 `device.telemetry` 统一建模。
4. 关键事件按策略物化为消息或通知。

### 7.2 指令下行

1. 用户或 Agent 发起 `device.command`。
2. `Messaging / IoT Plane` 校验设备绑定关系与授权。
3. `Route Plane` 查找设备当前 owner node。
4. `Link Plane` 将指令写入设备连接出站队列。
5. 设备回执走 `stream` 或 `message` 返回。

### 7.3 设计约束

- 设备不是外挂对象，而是统一主体体系的一部分。
- 设备命令必须和普通聊天一样经过 tenant、actor、capability 校验。

## 8. 核心链路六：节点排空与路由迁移

### 8.1 排空时序

1. `Ops Plane` 将目标节点标记为 `draining`。
2. `Route Plane` 停止为其分配新的 route。
3. 现存连接在新的 route owner 准备好之后逐步迁移。
4. 迁移完成后，旧节点释放 route ownership。
5. `Ops Plane` 校验无活动 route 后，节点进入 `drained`。

### 8.2 设计约束

- 排空不是简单断开连接，而是显式的迁移和接管流程。
- 迁移过程中必须依赖 epoch + fencing 防止双写。

## 9. 时序相关的不变量清单

- 单会话单主写。
- 单设备单权威 route owner。
- durable truth 总是早于 projection truth。
- 通知、审计、自动化都不能反向决定提交结果。
- stream 的恢复点必须独立于普通 message 的 read cursor。

## 10. 链路追踪字段建议

为了后续观测和问题定位，建议所有主链统一携带：

- `tenant_id`
- `trace_id`
- `request_id`
- `command_id`
- `event_id`
- `conversation_id`
- `message_id / stream_id / rtc_session_id`
- `principal_id`
- `device_id`
- `route_epoch`

## 11. 结论

只有把连接、消息、流、设备、迁移等核心时序全部明确下来，`craw-chat` 的六个 plane 才不只是概念分层，而是真正可指导重构和实现的工程架构。
