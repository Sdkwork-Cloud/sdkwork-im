# 152CJ Loop55补充 - 2026-04-11

- `control-plane-api` 当前已经把 shared-channel sync 的 failure boundary 明确物化为 durable pending backlog
  - backlog 记录在 `SocialControlState.pending_shared_channel_sync_requests`
  - ready pair 存在但 trigger 未配置时，请求会被 durable 保留，而不是静默丢弃
  - trigger dispatch 失败时，失败项与其后的未投递项都会进入 backlog，social durable truth 继续保持 committed
- control-plane 当前新增独立 operator surface：`POST /api/v1/control/social/runtime/repair-shared-channel-sync`
  - operator 可显式重放 backlog
  - 成功项会被清空
  - 失败项保留 `failureCount / lastError`，便于后续继续 repair
- `repair-derived-snapshot` 与 `repair-social-runtime-dir` 当前 replay social commit journal 时会保留 backlog
  - 这意味着 pending shared-channel sync 不再随着 snapshot rebuild 被误清空
  - backlog 当前被建模为 snapshot-backed operator state，而不是 social domain journal 事件
- `S07` 的 shared-channel auto-sync truth 当前已有三层边界
  - ready pair 自动发现
  - real consumer 装配：`local-minimal` 同进程 consumer + standalone public HTTP consumer
  - failure durability / operator repair：pending backlog + manual replay route
- 当前仍不能把 shared-channel sync 解释为 release-ready cross-service delivery semantics
  - 仍没有自动 retry scheduler
  - 仍没有 dead-letter / poison message 隔离
  - 仍没有 remote republish owner / lease / SLA
  - 仍没有 exactly-once 级别的 cross-service delivery guarantee
- As-Built 结论现在更新为：
  - 已闭合到 `shared-channel sync durable pending backlog + operator repair seam`
  - 仍待闭合到 `automatic retry / dead-letter / remote republish / delivery ownership semantics`
