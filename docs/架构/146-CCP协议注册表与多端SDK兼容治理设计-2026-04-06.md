# CCP 协议注册表与多端 SDK 兼容治理设计

## 1. 文档定位

本文档用于在 `143-145` 已经冻结的协议骨架之上，继续冻结：

- `CCP` 的协议注册表如何组织
- 多端客户端与 SDK 如何消费同一套协议标准
- 哪些能力可以灰度、哪些能力必须稳定
- 协议升级、废弃与兼容矩阵如何治理

如果没有这份文档，`CCP` 虽然定义了分层、握手和 envelope，但后续仍然容易出现：

- 每个客户端各做各的 envelope 兼容
- Web、App、桌面端、IoT 的支持能力漂移
- schema 命名和版本号无人管理
- SDK 无法判断什么能力是 stable，什么只是实验能力

因此，本文件的目标是把 `CCP` 从“设计得对”进一步推进到“可以长期治理”。

## 2. 治理目标

### 2.1 核心目标

- 让协议扩展有统一注册中心，而不是散落在不同模块文档中
- 让 SDK 和客户端知道自己应该支持哪些 binding、codec 和 capability
- 让协议升级变成可观测、可回滚、可灰度的过程
- 让产品线后续支持 Web、移动端、桌面端、服务端、IoT 时不出现分叉协议

### 2.2 非目标

- 不做重量级在线中心化配置系统
- 不在第一阶段建设独立数据库式 schema registry 服务
- 不要求所有能力一次性跨全平台支持

## 3. CCP 注册表模型

`CCP` 需要冻结 5 类注册表。

### 3.1 Protocol Registry

用于登记：

- 协议版本：`ccp/1.0`
- 可用 binding：`ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`、`ccp/mqtt/1`
- 可用 codec：`json`、`cbor`
- 协议级错误码
- 控制帧集合

这是最顶层的协议“总目录”。

### 3.2 Schema Registry

用于登记：

- schema 名称
- schema 主版本
- 所属 domain
- 对应 `kind / type`
- 发布阶段
- 兼容策略
- 推荐消费端

例如：

| schema | kind | type | stage | consumers |
| --- | --- | --- | --- | --- |
| `cc.control.hello.v1` | `control` | `hello` | `stable` | 全部 |
| `cc.message.send.v1` | `cmd` | `message.send` | `stable` | Web / App / Desktop / SDK |
| `cc.stream.delta.v1` | `stream` | `stream.delta` | `stable` | Web / App / Desktop / Agent |
| `cc.device.telemetry.v1` | `evt` | `device.telemetry` | `beta` | IoT / Edge / Backend |

### 3.3 Capability Registry

用于登记 feature/capability，而不是让客户端猜。

推荐登记：

- capability 名称
- 最低协议版本
- 最低 binding 要求
- 是否依赖特定 codec
- 是否为 stable
- 是否可灰度关闭

例如：

- `session.resume`
- `stream.patch`
- `rtc.signal`
- `agent.tool_call`
- `device.signature`
- `payload.cbor`

### 3.4 Compatibility Matrix Registry

用于登记：

- 客户端类型
- 支持的 binding
- 支持的 codec
- 支持的 capability 集
- 最低可兼容服务端版本
- 禁止使用的实验能力

### 3.5 Close / Error Registry

用于冻结：

- WebSocket close code / close reason
- MQTT 错误映射
- HTTP 状态与 CCP `error` kind 的映射
- SDK 恢复策略建议

这部分对商业 SDK 很关键，因为客户端需要稳定做分支判断，而不是依赖服务端日志猜语义。

## 4. Schema 生命周期

每个 schema 都必须带生命周期状态。

推荐状态为：

- `experimental`
- `beta`
- `stable`
- `deprecated`
- `removed`

### 4.1 `experimental`

- 仅灰度环境或指定 capability 下可用
- 不承诺跨版本稳定
- 默认不进入通用 SDK 稳定 facade

### 4.2 `beta`

- 结构基本稳定
- 可进入预览 SDK
- 仍允许在主版本不变前做有限调整

### 4.3 `stable`

- 可以进入公开 SDK 默认支持集合
- 必须遵守兼容承诺
- 删除或破坏必须走正式废弃流程

### 4.4 `deprecated`

- 不再推荐新增使用
- 服务端仍保留兼容期支持
- 需要在注册表中给出替代 schema

### 4.5 `removed`

- 从默认协商集中移除
- 仅在特别兼容层或旧环境开关下保留
- 文档需要明确最后支持版本

## 5. 多端客户端分型

协议治理必须承认客户端不是一类，而是多类。

推荐冻结以下客户端分型：

- `browser-web`
- `mobile-app`
- `desktop-app`
- `server-sdk`
- `iot-device`
- `edge-gateway`
- `admin-console`

## 6. 多端兼容矩阵

### 6.1 Browser Web

推荐支持：

- binding：`ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`
- codec：`json`
- 必需 capability：`session.resume`、`stream.delta`
- 可选 capability：`stream.patch`、`rtc.signal`

不要求：

- `cbor`
- `device.signature`

### 6.2 Mobile App

推荐支持：

- binding：`ccp/http/1`、`ccp/ws/1`
- codec：`json`
- 必需 capability：`session.resume`、`stream.delta`、`route.fencing`
- 可选 capability：`rtc.signal`

### 6.3 Desktop App

推荐支持：

- binding：`ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`
- codec：`json`
- 必需 capability：`session.resume`、`stream.delta`、`route.fencing`
- 可选 capability：`stream.patch`、`rtc.signal`、`agent.tool_call`

### 6.4 Server SDK

推荐支持：

- binding：`ccp/http/1`、按需 `ccp/sse/1`
- codec：`json`
- 必需 capability：`idempotent-write`、`trace-propagation`

服务端 SDK 的主要职责不是维持长连接 UI，而是提供：

- 稳定命令 API
- 稳定事件订阅能力
- 清晰错误映射

### 6.5 IoT Device

推荐支持：

- binding：`ccp/mqtt/1`，按需 `ccp/http/1`
- codec：`json` 或 `cbor`
- 必需 capability：`device.signature` 或 `device.credential`
- 可选 capability：`session.resume`

### 6.6 Edge Gateway

推荐支持：

- binding：`ccp/mqtt/1`、`ccp/http/1`
- codec：`json` 或 `cbor`
- 必需 capability：`device.signature`、`route.fencing`

## 7. SDK 产品化分层

为了避免 SDK 继续“手写 HTTP + 手写 WS + 各端自定义错误处理”，推荐统一 SDK 分层。

### 7.1 L1 Generated Contracts

职责：

- 生成 schema 对应 DTO
- 生成枚举、错误族、类型声明
- 不负责连接管理

### 7.2 L2 Transport Adapter

职责：

- `http`、`ws`、`sse`、`mqtt` 的具体适配
- codec 读写
- 认证头或票据绑定

### 7.3 L3 Session Runtime

职责：

- 握手
- `auth_bind`
- `session_resume`
- 心跳
- 重连
- `goaway` 处理

### 7.4 L4 App Facade

职责：

- 面向业务的消息发送、流式对话、会话查询、RTC 调用 API
- 隐藏 envelope 细节
- 对上暴露稳定开发者接口

结论是：

- DTO 不应直接等同于最终 SDK
- 连接运行时不应散落在每个 App 自己写
- facade 层必须建立在统一协议运行时之上

## 8. SDK 语言覆盖策略

第一阶段应从协议契约与 facade 一体化出发，面向以下语言规划：

- TypeScript / JavaScript
- Kotlin
- Swift
- Flutter / Dart
- Java
- Go
- Python
- C#
- Rust

原则如下：

- 所有语言共享同一份 schema registry 和 capability registry
- 不允许某语言 SDK 走私有 ad-hoc 协议
- 不允许在 SDK 内重新发明 envelope

## 9. 兼容与升级策略

### 9.1 服务端升级原则

- 先扩 capability，再开放默认支持
- 先灰度新 schema，再进入 stable
- 不允许上线即删除旧 schema

### 9.2 客户端升级原则

- 新客户端默认容忍未知可选字段
- 客户端仅在 capability 协商成功后启用新能力
- 未协商成功必须安全降级

### 9.3 降级路径

示例：

- `stream.patch` 不可用时，降级为 `stream.delta`
- `cbor` 不可用时，退回 `json`
- `ws` 不可用时，可局部退回 `http + sse`

## 10. 错误与关闭语义治理

推荐稳定以下内容：

- `CCP_*` 协议错误族
- WebSocket close code registry
- MQTT error mapping registry
- SDK 的默认重连策略建议

例如：

- `4001 / session.disconnect`：客户端应视为显式断开，不自动恢复旧连接
- `4002 / route.epoch.stale`：客户端应重新握手并尝试恢复
- `4003 / auth.expired`：客户端应刷新认证后重连

## 11. 文档与 CI 治理

为了避免注册表只存在文字、不进入工程约束，建议后续在 CI 中加入：

- schema name 唯一性检查
- schema 主版本单调递增检查
- capability matrix 完整性检查
- stable SDK facade 不引用 experimental schema 检查
- close code / error code 重复检查

## 12. 结论

`CCP` 的下一层治理已经明确：

- 用 Protocol Registry、Schema Registry、Capability Registry、Compatibility Matrix Registry、Close/Error Registry 统一治理
- 用生命周期状态约束 schema，而不是靠文档备注猜稳定性
- 用统一 SDK 分层避免多端重复发明 transport/runtime/facade
- 用兼容矩阵约束 Web、App、Desktop、Server、IoT、Edge 的能力边界

这样 `CCP` 才不只是“协议设计”，而是能够支撑长期商业化和多端产品线演进的协议平台。

继续向下的治理落点见：

- `148`：控制面中的 protocol profile、release channel、rollout policy 与 kill switch
- `149`：多 cell / 多 region 的协议升级、灾备兼容与回滚路径

- 第 3-10 节中的五类 registry、schema 生命周期、客户端分型、SDK 产品化分层、升级/降级治理，属于标准目标模型。

### C. 文档口径规则

- 写当前 `SDK / registry` 能力时，优先引用本文 `As-Built`、`148` 的治理现状，以及 `sdks/`、`tools/chat-cli/`、`services/control-plane-api/` 的真实资产。
- 写未来治理标准时，必须显式标记为 `目标态`、`治理目标` 或 `标准面`。
- 任何“已形成完整 SDK 治理闭环”的表述都必须先有 fresh code / test evidence，否则只能写成阶段性落地。
