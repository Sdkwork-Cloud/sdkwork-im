# 模块规划与边界

## 1. 文档目标

本文档用于将 `130-149` 的总体架构具体化为 Rust workspace 的模块规划，重点解决：

- 如何按 `sdkwork-api-router` 风格做 crate 分包
- 如何避免继续堆积巨型 service crate
- 哪些模块属于稳定契约，哪些模块属于运行时实现
- 如何为 AI、IoT、流式能力和连接层扩容预留清晰边界

## 2. Workspace 总体规划

建议工作区采用如下目录形态：

```text
craw-chat/
  bin/
  crates/
  docs/
  services/
  deployments/
  tools/
  tmp/
```

职责约定如下：

- `crates/`：可复用的内核、契约、领域、应用、适配器和运行时模块
- `services/`：最终可执行服务与 profile 组合
- `bin/`：跨操作系统安装、初始化、启动、停止、重启脚本
- `deployments/`：容器化、systemd、k8s 或私有化部署资产
- `tools/`：检查、测试、数据准备、压测和运维工具

## 4. 服务规划

### 4.1 第一阶段推荐服务

- `services/local-minimal-node`
- `services/gateway-service`
- `services/control-plane-service`

职责说明：

- `local-minimal-node`：单进程闭环验证与快速安装部署
- `gateway-service`：公共入口，组合 link、route、message、stream 的基础能力
- `control-plane-service`：租户、策略、审计、运维入口

### 4.2 第二阶段扩展服务

- `services/projection-worker`
- `services/agent-runtime-service`
- `services/iot-edge-service`

职责说明：

- `projection-worker`：异步投影与重建
- `agent-runtime-service`：Agent、tool call、流式输出编排
- `iot-edge-service`：MQTT、设备 uplink、设备控制与边缘接入

## 5. 关键边界定义

### 5.1 领域层边界

- `domain-*` 不允许依赖 `axum`、`tokio::net`、数据库驱动或 Redis 客户端
- `domain-*` 只表达语义，不负责 HTTP、SQL 或缓存格式

### 5.2 应用层边界

- `app-*` 负责编排，不承载 Web 协议细节
- `app-*` 通过端口访问存储，不直连具体适配器实现

### 5.3 接口层边界

- `interface-*` 只负责接入、鉴权、校验和协议转换
- `interface-*` 不直接实现复杂业务规则
- `interface-*` 必须复用 `ccp-binding-*`，不允许重复定义握手、envelope 或 close contract

### 5.4 运行时层边界

- `runtime-*` 负责连接、路由、恢复、队列和后台调度
- `runtime-*` 不擅自定义领域语义

### 5.5 服务层边界

- `services/*` 只做 profile 组合、依赖装配和进程入口
- 不允许继续把大部分业务堆在 `services/*/src/lib.rs`

## 6. 高风险模块整改方向

当前最需要拆分的高风险模块包括：

- `services/local-minimal-node/src/lib.rs`
- `services/conversation-runtime/src/lib.rs`
- `services/session-gateway/src/lib.rs`

整改后的标准目录形态建议为：

```text
src/
  lib.rs
  router.rs
  state.rs
  error.rs
  websocket/
  runtime/
  service/
  projection/
  recovery/
```

## 7. 依赖规则

- `contract-*` 可被所有上层依赖，但不得依赖具体 adapter。
- `domain-*` 只依赖 `kernel`、`error`、`contract-core` 等基础模块。
- `app-*` 可以依赖 `domain-*` 与 `storage-core` 端口。
- `interface-*` 可以依赖 `app-*` 与 `contract-*`。
- `runtime-*` 可以依赖 `app-*`、`storage-core` 与 `contract-*`，但不直接表达领域规则。
- `services/*` 可以依赖多个 crate 进行装配，但不应新增业务知识。

## 8. 扩展预留

为了支持后续商业化扩展，模块层面必须预留以下能力位：

- `topic / thread / card / workflow / knowledge` 作为协作扩展
- `agent handoff / tool call / structured stream` 作为 AI 扩展
- `device session / device twin / telemetry / command stream` 作为 IoT 扩展
- `search / embedding / archive / export` 作为查询与知识扩展

## 9. 结论

模块规划的目标不是把服务切得越多越好，而是把边界切清楚。`craw-chat` 后续重构必须以 `crate 先行、边界先行、服务装配后置` 为原则推进，否则连接层、消息层、AI 层和 IoT 层会再次缠绕在一起。
