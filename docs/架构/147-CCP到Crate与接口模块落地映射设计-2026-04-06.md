# CCP 到 Crate 与接口模块落地映射设计

## 1. 文档定位

本文档用于把 `143-146` 中冻结的协议设计，进一步收敛到 Rust workspace 的 crate 结构和接口模块边界上，解决以下问题：

- `CCP` 的不同层到底落到哪些 crate
- `interface-*`、`runtime-*`、`contract-*`、`app-*` 如何配合
- 如何避免把协议实现又堆回 `gateway-service` 或超大 `lib.rs`
- 多 transport binding 应如何共享一套协议骨架

如果没有这份文档，架构层虽然说清楚了 `CCP`，但代码层很容易退化为：

- `interface-ws` 自己写一套握手
- `interface-http` 自己写一套 envelope
- `runtime-link` 再复制一套 session 状态
- `gateway-service` 成为新的协议大杂烩

本文件的目标就是防止这种退化。

## 2. 总体设计结论

推荐采用：

`基础层 + CCP 协议基础设施层 + 业务契约层 + 应用层 + 接口层 + 运行时层 + 服务装配层`

其中最关键的新增结论是：

- `CCP` 应单独形成协议基础设施族，而不是仅作为 `contract-core` 中的一组类型
- transport binding 必须拆到独立 crate，而不是直接揉进 `interface-*`
- `interface-*` 负责接入与转换，不负责协议核心语义
- `runtime-*` 负责运行态协作，不负责定义稳定协议类型

## 3. 推荐的 CCP crate 族

### 3.1 协议基础核心

- `craw-chat-ccp-core`
- `craw-chat-ccp-control`
- `craw-chat-ccp-registry`

职责如下：

#### `craw-chat-ccp-core`

- envelope
- `kind / type`
- `scope / route / trace`
- 协议错误模型
- capability 基础类型

#### `craw-chat-ccp-control`

- `hello`
- `hello_ack`
- `auth_bind`
- `auth_ok`
- `session_resume`
- `session_resumed`
- `heartbeat`
- `goaway`
- `error`

#### `craw-chat-ccp-registry`

- protocol registry
- schema registry
- capability registry
- compatibility matrix
- close / error registry

### 3.2 编码层

- `craw-chat-ccp-codec`
- `craw-chat-ccp-codec-json`
- `craw-chat-ccp-codec-cbor`

职责如下：

#### `craw-chat-ccp-codec`

- codec trait
- envelope encode/decode 统一入口
- codec registry

#### `craw-chat-ccp-codec-json`

- JSON 具体实现
- 浏览器 / App 默认 codec

#### `craw-chat-ccp-codec-cbor`

- CBOR 具体实现
- IoT / 边缘设备优先 codec

### 3.3 绑定层

- `craw-chat-ccp-binding-http`
- `craw-chat-ccp-binding-ws`
- `craw-chat-ccp-binding-sse`
- `craw-chat-ccp-binding-mqtt`

职责如下：

- transport 与 envelope 的绑定
- transport 特有 metadata 归一化
- 控制帧与业务帧的读写顺序约束
- binding 能力限制声明

注意：

- 绑定层只负责“如何承载”
- 不负责“业务要做什么”

## 4. 业务契约层的重新细化

现有 `contract-*` 仍然保留，但要与 `ccp-*` 明确分工。

### 4.1 `ccp-*` 与 `contract-*` 的边界

`ccp-*` 负责：

- 协议骨架
- envelope
- 控制帧
- registry
- codec
- binding

`contract-*` 负责：

- 各领域 payload schema
- API 请求响应 DTO
- message / stream / rtc / agent / iot / admin 的稳定业务对象

### 4.2 推荐业务契约拆分

在现有基础上，建议把协议相关契约继续细化为：

- `craw-chat-contract-core`
- `craw-chat-contract-control`
- `craw-chat-contract-message`
- `craw-chat-contract-stream`
- `craw-chat-contract-rtc`
- `craw-chat-contract-agent`
- `craw-chat-contract-iot`
- `craw-chat-contract-notification`
- `craw-chat-contract-admin`

其中：

- `contract-control` 用于桥接 `ccp-control` 与业务层通用控制对象
- `contract-notification` 单独拆出，避免继续挤在 message 或 admin 中

## 5. 目标态 Interface 层与 CCP 的关系

`interface-*` 不再承担协议定义职责，而承担 4 件事：

- 建立 transport 连接
- 提取认证上下文
- 调用对应 `ccp-binding-*`
- 把解析后的命令转交给 `app-*`

### 5.1 `craw-chat-interface-http-public`

依赖：

- `craw-chat-ccp-binding-http`
- `craw-chat-ccp-codec-json`
- `craw-chat-app-*`

职责：

- HTTP 请求接入
- `Authorization` 解析
- request -> CCP envelope -> app command 转换

### 5.2 `craw-chat-interface-ws`

依赖：

- `craw-chat-ccp-binding-ws`
- `craw-chat-ccp-control`
- `craw-chat-runtime-link`
- `craw-chat-app-*`

职责：

- 建立 WebSocket 会话
- 完成 `hello / auth_bind / session_resume`
- 进入 active 后转发业务帧

### 5.3 `craw-chat-interface-sse`

依赖：

- `craw-chat-ccp-binding-sse`
- `craw-chat-ccp-codec-json`

职责：

- 下行事件流输出
- AI 流式结果推送
- 只读链路能力限制

### 5.4 `craw-chat-interface-mqtt`

依赖：

- `craw-chat-ccp-binding-mqtt`
- `craw-chat-ccp-codec-cbor`
- `craw-chat-runtime-iot`

职责：

- 设备接入
- 设备认证与签名校验
- 上下行命令和 telemetry 转换

## 6. 目标态 Runtime 层与 CCP 的关系

`runtime-*` 不是协议定义层，而是协议运行层。

### 6.1 `craw-chat-runtime-link`

依赖：

- `ccp-core`
- `ccp-control`
- `ccp-codec`
- `ccp-registry`

职责：

- 会话状态机
- 连接分片
- 出站队列
- 心跳超时
- `goaway` 驱动
- 恢复窗口治理

### 6.2 `craw-chat-runtime-route`

职责：

- route ownership
- route epoch
- fencing
- rebalance / drain / takeover

### 6.3 `craw-chat-runtime-gateway`

职责：

- ingress / egress 编排
- 命令确认
- 下行 fanout
- 协议错误映射

### 6.4 `craw-chat-runtime-agent`

职责：

- Agent 流式输出 materialization
- tool call 生命周期桥接
- stream 与 message 的协同投影

### 6.5 `craw-chat-runtime-iot`

职责：

- device session
- telemetry uplink
- command downlink
- device twin / edge sync

## 7. App 层如何消费协议

`app-*` 不直接面向 transport，而面向规范化后的命令和上下文。

推荐规则：

- `interface-*` 负责把原始 CCP 帧转换成 `app-*` 可消费的 command/query
- `app-*` 输出领域结果后，再由 `runtime-gateway` 或 `interface-*` 统一包装为 CCP `evt/ack/error`

这样做的好处是：

- `app-*` 不依赖 HTTP / WS / MQTT
- 同一用例可以被不同 binding 复用
- 协议变化与业务变化分层演进

## 8. 目标态 Service 装配层原则

`services/*` 只做装配，不定义协议逻辑。

### 8.1 `gateway-service`

- `interface-http-public`

- `interface-sse`
- `runtime-link`
- `runtime-route`
- `runtime-gateway`
- 相关 `app-*`

### 8.2 `iot-edge-service`

负责装配：

- `interface-mqtt`
- `runtime-iot`
- `runtime-route`
- 相关 `app-iot`

### 8.3 `local-minimal-node`

负责装配：

- 单进程可验证闭环
- 最小协议骨架
- 本地 memory/disk 适配器

## 10. 依赖方向硬约束

必须满足：

- `ccp-core` 不依赖 `axum`、`tokio-tungstenite`、MQTT 客户端或数据库驱动
- `ccp-binding-*` 可以依赖 transport 库，但不依赖 `app-*`
- `interface-*` 可以依赖 `ccp-binding-*` 和 `app-*`
- `runtime-*` 可以依赖 `ccp-*` 与 `storage-*`
- `contract-*` 不依赖 `binding-*`
- `domain-*` 不依赖 `ccp-binding-*`
- `services/*` 可以装配多个 crate，但不得承载具体协议实现细节

## 11. 测试与 CI

建议针对协议层新增以下测试层次：

### 11.1 `ccp-core` 单元测试

- envelope 序列化
- kind/type 合法性
- route / scope 不变量

### 11.2 `ccp-control` 协议测试

- `hello -> hello_ack`
- `auth_bind -> auth_ok`
- `session_resume -> session_resumed`
- 非法状态机跳转拒绝

### 11.3 `ccp-binding-*` 绑定测试

- HTTP 内容类型校验
- WebSocket close contract
- SSE 流输出稳定性
- MQTT 上下行 topic 映射

### 11.4 `interface-*` 适配测试

- 权威字段不可被客户端覆盖
- binding 到 app command 的转换正确
- capability 未协商时拒绝

## 12. 从当前代码到目标结构的迁移顺序

推荐迁移顺序如下：

1. 先建立 `ccp-core / ccp-control / ccp-codec` 基础骨架
2. 再建立 `ccp-binding-http / ws / sse / mqtt`
3. 再把 `interface-*` 改成调用 binding，而不是自行定义协议
4. 最后拆解 `runtime-link / runtime-gateway` 与 `services/*`

这样可以避免“一边改 transport，一边改业务，一边改服务装配”导致的大范围混乱。

## 13. 结论

`CCP` 的代码落地方向已经明确：

- `CCP` 单独形成协议基础设施 crate 族
- transport binding 单独成层
- `interface-*` 做接入与转换，不做协议定义
- `runtime-*` 做运行时协作，不做稳定契约定义
- `services/*` 只做装配，不再堆积协议实现

这份映射文档是后续真正进行 crate 重构时的直接依据，能够把“先进架构”落实为不会继续退化的工程结构。
