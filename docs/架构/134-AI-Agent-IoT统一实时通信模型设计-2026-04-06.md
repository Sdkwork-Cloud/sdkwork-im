# AI / Agent / IoT 统一实时通信模型设计

## 1. 文档目标

本文档用于定义 `craw-chat` 在 AI、Agent、IoT 三类新场景下的统一通信模型，确保系统不是“传统 IM + 若干外挂功能”，而是从协议和领域模型层面支持：

- 人与人
- 人与 Agent
- 人与设备
- Agent 与设备
- Agent 与 Agent

之间的统一实时通信。

## 2. 基本判断

AI 与 IoT 场景之所以容易把 IM 系统做乱，根本原因是：

- 把 Agent 当成普通用户补丁
- 把设备当成外部旁路系统
- 把流式输出当成文本消息补丁

`craw-chat` 必须避免这种设计。

系统应统一采用：

- 一等主体模型
- 一等流模型
- 一等扩展事件模型

## 3. 一等主体模型

系统统一主体：

- `user`
- `agent`
- `device`
- `bot`
- `system`

统一 actor / sender 结构必须至少具备：

- `id`
- `kind`
- `member_id`
- `device_id`
- `session_id`
- `metadata`

这样才能支持：

- 普通用户消息
- Agent 输出
- 设备遥测
- 系统广播
- Bot / workflow 执行结果

## 4. Message 与 Stream 的统一模型

### 4.1 Message

适合：

- 离散消息
- 卡片
- 文件引用
- 结构化通知
- 最终物化结果

### 4.2 Stream

适合：

- token streaming
- delta patch
- telemetry frame
- 长任务进度
- 音频转写增量
- 模型推理过程输出

### 4.3 统一原则

- Message 与 Stream 并列
- Stream 结束后可物化为 Message
- 同一会话中两者共存

## 5. AI / Agent 模型

### 5.1 Agent 作为会话主体

Agent 应可直接：

- 加入 conversation
- 接收历史上下文
- 输出普通消息
- 输出流式消息
- 发起工具调用
- 产出结构化结果

### 5.2 Agent 事件类型

建议标准化：

- `agent.response.started`
- `agent.response.delta`
- `agent.response.completed`
- `agent.tool_call.requested`
- `agent.tool_call.completed`
- `agent.handoff.started`
- `agent.handoff.accepted`
- `agent.handoff.closed`

### 5.3 Agent 结果物化

Agent 输出可物化为：

- 普通消息
- 卡片
- 文件
- 知识条目引用
- 工作流状态更新

## 6. IoT / 智能硬件模型

### 6.1 设备作为一等主体

设备必须具备：

- `device_session`
- `device_presence`
- `device_twin`
- `device_command_stream`
- `device_uplink_stream`

### 6.2 典型交互

#### 用户 -> 设备

- 控制消息
- 配置变更
- 音视频呼叫
- command stream

#### 设备 -> 用户

- telemetry stream
- 告警通知
- 结构化状态消息
- 媒体上报

#### Agent -> 设备

- 指令编排
- 任务执行
- 自动化反馈

#### 设备 -> Agent

- 状态上报
- 感知结果
- 异常事件

### 6.3 接入协议

建议支持：

- WebSocket
- MQTT
- HTTP / SSE
- 预留 QUIC / WebRTC DataChannel

## 7. 协作上下文与知识沉淀

系统必须支持：

- topic
- thread
- document reference
- task reference
- workflow reference
- AI summary
- context anchor

原则：

- 聊天不是终点
- 会话上下文是协作容器
- AI、任务、文档、设备都挂在会话上下文上

## 8. 流式事件标准建议

建议统一 frame：

- `message.publish`
- `message.edit`
- `message.recall`
- `stream.open`
- `stream.delta`
- `stream.patch`
- `stream.checkpoint`
- `stream.finalize`
- `stream.abort`
- `presence.update`
- `rtc.signal`
- `device.telemetry`
- `agent.tool_call`
- `agent.tool_result`

这套标准必须同时覆盖：

- 普通聊天
- AI 对话
- 设备上行
- RTC 信令
- 长任务流

## 9. 安全边界

### 9.1 Agent

Agent 默认不应自动拥有所有上下文权限。

必须明确：

- 可见上下文范围
- 可调用工具范围
- 可访问资源范围
- 可向哪些 conversation 输出

### 9.2 Device

设备必须明确：

- device identity
- owner binding
- tenant binding
- session binding
- command authorization

### 9.3 流式输出

流式输出必须支持：

- 中止
- 限流
- resume
- checkpoint
- 数据类型约束

## 10. 对存储与连接层的影响

### 10.1 对连接层的影响

连接层必须原生支持：

- 大量低带宽 IoT 设备
- 高频短帧 AI delta
- 多设备并发会话

### 10.2 对存储层的影响

存储层必须支持：

- stream checkpoint
- materialization
- large blob reference
- telemetry retention policy
- agent artifact retention

## 11. 推荐的实施优先级

### 第一阶段

- Message / Stream 统一模型冻结
- actor / sender 统一模型冻结
- conversation / member / message / stream 基线落地

### 第二阶段

- agent 作为一等主体落地
- tool call / tool result 标准落地
- AI 流式对话基线落地

### 第三阶段

- device 作为一等主体落地
- MQTT 与 device stream 落地
- 设备控制与 telemetry 统一接入

## 12. 结论

`craw-chat` 的 AI 与 IoT 设计必须从第一天统一在一个实时通信模型中，而不是分别做两套外挂系统。只有这样，系统才能真正成为 AI 时代的专业通信底座。
