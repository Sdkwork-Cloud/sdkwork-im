# 架构标准与总体设计

## 1. 总体架构

系统采用四层架构：

- `Domain Core`
- `Platform Contracts`
- `Runtime Services`
- `Deployment Profiles`

系统采用三平面部署：

- `Control Plane`
- `Data Plane`
- `Operations Plane`

## 2. 总体拓扑

```text
Client
  -> Edge Gateway
  -> Session Gateway
  -> Routing & Ownership
  -> Conversation Runtime / Streaming Core / RTC Signaling
  -> Commit Journal
  -> Projection Pipeline
  -> Query Views

Side Effects
  -> Notification
  -> Bot / Webhook
  -> Workflow
  -> Audit
  -> Media Processing

Control Plane
  -> Tenant / Auth / Policy / Quota / Route / Capability

Ops Plane
  -> Observability / Upgrade / Backup / Diagnostics / Rebalance
```

## 3. 核心标准

### 3.1 消息标准

- 同一 `conversation_id` 内严格有序。
- 不追求跨会话全局顺序。
- 命令与事件分离。
- durable truth 先提交，再投影。

### 3.2 流标准

- 流是原生能力。
- 流支持 `open / data / patch / checkpoint / complete / abort / error` 生命周期。
- 流既可瞬时，也可持久。
- 流可被消息引用，但不强制每帧都落为消息。

### 3.3 文件资源标准

- 文件资源统一用 `MediaResource` 语义建模。
- `MediaResource` 仅表示消息体中的文件相关资源结构。
- 大文件不进入消息主存，统一外置对象存储。

### 3.4 RTC 标准

- RTC 只统一信令层。
- 媒体传输独立部署。
- RTC 录制、转写等产物通过消息与资源体系回流。

### 3.5 多租户标准

- 所有持久化对象必须显式携带 `tenant_id`。
- 所有请求、事件、日志、审计必须 tenant-aware。
- 隔离至少覆盖：身份、配额、调度、数据、故障。

### 3.6 可插拔标准

- 领域语义固定。
- 基础设施实现可替换。
- 所有外部依赖通过标准端口接入。
- 不允许第三方 SDK 直接进入领域核心。

## 5. Cell 化部署原则

- 不做无限大的全局集群。
- 每个 Cell 作为独立故障域与扩展单元。
- Cell 内部完成路由、提交、投影、实时分发。
- SaaS 通过多个 Cell 形成区域池。
- 私有化默认一个或少量 Cell。

## 6. 分布式所有权原则

- 逻辑采用 `Virtual Shards`。
- 每个 shard 通过 `lease + epoch + fencing` 保证唯一 owner。
- 新节点加入采用 `bounded rebalance`。
- 下线节点采用 `graceful drain`。

## 7. Durable Truth 与 Query Truth

### 7.1 Durable Truth

以下对象的状态变化必须先写入 `CommitJournal`：

- 会话创建与成员变更
- 消息发送、编辑、撤回
- 已读推进
- 流开启、checkpoint、完成、终止
- RTC 呼叫关键生命周期
- 文件资源关键绑定事件

### 7.2 Query Truth

以下视图由投影构建：

- Timeline
- Conversation Summary
- Inbox
- Read Cursor View
- Device Sync Feed
- Stream Summary
- Notification View
- Audit View

## 8. 设计红线

- 不允许缓存成为消息真相源。
- 不允许副作用阻塞消息提交主路径。
- 不允许控制面事务进入热路径。
- 不允许同一会话跨 Region 多主写。
- 不允许把 RTC 媒体面塞入 IM 内核。
- 不允许把 AI 场景写死进基础协议。
