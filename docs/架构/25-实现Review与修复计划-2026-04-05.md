## 2. 已确认并修复的问题

### 2.1 高危：非成员可对会话执行写操作

问题表现：

- 非成员此前可以调用会话写路径，向任意已知 `conversationId` 发消息
- 非成员此前可以对任意已知 `messageId` 执行编辑和撤回

根因：

- `conversation-runtime` 在 `post_message / edit_message / recall_message` 中只尝试回填 `sender.memberId`
- 运行时没有强制要求当前操作者必须是会话有效成员

修复：

- 新增 `RuntimeError::PermissionDenied`
- 在 `conversation-runtime` 内统一校验 active member
- `add_member / remove_member / read_cursor_view / post_message / edit_message / recall_message` 全部接入成员校验

### 2.2 高危：非成员可读取私有会话视图

问题表现：

- 同租户但非会话成员此前可以读取：
  - 会话成员列表
  - 时间线
  - 会话摘要

根因：

- `local-minimal-node` 的 `list_members / get_timeline / get_conversation_summary / get_read_cursor`
- `projection-service` 的 `get_timeline / get_conversation_summary / get_read_cursor`
- 都缺少当前认证主体是否为会话 active member 的判断

修复：

- `conversation-runtime` 新增 `require_active_member(...)`
- `local-minimal-node` 新增 `ensure_conversation_member(...)`
- `projection-service` 新增 `is_active_member(...)` 和统一的访问校验

### 2.3 高危：通知查询未按 recipient 隔离

问题表现：

- 同租户下任意主体此前都可以查询和读取其他人的通知

根因：

- `notification-service` 的 `list_notifications(...)` 只按 `tenant` 过滤
- `get_notification(...)` 只按 `tenant + notificationId` 命中
- 未校验 `recipient_id == auth.actor_id`

修复：

- `list_notifications(...)` 改为 tenant + recipient 双重过滤
- `get_notification(...)` 改为只允许 recipient 自身读取
- 非 recipient 读取统一返回 not found，避免扩大枚举面

## 19. Review Cycle 6 实际完成结果

### 19.1 重要：`left -> rejoin` 未冻结导致 membership identity 与 read cursor 漂移

问题表现：

- 同一 principal 在 `left` 后重新通过 `add_member` 加入时，运行时此前复用了同一个 `memberId`
- `conversation-runtime` 会按这个 `memberId` 直接覆盖 read cursor
- `projection-service` 对同一个 `memberId` 只做 `or_insert`，不会重置 cursor
- 结果是 runtime 与 projection 对重入成员的已读状态可能产生漂移

根因：

- `memberId` 此前只由 `conversationId + principalId` 决定，没有 membership episode 概念
- `add_member(...)` 只阻止“仍然 active 的同 principal”，不会为历史 episode 生成新身份
- `remove_member(...)` 也没有防御 stale `memberId`，在保留历史成员记录后会存在误删当前 active 映射的风险

本轮标准与修复：

- 新增 `docs/架构/58-left-rejoin-membership-episode标准-2026-04-06.md`
- 冻结规则：
  - `left` 或 `removed` 之后再次通过 `add_member` 加入，必须创建新的 membership episode
  - 新 episode 必须生成新的 `memberId`
  - 历史 episode 保留原始状态与 cursor，不覆盖
  - `GET/POST /read-cursor` 只面向当前 active episode
  - stale `memberId` 不得再操作当前 active member
- 代码落地：
  - `services/conversation-runtime/src/lib.rs`
    - 新增 `next_member_episode(...)`
    - 新增 `member_episode_id(...)`
    - 新增 `ensure_current_active_member_target(...)`
    - `add_member(...)` 改为按 episode 生成新 `memberId`
    - `remove_member(...)` 改为拒绝 stale `memberId`
- 新增验证：
  - `services/conversation-runtime/tests/conversation_flow_test.rs`
    - `test_left_member_rejoin_creates_new_membership_episode`
    - `test_stale_member_id_cannot_remove_rejoined_active_member`
  - `services/local-minimal-node/tests/access_control_e2e_test.rs`
    - `test_left_member_rejoin_gets_new_member_identity_and_fresh_cursor`

- 同时新引入的 membership episode 标准也要求任何新的治理命令都必须防御 stale `memberId`。

根因：

- 代码层此前没有正式的 role mutation 命令、接口、事件和审计动作。
- `owner transfer` 只解决 owner 交接，不适合作为通用角色治理通道。
- projection 读模型此前也没有成员 role 变更事件的应用逻辑。

  - 仅当前 active `owner` 可发起

  - 目标角色只允许变更为 `admin / member / guest`
