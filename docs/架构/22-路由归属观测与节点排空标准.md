# 22-路由归属观测与节点排空标准

## 1. 目标

在完成跨节点 realtime 定向路由之后，运维面必须能回答两个问题：

- 当前哪些设备 route 归属到哪个节点
- 某个节点进入 drain / rebalance 时，当前处于什么生命周期阶段

本标准定义 route ownership 的可观测模型，以及节点排空状态在 `ops/cluster` 与 `ops/diagnostics` 中的最小暴露方式。

## 2. 设计原则

### 2.1 cluster 视图只放摘要

`GET /api/v1/ops/cluster` 是集群摘要视图，不应塞入大体量设备明细。  
它至少应返回：

- `drainStatus`
- `rebalanceState`
- `deviceRouteCount`

### 2.2 diagnostics 视图放明细

`GET /api/v1/ops/diagnostics` 是节点级诊断快照，可以返回该节点当前持有的 route ownership 明细，用于：

- 故障排查
- drain 前后对比
- route 粘连、错绑、未迁移排查

### 2.3 最小实现允许最终一致

当前本地最小实现中的 route ownership view 可以是最终一致的，但必须满足：

- 新绑定设备会尽快反映到 owner node 的 diagnostics
- cluster 摘要中的 `deviceRouteCount` 与 diagnostics 中的 route 数量一致
- 不得把 route ownership 当 durable 真相存储

## 3. 数据模型

### 3.1 RouteOwnershipView

```json
{
  "tenantId": "t_demo",
  "principalId": "u_demo",
  "deviceId": "d_pad",
  "ownerNodeId": "node_b",
  "connectionKind": "websocket",
  "boundAt": "2026-04-05T11:20:00Z"
}
```

### 3.2 ClusterNodeView 增强字段

```json
{
  "nodeId": "node_b",
  "profile": "local-minimal",
  "bindAddr": "127.0.0.1:18102",
  "drainStatus": "active",
  "rebalanceState": "stable",
  "deviceRouteCount": 1,
  "ownedScopes": ["conversation:*"],
  "services": [
    { "service": "conversation-runtime", "status": "ok" }
  ]
}
```

字段语义：

- `drainStatus`
  - `active`: 正常接收新连接与 route
  - `draining`: 已进入排空，不应再接收新的稳定 route
  - `drained`: 已完成排空，等待摘除或升级
- `rebalanceState`
  - `stable`
  - `moving_routes`
  - `awaiting_resume`
- `deviceRouteCount`: 当前节点持有的设备 route 数量

### 3.3 DiagnosticBundle 增强字段

诊断包中新增：

- `deviceRoutes`

用于返回当前节点 route ownership 明细。

## 4. API 标准

### 4.1 `GET /api/v1/ops/cluster`

返回集群摘要，每个节点至少包含：

- `drainStatus`
- `rebalanceState`
- `deviceRouteCount`

### 4.2 `GET /api/v1/ops/diagnostics`

返回节点级诊断包，至少包含：

- `drainStatus`
- `rebalanceState`
- `deviceRoutes`

## 5. 与节点排空的关系

节点排空的控制语义后续可以由 control plane API 承担，但在当前标准中，运维面必须至少能看见状态。

最小阶段约束：

1. 节点进入 `draining` 后，运维面必须能看见状态切换
2. route 迁移期间，`rebalanceState` 应能反映非 `stable`
3. 所有 route 迁移完成后，`deviceRouteCount` 应降为 0，然后节点才可视为 `drained`

## 6. 本地最小实现边界

当前阶段最小实现采用：

- `ops/cluster` 返回 route count 摘要
- `ops/diagnostics` 返回 route ownership 明细
- 节点生命周期状态先由 runtime 内部状态承载
- 尚未开放公网 drain 控制 API

这一步的目的不是完成完整控制面，而是先把运营可观测模型定下来，后续无论 route store 或 drain 编排如何替换，都不需要重做运维协议。
