# 152CJ Loop58 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“dead-letter 只是 poison backlog 隔离带”推进到“operator 可显式把 dead-letter request 回灌到 pending，并复用既有 repair route 完成恢复”的最小恢复边界

## 2. As-Built 事实
- `control-plane-api` 当前新增 `POST /api/v1/control/social/runtime/requeue-dead-letter-shared-channel-sync`
- `SocialControlState` 当前新增 `requeue_dead_letter_shared_channel_sync_requests()` 状态迁移
  - `dead_letter_shared_channel_sync_requests -> pending_shared_channel_sync_requests`
  - 迁移时保留 request payload、`failureCount` 与 `lastError`
- `requeue-dead-letter-shared-channel-sync` 响应当前显式暴露：
  - `pendingBefore`
  - `deadLetterBefore`
  - `requeued`
  - `pendingAfter`
  - `deadLetterAfter`
- route 当前写入 `control.social_runtime_shared_channel_sync_dead_letter_requeued` audit 事件
- 真正的 shared-channel sync 投递链路没有新增第二套实现
  - requeue 后仍由现有 `repair_shared_channel_sync(...)` 消费 pending backlog
  - 因此恢复语义仍完全落在 `control-plane-api` 内部，未把组合根耦合回 `conversation-runtime` crate 边界

## 3. 明确边界
- 本轮没有：
  - dead-letter 单条/筛选 requeue contract
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once delivery semantics
- 当前 requeue 保留 `failureCount / lastError`
  - 好处是保持 poison backlog 历史事实，不伪造“全新 request”
  - 代价是若环境依旧不健康，request 在下一次 repair 失败后会较快重新回到 dead-letter
- 因此 Loop58 证明的是“operator recovery boundary 已存在”，而不是“delivery ownership semantics 已闭环”
