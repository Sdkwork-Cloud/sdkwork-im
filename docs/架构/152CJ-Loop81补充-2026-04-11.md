# 152CJ Loop81 Addendum - 2026-04-11

## 1. As-Built Update
- Dead-letter promotion now treats ownership as pending-pool-only metadata.
- When a pending shared-channel sync request crosses the dead-letter failure threshold, owner metadata is cleared before the item enters the dead-letter bucket.
- This means both dead-letter transition directions are now ownership-neutral:
  - pending -> dead-letter clears owner metadata
  - dead-letter -> pending requeue clears owner metadata

## 2. Architectural Meaning
- Before Loop81, a claimed request that failed targeted republish enough times could become dead-lettered while still looking actively owned.
- After Loop81, dead-letter acts purely as a failure bucket and no longer preserves pending claim semantics.
- Active ownership remains authoritative only inside the pending pool, where claim, release, takeover, stale reclaim, and repair logic already operate.

## 3. Deferred Architecture Work
- automatic stale detection / timeout reclaim / scheduler
- release-ready exactly-once delivery semantics for shared-channel sync
- richer stale-age / SLA policy beyond the current manual and path-local reclaim seams
