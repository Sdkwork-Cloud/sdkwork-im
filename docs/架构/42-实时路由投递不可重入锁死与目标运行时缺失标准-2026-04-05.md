# 42-实时路由投递不可重入锁死与目标运行时缺失标准-2026-04-05

## 1. 背景

`publish_device_event(originNodeId, tenantId, principalId, deviceId, ...)` 在跨节点投递时需要先解析设备 route，再根据 owner node 查找 runtime。

如果 route 已存在但 target runtime 缺失，错误实现可能出现两类问题：

- 对同一个 runtime registry 锁做嵌套获取，导致线程锁死
- 把“target runtime 缺失”误当成“无 route”，偷偷 fallback 到 origin node 投递，造成错误节点收件

这两类问题都会直接破坏实时消息的正确性和可观测性。

## 2. 标准

### 2.1 route 已解析时不得回退到 origin node

当 `device route ownership` 已存在时：

- 必须优先以 route 的 `ownerNodeId` 为唯一投递目标
- 如果 owner node runtime 缺失，不得自动改投 origin node

### 2.2 target runtime 缺失必须显式暴露状态

当 route 存在但 `ownerNodeId` 对应 runtime 不存在时：

- `targetNodeId` 仍然返回 route 指向的 owner node
- `routeState` 必须标记为 `target_runtime_missing`
- `delivered = 0`

这保证诊断面能看到“目标是谁”和“为什么没有投递成功”。

### 2.3 只有无 route 才允许 `local_fallback`

`origin node fallback` 仅适用于：

- 当前设备不存在 route

此时：

- 若 origin runtime 存在，`routeState = local_fallback`
- 若 origin runtime 本身也不存在，`routeState = origin_runtime_missing`

### 2.4 runtime registry 读取不得产生可重入锁死

读取 runtime registry 时必须满足：

- 单次逻辑路径内不得在持锁期间再次尝试获取同一把锁
- 需要多分支读取时，应在一次持锁范围内完成查找，或显式先释放锁再进入下一步

## 3. 落地要求

- `session-gateway` 必须有回归测试覆盖 “route 指向的 target runtime 缺失” 场景
- 回归测试必须验证：
  - 不发生死锁
  - 不向 origin node 误投递
  - `routeState = target_runtime_missing`
  - `delivered = 0`
