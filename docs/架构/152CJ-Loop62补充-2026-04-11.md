# 152CJ Loop62 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“pending inventory + targeted republish 已存在，但缺少 owner contract”推进到“operator 先 claim pending request，再按 owner 边界 targeted republish”的最小 delivery ownership 语义

## 2. As-Built 事实
- `control-plane-api` 当前新增 `POST /api/v1/control/social/runtime/claim-pending-shared-channel-sync-targeted`
- pending/dead-letter inventory item 当前显式输出：
  - `requestKey`
  - `request`
  - `failureCount`
  - `lastError`
  - `ownerActorId`
  - `ownerActorKind`
- `PendingSharedChannelSyncRequest` 当前保留可选 owner 元数据，并随 runtime-dir snapshot 持久化
- `SocialControlRuntime::claim_pending_shared_channel_sync_targeted(...)` 当前会：
  - 只 claim 被选中的 pending request
  - 使用当前 control operator 的 `actorId / actorKind` 写入 owner
  - 对已被其他 operator 占有的 request 维持 conflict 计数而不改写 owner
- `SocialControlRuntime::republish_pending_shared_channel_sync_targeted(...)` 当前会：
  - 在 dispatch 前校验 selected pending item 是否归当前 operator 所有
  - 对 foreign-owned 或 unclaimed request 返回 `shared_channel_sync_owner_conflict`
  - 仅在 ownership 校验通过后才进入真实 remote runtime dispatch
  - 成功时清理对应 pending item；失败时继续沿用现有 failure budget / dead-letter 规则
- 现有 `repair-shared-channel-sync` 全量 operator route 仍保留，因此当前 operator surface 处于“全量 repair + claim-targeted republish”并存状态

## 3. 明确边界
- 本轮没有：
  - `claimedAt / leaseExpiresAt / SLA` 元数据
  - targeted unclaim / release / force-takeover 路由
  - ownership-aware `repair-shared-channel-sync`
  - automatic retry scheduler
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 单体内的 operator ownership boundary，而不是 release-ready cross-service lease / SLA orchestration
