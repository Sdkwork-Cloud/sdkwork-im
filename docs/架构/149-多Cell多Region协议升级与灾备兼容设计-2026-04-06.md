# 多 Cell 多 Region 协议升级与灾备兼容设计

## 1. 文档定位

本文档用于把 `CCP` 的协议升级、灰度发布、跨 cell / region 兼容和灾备切换策略冻结下来。

前面的文档已经回答了：

- 协议是什么
- 如何握手与协商
- 如何做版本兼容
- 如何做注册表与 SDK 治理
- 如何把协议落到 crate 和控制面

本文件补齐最后一块关键能力：

- 当系统已经进入多 cell、多 region、多租户和私有化交付阶段时，协议如何安全升级、切流、降级与回滚

## 2. 设计目标

### 2.1 核心目标

- 让协议升级不破坏正在运行的长连接和 SDK
- 让跨 cell / region 的切流与灾备保持协议兼容
- 让 SaaS、独占 cell 和私有化都拥有明确升级路径
- 让协议升级具备“先扩兼容、再切默认、最后回收旧版”的稳定节奏

### 2.2 非目标

- 不追求一次升级所有 region 同步切换
- 不在第一阶段建设复杂分布式配置一致性系统
- 不让灾备切换绑定单一云厂商能力

## 3. 升级总原则

### 3.1 协议升级三阶段

所有协议升级都必须遵循三阶段：

1. `Expand`
2. `Shift`
3. `Contract`

#### Expand

- 服务端先支持新旧能力并存
- registry 和 compatibility matrix 先补齐
- SDK 先具备向后兼容能力

#### Shift

- 默认协商结果逐步切向新 capability / 新 schema
- 通过 rollout policy 扩大流量
- 持续观测失败率和降级率

#### Contract

- 停止新流量使用旧 schema
- 标记旧 schema 为 deprecated
- 在长兼容窗口后移除

### 3.2 长连接优先原则

即时通讯的主风险不只是 HTTP 请求失败，而是：

- 现有长连接上的握手与恢复断裂
- 重连时协商结果不一致
- 不同 cell 的 close contract 不一致

因此协议升级必须优先保护：

- `hello / auth_bind / session_resume`
- `routeEpoch`
- close / error contract
- capability negotiation

## 4. Cell / Region 分层发布模型

### 4.1 Cell 是最小协议发布域

推荐把 `cell` 视为协议发布、回滚和故障隔离的最小单元。

原因：

- cell 已经是部署域、扩容域和故障域
- 协议升级出问题时可以在 cell 级快速止损
- 不需要一开始就 region 级同步切换

### 4.2 Region 是次级汇聚域

同一 region 内可包含多个 cell。

region 主要负责：

- 汇总 rollout 状态
- 控制跨 cell 流量导向
- 作为灾备切换的组织层

### 4.3 发布顺序建议

建议固定顺序如下：

1. `local / test`
2. `private pre-release cell`
3. `shared canary cell`
4. `shared beta cells`
5. `shared stable cells`
6. `dedicated cells`
7. `private lts bundle`

## 5. 兼容矩阵在升级中的作用

升级不能只看服务端版本，还必须看客户端矩阵。

每次协议变更都必须回答：

- 哪些客户端类型支持
- 哪些 SDK 版本支持
- 哪些租户需要保留旧版
- 哪些 region 尚未准备好切换

### 5.1 必须覆盖的矩阵维度

- `protocol version`
- `binding`
- `codec`
- `capability set`
- `schema stage`
- `client type`
- `sdk version`
- `deployment profile`

## 6. 典型升级场景

### 6.1 新 capability 上线

例如上线 `stream.patch`。

标准流程：

1. 服务端先支持 `stream.patch`，但默认关闭
2. registry 标记为 `beta`
3. canary cell 对部分 tenant 打开
4. SDK 在协商成功后启用
5. 观测稳定后提升为 `stable`

### 6.2 新 schema 主版本上线

例如 `cc.message.send.v2`。

标准流程：

1. 新旧 schema 并存
2. 服务端优先同时消费 `v1 / v2`
3. SDK 逐步切换到默认发 `v2`
4. 服务端统计 `v1` 使用率
5. 低于阈值后标记 `v1 deprecated`

### 6.3 新 codec 上线

例如在设备侧启用 `cbor`。

标准流程：

1. 先通过 capability 协商开放 `payload.cbor`
2. 只对 `iot-device / edge-gateway` 打开
3. 浏览器与旧 SDK 继续使用 `json`
4. 不允许因为 `cbor` 上线而影响 `json` 兼容链路

## 7. 灾备切换中的协议要求

灾备时最容易被忽略的问题是：新 region 不仅要能运行，还要能讲同一种协议。

### 7.1 灾备 region 必须满足

- 支持相同的 `protocol profile`
- 支持相同的 close / error contract
- 支持兼容窗口内的旧 schema
- 支持相同的 `session resume` 语义

### 7.2 不允许的做法

- 主 region 已升级新 close code，灾备 region 仍保留旧含义
- 主 region 默认启用新 capability，灾备 region 不认识
- 主 region 允许旧 SDK，灾备 region 拒绝握手

## 8. 回滚模型

协议升级必须有明确回滚等级。

### 8.1 Capability 回滚

最轻量：

- 关闭单个 capability
- 不需要改变协议主版本
- 用 kill switch 即可

### 8.2 Profile 回滚

中等等级：

- 回退一个 protocol profile
- 适用于某一 cell / tenant / region

### 8.3 Bundle 回滚

更高等级：

- 私有化或 region 级回退整套 registry bundle
- 适用于控制面配置或兼容矩阵整体误发

### 8.4 服务端二进制回滚

最高等级：

- 回退实际部署二进制
- 只在 Expand 阶段兼容已准备好的前提下允许快速回退

## 9. 长连接升级策略

### 9.1 不强制踢全量连接

发布新协议能力时，不应默认踢掉全量连接。

优先策略：

- 新连接使用新协商结果
- 老连接在自然重连时升级
- 仅在高风险场景使用 `goaway` 触发迁移

### 9.2 需要强制迁移的场景

- close contract 发生必须统一的安全修正
- route fencing 语义必须整体切换
- 存在高危协议漏洞

### 9.3 迁移方式

- 通过 `goaway` 给出明确原因
- 客户端重新 `hello`
- 若支持则执行 `session_resume`

## 10. 私有化升级策略

私有化不应跟随 SaaS 每一轮协议实验。

推荐模式：

- SaaS 走 `canary -> beta -> stable`
- 私有化走 `lts -> private bundle`

交付要求：

- 提供兼容矩阵说明
- 提供回滚 bundle
- 提供升级前检查脚本
- 提供升级后协议观测基线

## 11. 观测与门禁

协议升级必须配合门禁指标。

### 11.1 发布门禁

至少检查：

- 握手成功率
- `auth_bind` 成功率
- `session_resume` 成功率
- close code 异常占比
- capability 协商失败率
- schema decode 失败率

### 11.2 回滚触发建议

建议触发回滚的情况：

- canary cell 握手失败率明显上升
- `session_resume` 失败率超阈值
- 某 SDK 版本异常集中报错
- 某租户出现大量 close / error 异常

## 12. 与部署、控制面、灾备文档的关系

本文档与以下文档配套：

- `137`：定义 cell / region / deployment profile
- `138`：定义高可用与恢复原则
- `142`：定义控制面与 feature flag / runtime control config
- `148`：定义 protocol registry、release channel 和 rollout policy

## 13. 结论

`CCP` 的协议升级必须被设计成：

- 以 cell 为最小发布域
- 以 region 为汇聚与灾备域
- 以 compatibility matrix 为判断依据
- 以 expand / shift / contract 为固定升级节奏
- 以 kill switch / profile rollback / bundle rollback 为回滚手段

只有这样，`craw-chat` 在进入多集群、多租户和私有化交付阶段后，协议层才不会成为最大的不确定性来源。
