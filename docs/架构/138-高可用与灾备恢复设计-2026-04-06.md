# 高可用与灾备恢复设计

## 1. 文档目标

本文档用于定义 `craw-chat` 在高可用、故障隔离、恢复和灾备方面的架构原则，确保系统不仅能运行，还能在异常情况下：

- 快速止损
- 正确恢复
- 保持权威数据不被破坏
- 支撑 SaaS 与私有化交付

## 2. 高可用设计总原则

- 连接可中断，但真相不能错乱。
- route 可以重建，但 fencing 不能失效。
- projection 可以滞后，但 durable truth 不能缺失。
- 控制面可以降级，但消息热链不能依赖控制面强事务。
- 恢复必须以权威数据源为准，而不是以内存态为准。

## 3. 故障域模型

系统至少区分以下故障域：

- 进程级故障
- 节点级故障
- plane 级故障
- 存储级故障
- cell 级故障
- region 级故障

设计目标是让故障尽可能停留在小故障域内，不向整个系统扩散。

## 4. 各平面的高可用要求

### 4.1 Link Plane

故障表现：

- 连接中断
- 心跳异常
- 出站队列丢失

恢复策略：

- 客户端主动重连
- route epoch 重新协商
- `sync feed + stream checkpoint` 补偿恢复

### 4.2 Route Plane

故障表现：

- route lookup 超时
- owner 不明确
- drain 过程卡住

恢复策略：

- 以权威 route store 重建
- 使用 `epoch + fencing` 防止陈旧 owner 继续工作
- 支持显式 takeover

### 4.3 Messaging / Stream Plane

故障表现：

- durable commit 延迟升高
- write path 不稳定
- AI / stream backlog 堆积

恢复策略：

- 优先保护 durable write
- 非关键副链降级
- 按需暂停部分低优先级 stream delta 下发

### 4.4 Projection Plane

故障表现：

- timeline / inbox / summary 滞后
- 未读数更新延迟

恢复策略：

- 投影可滞后但必须可重建
- lag 必须可观测
- 支持 replay 和 targeted rebuild

## 5. 典型故障场景

## 5.1 连接节点故障

处理原则：

1. 连接断开
2. 客户端重连到新 link node
3. route plane 判断当前 owner
4. 旧 route 通过 fencing 失效
5. 客户端补拉 `sync feed` 和 `stream checkpoint`

## 5.2 Redis 短暂不可用

处理原则：

- route、presence、resume hint 读写会受影响
- 允许退化到更慢的权威存储和恢复路径
- 不允许因为 Redis 不可用就宣告消息 durable truth 丢失

## 5.3 PostgreSQL 压力或故障

处理原则：

- 优先保护消息与流的提交真相
- 暂停部分非关键投影和后台任务
- 结合备份、PITR、replay 恢复

## 5.4 Projection Worker 故障

处理原则：

- 不影响消息提交结果
- 通过 lag 和 backlog 指标发现问题
- 恢复后按事件重放重建视图

## 5.5 对象存储异常

处理原则：

- 不影响纯文本消息链路
- 媒体上传、附件物化和导出可降级
- 引用一致性必须后续补偿校验

## 6. RTO / RPO 设计建议

不同 profile 的目标可以不同，但原则建议如下：

- `Local Minimal`：以开发闭环为目标，不承诺严格 RTO/RPO
- `Private Standard`：要求支持标准备份恢复和 projection rebuild
- `SaaS Shared / Dedicated`：要求具备节点级快速恢复、存储级恢复和租户级恢复能力

更重要的是恢复顺序必须被文档化和脚本化，而不是只写一个抽象指标。

## 7. 恢复流程标准

推荐恢复顺序：

1. 恢复配置与控制面最小能力
2. 恢复 metadata
3. 恢复 `MessageLog / Stream checkpoint`
4. 重建 route 热状态
5. 重放 projection
6. 对齐对象存储引用和审计状态
7. 恢复对外服务

## 8. 备份与重建能力

必须具备：

- metadata snapshot
- message log replay
- stream checkpoint restore
- object storage backup
- tenant-scoped recovery
- projection rebuild

## 9. 演练与观测

建议固定纳入运维演练：

- link node 故障演练
- route takeover 演练
- projection rebuild 演练
- object storage 引用一致性校验演练
- tenant 级恢复演练

必须观察的关键指标：

- reconnect success rate
- resume success rate
- route migration success rate
- projection lag
- replay duration
- drain completion time

## 10. 结论

`craw-chat` 的高可用不是“所有组件都双机热备”这么简单，而是要把连接恢复、route fencing、durable truth、projection rebuild 和 tenant 级恢复这五件事设计成完整闭环。
