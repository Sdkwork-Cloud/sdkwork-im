## 2. 实施总原则

- 先冻结文档，再驱动实现
- 先清边界，再做性能
- 先连接与消息主链路，再扩 AI / IoT
- 先抽象，再替换底层
- 先做到可安装、可恢复、可观测，再追求极限复杂能力
- 所有新增代码必须符合代码治理标准

## 3. 阶段划分

### 阶段 0：文档冻结与架构校准

目标：

- 冻结 `130-149` 总纲与补充专项文档
- 统一 `02-09` 的基础说明口径
- 正式冻结 `CCP` 的协议分层、传输绑定、握手、版本与安全边界
- 正式冻结 `registry / compatibility matrix / ccp-* crate family` 的治理口径
- 明确新旧文档优先级和实施顺序

交付物：

- `docs/架构/README.md`
- `docs/架构/130-149`
- 已对齐的 `docs/架构/02-09`

验收标准：

- 无互相冲突的架构口径
- 实现团队可以直接按文档推进

### 阶段 1：工程骨架重构

目标：

- 按 `kernel / contract / domain / app / interface / storage / runtime / service` 重构 workspace

重点动作：

- 新建基础 crate
- 将超大 service crate 拆分为多模块目录
- 明确 `services/*` 只做启动与装配
- 落地 `lib.rs` 和单文件长度约束

验收标准：

- 新 workspace 拓扑可编译
- 高风险 `lib.rs` 不再承载大块业务逻辑

### 阶段 2：协议与存储抽象冻结

目标：

- 冻结统一的：
  - `CCP protocol / envelope / schema / capability negotiation`
  - `sender / actor`
  - `conversation / member`
  - `message / stream`
  - `MetadataStore / RouteStore / MessageLog / StreamStore / BlobStore`

重点动作：

- 契约 crate 落地
- `ccp-*` 协议基础设施 crate 落地
- 存储 core 端口落地
- 消息和流的 envelope 统一
- 冻结 `ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`、`ccp/mqtt/1`
- 冻结 `hello / auth_bind / session_resume / heartbeat / goaway / error`

验收标准：

- API、事件、持久化模型不再漂移
- 协议版本与 schema 版本的演进规则不再漂移
- 存储层具备可替换性

### 阶段 3：连接层与路由层重构

目标：

- 建立真正的 `Link Plane` 与 `Route Plane`
- 建立 `resume / reconnect / fencing / drain` 统一模型

重点动作：

- 从 `session-gateway` 抽出连接热路径
- 建立连接分片、出站队列、背压控制
- 建立 route ownership 与节点排空模型

验收标准：

- 节点排空与 route 迁移语义稳定
- 连接层可独立扩容

### 阶段 4：消息主链路迁移

目标：

- 完整迁移：
  - conversation
  - member
  - message
  - read-cursor
  - presence
  - device sync

重点动作：

- 建立 `domain-conversation / domain-message / app-conversation`
- 对齐外部 API 与内部命令模型
- 统一 sender、member、read-cursor 和多端补偿逻辑

验收标准：

- 主链路可运行
- 编辑、撤回、已读、多端同步语义稳定

### 阶段 5：流与 RTC 基线

目标：

- 建立标准流能力与 RTC 信令能力

重点动作：

- `stream.open / delta / checkpoint / finalize / abort` 落地
- `rtc.invite / accept / reject / end` 落地
- 流与消息、RTC 与消息的桥接规则落地

验收标准：

- 普通流、AI 流、RTC 信令回流至少一条主路径可验证

### 阶段 6：AI / Agent 基线

目标：

- Agent 作为一等主体进入系统
- 支持 tool call / tool result
- 支持流式 AI 输出并物化

重点动作：

- 冻结 agent 事件模型
- 建立 AI stream materialization 规则
- 建立 handoff 基线

验收标准：

- 人与 Agent 对话可跑通
- Agent 输出可写回 timeline 和通知系统

### 阶段 7：IoT / 智能硬件接入

目标：

- Device 成为一等主体
- MQTT / device stream / command stream 基线落地

重点动作：

- 引入 device session / device presence / device twin
- 统一用户、Agent、设备的实时通信模型

验收标准：

- 至少一种智能硬件接入路径跑通

### 阶段 8：性能与高可用打磨

目标：

- 在不破坏上层协议和产品能力的前提下，下探连接密度、吞吐和恢复能力

重点动作：

- 压测连接数、消息吞吐、流式吞吐
- 优化 shard、队列、route、projection 和缓存策略
- 建立 SLO、故障演练和恢复脚本

验收标准：

- 单机连接能力和恢复能力有明确量化结果
- 节点排空、重平衡、恢复链路经验证

### 阶段 9：自研热路径演进

目标：

- 在确有规模和收益时，演进自研 `MessageLog / StreamStore`

重点动作：

- segment log
- stream segment storage
- compaction / archive / replay 优化

验收标准：

- 热路径可替换
- 上层 API 和协议不受影响

## 4. 当前绝对优先级

当前优先级必须严格如下：

1. 文档口径统一
2. crate 拆分与代码治理落地
3. 协议和存储抽象冻结
4. 连接与路由重构
5. 消息主链路迁移
6. 流与 RTC
7. AI / Agent
8. IoT
9. 性能与高可用

## 5. 当前工程整改重点

优先处理的高风险文件：

- `services/local-minimal-node/src/lib.rs`
- `services/conversation-runtime/src/lib.rs`
- `services/session-gateway/src/lib.rs`

整改要求：

- 目录化拆分
- 单文件不超过 `1000` 行
- 推荐控制在 `200-500` 行
- 路由、领域、运行时、存储必须分离

## 6. 验收门禁

### 6.1 架构门禁

- 文档与代码边界一致
- 不新增新的超大 `lib.rs`
- 不引入与基线相冲突的重型默认依赖

### 6.2 功能门禁

- 消息、流、已读、多端恢复路径可验证
- `tenant`、`sender`、`actor` 权威字段来源一致
- AI、IoT 扩展不破坏普通 IM 体验

### 6.3 工程门禁

- `cargo fmt --check`
- `cargo clippy --all-targets --all-features -D warnings`
- `cargo test --workspace`
- 文件长度与模块边界检查

## 7. 技术路线结论

当前最优实施路线不是：

- 一开始就重度微服务化
- 一开始就引入大量重中间件
- 一开始就自建通用分布式数据库

而是：

`连接优先 + 协议优先 + 抽象优先 + 组件最小依赖 + 热路径后续自研演进`

## 8. 结论

本实施计划的目标不是简单“补功能”，而是把 `craw-chat` 从当前工程态重构为一套真正具备连接优先、分层扩容、AI 可扩展、IoT 可接入和商业化可控能力的下一代实时通信底座。

## 2026-04-09 增补：社交 / 空间 / 会话 DDD 主线

### A. 当前状态

- 当前仓库主线已覆盖连接、消息、流、RTC、provider 治理与 step/evidence 体系，但好友、空间、群组、频道 durable truth 仍未独立落库。
- `conversation-runtime` 仍是消息写入与时序 owner；`control-plane-api` 是最适合承接社交与治理写入口的现有服务。
- 因此当前最优顺序不是继续泛化 `conversation`，而是先补齐社交与组织真相，再把 `conversation` 收敛为运行时容器。

### B. 推荐执行顺序

1. 在 `im-domain-core` 与 `control-plane-api` 落地 `user / actor / friend_request / friendship / user_block / direct_chat`。
2. 落地 `space / space_member / space_role / chat_group / group_member / group_role / chat_channel / channel_access_rule / channel_subscription`。
3. 让 `conversation-runtime` 通过 `business_type + business_id` 绑定 `direct_chat / chat_channel / thread`，不再自持关系真相。
4. 补齐 `invitation / membership_request / ban_record / mute_setting / audit_event / outbox_event` 治理闭环。
5. 最后接入 `thread / external_connection / external_member_link / reaction / pin / retention / history_visibility` 高阶能力。

### C. 门禁

- 未形成 `friendship / space / group / channel` durable truth 前，不判定“群组与好友能力已完成”。
- `conversation_member` 只能表达 runtime membership sync，不得替代组织权限真相。
- 每轮架构迭代都必须同步回写 `150CJ / 151CJ / 152CJ / 02 / 03 / 05 / 09`。
- 文档若同时描述现状与目标态，必须显式分层，禁止再用统一话术掩盖差异。

### D. 非目标

- 现阶段不为了命名洁癖大规模重命名已有 crate。
- 现阶段不把现有服务强拆成过细微服务。
- 现阶段不在 durable truth 未稳定前追求复杂查询视图、全量搜索闭环或超前存储自研。
