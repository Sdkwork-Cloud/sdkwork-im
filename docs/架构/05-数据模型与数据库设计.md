# 数据模型与数据库设计

## 1. 数据层总原则

- durable truth 与 query truth 分离
- 控制面元数据与消息时间线分离
- 文件资源与消息正文分离
- 可重建投影与不可替代真相分离

## 2. 核心聚合根

- `Tenant`
- `UserPrincipal`
- `Conversation`
- `Message`
- `StreamSession`
- `RtcSession`
- `MediaAsset`
- `NotificationTask`
- `AutomationExecution`

## 3. Durable Truth 模型

### 3.1 Conversation

核心字段：

- `tenant_id`
- `conversation_id`
- `conversation_type`
- `routing_key`
- `virtual_shard_id`
- `owner_epoch`
- `message_seq`
- `member_epoch`
- `status`
- `settings_json`

推荐 `conversation_type` 取值：
- `direct`：单聊
- `group`：群聊
- `agent_dialog`：用户与 agent 主会话
- `agent_handoff`：agent 与人工或其他 agent 交接会话
- `system_channel`：系统通知或平台广播通道

补充标准（2026-04-06）：

- 会话创建时，首个成员记录的 `principal_kind` 必须来自认证上下文中的 `actor_kind`
- 不能把所有创建者统一固化为 `user`
- `conversation_type` 不允许写入白名单之外的任意字符串
- 虽然数据模型保留 `agent_dialog / agent_handoff / system_channel`，但当前 generic create 入口只开放 `group / direct`
- `agent_dialog` 已经通过 dedicated create command 开放写路径：`POST /api/v1/conversations/agent-dialogs`
- `agent_handoff / system_channel` 仍需等待 dedicated create command 冻结后再开放写路径

## 4. 消息内容结构

### 4.1 Sender

消息发送者统一建模为 `sender` 对象，而不是散落的 `senderId` / `senderKind` 字段：

```json
{
  "id": "u_xxx",
  "kind": "user",
  "memberId": "cm_xxx",
  "deviceId": "d_xxx",
  "sessionId": "s_xxx",
  "metadata": {
    "role": "owner"
  }
}
```

说明：
- 单聊通常使用 `kind=user`，`memberId` 可为空。
- 群聊建议带上 `memberId`，用于关联成员身份、角色与禁言快照。
- agent 对话建议使用 `kind=agent`，并在 `metadata` 中预留 `agentId`、`handoffSessionId`、`toolCallId` 等扩展字段。
- system、webhook、workflow 等平台账号也统一复用该结构。

### 4.2 MessageBody

```json
{
  "summary": "optional summary",
  "parts": [],
  "render_hints": {}
}
```

### 4.3 ContentPart

支持以下 part：

- `text`
- `data`
- `media`
- `signal`
- `stream_ref`

### 4.4 MediaPart

`MediaPart` 内只引用资源，不直接承载大文件内容：

```json
{
  "kind": "media",
  "media_asset_id": "ma_xxx",
  "resource": {
    "uuid": "res_xxx",
    "url": "https://...",
    "type": "image",
    "mimeType": "image/png",
    "size": 1024,
    "name": "demo.png",
    "extension": "png",
    "tags": {},
    "metadata": {},
    "prompt": "optional"
  }
}
```

### 4.5 DataPart

用于用户自定义结构化消息：

```json
{
  "kind": "data",
  "schema_ref": "custom.biz.data.v1",
  "encoding": "json",
  "payload": {}
}
```

### 4.6 SignalPart

用于 RTC 或其他控制信令：

```json
{
  "kind": "signal",
  "signal_type": "rtc.offer",
  "schema_ref": "rtc.signal.v1",
  "payload": {}
}
```

### 4.7 StreamRefPart

```json
{
  "kind": "stream_ref",
  "stream_id": "st_xxx",
  "stream_type": "custom.delta.text",
  "state": "completed"
}
```

## 6. 查询投影建议

### 6.1 Timeline View

主键建议：

- partition: `tenant_id + conversation_id + bucket`
- clustering: `message_seq`

### 6.2 Inbox View

主键建议：

- partition: `tenant_id + user_id`
- clustering: `last_activity_at desc`

核心字段建议：

- `tenant_id`
- `principal_id`
- `member_id`
- `conversation_id`
- `conversation_type`
- `message_count`
- `last_message_id`
- `last_message_seq`
- `last_sender_id`
- `last_sender_kind`
- `last_summary`
- `unread_count`
- `last_activity_at`

说明：

- `inbox view` 是查询投影，不是 durable truth。
- `unread_count` 建议由 `conversation summary.last_message_seq - read cursor.read_seq` 推导。
- `last_activity_at` 在已有消息时取最后一条消息时间；无消息时可退化为成员加入时间或会话创建时间。
- `conversation_type` 来自 `conversation.created` 事件投影，供客户端快速区分群聊、单聊、agent 对话等类型。

### 6.3 Read Cursor View

主键建议：

- partition: `tenant_id + conversation_id`
- clustering: `member_id`

核心字段建议：

- `tenant_id`
- `conversation_id`
- `member_id`
- `principal_id`
- `read_seq`
- `last_read_message_id`
- `updated_at`
- `unread_count`

说明：

- `read_seq` 表示该成员已经确认读取到的最大 `message_seq`，只能单调递增。
- `last_read_message_id` 作为客户端和审计友好的辅助字段，允许为空。
- `unread_count` 是查询视图字段，不是 durable truth；通常由 `conversation summary.last_message_seq - read_seq` 计算或投影生成。
- 已读游标默认绑定到 `ConversationMembership.member_id`，保证用户、agent、bot 在会话内均能统一推进已读状态。

### 6.4 Device Sync Feed

主键建议：

- partition: `tenant_id + user_id + device_id`
- clustering: `sync_seq`

核心字段建议：

- `tenant_id`
- `principal_id`
- `device_id`
- `sync_seq`
- `origin_event_id`
- `origin_event_type`
- `conversation_id`
- `message_id`
- `message_seq`
- `member_id`
- `read_seq`
- `last_read_message_id`
- `actor_id`
- `actor_device_id`
- `summary`
- `occurred_at`

与已读关系：

- `read cursor` 负责成员级高水位。
- `device sync feed` 负责把成员级已读推进同步到各终端设备。
- `device sync feed` 还负责把会话消息、系统信令、RTC 信令等需要多端补偿的事件按设备维度扇出。
- `sync_seq` 仅在 `tenant_id + principal_id + device_id` 分区内单调递增，用于断线续传和补拉。
- `origin_event_type` 一期至少覆盖 `message.posted`、`conversation.read_cursor_updated`，后续可扩展 `notification.dispatched`、`presence.event`、`session.resume`。
- `actor_device_id` 用于标记事件来源设备，消费端可据此做本机去重，但服务端不依赖客户端去重保证正确性。
- 最小实现阶段允许通过设备注册表把事件投递到“已知设备集合”；未注册设备不承诺接收历史补偿流。

### 6.5 Session Resume 与 Presence Snapshot

`session resume` 和 `presence snapshot` 属于接入层与查询层协同能力，不直接承担 durable truth。

- `session resume` 的输入真相来自：
  - 认证上下文中的 `tenant_id / actor_id / session_id / device_id`
  - 设备补偿流当前最新 `sync_seq`
  - 客户端上送的最近已消费 `last_seen_sync_seq`
- `presence snapshot` 的输入真相来自：
  - 已注册设备集合
  - 接入层最近一次 `resume / heartbeat / disconnect` 记录
  - 可选缓存中的在线状态提示

建议标准结构：

```json
{
  "tenantId": "t_xxx",
  "actorId": "u_xxx",
  "sessionId": "s_xxx",
  "deviceId": "d_xxx",
  "resumeRequired": true,
  "resumeFromSyncSeq": 14,
  "latestSyncSeq": 19,
  "resumedAt": "2026-04-05T10:00:20Z",
  "presence": {
    "tenantId": "t_xxx",
    "principalId": "u_xxx",
    "currentDeviceId": "d_xxx",
    "devices": [
      {
        "deviceId": "d_phone",
        "platform": null,
        "sessionId": "s_phone",
        "status": "online",
        "lastSyncSeq": 19,
        "lastResumeAt": "2026-04-05T10:00:20Z",
        "lastSeenAt": "2026-04-05T10:00:20Z"
      },
      {
        "deviceId": "d_pad",
        "platform": null,
        "sessionId": "s_pad",
        "status": "offline",
        "lastSyncSeq": 12,
        "lastResumeAt": "2026-04-05T09:50:00Z",
        "lastSeenAt": "2026-04-05T09:51:00Z"
      }
    ]
  }
}
```

字段约束：

- `resumeRequired = latestSyncSeq > lastSeenSyncSeq`
- `resumeFromSyncSeq` 表示第一个待补拉序号，建议计算为：
  - 若 `latestSyncSeq == 0`，返回 `0`
  - 若 `resumeRequired == false`，返回 `latestSyncSeq`
  - 否则返回 `lastSeenSyncSeq + 1`
- `presence.devices[]` 至少覆盖当前主体已注册设备集合；未处于在线态的设备也必须返回，状态标记为 `offline`
- `lastSyncSeq` 表示该设备最近一次成功恢复时看到的最新补偿流水位，不等同于实时 ACK 真相
- `lastSeenAt` 由 `session.resume`、`presence.heartbeat`、`session.disconnect` 推进，用于表征最近一次接入层看到该设备的时间
- 一期实现可不单独创建持久化表，允许由缓存或接入层内存态维护，后续再平滑替换为独立 presence store

## 7. 事件表意标准

### 7.1 Message Mutation 事件

标准 IM 需要支持“已发送消息的后续变更”，一期先定义两类事件：

- `message.edited`
- `message.recalled`

约束：

- 两类事件都绑定已有 `message_id`
- 不创建新消息，不改变原 `message_seq`
- 查询面必须按事件重放得到最新消息视图
- 多端补偿流可继续沿用 `device sync feed`，通过 `origin_event_type` 透出编辑或撤回事实

建议 `message.edited` 载荷：

```json
{
  "tenantId": "t_xxx",
  "conversationId": "c_xxx",
  "messageId": "msg_c_xxx_12",
  "messageSeq": 12,
  "body": {
    "summary": "edited summary",
    "parts": [
      {
        "kind": "text",
        "text": "edited body"
      }
    ],
    "renderHints": {}
  },
  "editor": {
    "id": "u_xxx",
    "kind": "user",
    "deviceId": "d_xxx",
    "sessionId": "s_xxx"
  },
  "editedAt": "2026-04-05T10:00:30Z"
}
```

建议 `message.recalled` 载荷：

```json
{
  "tenantId": "t_xxx",
  "conversationId": "c_xxx",
  "messageId": "msg_c_xxx_12",
  "messageSeq": 12,
  "recalledBy": {
    "id": "u_xxx",
    "kind": "user",
    "deviceId": "d_xxx",
    "sessionId": "s_xxx"
  },
  "recalledAt": "2026-04-05T10:00:40Z"
}
```

查询面最小标准：

- `timeline` 中被编辑的消息应反映最新 `summary`
- `timeline` 中被撤回的消息应保留原位置，但 `summary` 可标准化为 `[recalled]`
- 如果最后一条消息被编辑或撤回，`conversation summary` 与 `inbox` 也必须同步更新

- durable truth 不依赖 projection 表
- 允许 projection 重建
- 允许多份物化视图
- 允许读模型按 profile 差异优化
