# 数据模型与数据库设计

## 1. 文档目标

本文档用于把当前推荐架构下的数据模型、持久化边界和数据库设计收敛为一套统一方案，重点回答：

- 哪些数据是权威真相
- 哪些数据是可重建投影
- `sender`、`actor`、`device`、`member` 如何统一建模
- 消息、流、RTC、AI、IoT 数据如何共存
- 第一阶段在 `PostgreSQL + Redis + Object Storage` 上如何落地

## 2. 数据设计总原则

- `durable truth` 与 `query truth` 分离
- 会话内顺序真相与查询投影分离
- 热状态与长期元数据分离
- 大对象与业务元数据分离
- 流与消息并列建模，但允许流最终物化为消息
- 领域语义优先稳定，底层物理存储可以后续演进

## 3. 认证主体与业务主体模型

### 3.1 认证主体（Principal）

系统主体统一为 `Principal`，推荐 `kind` 包括：

- `user`
- `agent`
- `device`
- `bot`
- `system`

### 3.2 设备主体（Device）

设备不是用户附属字段，而是一等对象，至少包括：

- `tenant_id`
- `device_id`
- `principal_id`
- `device_type`
- `platform`
- `status`
- `registered_at`
- `last_seen_at`

### 3.3 容器成员（Member）

成员是会话内主体关系，不等同于主体本身：

- `tenant_id`
- `conversation_id`
- `member_id`
- `principal_id`
- `principal_kind`
- `role`
- `state`
- `episode_no`
- `joined_at`
- `left_at`

说明：

- `member_id` 是会话内稳定主键
- `left -> rejoin` 必须创建新的 membership episode
- `member_id` 用于群聊角色、禁言、已读游标和 sender 快照

## 6. 消息模型

### 6.1 Message 的逻辑模型

消息是离散语义载体，建议核心字段：

- `tenant_id`
- `conversation_id`
- `message_id`
- `message_seq`
- `message_type`
- `delivery_mode`
- `sender_snapshot_json`
- `client_msg_id`
- `stream_session_id`
- `rtc_session_id`
- `body_json`
- `attributes_json`
- `occurred_at`
- `committed_at`

### 6.2 Sender 统一结构

对外协议和内部事件统一使用 `sender` 对象，而不是平铺 `senderId`：

```json
{
  "id": "u_xxx",
  "kind": "user",
  "memberId": "cm_xxx",
  "deviceId": "d_xxx",
  "sessionId": "s_xxx",
  "metadata": {
    "role": "owner"
  }
}
```

说明：

- `id` 表示主体 ID
- `kind` 表示主体类型
- `memberId` 表示会话内身份快照
- `deviceId / sessionId` 用于多端追踪和审计
- `metadata` 用于扩展 `agentId / toolCallId / handoffSessionId / deviceTwinVersion`

### 6.3 MessageBody

```json
{
  "summary": "optional summary",
  "parts": [],
  "render_hints": {}
}
```

### 6.4 ContentPart 类型

支持以下 part：

- `text`
- `data`
- `media`
- `signal`
- `stream_ref`
- `card_ref`

## 9. Drive-backed media usage model

IM 不拥有文件生命周期表。`sdkwork-drive` 负责上传会话、节点身份、版本、存储提供方事实、权限和访问 URL；IM 只在消息内容中持久化 Drive 引用和渲染所需的 `MediaResource` 使用快照。

- `message_id`
- `part_index`
- `drive_json` (`DriveReference`: `driveUri/spaceId/nodeId/versionId`)
- `resource_json` (`MediaResource`: `kind/source=drive/uri/fileName/mimeType/sizeBytes/...`)
- `media_role`
- `schema_ref`
- `encoding`
- `payload`

## 12. 事件表意标准

### 12.1 Message Mutation

标准消息变更至少支持：

- `message.posted`
- `message.edited`
- `message.recalled`

约束：

- 编辑和撤回都通过追加事件表达
- 不允许原地覆盖 durable message log
- 投影必须可通过重放获得正确最新视图

### 12.2 Stream Lifecycle

标准流事件至少支持：

- `stream.opened`
- `stream.delta_appended`
- `stream.checkpointed`
- `stream.finalized`
- `stream.aborted`

### 12.3 RTC Lifecycle

标准 RTC 事件至少支持：

- `rtc.invited`
- `rtc.accepted`
- `rtc.rejected`
- `rtc.ended`

## 13. 分区与索引建议

### 13.1 PostgreSQL

- 大表按 `tenant_id` 和时间做联合分区或归档
- `message_log_entries` 按 `tenant_id + conversation_id + ordering_seq` 建主索引
- `stream_frames` 按 `tenant_id + stream_id + frame_seq` 建主索引
- `projection_inbox` 按 `tenant_id + principal_id + last_activity_at desc` 建查询索引

### 13.2 Redis

- route、presence、resume、quota 明确分 key space
- TTL 状态与非 TTL 状态严格区分
- 热租户与热点设备支持统计和限流

## 14. 结论

`craw-chat` 的数据设计已经明确从“用户消息系统”演进为“主体、消息、流、设备、投影并列”的统一模型。第一阶段基于 PostgreSQL、Redis 和对象存储即可稳定落地，同时为后续自研 `MessageLog / StreamStore / RouteStore` 预留清晰边界。

## 2026-04-09 增补：IM DDD 命名与演进基线

### A. 命名标准

- 自然人主体使用 `user`，不用 `account` 作为 IM 主业务名词。
- 任意可发言、可治理、可被路由的参与者使用 `actor`。
- `member` 只表示 `actor` 在 `space / group / channel / conversation / thread` 等容器中的关系。
- `conversation` 只表示消息容器与时序，不直接等同好友关系、群组或空间。
- `principal` 仅保留为认证抽象；IM 主业务文档优先使用 `user / actor / member`。

### B. 当前 durable truth 与目标 durable truth

- 当前已稳定：`conversation / conversation_member / message / read_cursor / stream / rtc / media / projection`
- 当前仍在治理主线但非完整社交模型：`provider_policy / provider_binding / audit / ops snapshot`
- 当前未完整落地：`friend_request / friendship / user_block / direct_chat / space / space_member / chat_group / group_member / chat_channel / channel_access_rule / invitation / membership_request / ban_record / thread / external_connection`
- 因此数据库文档必须把“当前已实现表”和“目标标准表”分开写，不能把目标态误写成现状。

### C. 目标标准模型

- 社交：`im_friend_request / im_friendship / im_friendship_event / im_user_block / im_direct_chat / im_contact_projection`
- 空间与群组：`im_space / im_space_role / im_space_member / im_chat_group / im_group_role / im_group_member / im_chat_channel / im_channel_access_rule / im_channel_subscription`
- 治理：`im_invitation / im_membership_request / im_ban_record / im_mute_setting / im_report_ticket / im_audit_event / im_outbox_event`
- 进阶扩展：`im_chat_thread / im_thread_subscription / im_external_connection / im_external_member_link / im_message_reaction / im_pin_record / im_history_visibility_policy / im_retention_policy`

### D. 性能与分层规则

- 好友、空间、群组、频道、线程都先写 durable truth，再异步投影联系人、目录、徽标、搜索与侧边栏视图。
- `conversation_id + seq` 继续作为消息时序主键；社交与组织关系不得依赖消息表反推。
- 权限采用 `role + access rule overwrite`，不把频道权限压扁为会话成员快照。
- 外部协作单独建模连接与成员映射，不复用普通成员唯一键。
- 历史可见性、保留策略、legal hold 归治理域，不混入消息热表。
