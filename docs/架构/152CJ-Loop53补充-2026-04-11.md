# 152CJ Loop53补充 - 2026-04-11

- `control-plane-api` 当前除了完整 `build_app*` 之外，还显式提供了 embedded control surface builder
  - embedded builder 不再携带 `/healthz`，因此可以被 `local-minimal-node` 直接 merge 到已有 router，而不会制造根路径冲突
  - embedded builder 已补齐 `runtime_dir + governance_sinks + shared_channel_sync_trigger` 组合装配，允许 file-backed social runtime 与 injected sync trigger 同时成立
- `local-minimal-node` 现在是 `control-plane-api` 的真实组合根之一
  - 默认/公开装配面都 merge 了 `/backend/v3/api/control/*`
  - merged control-plane surface 与 local-minimal 复用同一 `RealtimeClusterBridge / OpsRuntime / AuditRuntime`
  - `OpsRuntime` 的 service inventory 现在显式包含 `control-plane-api`
- `SharedChannelLinkedMemberSyncTrigger` 当前已有一个真实同进程 consumer 形态
  - 该 consumer 位于 `local-minimal-node`
  - 其行为不是二次 HTTP，而是直接调用 `ConversationRuntime::sync_shared_channel_linked_member(...)`
  - 这保持了 `control-plane-api` 不依赖 `conversation-runtime`，同时让组合根承担跨 bounded-context 接线责任
- `projection-service` 的 as-built history access 口径本轮得到一条关键修正
  - `timeline_from_auth_context(...)` 不再只认 active member
  - 带完整 shared/external anchor 的 `MembershipState::Linked` 当前可通过 `can_read_shared_history()` 读取 local-minimal timeline
  - 因此 local-minimal merged app 上的 shared linked actor 已能沿 `control-plane durable truth -> runtime sync -> projection timeline read` 完成一条真实主链
- 当前仍然不能把系统解释为“所有 control-plane 装配面都自动 materialize runtime linked member”
  - standalone `control-plane-api::build_app*` / `main.rs` 仍未注入真实 consumer
  - 当前仍没有 outbox / retry / remote republish，因此并非跨服务 release-ready orchestration
  - projection 的其他 read surfaces 是否也该对 linked member 放宽，当前尚未一并宣称
- As-Built 结论现在更新为：
  - 已闭合到 `local-minimal embedded real consumer wiring`
  - 仍待闭合到 `standalone control-plane consumer wiring / cross-service outbox boundary`
