# CCP 控制面注册表与协议发布治理设计

## 1. 文档定位

本文档用于把 `CCP` 的协议注册表、能力开关、租户灰度、节点下发与运维审计，正式接入 `Control Plane` 体系。

`146` 已经定义了：

- Protocol Registry
- Schema Registry
- Capability Registry
- Compatibility Matrix Registry
- Close / Error Registry

但如果这些注册表只停留在文档层，而不进入控制面治理，就会出现几个问题：

- 服务端版本虽然支持某能力，但没有统一发布路径
- 多租户的灰度与回滚无法审计
- 不同 cell 的协议配置漂移
- 私有化客户无法拿到一份可控的协议发行包

本文档解决的是“协议如何被治理、发布、下发和回滚”。

## 2. 设计目标

### 2.1 核心目标

- 把 `CCP` 注册表纳入控制面权威配置体系
- 支持协议能力的全局、cell、tenant、client-segment 四级治理
- 支持协议灰度、冻结、回滚和紧急 kill switch
- 让 SaaS 与私有化都能使用同一套协议发布模型

### 2.2 非目标

- 不在第一阶段建设重量级中心化 Schema Registry 服务
- 不让控制面参与每一条消息的同步热路径决策
- 不把运行时节点变成配置拼装器

## 3. 协议治理对象模型

控制面中必须显式建模以下对象。

### 3.1 Protocol Profile

用于描述一个可发布的协议能力集合。

建议字段：

- `profileId`
- `protocolVersion`
- `allowedBindings`
- `allowedCodecs`
- `enabledCapabilities`
- `defaultSchemas`
- `deprecatedSchemas`
- `closeContractVersion`
- `sdkCompatibilityBaseline`

### 3.2 Release Channel

用于定义一组协议能力面向谁开放。

建议内建：

- `canary`
- `beta`
- `stable`
- `lts`
- `private`

### 3.3 Rollout Policy

用于定义如何发布。

建议字段：

- `scope`
- `targetPercent`
- `orderedWaves`
- `successThreshold`
- `rollbackThreshold`
- `ttl`
- `operatorApprovalRequired`

### 3.4 Tenant Protocol Policy

用于覆盖租户级协议能力。

建议字段：

- `tenantId`
- `protocolProfileId`
- `allowedExperimentalSchemas`
- `disabledCapabilities`
- `clientSegmentPolicy`
- `effectiveFrom`
- `effectiveTo`

### 3.5 Client Segment Policy

用于控制协议能力按客户端群体启用。

建议维度：

- `clientType`
- `platform`
- `sdkVersion`
- `region`
- `deploymentProfile`

## 4. 权威来源与优先级

协议治理必须遵循与配置治理一致的权威优先级。

推荐顺序如下：

1. `Emergency Kill Switch`
2. `Operator Rollout Override`
3. `Tenant Protocol Policy`
4. `Cell / Region Release Channel`
5. `Global Stable Baseline`

约束如下：

- 同一个 capability 只能有一个最终生效结果
- override 必须带审计与 TTL
- 紧急 kill switch 默认优先级最高

## 5. 控制面如何下发协议结果

运行时节点不应自己拼协议配置，而应消费控制面输出的“已计算结果”。

推荐产物如下：

### 5.1 Runtime Effective Protocol Snapshot

用于节点直接消费的只读快照。

建议内容：

- 生效的 `protocol profile`
- 生效的 capability 集合
- 生效的 schema allowlist / denylist
- 生效的 binding allowlist
- close / error contract 版本
- 配置版本号

### 5.2 Tenant Effective Protocol Snapshot

用于：

- 公共网关
- 鉴权层
- SDK 协商结果

它应能在不查询全量控制面对象的情况下，快速判断：

- 这个租户能否启用 `stream.patch`
- 这个租户是否允许 `device.signature`
- 这个租户是否仍需要保留旧 schema 兼容

## 6. 协议发布通道设计

### 6.1 Channel 模型

协议能力不直接“全量上线”，而通过通道发布：

- `canary`：少量节点、少量租户、少量 SDK 版本
- `beta`：可控扩大范围
- `stable`：默认生产发布
- `lts`：用于私有化和长维护周期客户
- `private`：客户专属协议配置包

### 6.2 推荐发布顺序

建议固定为：

1. 内部开发 / local
2. canary cell
3. shared cell beta
4. shared cell stable
5. dedicated cell
6. private delivery pack

## 7. Kill Switch 与冻结策略

对高风险协议能力必须内建 kill switch。

建议覆盖：

- `stream.patch`
- `rtc.signal`
- `agent.tool_call`
- `device.signature`
- `payload.cbor`
- 新增 experimental schema

Kill Switch 的规则：

- 可以全局关闭
- 可以按 cell 关闭
- 可以按 tenant 关闭
- 必须可审计
- 必须可设置 TTL

## 8. 私有化与离线交付

私有化客户不能依赖云端控制面在线判断协议能力，因此需要可交付的协议发行包。

### 8.1 Protocol Release Bundle

建议把以下内容打包为离线发行物：

- protocol profiles
- schema registry snapshot
- capability registry snapshot
- compatibility matrix snapshot
- close / error registry snapshot
- 发布说明与兼容说明

### 8.2 私有化治理原则

- 私有化默认使用 `lts` 或 `private` channel
- 不强制客户跟随 SaaS 的实验能力
- 所有协议变更都应以 bundle 版本为单位交付

## 9. 运维操作模型

控制面至少支持以下操作：

- 创建 protocol profile
- 发布 rollout policy
- 预览影响范围
- 下发到指定 cell / tenant
- 查看协议配置漂移
- 触发紧急回滚
- 导出协议发行包

### 9.1 发布前检查

协议发布前必须自动验证：

- registry 完整性
- schema 依赖是否闭合
- compatibility matrix 是否覆盖目标客户端
- kill switch 是否存在
- 回滚路径是否存在

### 9.2 发布后观测

必须观测：

- 握手成功率
- capability negotiation 成功率
- 新旧 schema 使用占比
- close code / error code 异常分布
- 租户或 cell 的失败集中度

## 10. 控制面与热路径边界

这是最重要的边界之一。

控制面可以决定：

- 什么能力允许启用
- 哪个 tenant / cell 可以用
- 哪个 channel 当前生效

但控制面不直接参与：

- 每条消息的同步写入
- 每条流帧的实时选择
- 每次 ACK 的即时决策

运行时只消费已经冻结的 `effective snapshot`，而不是在热路径同步查控制面。

## 11. 数据模型建议

建议后续控制面数据模型至少有：

- `protocol_profiles`
- `protocol_rollouts`
- `tenant_protocol_policies`
- `client_segment_policies`
- `protocol_release_bundles`
- `protocol_audit_records`

## 12. 与既有文档的关系

本文档补齐了：

- `142` 中的 feature flags / capability enablement / runtime control config
- `146` 中的 registry / compatibility matrix / SDK 治理
- `149` 将继续补充多 cell / 多 region 的升级与回滚流程

## 13. 结论

`CCP` 的协议治理不能只停留在 schema 和兼容矩阵文档里，必须进入 `Control Plane`，形成：

- 可发布
- 可灰度
- 可审计
- 可回滚
- 可私有化打包

的完整协议运营体系。

## 2026-04-09 增补：协议发布治理的目标控制面与当前实现边界

### A. 当前实现真相

- 当前已存在 `protocol registry / protocol governance` read surface、kill switch baseline、effective snapshot 概念与部分 operator validation path。
- 当前热路径仍以 runtime 消费已计算结果为主，还没有把本文全部 `rollout / channel / bundle / tenant-segment` 模型完整做成商用控制面产品。
- 文末 `As-Built` 段落才代表当前真实落地，不应以前文目标设计替代现状描述。

### B. 哪些内容是目标态

- 第 3-11 节中的 governance object model、release channel、rollout policy、private delivery bundle、运维操作模型，属于目标控制面标准。
- 这些章节定义的是面向 `SaaS / 私有化 / 多 cell` 的终局治理方式，不等于当前控制面已经完整兑现。
- 当前已落地的是最小治理骨架，不是完整协议发布平台。

### C. 文档口径规则

- 写当前发布治理能力时，必须优先引用本文 `As-Built`、`146`、`152CJ` 以及 `control-plane-api` 的真实 API 与测试。
- 写未来控制面方案时，必须显式标记 `目标态` 或 `标准目标面`。
- 若治理对象仍未落地到 durable truth，不得写成“已支持租户级 / client-segment 级 / release bundle 级治理”，只能写成目标设计。
