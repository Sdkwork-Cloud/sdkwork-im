# 152CJ Loop66 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“operator 可看见 `claimedAt` 与 manual takeover，但看不见 lease deadline”推进到“pending/dead-letter inventory 显式暴露 `leaseExpiresAt`”的最小 lease-metadata seam

## 2. As-Built 事实
- `control-plane-api` 当前为 `PendingSharedChannelSyncRequest` 新增 durable `leaseExpiresAt`
- `SocialControlRuntime::claim_pending_shared_channel_sync_targeted(...)` 当前会在首次 owner claim 时同时写入 `claimedAt` 与固定 `15m` lease window 派生出的 `leaseExpiresAt`
- 同 owner 重复 claim 当前保留原 `claimedAt / leaseExpiresAt`，不把重复 claim 解释成 lease refresh
- `SocialControlRuntime::takeover_pending_shared_channel_sync_targeted(...)` 当前会在 foreign takeover 时一并刷新 `claimedAt / leaseExpiresAt`
- `SocialControlRuntime::release_pending_shared_channel_sync_targeted(...)` 当前会在 owner release 时清空 `ownerActorId / ownerActorKind / claimedAt / leaseExpiresAt`
- `GET /api/v1/control/social/runtime/pending-shared-channel-sync` 与 `GET /api/v1/control/social/runtime/dead-letter-shared-channel-sync` 当前都通过同一 inventory item 结构显式返回 `leaseExpiresAt`

## 3. 明确边界
- 本轮没有：
  - stale-age threshold / SLA
  - timeout reclaim / automatic stale detection
  - background retry scheduler
  - ownership-aware `repair-shared-channel-sync`
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 单体内的 lease-metadata visibility seam，而不是 release-ready stale-claim policy engine
