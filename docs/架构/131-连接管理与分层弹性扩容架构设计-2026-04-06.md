# 连接管理与分层弹性扩容架构设计

## 1. 文档目标

本文档用于详细定义 `craw-chat` 的连接管理层、在线路由层与分层扩容模型，解决以下关键问题：

- 如何支撑高连接密度
- 如何将连接治理与业务逻辑解耦
- 如何实现按层扩容，而不是全系统一起扩容
- 如何支持多设备、AI 流式输出、IoT 设备长连
- 如何在后续逐步演进到更高单机能力和更高集群规模

本设计是 `130-连接优先的AI时代即时通讯架构蓝图-2026-04-06.md` 的专项细化文档。

## 2. 总体目标

连接层的核心目标分为两级：

### 2.1 第一落地目标

- 单机 `10 万级` 长连接能力
- 节点排空与重平衡可控
- 设备 resume / reconnect 正确
- 慢连接不拖垮整体节点
- Link Plane 可独立扩容

### 2.2 设计上限目标

- 架构上预留 `50 万级以上` 单机长连接的演进空间
- 可替换更专用的连接运行时实现
- 可演进到更底层协议栈与更高性能网络模型

## 3. 分层模型

系统的扩容不是按“服务个数”定义，而是按压力面定义。

### 3.1 Link Plane

职责：

- WebSocket / HTTP / SSE / MQTT / QUIC 接入
- 握手
- 鉴权
- 心跳
- 背压
- 本机连接分片
- 本机发送队列

扩容触发条件：

- 在线连接数升高
- 握手 TPS 升高
- 出站带宽升高
- 心跳与保活压力升高

### 3.2 Route Plane

职责：

- device -> session -> node 归属
- route lookup
- fencing
- drain
- rebalance
- failover

扩容触发条件：

- 在线设备数升高
- 会话迁移频率升高
- 路由查询与写入压力升高

### 3.3 Messaging Plane

职责：

- 会话
- 成员
- 消息提交
- 已读推进
- 广播与 fanout 编排

扩容触发条件：

- 消息写入 TPS 升高
- 群聊 fanout 压力升高
- 多租户热点会话升高

### 3.4 Stream / AI Plane

职责：

- stream.open / append / checkpoint / finalize
- LLM token streaming
- 转写流
- telemetry 流
- 长任务渐进输出

扩容触发条件：

- LLM 对话并发升高
- 流帧吞吐升高
- 长任务渐进输出增多

### 3.5 Projection Plane

职责：

- inbox
- timeline
- unread
- summary
- topic/thread
- 搜索与知识投影

扩容触发条件：

- 查询流量升高
- 读写比失衡
- AI/搜索投影任务升高

### 3.6 Storage Plane

职责：

- MetadataStore
- RouteStore
- MessageLog
- StreamStore
- BlobStore

扩容触发条件：

- 元数据事务压力升高
- 热路由状态膨胀
- 消息日志写入膨胀
- 大对象存储容量与吞吐升高

## 4. Link Runtime 设计

### 4.1 组件清单

建议将连接运行时抽象为独立模块：

- `acceptor`
- `protocol_upgrade`
- `auth_pipeline`
- `connection_registry`
- `shard_dispatcher`
- `outbound_queue`
- `resume_window_manager`
- `heartbeat_manager`
- `backpressure_controller`
- `connection_metrics`

### 4.2 连接分片

本机连接必须按 shard 管理，避免全局锁。

分片建议：

- 按 `device_id`
- 或按 `principal_id + device_id`
- 或按连接内部生成的 `conn_hash`

约束：

- 单连接只归属于一个 shard
- 连接生命周期只在归属 shard 内修改
- shard 间仅传递轻量事件，不共享大状态对象

### 4.3 出站发送队列

每个连接必须具备独立的有界出站队列。

优先级建议：

- P0: 控制帧、ACK、disconnect、resume、route update
- P1: RTC signal、presence、session lifecycle
- P2: 普通消息
- P3: 流式 delta、大对象引用提示

治理策略：

- 队列满时触发背压
- 慢消费者优先降级低优先级事件
- 极端情况下主动关闭异常连接

### 4.4 背压模型

必须同时支持：

- per-connection 背压
- per-shard 背压
- per-node 背压
- per-tenant 背压

背压动作：

- 降低低优先级事件发送频率
- 暂停部分 stream delta 下推
- 延后次级投影通知
- 对热点租户做局部限流

### 4.5 Resume / Reconnect

每个设备连接必须具备：

- `session_id`
- `device_id`
- `resume_token`
- `last_seen_event_seq`
- `last_seen_stream_checkpoint`

恢复原则：

- session 恢复与消息补偿分离
- session 恢复与 stream 恢复分离
- 恢复只依赖权威状态，不依赖单点内存缓存

## 5. Route Plane 设计

### 5.1 路由模型

建议统一模型：

- `tenant_id`
- `principal_id`
- `device_id`
- `session_id`
- `owner_node_id`
- `route_epoch`
- `bound_at`
- `drain_state`

### 5.2 Fencing

为避免陈旧连接继续投递，必须引入 fencing：

- 每次 route 绑定带 `epoch`
- 新 route 绑定会提升 epoch
- 老连接看到更高 epoch 必须主动退出
- 所有投递都要检查 route epoch 是否匹配

### 5.3 Drain 与 Rebalance

节点不能粗暴下线，必须支持：

- `active`
- `draining`
- `drained`

`draining` 阶段规则：

- 不再接收新连接
- 允许已有连接平滑迁移
- 允许短时保留路由，直到目标节点接管

### 5.4 RouteStore 建议

第一阶段建议：

- Redis 承担热点 route state
- 持久化元数据进入 PostgreSQL

长期演进：

- 保留 RouteStore 抽象
- 后续可切换自研 route directory 或轻量内嵌一致性层

## 6. 分层扩容策略

### 6.1 横向扩容粒度

扩容单位不应是整套系统，而应是：

- `link node`
- `route node`
- `messaging node`
- `stream node`
- `projection worker`
- `admin/control node`

### 6.2 典型扩容场景

#### 场景 A：在线人数暴涨

优先扩：

- Link Plane
- Route Plane

无需立即扩：

- Projection Plane
- Storage Plane

#### 场景 B：群聊与频道 fanout 暴涨

优先扩：

- Messaging Plane
- Projection Plane

可能联动：

- Link Plane

#### 场景 C：AI 流式对话暴涨

优先扩：

- Stream / AI Plane
- Link Plane

必要时联动：

- ObjectStore
- Projection Plane

#### 场景 D：IoT 设备大规模在线

优先扩：

- Link Plane
- Route Plane
- Stream / AI Plane

### 6.3 热点治理

必须考虑：

- 热租户
- 热群组
- 热频道
- 热设备
- 热 agent 会话

治理手段：

- tenant lane
- scoped throttling
- hot shard rebalance
- high-fanout 分离处理
- stream 降级策略

## 7. Cell 与区域扩展

### 7.1 Cell 原则

一个 Cell 应作为：

- 故障域
- 扩展单元
- 部署单元
- 运维单元

### 7.2 SaaS 形态

建议：

- 小租户共享 cell
- 大租户可进入 dedicated cell
- 特高负载 tenant 可独占 link lane / route lane / storage lane

### 7.3 私有化形态

建议：

- 默认单 cell 起步
- 保留多 cell 模式接口
- 与云版本保持同一协议与模块边界

## 8. 可观测性与 SLO

连接层必须提供以下指标：

- 当前连接数
- 活跃连接数
- 握手成功率
- 握手耗时
- 平均心跳延迟
- 每连接积压长度
- shard 负载分布
- route lookup 延迟
- reconnect 成功率
- resume 成功率
- 主动断开原因分布

建议核心 SLO：

- 连接建立成功率
- 在线消息投递成功率
- reconnect 恢复成功率
- 路由迁移成功率
- 节点排空完成时间

## 9. 与 AI / IoT 的关系

连接层必须直接支持：

- LLM token streaming
- AI tool result streaming
- 设备 telemetry streaming
- 设备命令回执
- 多端实时 presence

因此连接层协议绝不能只围绕“发送一条静态消息”设计。

## 10. 实施建议

第一阶段优先事项：

- 从现有 `session-gateway` 与 `local-minimal-node` 中抽离连接热路径
- 建立 `runtime-link` 与 `runtime-route` 边界
- 统一连接 frame 模型
- 统一 session / route / resume / reconnect 语义
- 建立连接层指标与诊断接口

后续阶段：

- 引入 MQTT
- 引入 QUIC 预留
- 更强 shard 调度
- 更高连接密度优化

## 11. 结论

`craw-chat` 的扩容能力必须首先建立在“连接与路由是独立 plane”的前提上。只有这样，系统才能同时具备：

- 高连接密度
- 多设备一致性
- AI 流式能力
- IoT 设备接入
- 分层独立扩容
- 商业化可控演进
