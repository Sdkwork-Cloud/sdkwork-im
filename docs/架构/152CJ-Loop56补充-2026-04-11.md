# 152CJ Loop56 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“只有 `durable pending backlog + operator repair`”推进到“当 trigger 恢复健康且再次出现 ready pair 写入时，控制面会先自动重放 backlog，再投递本次新请求”的最小自愈边界

## 2. As-Built 事实
- `control-plane-api::dispatch_shared_channel_sync_requests(...)` 当前不再只看本次新生成的 request
- dispatch 前会先读取 `SocialControlState.pending_shared_channel_sync_requests`
- dispatch 队列当前按“既有 pending backlog -> 本次 ready-pair request”的顺序构造，并按 request key 去重
- 对于成功投递的 backlog 项：
  - 仍沿用既有成功路径的 best-effort 清理逻辑
  - 不引入新的 journal event / outbox topic / lease owner
- 对于再次失败的项：
  - 失败项与其后的未投递项会继续 durable 写回 `pending_shared_channel_sync_requests`
  - social durable truth 保持 committed，不做回滚

## 3. 明确边界
- 本轮自动重试不是后台 scheduler；它只发生在“trigger 已恢复健康，并且又发生一次新的 ready pair 写入”这一同步入口
- `POST /api/v1/control/social/runtime/repair-shared-channel-sync` 仍然保留，作为显式 operator replay surface
- 当前仍未实现：
  - dead-letter / poison backlog 隔离
  - remote republish SLA / lease owner / delivery ownership contract
  - release-ready exactly-once cross-service delivery semantics

## 5. 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop56 只补齐了“同步入口上的最小自动重试”这一条真相
- 更强的自动 delivery 闭环仍收敛为 `automatic retry scheduler / dead-letter / remote republish / delivery ownership semantics`
