# 权限能力模型与协议演进设计

## 1. 文档目标

本文档用于定义 `craw-chat` 的授权层次、能力模型和协议演进规则，确保系统在支持用户、Agent、设备、Bot 等多类主体后，仍然能够：

- 明确谁可以做什么
- 让 API 与事件模型稳定演进
- 保持客户端、服务端和 SDK 长期兼容

## 2. 授权模型总原则

- 身份先于能力，能力先于命令执行。
- `tenant`、`actor`、`device`、`member`、`session` 是不同层级的约束，不可混为一谈。
- 权限判断必须以权威上下文和服务端关系为准，不能信任客户端自带声明。
- AI、IoT 是扩展能力域，但不能绕过通用授权模型。

## 3. 授权层次

### 3.1 Tenant 级

用于判断：

- 租户是否启用某能力
- 租户配额是否允许
- 租户是否被限流或冻结

### 3.2 Principal 级

用于判断：

- 当前主体是谁
- 主体类型是什么
- 是否具备全局基础权限

### 3.3 Device / Session 级

用于判断：

- 当前请求是否来自已绑定设备
- 是否允许该 session 执行某些敏感操作
- 是否存在更高 epoch 的会话 fencing

### 3.4 Membership 级

用于判断：

- 是否是该会话成员
- 当前 `member role / state` 是否允许某操作

### 3.5 Conversation Capability 级

用于判断：

- 在特定会话内，当前主体能否执行指定命令

## 4. 能力模型设计

建议将能力显式建模为 `capability`，而不是散落在各 handler 里的 if-else。

### 4.1 基础能力

- `conversation.read`
- `message.post`
- `message.edit_own`
- `message.recall_own`
- `read_cursor.update`
- `media.attach`

### 4.2 成员治理能力

- `member.add`
- `member.remove`
- `member.leave`
- `member.role.change`
- `member.owner.transfer`

### 4.3 流与 RTC 能力

- `stream.open`
- `stream.abort`
- `rtc.create`
- `rtc.invite`
- `rtc.control`

### 4.4 AI 与 Agent 能力

- `agent.invoke`
- `agent.handoff`
- `agent.tool.approve`
- `agent.context.read`

### 4.5 IoT 能力

- `device.bind`
- `device.command.send`
- `device.telemetry.read`
- `device.twin.manage`

## 5. 默认角色与能力关系

### 5.1 owner

- 拥有会话治理和高风险成员管理能力

### 5.2 admin

- 拥有部分会话治理能力，但不具备 owner 专有操作

### 5.3 member

- 拥有基础读写能力和已读推进能力

### 5.4 guest / observer

- 默认只读或受限发言

### 5.5 agent / bot / device

- 不自动拥有与用户相同的全部权限
- 必须显式映射到 capability profile

## 6. 权限解析时序

推荐统一解析流程：

1. `Link / Interface` 层完成认证和权威上下文生成
2. `Policy / Auth` 层解析 tenant policy 与 actor profile
3. `Messaging / App` 层解析 conversation membership
4. `Capability Resolver` 生成当前命令的能力快照
5. 命令执行前完成 capability gate
6. 执行结果与拒绝结果都写入审计锚点

## 7. 协议演进原则

### 7.1 API 演进

- 路径稳定优先
- 新能力优先通过新增字段、新增类型、新增命令表达
- 尽量避免破坏式字段修改

### 7.2 Event 演进

- 使用 `event_version`
- payload 内新增字段优先采用向后兼容策略
- 语义变化较大时升级事件类型或版本

### 7.3 Schema 演进

- `schema_ref` 必须稳定且显式版本化
- stream / signal / data payload 都必须有清晰 schema 归属

### 7.4 SDK 演进

- SDK 必须围绕稳定 contract 生成或封装
- 不允许不同语言 SDK 各自发明不同的基础字段语义

## 8. 握手与能力协商

为了支持客户端长期演进，建议在连接建立或初始化阶段支持：

- protocol version
- feature flags
- supported stream types
- supported signal types
- optional capability hints

注意：

- 协商结果只能决定“能否使用某个新能力”
- 不能覆盖服务端权威授权判断

## 9. 与 sender / tenant 规则的关系

- `tenantId` 不由客户端业务体提交
- `sender` 不由客户端随意声明完整快照
- 服务端根据认证上下文、membership 和 session 状态生成最终 `sender snapshot`
- 这样才能确保审计、回放、多端同步和权限判断保持一致

## 10. 结论

`craw-chat` 要想长期保持架构稳定，必须把“授权能力模型”和“协议演进规则”一起冻结。否则系统很快会演变为：接口越来越多、例外越来越多、AI 和 IoT 场景各写一套权限逻辑，最终失去可维护性。
