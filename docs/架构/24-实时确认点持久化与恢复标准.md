# 24-实时确认点持久化与恢复标准

## 1. 目标

本标准定义 realtime 设备窗口的确认点持久化模型，以及 runtime 重建后的最小恢复语义。

本标准只解决以下 durable 状态：

- `latestRealtimeSeq`
- `ackedThroughSeq`
- `trimmedThroughSeq`

不解决以下内容的 durable 存储：

- realtime event window 本身
- WebSocket/TCP 连接对象
- transport 层的在线会话迁移

目标是先把 checkpoint 边界冻结下来，使 SaaS 与私有化部署都能在不改变业务协议的前提下，自由替换 checkpoint 后端。

## 2. 核心原则

### 2.1 checkpoint 是设备级 durable 元数据

checkpoint 的归属键必须是：

- `tenantId`
- `principalId`
- `clientRouteId`

也就是 `tenantId + principalId + clientRouteId`。

checkpoint 不属于某条 WebSocket 连接，不属于某次 HTTP 请求，也不属于某个节点本地进程。

### 2.2 checkpoint 与 event window 解耦

checkpoint 的职责是记录设备窗口已经推进到哪里，而不是保存窗口内所有事件。

因此：

- checkpoint store 必须可持久化
- realtime event window 可以继续保留当前的内存实现
- runtime 重建后，即便旧窗口事件已经丢失，也必须能恢复 checkpoint 状态

### 2.3 `trimmedThroughSeq` 不能大于 `ackedThroughSeq`

checkpoint 必须满足以下不变量：

- `0 <= ackedThroughSeq <= latestRealtimeSeq`
- `0 <= trimmedThroughSeq <= ackedThroughSeq`

任何实现都不得写出违反该约束的 checkpoint。

### 2.4 latest seq 是 checkpoint 的一部分

虽然 `latestRealtimeSeq` 也可以从窗口内存推导，但为了保证 runtime 重建与多节点迁移时的状态连续性，本标准要求：

- `latestRealtimeSeq` 必须作为 checkpoint record 的字段保存

否则 runtime 重建后无法知道当前设备 realtime 序列推进到了哪里。

## 3. 数据模型

### 3.1 `RealtimeCheckpointRecord`

最小持久化记录如下：

```json
{
  "tenantId": "t_demo",
  "principalId": "u_demo",
  "clientRouteId": "d_pad",
  "latestRealtimeSeq": 42,
  "ackedThroughSeq": 40,
  "trimmedThroughSeq": 40,
  "updatedAt": "2026-04-05T12:30:00Z"
}
```

字段要求：

- `tenantId`
- `principalId`
- `clientRouteId`
- `latestRealtimeSeq`
- `ackedThroughSeq`
- `trimmedThroughSeq`
- `updatedAt`

### 3.2 durable key

推荐 durable key 采用设备主键展开后的稳定格式，例如：

```text
realtime-checkpoint:t_demo:u_demo:d_pad
```

标准只要求键空间稳定、可枚举、可按设备唯一定位，不强制具体存储实现的物理编码。

## 4. 写入语义

### 4.1 publish 后写入 latest seq

当 `publish_scope_event` 实际命中设备并推进 `latestRealtimeSeq` 后，runtime 必须把新的 checkpoint 写入 store。

### 4.2 ack 后写入 ack/trim

当 `ack_events` 推进了 `ackedThroughSeq` 或 `trimmedThroughSeq` 后，runtime 必须把新的 checkpoint 写入 store。

### 4.3 route migration / state handoff 后写入

当设备 route 迁移并恢复到目标 runtime 时，目标 runtime 必须把迁移后的 checkpoint 再次写入 store，确保：

- checkpoint durable 状态与新 owner node 上的内存状态一致

## 5. 读取与恢复语义

### 5.1 runtime 初始化或首次访问设备时可恢复 checkpoint

当 runtime 首次见到某个客户端路由状态时，应尝试从 checkpoint store 恢复：

- `latestRealtimeSeq`
- `ackedThroughSeq`
- `trimmedThroughSeq`

### 5.2 恢复后不强制恢复旧 event window

当前阶段允许 runtime 恢复 checkpoint，但不恢复旧的 event window。

这意味着 runtime 重建后：

- `window_checkpoint` 必须返回持久化恢复的 seq 状态
- `list_events` 可以返回空窗口
- 后续新的 realtime publish 必须从恢复后的 `latestRealtimeSeq + 1` 继续递增

### 5.3 notifier 恢复要求

设备 notifier 在 runtime 重建后重建即可，但其初始值必须至少等于恢复后的 `latestRealtimeSeq`。

## 6. 平台契约标准

### 6.1 `RealtimeCheckpointStore`

平台契约必须提供可替换的 checkpoint store 接口，至少支持：

- 按设备 key 读取 checkpoint
- 保存或覆盖 checkpoint

当前标准不强制：

- 范围扫描
- 条件更新
- compare-and-swap
- lease / fencing

这些能力后续可以通过更高级的 route lease 标准继续补充。

### 6.2 local-memory 适配器

最小可安装闭环必须提供 `local-memory` 适配器实现，便于：

- 单机开发
- 离线测试
- 本地最小部署

但该实现不得成为协议耦合点，后续必须能无痛替换成：

- Redis / Valkey
- FoundationDB / Etcd / Consul 风格元数据存储
- 商业版 route directory / metadata service

## 7. 与当前实时能力的关系

### 7.1 不改变现有 HTTP / WebSocket 语义

以下接口与协议保持不变：

- `POST /im/v3/api/realtime/subscriptions/sync`
- `GET /im/v3/api/realtime/events`
- `POST /im/v3/api/realtime/events/ack`
- `/im/v3/api/realtime/ws`

checkpoint durable 只是把现有语义的状态边界做成可恢复、可替换的内部实现。

### 7.2 不替代 route migration 的 runtime handoff

checkpoint store 不是 route migration 的唯一状态来源。

因为当前 event window 仍在内存里，route migration 期间仍然需要：

- subscriptions handoff
- window handoff
- checkpoint handoff

checkpoint store 的作用是：

- 保证 ack/trim/latest seq 不会随着 runtime 重建而完全丢失

## 8. 当前落地边界

本阶段允许：

- checkpoint 使用 `local-memory` store
- realtime event window 继续使用 runtime 内存结构
- runtime 恢复后只恢复 checkpoint，不恢复旧窗口事件

本阶段不要求：

- checkpoint 的分布式 CAS
- checkpoint 与 route lease 的联合 fencing
- checkpoint 与 durable realtime event log 的统一恢复

## 9. 后续演进方向

在本标准之后，建议继续冻结：

1. route lease / epoch / fencing 标准
2. route store 与 checkpoint store 的一致性边界
3. durable realtime event log / replay 标准
4. transport cutover / resume 编排标准

这些后续标准可以增强集群强一致与故障恢复能力，但不得推翻本标准定义的 checkpoint record 与读取/写入语义。
