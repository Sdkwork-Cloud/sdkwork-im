# 部署拓扑与容量规划设计

## 1. 文档目标

本文档用于定义 `craw-chat` 的部署拓扑形态、扩容单元和容量规划方法，确保架构不只“可画图”，而且能指导：

- 单机验证
- 小规模生产落地
- SaaS 标准化部署
- 大租户独占部署
- 私有化与受限环境交付

## 2. 部署设计总原则

- 扩容单位必须按压力面拆分，而不是整套服务一起复制。
- 一个 `Cell` 同时承担故障域、扩展域、部署域和运维域的角色。
- 初期允许服务进程合并部署，但边界必须按 plane 保持清晰。
- 部署拓扑必须兼容：
  - `Local Minimal`
  - `Private Standard`
  - `SaaS Shared`
  - `SaaS Dedicated`

## 3. 目标基础拓扑形态

### 3.1 Local Minimal

适用场景：

- 开发
- 最小安装验证
- 集成测试
- 演示环境

推荐形态：

- 单进程或单节点
- `local-minimal-node`
- 本地内存或本地磁盘适配器

### 3.2 Private Standard

适用场景：

- 标准私有化
- 中小规模企业部署

推荐形态：

- `gateway-service`
- `control-plane-service`
- `PostgreSQL`
- `Redis`
- `Object Storage`

### 3.3 SaaS Shared Cell

适用场景：

- 公有云多租户
- 中小租户共享资源池

推荐形态：

- 多个 `link nodes`
- 至少一个 `route service`
- 至少一个 `message/stream runtime`
- 独立 `projection workers`
- 共享 `PostgreSQL / Redis / Object Storage`

### 3.4 SaaS Dedicated Cell

适用场景：

- 大租户
- 高 SLA
- 强隔离

推荐形态：

- 独占 cell
- 可独占 link lane、route lane、storage lane
- 可单独调优连接密度、缓存和投影策略

### 3.5 协议发布域

从 `CCP` 协议治理视角看，部署拓扑还必须具备以下角色划分：

- `cell`：最小协议发布域
- `region`：协议汇聚域与灾备切换域
- `private deployment`：协议 bundle 交付域

这意味着部署设计不仅仅是算力和存储布局，还必须能承载：

- protocol profile 下发
- capability rollout
- kill switch 生效
- profile / bundle 回滚

### 3.6 provider/plugin 部署位

部署拓扑还必须显式预留 provider/plugin 部署位：

- RTC provider control callback 与 credential issuer
- `S3` 兼容对象存储 endpoint / gateway
- `iot-access` 节点
- `iot-protocol` 适配节点

当前冻结要求：

- RTC 默认 provider 为 `火山引擎`，但同一拓扑必须允许切换到 `阿里云`、`腾讯云`
- 对象存储必须按 `S3` 标准接入，便于在 `阿里云 / 腾讯云 / 火山引擎 / AWS / Google / Microsoft` 间替换
- IoT 必须为 `MQTT` 与开源 `小智协议` 预留独立协议适配位，避免与消息/流 runtime 混部失控

## 4. Plane 与部署关系

### 4.1 Link Plane

特点：

- 高连接密度
- 网络和内存敏感
- 水平扩容频率高

部署建议：

- 优先独立为 `link nodes`
- 连接数上升时优先横向扩容

### 4.2 Route Plane

特点：

- 热状态密集
- 依赖 fencing 与 route ownership 正确性

部署建议：

- 可与 gateway 同机房部署
- 但逻辑上保持独立 plane

### 4.3 Messaging / Stream Plane

特点：

- CPU、存储写入和有序约束敏感

部署建议：

- 与 link nodes 分离
- 可按消息负载和流负载分别扩容

### 4.4 Projection Plane

特点：

- 可异步
- 可重建
- 更偏向吞吐与批处理

部署建议：

- 独立 worker
- 允许滞后，但必须可观测

## 5. 容量规划方法

## 5.1 核心输入指标

容量规划至少要掌握以下指标：

- 日活主体数
- 峰值在线设备数
- 人均设备数
- 峰值消息 TPS
- 峰值流式帧 TPS
- 平均消息体大小
- 平均流帧大小
- 热群数量
- AI 对话并发数
- IoT 设备在线数与 telemetry 频率

## 5.2 连接容量基线

当前规划基线：

- 单机落地目标：`10 万级长连接`
- 后续架构保留：`50 万级以上长连接 / 单机` 演进空间

容量规划示意公式：

`连接总量 = 峰值在线主体数 × 人均设备数 × 安全系数`

安全系数建议：

- 日常生产：`1.3 - 1.5`
- 大促或重大活动：`1.5 - 2.0`

## 5.3 消息写入容量

容量规划示意公式：

`消息写入带宽 = 峰值消息 TPS × 平均消息大小`

需要同时评估：

- durable write 吞吐
- route lookup 压力
- fanout 扇出量
- projection backlog

## 5.4 流式容量

容量规划示意公式：

`流式输出带宽 = 峰值流帧 TPS × 平均流帧大小`

重点关注：

- AI token streaming 高频短帧
- 设备 telemetry 高频持续帧
- checkpoint 频率
- materialization 触发比例

## 5.5 存储容量

至少分开估算：

- 元数据容量
- 消息日志容量
- 流式 frame 容量
- 投影容量
- 对象存储容量

对于对象存储，建议单独考虑：

- 热资源保留期
- 冷归档周期
- 导出与备份容量

## 6. 扩容触发信号

### 6.1 Link Plane 扩容信号

- 当前连接数接近单节点安全水位
- 握手延迟上升
- 心跳超时上升
- 出站队列积压上升

### 6.2 Route Plane 扩容信号

- route lookup 延迟升高
- route write 冲突升高
- drain / rebalance 耗时上升

### 6.3 Messaging / Stream Plane 扩容信号

- durable commit 延迟升高
- 消息/流 TPS 持续高位
- AI / telemetry 流占用明显升高

### 6.4 Projection Plane 扩容信号

- projection lag 升高
- inbox / summary 更新时间明显滞后

## 7. 协议发布容量补充

除连接和消息容量外，还必须规划协议发布容量：

- canary cell 数量
- beta cell 数量
- region 内并行 rollout 波次
- 每个 cell 可承受的升级连接抖动窗口

推荐要求：

- 至少保留一个 canary cell
- shared region 内至少保留一个非升级 stable cell
- dedicated cell 的协议升级晚于 shared stable
- private deployment 默认使用 `lts / private bundle`

## 8. 热点治理

热点对象至少包括：

- 热租户
- 热群聊
- 热频道
- 热 Agent 会话
- 热 IoT 设备流

治理手段包括：

- tenant lane
- scoped throttling
- hot shard rebalance
- high-fanout 专门处理
- 流式降级和限流

## 9. 部署建议矩阵

| 场景 | 推荐部署重点 |
| --- | --- |
| 本地闭环 | 单节点、最小依赖、快速安装 |
| 私有化标准版 | gateway + control plane + PostgreSQL + Redis + Object Storage |
| 公有云 SaaS 共享池 | 多 link nodes + 独立 route/message/projection plane |
| 大租户独占 | 独立 cell、独立 lane、独立容量规划 |
| AI/IoT 高密度场景 | 单独拉起 stream/iot runtime、iot protocol adapter、rtc provider callback 处理链，避免挤压普通消息链路 |

## 10. 结论

`craw-chat` 的部署与容量规划必须从第一天建立“按 plane 扩容、按 cell 隔离、按热点治理、按协议发布域治理”的能力，否则再好的协议和领域模型都会在上线后被单点瓶颈或升级风险拖垮。

## 2026-04-09 增补：部署标准与当前可运行形态

### A. 当前可运行形态

- 当前最真实、最可验证的运行闭环仍是 `local-minimal-node` 与现有服务组合，而不是完整 SaaS Shared Cell 或完整 Private Standard 产品包。
- 文中的 `gateway-service / control-plane-service / projection workers / route service / message/stream runtime` 更接近目标拓扑角色，不完全等于当前仓库中的实际服务名。

### B. 目标部署标准

- 第 3 节至第 9 节主要描述目标部署标准、容量规划方法与长期交付形态。
- 当前若要描述 as-built 部署，应优先引用 `local-minimal-node`、`session-gateway`、`conversation-runtime`、`control-plane-api`、`projection-service`、`streaming-service`、`rtc-signaling-service` 等真实服务。

### C. 文档口径规则

- `Local Minimal` 可以视为当前可验证 profile。
- `Private Standard / SaaS Shared Cell / SaaS Dedicated Cell` 在当前文档体系中属于目标标准，不应简写为“当前已产品化部署形态”。
