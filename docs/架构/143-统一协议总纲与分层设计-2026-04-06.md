# 统一协议总纲与分层设计

## 1. 文档定位

本文档用于正式冻结 `craw-chat` 的统一协议基线，回答三个核心问题：

- 是否要自研一套完全独立的传输协议
- 协议的“自主知识产权”应该沉淀在哪一层
- 后续消息、流式输出、RTC、Agent、IoT 应当如何共享同一套协议骨架

结论明确如下：

- 不自研新的 TCP 级或 UDP 级底层传输协议
- 保留标准传输：`HTTP`、`WebSocket`、`SSE`、`MQTT`，后续扩展 `QUIC`
- 在标准传输之上，自主定义统一的应用协议族：`CCP = Craw Chat Protocol`

这意味着 `craw-chat` 的协议自主权，不放在“重造传输栈”，而放在以下长期稳定资产上：

- 握手与协商模型
- 会话与恢复模型
- 通用 envelope
- 领域 schema 体系
- 版本兼容规则
- 权威字段归属与安全边界

本文档是 `130-147` 基线中的协议总纲，配套细化文档为：

- `144`：CCP 传输绑定与握手协商设计
- `145`：CCP 数据协议、版本兼容与安全设计
- `146`：CCP 协议注册表与多端 SDK 兼容治理设计
- `147`：CCP 到 crate 与接口模块落地映射设计

## 2. 为什么不自研底层传输协议

从架构先进性、商业可控性和工程落地性综合看，自研底层传输协议不是当前最优解。

### 2.1 原因

- 浏览器、移动端、桌面端、IoT 设备天然支持 `HTTP / WebSocket / SSE / MQTT`，生态成熟。
- TLS、代理、负载均衡、WAF、链路观测、证书体系都围绕标准传输构建，自研传输会抬高接入和运维成本。
- 即时通讯产品真正长期稳定的不是 socket 字节格式，而是“身份、会话、消息、流、兼容、安全”的语义层协议。
- 未来要支持 AI 流式输出、智能硬件接入和企业协作扩展时，应用协议的可演进性比底层传输的差异化更有价值。

### 2.2 自主权应沉淀的位置

`craw-chat` 的协议自主权应沉淀在：

- `CCP` 协议族命名空间
- 通用 envelope 与控制帧
- `message / stream / rtc / agent / device / ops` 领域 schema
- 版本协商与兼容矩阵
- sender / actor / authority 的服务端归属规则

这样既不被某一种开源传输组件绑定，也不会把系统锁死在某一种语言或某一类客户端上。

## 3. CCP 的定义

`CCP = Craw Chat Protocol`，是 `craw-chat` 的统一应用协议族。

它不是某一种单独的物理编码，也不是某一种单独的网络连接方式，而是一组稳定契约：

- 统一协议版本号
- 统一握手帧
- 统一 envelope
- 统一 kind/type 分类
- 统一 schema 命名规则
- 统一兼容与废弃策略
- 统一安全与权威字段边界

因此：

- `JSON` 不是协议本身，只是 `CCP` 的一种编码
- `WebSocket` 不是协议本身，只是 `CCP` 的一种传输绑定
- `MQTT` 不是协议本身，只是 `CCP` 的一种设备接入绑定

## 4. CCP 的设计目标

### 4.1 核心目标

- 让消息、流、RTC、Agent、IoT 共用一套协议骨架
- 让浏览器、App、桌面端、服务间、设备端可以按需选择不同传输
- 让 envelope、握手、版本、安全边界从第一天冻结，避免后期全量返工
- 让领域 payload 可以在不破坏主协议的前提下持续演进
- 让“流式聊天输出”成为协议内建能力，而不是业务层补丁

### 4.2 非目标

- 不在第一阶段自研自定义 TCP 栈
- 不在第一阶段冻结所有业务 payload 的字段细节
- 不让客户端自带权威身份字段覆盖服务端上下文
- 不把协议设计成只适用于 AI，必须兼容普通 IM 主路径

## 5. CCP 分层模型

`CCP` 采用五层协议结构，加一层安全基座。

| 层级 | 名称 | 作用 | 当前冻结内容 |
| --- | --- | --- | --- |
| `L0` | Security Layer | 负责传输安全、证书、信道可信性 | `TLS 1.3`，内部按场景支持 `mTLS` |
| `L1` | Transport Binding | 将 CCP 绑定到不同传输介质 | `ccp/http/1`、`ccp/ws/1`、`ccp/sse/1`、`ccp/mqtt/1`，预留 `ccp/quic/1` |
| `L2` | Handshake / Session | 完成版本协商、认证绑定、恢复、保活、关闭 | `hello`、`hello_ack`、`auth_bind`、`auth_ok`、`session_resume`、`session_resumed`、`heartbeat`、`goaway`、`error` |
| `L3` | Common Envelope | 统一帧头、路由、追踪、ack、安全归属 | `protocol`、`kind`、`type`、`id`、`schema`、`scope`、`route`、`flags`、`payload` |
| `L4` | Domain Schemas | 领域级命令、事件、流式帧和信令 | `message.*`、`conversation.*`、`stream.*`、`rtc.*`、`notification.*`、`agent.*`、`device.*`、`ops.*` |
| `L5` | Governance | 能力协商、兼容矩阵、灰度与废弃治理 | `feature flags`、`capability negotiation`、`compatibility matrix`、`deprecation rules` |

## 6. L0-L5 的职责边界

### 6.1 L0 安全层

L0 不承载业务语义，只解决“信道是否可信”。

冻结结论：

- 公网与客户端流量强制 `TLS 1.3`
- 内部服务间流量按场景采用 `mTLS`
- 设备与边缘网关场景支持后续追加“载荷签名扩展”，但不取代 TLS

### 6.2 L1 传输绑定层

L1 负责让同一套 CCP 协议语义可以落在不同传输之上。

冻结结论：

- `ccp/http/1`：面向 REST 命令、短连接查询、上传初始化、补偿拉取
- `ccp/ws/1`：面向主实时链路，承担双向消息、流、ACK、RTC 信令
- `ccp/sse/1`：面向服务端单向流输出与受限网络环境的实时下行
- `ccp/mqtt/1`：面向 IoT / 边缘硬件接入
- `ccp/quic/1`：未来用于更低时延、多路复用和弱网优化

### 6.3 L2 握手与会话层

L2 解决“这条连接是谁、说什么版本、是否可恢复、是否可以继续传输”。

冻结结论：

- 所有长连接绑定都必须先完成 `hello -> auth_bind -> active`
- 恢复模型通过 `session_resume` 完成，而不是用业务命令模拟
- `heartbeat`、`goaway`、`error` 必须是统一控制帧，不允许每个 transport 自搞一套

### 6.4 L3 通用 envelope 层

L3 是整个协议族的真正核心。消息、流、RTC、Agent、Device 全部通过统一 envelope 传输。

冻结结论：

- 所有 domain payload 都必须放在统一 envelope 内
- envelope 稳定优先级高于具体 payload 稳定性
- trace、request、ack、route、scope 等元信息必须统一建模

### 6.5 L4 领域 schema 层

L4 承载真正的业务语义，但不允许随意破坏上层 envelope。

冻结结论：

- 领域 schema 按命名空间治理，不允许裸类型名扩散
- 新领域能力优先通过新增 schema / type 扩展，而不是破坏老 schema
- `stream.*` 是一等领域，不是 `message.*` 的附属字段

### 6.6 L5 治理层

L5 用于解决“如何升级而不打崩客户端”的问题。

冻结结论：

- 协议支持 capability negotiation
- 支持 feature flag 粒度的渐进发布
- 保留兼容矩阵与废弃周期，不允许直接“线上拍脑袋切版本”

## 7. CCP 与系统主体模型的关系

CCP 不是只服务“用户聊天”的协议，它要承载所有一等主体。

统一主体包括：

- `user`
- `agent`
- `device`
- `bot`
- `system`

统一能力对象包括：

- 普通消息
- 流式输出
- RTC 信令
- 协作事件
- AI 工具调用
- 设备遥测与命令
- 运维控制与诊断事件

这保证了未来无论是：

- 用户与用户单聊
- 用户与 Agent 对话
- 用户在群聊里收到设备消息
- Agent 对 IoT 设备下发命令

都不需要重新造第二套实时协议。

## 8. 当前阶段明确冻结的协议资产

本轮文档冻结后，以下内容视为主基线：

### 8.1 已冻结

- `CCP` 协议族命名
- 传输绑定命名规则
- 握手控制帧集合
- 通用 envelope 基本字段
- `kind` 分类体系
- 双版本治理方向
- 权威字段不信任客户端的安全原则
- 编码与传输解耦原则

### 8.2 暂不冻结到字段级

- 每一种业务消息 `payload` 的最终字段全集
- MQTT 最终 topic 模板的所有部署细节
- QUIC 绑定中的具体流分配策略
- 未来二进制 codec 的线格式

这是一种刻意的设计取舍：

- 先冻结骨架，避免架构反复摇摆
- 再让具体业务 schema 在骨架内安全迭代

## 9. 传输矩阵与推荐使用场景

| 传输 | 推荐角色 | 典型用途 | 是否主路径 |
| --- | --- | --- | --- |
| `HTTP` | 浏览器 / App / 服务端 | 命令提交、查询、上传初始化、补偿拉取 | 是 |
| `WebSocket` | 浏览器 / App / 桌面端 | 双向实时消息、ACK、流式输出、RTC 信令 | 是 |
| `SSE` | 浏览器 / 服务端 | 服务端向客户端持续推送、AI 结果输出 | 是 |
| `MQTT` | 设备 / 边缘节点 | 遥测上报、命令下发、低功耗实时通道 | 是 |
| `QUIC` | 未来高性能客户端 | 弱网优化、低延迟多路复用 | 预留 |

## 10. 第一阶段落地策略

### 10.1 先冻结，再实现

第一阶段的重点不是把所有 transport 一次性实现满，而是先统一口径：

- 所有公网 API 与实时链路都必须明确自己属于哪一种 `CCP binding`
- 所有命令、事件和流都必须能映射到统一 envelope
- 所有认证、恢复和权威字段都要服从同一套协议治理

### 10.2 编码先简单，语义先稳定

第一阶段推荐：

- 浏览器 / App 优先 `JSON`
- 设备 / 边缘节点支持 `JSON` 或 `CBOR`
- 暂不为了“看起来先进”过早引入复杂二进制协议

### 10.3 自主知识产权沉淀路线

优先沉淀：

- 自有协议命名空间
- 自有 envelope
- 自有 schema 体系
- 自有 sender / actor / route 语义
- 自有兼容治理规则

而不是优先沉淀：

- 自研 socket 协议
- 自研浏览器不兼容传输
- 自研复杂集群控制协议

## 11. 结论

`craw-chat` 的协议路线已经明确收敛为：

`标准传输 + 自有应用协议族 + 双版本治理 + 服务端权威字段边界 + 面向消息/流/AI/IoT 的统一协议骨架`

后续所有协议相关文档、网关 API、SDK 契约、连接运行时和设备接入方案，都必须以 `CCP` 为共同母体推进，不再接受“每个模块自己定义一套帧格式和安全规则”的做法。
