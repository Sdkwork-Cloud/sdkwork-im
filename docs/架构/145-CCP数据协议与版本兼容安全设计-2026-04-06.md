# CCP 数据协议与版本兼容安全设计

## 1. 文档定位

本文档用于冻结 `CCP` 的通用数据协议结构，重点回答：

- 帧如何统一封装
- 如何区分控制帧、命令帧、事件帧、流式帧和错误帧
- 协议版本与业务 schema 版本如何共存
- 哪些字段由客户端提供，哪些字段必须由服务端权威生成
- 如何在不破坏旧客户端的前提下持续演进

本文档与 `143`、`144` 一起构成协议层主基线。

## 2. CCP 通用 envelope

所有 CCP 帧都必须包裹在统一 envelope 中。

推荐结构如下：

```json
{
  "protocol": "ccp/1.0",
  "kind": "control",
  "type": "hello",
  "id": "fr_01H...",
  "ackId": null,
  "traceId": "tr_01H...",
  "requestId": "req_01H...",
  "ts": "2026-04-06T22:00:00Z",
  "schema": "cc.control.hello.v1",
  "scope": {
    "kind": "session",
    "id": "sess_xxx"
  },
  "route": {
    "sessionId": "s_xxx",
    "deviceId": "d_xxx",
    "routeEpoch": 12
  },
  "flags": [
    "replayable"
  ],
  "payload": {}
}
```

## 3. 字段语义

| 字段 | 作用 | 归属 |
| --- | --- | --- |
| `protocol` | CCP 协议版本 | 协商后固定 |
| `kind` | 帧大类 | 协议定义 |
| `type` | 具体帧类型 | 协议定义 |
| `id` | 当前帧唯一 ID | 发送方生成，服务端可二次校验 |
| `ackId` | 被确认帧 ID | 由 ACK 语义使用 |
| `traceId` | 分布式追踪 | 由接入层透传或补齐 |
| `requestId` | 请求关联 ID | 用于端到端诊断 |
| `ts` | 帧时间戳 | 客户端可提供，服务端需区分是否权威 |
| `schema` | payload schema 标识 | 协议定义 |
| `scope` | 当前帧所属范围 | 服务端按领域校验 |
| `route` | 当前投递路由信息 | 由路由面维护 |
| `flags` | 可选能力标记 | 协商后使用 |
| `payload` | 领域数据 | 由 schema 约束 |

## 4. `kind` 分类冻结

当前阶段冻结以下 `kind` 分类：

- `control`
- `cmd`
- `evt`
- `stream`
- `ack`
- `error`

### 4.1 `control`

用于连接、握手、恢复、保活、关闭等协议控制面行为。

典型 `type`：

- `hello`
- `hello_ack`
- `auth_bind`
- `auth_ok`
- `session_resume`
- `session_resumed`
- `heartbeat`
- `goaway`

### 4.2 `cmd`

用于客户端或上游发起“希望系统做一件事”的命令。

典型 `type`：

- `message.send`
- `conversation.create`
- `stream.open`
- `rtc.invite`
- `device.command`

### 4.3 `evt`

用于服务端或领域层对外发布“事情已经发生”的事实。

典型 `type`：

- `message.posted`
- `conversation.member_joined`
- `notification.dispatched`
- `device.telemetry_received`

### 4.4 `stream`

用于流式输出和增量帧传输。它是一等类型，不从属于 `message`。

典型 `type`：

- `stream.open`
- `stream.delta`
- `stream.patch`
- `stream.checkpoint`
- `stream.finalize`
- `stream.abort`

### 4.5 `ack`

用于协议级确认，而不是业务上“已读”的替代品。

典型用途：

- 确认命令已接收
- 确认窗口推进
- 确认特定帧或特定序号

### 4.6 `error`

用于协议级错误，不应和业务领域错误随意混用。

典型用途：

- 协议版本不支持
- 认证失败
- binding 与 envelope 不匹配
- routeEpoch 过期

## 5. Schema 命名规则

`schema` 字段必须显式版本化，推荐命名格式：

`cc.<domain>.<name>.v<major>`

示例：

- `cc.control.hello.v1`
- `cc.message.send.v1`
- `cc.message.posted.v1`
- `cc.stream.delta.v1`
- `cc.rtc.signal.v1`
- `cc.agent.tool_call.v1`
- `cc.device.telemetry.v1`

命名规则冻结如下：

- `cc` 是 `craw-chat` 的协议命名空间前缀
- `<domain>` 表示领域分区
- `<name>` 表示语义动作或事实
- `v<major>` 表示 schema 主版本

## 6. 双版本治理策略

`CCP` 采用双版本治理：

- 协议版本：控制 handshake、envelope、兼容规则
- schema 版本：控制具体 payload 结构

### 6.1 协议版本

推荐表达为：

- `ccp/1.0`
- 后续可扩展为 `ccp/1.1`
- 不兼容时进入 `ccp/2.x`

规则如下：

- `major` 不兼容变更必须拒绝并回退
- `minor` 兼容变更通过握手协商取双方共同最高值

### 6.2 Schema 版本

规则如下：

- 每个 schema 独立版本化
- 主版本变化通过新的 schema 名称表达，例如从 `cc.message.send.v1` 演进到 `cc.message.send.v2`
- 向后兼容的小扩展通过新增可选字段与 `feature_flag` 协商完成

### 6.3 兼容原则

- 客户端必须忽略未知的可选字段
- 服务端必须容忍旧客户端缺少新增的可选字段
- 不允许复用旧字段表达新语义
- 不允许在不改 schema 版本的情况下破坏字段含义

## 7. Capability 与 Feature Flag

新能力的推荐扩展路径是：

1. 新增 schema
2. 新增 `type`
3. 新增 `feature_flag`

而不是：

- 在旧 payload 中插入一组语义不清的新字段
- 让客户端猜测当前服务端是否支持某行为

示例能力：

- `stream.patch`
- `payload.cbor`
- `device.signature`
- `agent.tool_result`
- `route.fencing`

## 8. 服务端权威字段边界

这是协议安全设计的核心基线。

客户端请求体不得作为以下字段的权威来源：

- `tenantId`
- `actorId`
- `actorKind`
- `sender`
- `routeEpoch`
- `serverTs`

这些字段必须来自：

- `JWT`
- 会话上下文
- 设备绑定关系
- 成员关系
- 路由归属关系

### 8.1 为什么 `sender` 不能信任客户端

即时通讯里，发送者身份决定：

- 谁在发消息
- 是否具备该会话成员资格
- 是否允许使用该能力
- 审计日志如何归属

因此客户端只能表达“我想发什么”，不能表达“我是以谁的身份发的”。

`sender` 必须由服务端根据认证上下文、成员关系和当前连接状态补齐。

### 8.2 与既有 API 的一致性

这与已经冻结的 API 原则完全一致：

- 请求体不再显式提交 `tenantId`
- 平铺 `senderId` 已被 `sender` 结构替代
- `actor` 与 `sender` 都服从服务端推导

## 9. Auth 模式

当前阶段冻结以下认证模式：

- `Bearer token`
- `session ticket / resume ticket`
- `device credential`
- `device signature`

适用关系如下：

| 模式 | 主要场景 |
| --- | --- |
| `Bearer token` | App、桌面端、浏览器主路径 |
| `session ticket / resume ticket` | 长连接恢复、短时重连 |
| `device credential` | IoT 设备注册与长期接入 |
| `device signature` | 设备高信任场景、边缘网关场景 |

## 10. 重放、防重与 fencing

协议层必须内建最小防重放和幂等约束。

### 10.1 帧 ID

- 每个 `cmd` 帧都必须有唯一 `id`
- 同一连接或同一恢复窗口内不允许重复消费同一命令 ID

### 10.2 幂等键

写请求建议带 `idempotency key`，用于：

- 断线重试
- HTTP 超时重发
- 设备弱网重复上报

### 10.3 Resume Token

恢复必须使用 `resume token`，不能只靠 `sessionId` 猜恢复资格。

### 10.4 Route Epoch

`routeEpoch` 用于连接归属 fencing：

- 老连接拿着过期 epoch 继续写时，必须被拒绝
- 节点迁移后，旧节点不能继续宣称自己持有路由

## 11. Codec 策略

协议语义固定，物理编码可协商。

### 11.1 第一阶段推荐

- 浏览器：`JSON`
- App / Native：`JSON`
- 设备 / IoT：`JSON` 或 `CBOR`

### 11.2 后续扩展

后续可加入：

- `Protobuf`
- 自定义二进制 codec

但扩展规则必须是：

- 先保留相同的 CCP 语义
- 再增加新的 codec binding

即：

- `JSON` 不是协议
- `CBOR` 不是协议
- 真正的协议是 `CCP envelope + schema + compatibility rules`

## 12. 推荐内容类型

第一阶段推荐：

- `application/ccp+json`
- `application/ccp+cbor`

其中：

- `HTTP` 以内容类型表达 codec
- `WebSocket / SSE / MQTT` 通过握手协商 codec

## 13. 错误模型

协议错误建议保留稳定的错误族，而不是直接把内部异常透给客户端。

推荐错误族：

- `CCP_UNSUPPORTED_PROTOCOL`
- `CCP_UNSUPPORTED_CODEC`
- `CCP_AUTH_REQUIRED`
- `CCP_AUTH_FAILED`
- `CCP_RESUME_REJECTED`
- `CCP_ROUTE_EPOCH_STALE`
- `CCP_CAPABILITY_NOT_NEGOTIATED`
- `CCP_SCHEMA_INCOMPATIBLE`

业务错误仍由领域层返回业务码，但外层 envelope 必须明确其属于 `error` kind。

## 14. 示例

### 14.1 消息发送命令

```json
{
  "protocol": "ccp/1.0",
  "kind": "cmd",
  "type": "message.send",
  "id": "fr_01HMESSAGE",
  "traceId": "tr_01H123",
  "requestId": "req_01H123",
  "ts": "2026-04-06T22:05:00Z",
  "schema": "cc.message.send.v1",
  "scope": {
    "kind": "conversation",
    "id": "c_xxx"
  },
  "route": {
    "sessionId": "s_xxx",
    "deviceId": "d_xxx",
    "routeEpoch": 18
  },
  "flags": [
    "idempotent"
  ],
  "payload": {
    "parts": [
      {
        "kind": "text",
        "text": "hello"
      }
    ]
  }
}
```

说明：

- `tenantId` 不在客户端 payload 中出现
- `sender` 不由客户端显式提供
- 这些信息由网关在认证完成后从上下文推导

```json
  },

  }

```

## 15. 演进与废弃规则

推荐治理周期如下：

- 新能力先通过 `feature_flag` 灰度
- 能力稳定后进入默认协商集
- 废弃旧 schema 时至少跨一个稳定版本周期
- 服务端必须在观测面统计旧版本使用率后再移除支持

## 16. 结论

`CCP` 的数据协议设计已经明确：

- 使用统一 envelope
- 使用 `control / cmd / evt / stream / ack / error` 六类帧
- 使用协议版本与 schema 版本双轨治理
- 使用服务端权威字段边界保障安全
- 使用编码协商而非编码绑定协议本身

后续 SDK、网关、消息投递、流式对话、设备接入和 AI 扩展都必须在这套数据协议基线内迭代。
