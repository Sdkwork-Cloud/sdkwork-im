# 21-跨节点实时路由与设备连接归属标准

## 1. 目标

本标准定义多节点集群下 realtime 下行的最小正确性边界，解决以下核心问题：

- 设备连接挂在节点 B，但业务写路径发生在节点 A 时，如何把事件送到正确节点
- HTTP 轮询、WebSocket、未来自定义长连接如何共享同一套设备归属模型
- 多节点下如何避免“每个节点都广播一遍”的粗放 fanout
- 后续如何把本地最小实现演进到 MQ bridge / shared route store，而不推翻协议层

本标准只解决在线下行的跨节点路由，不解决 durable 历史补偿。历史补偿仍走：

1. `session.resume`
2. `sync-feed`
3. timeline / projection / stream frame 补拉

## 2. 设计选型

### 2.1 备选方案

方案 A：全局广播再由各节点本地过滤  
优点：实现简单。  
缺点：租户规模和节点数一大，带宽与 CPU 会被无效 fanout 吃掉，不适合商业化集群。

方案 B：按设备连接归属做定向桥接  
优点：只把事件送到设备连接所在节点，最符合 IM 数据面的扩展方向。  
缺点：需要维护设备归属目录。  
结论：本标准采用该方案。

方案 C：按 tenant 或 principal 建共享实时总线  
优点：比全局广播更好。  
缺点：粒度仍过粗，热点租户或热点主体会造成无效消费。  
结论：可作为桥接底层实现候选，不作为上层语义标准。

## 3. 核心原则

### 3.1 设备是 realtime 下行的最小路由单元

realtime event window 仍然按：

- `tenantId`
- `principalId`
- `deviceId`

建模。  
跨节点路由也必须精确到这一级，不能退化为“给用户发”或“给租户发”。

### 3.2 设备归属与订阅分离

设备归属回答的是：

- 这个设备当前连接在哪个节点

订阅回答的是：

- 这个设备愿意接收哪些 scope / event type

二者必须独立：

- 路由层先找到 owner node
- owner node 再按本地订阅过滤决定是否写入 realtime window

### 3.3 latest bind wins

同一设备在同一时刻只允许有一个逻辑 owner node。  
如果同一设备随后又通过其他入口绑定到新节点，则采用最新绑定覆盖旧归属。

这条规则是当前最小实现的稳定边界。更复杂的多连接并发治理留到后续版本。

## 4. 数据模型

### 4.1 Device Route

设备归属目录至少包含：

```json
{
  "tenantId": "t_demo",
  "principalId": "u_demo",
  "deviceId": "d_pad",
  "ownerNodeId": "node_b",
  "connectionKind": "websocket",
  "boundAt": "2026-04-05T11:20:00Z"
}
```

字段含义：

- `ownerNodeId`: 当前负责该设备在线 realtime 下行的节点
- `connectionKind`: 当前绑定来源，例如 `websocket`、`http_poll`、`http`
- `boundAt`: 最近一次归属刷新时间

### 4.2 Route Publish Result

路由层发布结果至少要能表达：

- 实际命中的目标节点
- 是否命中显式 route
- 是否退化到 origin node fallback
- 实际写入了多少事件

这类结果主要用于测试与运维观测，不强制暴露为公网 API。

## 5. 路由流程

### 5.1 设备绑定

当设备通过以下任一路径与某节点建立实时会话关系时，节点必须刷新设备归属：

- `realtime/ws`
- `realtime/subscriptions/sync`
- `realtime/events`
- `realtime/events/ack`
- `sessions/resume`
- `devices/register`

### 5.2 事件发布

业务节点发布 realtime 事件时按以下顺序执行：

1. 根据 `tenantId + principalId + deviceId` 查询设备归属
2. 若找到 owner node，则把事件桥接到 owner node 的 realtime runtime
3. owner node 在本地按设备订阅过滤
4. 命中订阅后写入该设备 realtime window
5. WebSocket / HTTP 轮询继续复用 owner node 本地窗口进行下行

### 5.3 fallback 规则

如果未找到设备归属，则允许最小实现退化为：

- 尝试投递到 origin node 本地 runtime

目的：

- 保持单节点 profile 与现有行为兼容
- 避免 route store 暂时空缺时完全没有在线投递

但要明确：

- fallback 不代表跨节点一定成功
- route 缺失下的可靠恢复仍必须靠 durable 路径

## 6. 与 ack / trim 的关系

ack 仍然只发生在 owner node 的设备窗口上。  
因此：

- `ackedThroughSeq`
- `trimmedThroughSeq`
- `latestRealtimeSeq`

都以 owner node 上的那份设备窗口为准。

路由层不解释 ack 语义，也不存储 ack 本体；它只负责把设备请求和业务事件导向正确节点。

## 7. 与 WebSocket / HTTP 的关系

WebSocket 与 HTTP 轮询都必须绑定到同一设备归属模型：

- WebSocket 建连会刷新 route
- `GET /realtime/events` 也会刷新 route
- `POST /realtime/events/ack` 也会刷新 route

这意味着 transport 可以切换，但设备 owner 的抽象不变。

## 8. 本地最小实现边界

当前阶段采用内存内共享 bridge 落地，标准化边界如下：

- route directory 为进程内共享状态
- node runtime registry 为进程内共享状态
- bridge 精确按 `deviceId` 做定向投递
- owner node 的 `RealtimeDeliveryRuntime` 仍然是真正写窗口的地方

这不是最终集群实现，但接口边界已经固定，后续可以替换为：

- MQ bridge
- shared metadata store
- cell 内 route directory service
- sidecar / gateway mesh

而不需要改业务事件模型和客户端协议。

## 9. 最小落地要求

一期代码必须满足：

1. 单节点行为不回归
2. 两个节点共享 route bridge 时，节点 A 产生的业务 realtime 事件可以写入节点 B 上已绑定设备的窗口
3. owner node 的 HTTP `realtime/events` 能读取到远端桥接过来的事件
4. WebSocket 继续绑定 owner node 本地窗口，不单独维护第二套缓冲

## 10. 后续演进

下一阶段应继续推进：

- route directory 持久化
- route lease / epoch / fencing
- 节点 drain 时的 route 迁移
- ack checkpoint 持久化
- route ownership view 接入 `ops/cluster` 与 diagnostics
- bridge 背压、队列深度、慢节点熔断与丢弃策略
