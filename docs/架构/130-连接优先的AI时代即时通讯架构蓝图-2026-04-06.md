## 2. 产品目标与定位

`craw-chat` 的目标不是做“又一个普通 IM 服务”，而是构建一套：

- 消费级消息内核标准
- 企业协作与知识沉淀能力
- AI / Agent 原生流式交互能力
- IoT 与智能硬件接入能力
- 高连接密度与分层弹性扩容能力
- 具备自主知识产权与商业可控性的即时通讯底座

系统的最终定位是：

`消费级消息内核 + 企业协作上下文 + AI/Agent 扩展平台 + 智能硬件连接底座`

这意味着系统必须同时满足以下目标：

- 在消息体验上接近 Telegram / WhatsApp / WeChat 这类头部 IM 的标准。
- 在协作承载能力上具备接近 Lark / Feishu / Zulip 的上下文化能力。
- 在扩展能力上具备接近 OpenIM / Matrix 的平台化潜力。
- 在连接、路由、消息热路径设计上吸收 WuKongIM 的高性能内核思路。

## 4. 核心设计原则

### 4.1 连接优先

即时通讯系统最重要的能力之一是：

- 单机连接数
- 单机并发处理能力
- 弹性扩容能力
- 连接生命周期治理能力

因此系统必须首先设计 `连接管理层`，而不能把连接能力作为业务 API 的附属物。

### 4.2 消息与流并列

系统必须把 `Message` 与 `Stream` 作为两个一等公民，而不是把流式输出降级为普通消息的一个字段。

### 4.3 先做抽象，再换底层

系统的核心知识产权应放在：

- 协议模型
- 事件模型
- 消息模型
- 流模型
- 路由模型
- 存储抽象

而不是第一阶段就自建通用数据库。

### 4.4 分层扩容

不同压力面必须能够分开扩容，而不是所有节点同时承担所有能力。

### 4.5 协作与 AI 原生预留

AI / Agent / Workflow / Card / Bot / IoT 必须从第一天预留扩展位，而不是后续外挂。

### 4.6 代码结构治理

不允许通过单个超大 `lib.rs` 或少数几个巨型文件来承载系统复杂度，必须通过模块化、边界化、目录化治理复杂度。

## 5. 总体架构

系统采用 `六个 plane + 两类横切层` 的总体模型。

### 5.1 六个 plane

#### Link Plane

负责：

- WebSocket / HTTP / SSE / MQTT / QUIC 接入
- `CCP` 协议绑定、握手与版本协商
- 连接认证
- 心跳与保活
- 背压治理
- 慢连接治理
- resume / reconnect
- 连接分片与出站发送队列

#### Route Plane

负责：

- principal / device / session / node 的在线归属
- 路由查找
- 会话 fencing
- 节点 drain
- 重平衡
- failover 接管

#### Messaging Plane

负责：

- 会话
- 成员
- 消息
- 已读
- 群聊
- 单聊
- 频道
- 广播
- 离线补偿
- 多设备同步

#### Stream / AI Plane

负责：

- 流式消息
- LLM stream 输出
- 实时转写
- 长任务进度流
- 结构化 patch 流
- 设备 telemetry 流
- Agent 工具调用过程流

#### Projection Plane

负责：

- timeline
- inbox
- conversation summary
- unread
- topic/thread 索引
- AI 上下文视图
- 搜索与知识视图

#### Storage Plane

负责：

- 元数据
- 缓存
- 对象存储
- 消息日志
- 流式帧存储
- 冷热分层

### 5.2 横切层

#### Control Plane

负责：

- 租户
- 身份
- 策略
- 配额
- 能力开关
- 路由策略
- 版本治理

#### Ops Plane

负责：

- 观测
- 诊断
- 备份恢复
- 运维控制
- 扩缩容
- 排空
- 升级

## 6. 连接管理层设计

连接管理层是本系统的核心能力层之一，目标是实现：

- 高连接密度
- 低内存占用
- 高并发处理
- 低尾延迟
- 灵活扩容
- 路由迁移可控

### 6.1 连接运行时组件

建议形成独立运行时：

- `acceptor`
- `handshake pipeline`
- `auth pipeline`
- `connection registry`
- `shard dispatcher`
- `outbound queue`
- `heartbeat manager`
- `resume window manager`
- `frame codec`
- `backpressure controller`
- `connection metrics`

### 6.2 连接治理策略

- 一个连接只归属于一个本机 shard
- 避免全局锁
- 每连接独立发送队列
- 慢消费者单独治理
- 有界缓存，不允许无限堆积
- 控制帧、ACK、RTC 信令优先于普通消息
- 连接层尽量轻状态，不承载重业务判断

### 6.3 单机目标

建议采用双层目标：

- 第一落地目标：`10 万级长连接 / 单机`
- 设计上限目标：`50 万级以上长连接 / 单机`

前者用于当前实现基线，后者用于架构保留空间与运行时设计约束。

### 6.4 分层扩容

连接层必须支持独立扩容：

- 增加 Link Plane 节点，不要求同步扩 Metadata / Projection
- Route Plane 可独立扩容，用于支撑更多设备与会话归属
- 消息、流、投影、AI 节点按压力面单独扩容

## 7. 消息、流、RTC 与协作模型

### 7.1 主体模型

系统统一主体：

- `user`
- `agent`
- `device`
- `system`
- `bot`

统一使用可扩展的 `sender` / `actor` 结构，不再依赖单一 ID 字段。

### 7.2 消息模型

消息负责离散语义：

- 文本
- 富文本
- 媒体
- 卡片
- 命令
- 系统事件
- 协作挂件引用

### 7.3 流模型

流负责渐进式语义：

- `stream.open`
- `stream.delta`
- `stream.patch`
- `stream.checkpoint`
- `stream.finalize`
- `stream.abort`

统一字段建议包括：

- `stream_id`
- `stream_type`
- `scope_kind`
- `scope_id`
- `sender`
- `schema_ref`
- `frame_seq`
- `resume_token`
- `ack_policy`
- `materialize_target`

### 7.4 适用场景

流模型必须可直接承载：

- LLM 对话 token streaming
- 实时 AI 回复
- 实时语音转文本
- 长任务进度
- Agent 多阶段输出
- 设备连续数据
- 游戏或应用自定义数据流

### 7.5 RTC 模型

RTC 只进入信令层，不把媒体面塞进 IM 内核。

RTC 核心职责：

- session create / invite / accept / reject / close
- signal relay
- conversation 绑定
- artifact / recording 回流消息体系

### 7.6 协作上下文

系统必须支持：

- topic
- thread
- message card
- document reference
- task reference
- workflow reference
- AI summary
- knowledge context anchor

## 8. AI / Agent 架构

AI / Agent 不是外挂服务，而是系统中的一类主体与能力域。

### 8.1 核心能力

- agent 作为一等主体加入会话
- agent 输出支持流式消息
- tool call / tool result 可作为结构化事件
- AI 结果可物化为消息、卡片、文件、知识片段
- AI 上下文可挂在 conversation / topic / thread / document

### 8.2 约束原则

- AI 不侵入消息内核协议基础语义
- AI 能力通过扩展协议、事件类型、卡片和流模型接入
- AI 不得破坏普通 IM 体验和热路径稳定性

## 9. IoT 与智能硬件接入架构

系统未来必须支持“小智类”智能硬件接入，因此设备必须是一等主体。

### 9.1 协议接入

- MQTT
- WebSocket
- HTTP / SSE
- 后续预留 QUIC / WebRTC DataChannel

### 9.2 设备能力

- device session
- device presence
- uplink stream
- command stream
- device twin
- user / agent / device 会话绑定

### 9.3 目标场景

- 智能音箱
- 陪伴机器人
- 车载终端
- 可穿戴设备
- 工业终端
- 智能摄像头与家庭网关

## 10. 存储架构与演进路线

### 10.1 第一阶段推荐

- `PostgreSQL`
- `Redis`
- `S3-compatible Object Storage`

这是当前最合理的商业化与可控平衡点。

### 10.2 为什么不建议第一阶段全自建数据库

- 成本过高
- 正确性复杂度极大
- 会拖慢产品落地
- 不符合“先做核心知识产权，再逐步替换底层”的策略

### 10.3 建议自研的层

建议自研：

- `MessageLog` 抽象
- `StreamStore` 抽象
- `RouteStore` 抽象
- `ProjectionRebuild` 机制
- 统一 frame / event 协议

### 10.4 三阶段演进

#### 第一阶段

- `PostgreSQL + Redis + Object Storage`
- 自有协议模型、自有消息/流抽象、自有路由抽象

#### 第二阶段

- 引入自研 `MessageLog`
- 引入自研 `StreamStore`
- 底层仍可落到 PostgreSQL / 本地段文件 / 对象存储

#### 第三阶段

- 如规模确实需要，再将最热路径演进到自研段文件、副本、压缩和冷热分层

## 11. 技术选型建议

### 11.1 服务端语言与核心框架

- Rust
- Tokio
- Axum
- Serde
- Tracing
- OpenTelemetry

### 11.2 连接与协议

- 保留标准传输：`HTTP / WebSocket / SSE / MQTT`，后续演进 `QUIC`
- 在标准传输之上统一定义 `CCP = Craw Chat Protocol`
- `ccp/ws/1` 作为主实时链路绑定
- `ccp/http/1` 作为命令、查询与补偿拉取绑定
- `ccp/sse/1` 作为单向流输出绑定
- `ccp/mqtt/1` 作为 IoT 设备接入绑定
- 统一冻结 `hello / auth_bind / session_resume / heartbeat / goaway / error` 控制帧
- 统一冻结 envelope、schema、capability negotiation 与兼容规则

### 11.3 存储与缓存

- PostgreSQL
- Redis
- S3-compatible Object Storage

### 11.4 观测

- Prometheus
- OpenTelemetry
- Loki-compatible logs

### 11.5 不建议第一阶段强依赖

- Kafka
- etcd
- 重型 RPC 网格
- 自研通用数据库

这些能力应通过抽象层预留，不作为第一阶段前置依赖。

## 13. 代码结构治理标准

### 13.1 硬约束

- `lib.rs` 只做模块声明、导出和极少量 facade
- 单文件不超过 `1000` 行
- 推荐单文件控制在 `200-500` 行
- 超过 `700` 行视为预警
- 路由、领域、存储、DTO、运行时不得混写在一个文件

### 13.2 模块拆分原则

推荐目录化拆分：

- `aggregate/`
- `command/`
- `event/`
- `policy/`
- `service/`
- `handler/`
- `router/`
- `runtime/`
- `port/`
- `adapter/`

### 13.3 CI 约束

后续必须加入：

- `cargo fmt --check`
- `cargo clippy --all-targets --all-features -D warnings`
- 自定义 `file-length-check`
- 轻量 `module-boundary-check`

## 16. 商业可控与自主知识产权策略

系统必须优先拥有以下自主资产：

- 自有协议标准
- 自有 `CCP` 应用协议族
- 自有协议注册表与兼容矩阵
- 自有协议发布、灰度与回滚治理模型
- 自有消息与流模型
- 自有路由与连接模型
- 自有存储抽象
- 自有 SDK 契约
- 自有 AI / Agent / IoT 接入规范

基础组件可以使用成熟开源实现，但系统架构必须保证：

- 可插拔
- 可替换
- 可自研逐步替换
- 不因单一组件而失去商业主动权

## 17. 结论

`craw-chat` 的推荐终局不是传统聊天后端，而是：

`连接优先、流式原生、消息稳定、协作可挂载、AI 可对话、设备可接入、组件可替换、商业可控的下一代即时通讯平台`

后续所有架构设计、代码拆分、存储演进、部署与扩容策略，均应以本文档为总纲推进。
