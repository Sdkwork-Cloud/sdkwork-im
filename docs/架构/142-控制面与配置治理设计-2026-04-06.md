# 控制面与配置治理设计

## 1. 文档目标

本文档用于定义 `craw-chat` 的控制面职责、配置分层和变更治理规则，确保系统在支持多租户、分层扩容、私有化部署和复杂能力开关后，仍然能够：

- 清楚区分控制面与数据面
- 安全地下发策略和配置
- 支持能力开关、配额和租户治理
- 避免配置漂移和运行期混乱

## 2. 控制面总原则

- 控制面负责“策略、治理、编排、配置”，不负责热路径数据提交。
- 控制面强事务不得进入消息与流的同步主链。
- 控制面写操作必须具备独立授权、审计和回放能力。
- 运行时只消费已冻结或已验证的控制面结果。

## 3. 控制面职责边界

控制面至少负责：

- tenant lifecycle
- capability enablement
- protocol registry management
- protocol release channel
- provider registry management
- provider selection policy
- principal profile policy
- rollout policy
- quota 管理
- route / drain / rebalance 策略
- deployment profile
- feature flags
- operator action
- compliance policy

不负责：

- 消息提交
- stream frame append
- timeline 直接写入
- 连接热路径状态持久化

## 4. 配置分层模型

建议把配置分为四层：

### 4.1 Build-time Config

用于：

- 编译特性
- profile 组合
- 条件编译

### 4.2 Deploy-time Config

用于：

- 节点端口
- 外部依赖地址
- 存储连接
- 日志级别

### 4.3 Runtime Control Config

用于：

- feature flags
- protocol profile
- provider profile
- principal profile profile
- quota
- routing strategy
- drain policy
- capability profile

### 4.4 Tenant Policy Config

用于：

- 租户级能力开关
- 租户级 provider 选择
- 租户级用户模块形态选择
- 租户级 retention policy
- 租户级安全与合规策略

## 5. 配置优先级建议

建议优先级从高到低为：

1. 运行时紧急控制配置
2. tenant policy
3. deploy-time config
4. build-time default

但必须保证：

- 同一配置项只有一个权威写入口
- 紧急 override 必须有 TTL 或显式清理机制
- 任何 override 都必须可审计

## 6. Feature Flag 设计

feature flag 应至少支持以下类型：

- 全局开关
- profile 开关
- tenant 开关
- capability 开关
- 灰度开关
- protocol channel 开关

应用场景包括：

- 新 stream 类型灰度
- AI capability 灰度
- IoT 协议接入灰度
- RTC provider 灰度切换
- 小智协议灰度放量
- 高风险治理动作开关

## 7. Quota 与策略治理

控制面应能够下发并管理：

- 连接数配额
- 消息 TPS 配额
- stream 吞吐配额
- 媒体容量配额
- AI token / agent invoke 配额
- IoT telemetry 频率限制

## 8. Route / Drain / Rebalance 编排

控制面必须显式负责：

- 节点标记为 `active / draining / drained`
- route migration 编排
- rebalance 策略下发
- operator action 审计
- provider drain / freeze / rollback 编排

对于协议治理，控制面还必须负责：

- protocol profile 发布
- cell / region rollout 编排
- kill switch 下发
- private bundle 导出

但运行期具体迁移动作仍由 `Route Plane` 与 `Runtime` 执行。

## 9. 私有化与本地配置治理

私有化和本地模式必须支持：

- 模板化配置初始化
- `bin/init-config-*` 自动生成基础配置
- 环境变量与配置文件并存
- 密钥和敏感参数不直接写死在脚本中

## 10. 配置变更安全要求

- 任何控制面写操作都必须走独立授权模型
- 配置变更必须有审计记录
- 高风险配置变更应支持 dry-run 或 preview
- 对影响连接层、route 层和安全层的变更应支持分阶段发布
- 对协议 registry、close contract、capability matrix 的变更必须支持预览与回滚

## 11. 配置漂移治理

系统应具备：

- 运行时配置摘要导出
- 节点实际配置与期望配置对比
- 配置版本号
- 配置变更时间线
- protocol profile 与 effective snapshot 对比
- provider profile 与 effective provider snapshot 对比
- cell / tenant 级协议配置漂移检测

## 12. 与 CCP 协议治理的关系

从 `CCP` 角度看，控制面必须承接：

- `146` 的 registry / compatibility matrix
- `148` 的 protocol profile / release channel / rollout policy
- `149` 的 cell / region 升级与回滚策略

运行时只能消费控制面输出的 `effective protocol snapshot`，而不能在热路径直接拼协议配置。

## 13. 结论

`craw-chat` 的控制面不是附属后台，而是保证多租户治理、能力开关、扩缩容和私有化可交付的关键层。只有把控制面与配置治理单独冻结，整个架构才真正具备长期可维护和可运营能力。

## 2026-04-09 增补：控制面标准、行业校准与当前实现边界

### A. 当前实现真相

- 当前已存在 `control-plane-api` 的只读 registry / governance / provider snapshot 入口，以及部分 runtime 对 `effective snapshot` 的真实消费。
- 当前仍未形成完整的 durable truth 控制面产品：`tenant override`、`release bundle`、异步传播、全量审计与多 region 运维编排仍在后续阶段。
- 因此前文不能被理解为“控制面已完整兑现”，文末 `As-Built` 才是当前实现证据。

### B. 行业校准规则

- 对标 `Slack / Teams / Discord`，管理员策略、工作区/团队/服务器级治理、发布开关与外部协作开关都应属于 control plane，而不是 runtime 热路径临时判断。
- runtime 只消费已计算好的 `effective policy / effective snapshot`，不在热链路临时拼配置。
- 私有化、SaaS、多 cell 交付必须共享同一套治理模型，只允许交付形态不同，不允许治理语义分叉。

### C. 本文哪些是目标态

- 第 2-13 节中的职责边界、配置分层、quota、route/drain/rebalance、配置漂移治理，属于目标控制面标准。
- 这些章节定义的是“商用 IM 控制面应长成什么样”，不等于当前仓库已经具备完整运营后台与配置真源。

### D. 文档口径规则

- 写当前控制面能力时，以本文 `As-Built`、`148`、`152CJ` 与 `services/control-plane-api` 的真实 API / 测试为准。
- 写未来控制面能力时，必须显式标记为 `目标态` 或 `标准目标面`。
- 若未形成 durable truth 与写接口闭环，不得写成“已具备完整控制面治理”。
