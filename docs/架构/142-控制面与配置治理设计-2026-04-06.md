# 控制面与配置治理设计

## 1. 文档目标

本文档用于定义 `craw-chat` 的控制面职责、配置分层和变更治理规则，确保系统在支持多租户、分层扩容、私有化部署和复杂能力开关后，仍然能够：

- 清楚区分控制面与数据面
- 安全地下发策略和配置
- 支持能力开关、配额和租户治理
- 避免配置漂移和运行期混乱

## 2. 控制面总原则

- 控制面负责“策略、治理、编排、配置”，不负责热路径数据提交。
- 控制面强事务不得进入消息与流的同步主链。
- 控制面写操作必须具备独立授权、审计和回放能力。
- 运行时只消费已冻结或已验证的控制面结果。

## 3. 控制面职责边界

控制面至少负责：

- tenant lifecycle
- capability enablement
- quota 管理
- route / drain / rebalance 策略
- deployment profile
- feature flags
- operator action
- compliance policy

不负责：

- 消息提交
- stream frame append
- timeline 直接写入
- 连接热路径状态持久化

## 4. 配置分层模型

建议把配置分为四层：

### 4.1 Build-time Config

用于：

- 编译特性
- profile 组合
- 条件编译

### 4.2 Deploy-time Config

用于：

- 节点端口
- 外部依赖地址
- 存储连接
- 日志级别

### 4.3 Runtime Control Config

用于：

- feature flags
- quota
- routing strategy
- drain policy
- capability profile

### 4.4 Tenant Policy Config

用于：

- 租户级能力开关
- 租户级 retention policy
- 租户级安全与合规策略

## 5. 配置优先级建议

建议优先级从高到低为：

1. 运行时紧急控制配置
2. tenant policy
3. deploy-time config
4. build-time default

但必须保证：

- 同一配置项只有一个权威写入口
- 紧急 override 必须有 TTL 或显式清理机制
- 任何 override 都必须可审计

## 6. Feature Flag 设计

feature flag 应至少支持以下类型：

- 全局开关
- profile 开关
- tenant 开关
- capability 开关
- 灰度开关

应用场景包括：

- 新 stream 类型灰度
- AI capability 灰度
- IoT 协议接入灰度
- 高风险治理动作开关

## 7. Quota 与策略治理

控制面应能够下发并管理：

- 连接数配额
- 消息 TPS 配额
- stream 吞吐配额
- 媒体容量配额
- AI token / agent invoke 配额
- IoT telemetry 频率限制

## 8. Route / Drain / Rebalance 编排

控制面必须显式负责：

- 节点标记为 `active / draining / drained`
- route migration 编排
- rebalance 策略下发
- operator action 审计

但运行期具体迁移动作仍由 `Route Plane` 与 `Runtime` 执行。

## 9. 私有化与本地配置治理

私有化和本地模式必须支持：

- 模板化配置初始化
- `bin/init-config-*` 自动生成基础配置
- 环境变量与配置文件并存
- 密钥和敏感参数不直接写死在脚本中

## 10. 配置变更安全要求

- 任何控制面写操作都必须走独立授权模型
- 配置变更必须有审计记录
- 高风险配置变更应支持 dry-run 或 preview
- 对影响连接层、route 层和安全层的变更应支持分阶段发布

## 11. 配置漂移治理

系统应具备：

- 运行时配置摘要导出
- 节点实际配置与期望配置对比
- 配置版本号
- 配置变更时间线

## 12. 结论

`craw-chat` 的控制面不是附属后台，而是保证多租户治理、能力开关、扩缩容和私有化可交付的关键层。只有把控制面与配置治理单独冻结，整个架构才真正具备长期可维护和可运营能力。
