# 产品设计与需求范围

## 1. 产品定位

`craw-chat` 定位为一套面向 `SaaS` 与 `私有化` 双交付模式的统一通信内核产品，面向以下场景：

- 企业协同通信
- 平台型 IM
- Bot 与自动化平台
- 应用内嵌消息中心
- RTC 信令协作
- 用户可定义的结构化数据流传输
- AI 时代的多模态消息承载

产品不是单纯的聊天服务器，而是统一承载：

- 离散消息
- 流式消息
- 文件资源引用
- 结构化数据
- 事件通知
- RTC 信令
- 自动化扩展

## 2. 目标用户

- SaaS 平台租户
- 企业私有化客户
- 平台开发者
- App 内嵌聊天能力的业务团队
- 机器人与工作流接入方

## 3. 产品目标

### 3.1 业务目标

- 支持单聊、群聊、频道、系统消息
- 支持多端同步与离线漫游
- 支持通知、机器人、Webhook、工作流
- 支持用户自定义数据流
- 支持 RTC 信令
- 支持图片、视频、音频、文件等资源引用
- 支持流式消息发送与接收

### 3.2 非功能目标

- 一期支持 `50 万在线`
- 架构边界预留到 `500 万在线`
- 支持在线扩容与节点平滑下线
- 支持多租户治理
- 支持强审计、强安全、强隔离
- 支持 `SaaS / 私有化 / Air-Gapped` 差异化交付

## 4. 一期能力边界

### 4.1 一期必须支持

- 账号与设备接入
- 单聊、群聊、频道
- 消息发送、编辑、撤回
- 已读、送达、多端同步
- 文件资源消息
- 流式消息
- 用户自定义结构化数据消息
- 基础通知
- RTC 信令
- Bot / Webhook 基础扩展
- 审计与可观测

### 4.2 二期增强

- 大群专项优化
- 搜索与归档
- 多 Region 主动治理
- 高级工作流平台
- 细粒度风控与内容治理
- AI 扩展能力库

## 5. 核心设计原则

- 消息语义优先于基础设施实现。
- 会话内顺序优先于跨会话全局顺序。
- 投影可丢失可重建，提交真相不可模糊。
- 流与消息要统一但不混淆。
- 文件资源与消息体分层表达。
- RTC 只共享信令面，不侵入媒体面。
- AI 只是一种上层生产者，不绑架底层架构。
