# 152CJ Loop78 Addendum - 2026-04-11

## 1. As-Built Update
- The next-write retry failure persistence path is now stale-aware.
- When a stale claimed pending request is retried and fails again, the backlog rewrite clears stale owner metadata before persisting the failed item.
- This stale reclaim behavior now aligns three failure persistence seams:
  - runtime-level pending backlog persistence
  - operator repair retry failure
  - targeted republish retry failure

## 2. Architectural Meaning
- Loop77 made next-write auto-retry respect active ownership at queue time.
- Loop78 closes the remaining consistency hole at write-back time:
  - queue preparation could already allow stale work back into retry
  - but failure persistence could still preserve obsolete owner metadata
- After Loop78, stale owner metadata is no longer able to survive a failed retry round just because the retry came from the next-write path instead of explicit repair.

## 3. Deferred Architecture Work
- automatic stale detection / timeout reclaim / scheduler
- release-ready exactly-once delivery semantics for shared-channel sync
- richer operator-visible stale-age / SLA policy beyond the current lease-status surface
