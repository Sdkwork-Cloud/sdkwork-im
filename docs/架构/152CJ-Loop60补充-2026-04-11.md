# 152CJ Loop60 补充 - 2026-04-11

## 1. 本轮唯一架构增量
- 把 `shared-channel sync` 从“dead-letter 只有全量回灌、缺少可读 inventory 与定点恢复”推进到“operator 可读取 dead-letter inventory，并按 `requestKey` 定点 requeue”的恢复语义

## 2. As-Built 事实
- `control-plane-api` 当前新增 `GET /backend/v3/api/control/social/runtime/dead-letter-shared-channel-sync`
- inventory item 当前显式输出：
  - `requestKey`
  - `request`
  - `failureCount`
  - `lastError`
- `requestKey` 当前与 `shared_channel_sync_request_key(...)` 使用同一稳定 key 规则：
  - `tenant_id`
  - `conversation_id`
  - `shared_channel_policy_id`
  - `external_connection_id`
  - `local_actor_id`
  - `local_actor_kind`
  - `external_member_id`
- `control-plane-api` 当前新增 `POST /backend/v3/api/control/social/runtime/requeue-dead-letter-shared-channel-sync-targeted`
- `SocialControlState::requeue_selected_dead_letter_shared_channel_sync_requests(...)` 当前会：
  - 只迁移被选中的 dead-letter request
  - 把被选中 request 的 `failureCount` 重置为 `0`
  - 保留原 request payload 与 `lastError`
  - 保持未被选中的 dead-letter request 继续隔离
- 现有 `requeue-dead-letter-shared-channel-sync` 全量 route 仍保留，因此 operator 当前拥有“全量回灌 + 定点回灌”两条最小恢复路径

## 3. 明确边界
- 本轮没有：
  - inventory filter / pagination / search language
  - automatic retry scheduler
  - remote republish ownership / lease / SLA
  - release-ready exactly-once delivery semantics
- 当前仍是 `control-plane-api` 单体内的 operator ownership boundary，而不是 release-ready cross-service delivery orchestration
