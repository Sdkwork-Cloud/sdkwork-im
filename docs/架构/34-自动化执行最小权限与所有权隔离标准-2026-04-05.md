# 自动化执行最小权限与所有权隔离标准 - 2026-04-05

## 1. 问题定义

本轮 review 发现 `automation-service` 与 `sdkwork-im-server` 聚合自动化入口存在两个高风险缺陷：

- `POST /im/v3/api/automation/executions` 只做 Bearer 认证，没有最小权限校验，任意普通登录用户都可以触发 workflow / webhook 级自动化执行。
- 自动化执行记录只按 `tenant_id + execution_id` 存取，没有 owner principal 维度。同租户中，其他具备读取入口的主体只要知道 `executionId`，就可能读到不属于自己的执行结果。

这两个缺陷在多租户 SaaS 场景中会直接削弱租户内最小授权边界，也会破坏 user / bot / agent / workflow 之间的主体隔离。

## 2. 标准结论

### 2.1 自动化执行最小权限

对外 HTTP / Gateway 入口统一按以下最小权限收口：

- `POST /im/v3/api/automation/executions`
  - 必须具备 `automation.execute`
- `GET /im/v3/api/automation/executions/{execution_id}`
  - 必须具备 `automation.read`

权限来源统一继承现有 `AuthContext` 规则，可来自：

- JWT claims: `permissions` / `perms` / `scope` / `scp`
- trusted headers: `x-permissions` / `x-scope` / `x-scopes`

通配符仍按统一权限模型生效：

- `*`
- `tenant.admin`
- `automation.*`

未满足权限时必须返回：

- `403`
- `code = permission_denied`

### 2.2 所有权隔离

自动化执行记录必须按如下逻辑隔离：

- 请求体不接受 `tenantId`、`principalId`、`principalKind`
- `tenant_id`、`principal_id`、`principal_kind` 一律从认证上下文推导
- 执行记录的主隔离键至少包含：
  - `tenant_id`
  - `principal_id`
  - `execution_id`

推荐逻辑键：

- `tenant_id + principal_id + execution_id`

推荐事件 / 响应负载显式保留：

- `principalId`
- `principalKind`

这样可以同时满足：

- 执行结果按 owner principal 隔离
- 审计与事件回放能够还原执行归属
- 后续扩展到 `user` / `agent` / `bot` / `workflow` / `system` 时不需要再改数据模型

### 2.3 跨主体读取行为

当调用方读取不属于自己的 `executionId` 时：

- 返回 `404 automation_execution_not_found`

原因：

- 避免把 `executionId` 变成同租户内的枚举探针
- 与媒体资源隔离策略保持一致
- 将“无权限读取”与“资源不存在”合并为不可区分的外部表现

## 3. 落地要求

本标准必须同时落在两个入口层级：

- 独立服务：`services/automation-service`
- 聚合节点：`services/sdkwork-im-gateway`

实现上优先收敛到共享 runtime，而不是只在单个 handler 打补丁。这样做的原因是：

- `sdkwork-im-server` 直接复用 `AutomationRuntime`
- 只有把权限和 owner 隔离收进 runtime，独立服务与聚合节点才能共享同一套安全边界
- 后续如果增加私有化 profile / 管理面 profile，也不会因为入口差异重新引入漏洞

## 4. 本轮实现要求

本轮标准落地后，至少要满足以下行为：

1. 普通 Bearer 调用自动化执行接口返回 `403 permission_denied`
2. 具备 `automation.execute` 的 Bearer 可以提交自动化执行
3. 仅具备 `automation.execute` 而不具备 `automation.read` 的 Bearer，不可查询执行详情
4. A principal 创建的执行，B principal 即便具备 `automation.read` 也不能读取，返回 `404 automation_execution_not_found`
5. `sdkwork-im-server` 聚合入口与独立 `automation-service` 行为保持一致
6. 自动化完成后生成的 `automation.result` 通知仍然只投递给当前调用者自身
