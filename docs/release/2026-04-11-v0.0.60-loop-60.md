# v0.0.60 - Loop 60 - 2026-04-11

## 本轮目标
- 把 `shared-channel sync` 从“dead-letter 只有全量 requeue，缺少 inventory 与定点恢复契约”推进到“operator 可读取 dead-letter inventory，并按 `requestKey` targeted requeue”的最小可审阅边界

## 本轮实现
- `services/control-plane-api/src/lib.rs`
  - 新增 `GET /api/v1/control/social/runtime/dead-letter-shared-channel-sync`
  - inventory item 当前显式返回 `requestKey / request / failureCount / lastError`
  - 新增 `POST /api/v1/control/social/runtime/requeue-dead-letter-shared-channel-sync-targeted`
  - `requeue_selected_dead_letter_shared_channel_sync_requests(...)` 当前允许按 `requestKey` 只回灌被选中的 dead-letter request
  - 被选中 request 在 targeted requeue 时会把 `failureCount` 重置为 `0`，同时保留 `lastError`
  - 未被选中的 dead-letter request 会继续停留在 `dead_letter_shared_channel_sync_requests`
  - 现有“全量 dead-letter requeue” route 保持不变
- `services/control-plane-api/tests/social_external_collaboration_test.rs`
  - 新增 dual-service e2e，证明 inventory 可列出两条 dead-letter request 的稳定 `requestKey`
  - targeted requeue 后只有被选中的 actor 恢复 shared history 可读；未被选中的 actor 仍然 `403`

## 风险与未闭环
- 当前 dead-letter inventory 还没有 filter / pagination / search contract
- 当前 targeted requeue 会保留 `lastError`，只重置 `failureCount`
- 当前仍没有 automatic retry scheduler
- 当前没有 remote republish ownership / lease / SLA，也没有 release-ready exactly-once semantics
- `https://` public runtime target 仍未进入本轮 truth

## 结论
- `S07` 仍是 `not_closed / local_closure`
- Loop60 已把 `shared-channel sync` 补齐到 `durable pending backlog + operator repair + next healthy ready-pair write auto-retry + repeated-failure dead-letter + dead-letter inventory + dead-letter targeted/all requeue + failure-budget rearm`
- 下一主缺口收敛为 `remote republish / delivery ownership semantics`
