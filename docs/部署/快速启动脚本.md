## 2. 职责边界

- `install-local`
  - 检查 `cargo`
  - 创建 `.runtime/local-minimal` 下的 config、logs、pids 目录
  - 离线构建 `local-minimal-node`
- `init-config-local`
  - 初始化或重写 `local-minimal.env`
  - 固化 `CRAW_CHAT_BIND_ADDR`、`CRAW_CHAT_RUNTIME_DIR` 与本地 bearer secret
- `start-local`
  - 自动先执行 `install-local`
  - 写出日志文件与 PID 文件
  - 默认后台启动，可切换前台模式
  - 等待 `healthz` 成功后才返回
- `status-local`
  - 输出配置文件、bind 地址、日志路径、PID、health 结果
  - 给出后续 `inspect/repair/list/archive/prune/preview/restore` 操作入口
- `restart-local`
  - 严格串联 `stop-local` 与 `start-local`
  - 只有停止成功后才允许启动新实例
- `stop-local`
  - 只停止受管 `local-minimal-node`
  - 清理 PID 文件并保留其他非受管进程安全
- `deploy-local`
  - 用于 Docker 本地部署
  - Windows 复用 `deployments/scripts/bootstrap-local.ps1`
  - 支持 `local-minimal` / `local-default` 两个 profile 入口
  - PowerShell / CMD 使用 `-ProfileName <local-minimal|local-default>`
  - Bash 使用 `--profile <local-minimal|local-default>`
  - 需要重复执行非默认端口 smoke 时，可显式传入 `-SmokeBaseUrl <url>` / `--smoke-base-url <url>`
  - smoke 脚本会优先从 `CRAW_CHAT_PUBLIC_BEARER_HS256_SECRET` 或本地 config 生成 HS256 bearer；Docker `local-minimal` profile 内置 dev secret `local-minimal-public-dev-secret`
  - 当前 `local-default` 仍复用 `local-minimal` 的 compose 服务合同与 smoke 脚本
- `open-chat-test`
  - 打开 chat 测试窗口，验证本地闭环可交互
- `inspect-runtime-local` / `repair-runtime-local`
  - 面向运行目录检查与缺失文件修复
- `list-runtime-backups-local` / `archive-runtime-backup-local` / `prune-runtime-archives-local`
  - 面向快照目录盘点、归档与保留期清理
- `preview-runtime-restore-local` / `restore-runtime-local`
  - 面向恢复前预演与显式恢复执行
- 运行时运维脚本同样支持 `--profile <local-minimal|local-default>` / `-ProfileName <local-minimal|local-default>`
  - 选择 `local-default` 时优先读取 `.runtime/local-default/config/local-default.env`；若尚未落地，则回退到 `.runtime/local-minimal/config/local-minimal.env` 与 `.runtime/local-minimal` 运行目录合同

## 3. 推荐执行顺序

### 3.1 本地进程闭环

1. `install-local`
2. `init-config-local`
3. `start-local`
4. `status-local`
5. `open-chat-test`
6. `inspect-runtime-local`
7. 需要时执行 `repair-runtime-local`、`list-runtime-backups-local`、`archive-runtime-backup-local`
8. 恢复演练时先执行 `preview-runtime-restore-local`，再执行 `restore-runtime-local`
9. 例行治理执行 `prune-runtime-archives-local`
10. 需要重载时执行 `restart-local`
11. 收尾执行 `stop-local`

### 3.2 Docker 闭环

1. `deploy-local`
   - 默认执行 `local-minimal`
   - 需要兼容入口时可显式选择 `local-default`
2. `status-local` 或 `tools/smoke/local_stack_smoke.*`
3. `open-chat-test`
4. 需要排障时切回运行时运维闭环命令

## 5. 运行产物

脚本会在项目根目录生成以下运行时产物：

- `.runtime/local-default/config/local-default.env`
- `.runtime/local-minimal/config/local-minimal.env`
- `.runtime/local-minimal/logs/local-minimal-node.out.log`
- `.runtime/local-minimal/logs/local-minimal-node.err.log`
- `.runtime/local-minimal/pids/local-minimal-node.pid`
- `.runtime/local-minimal/backups/`
- `.runtime/local-minimal/archive/`

当前 `local-default` 仍复用 `.runtime/local-minimal` 运行目录合同，因此 logs、pid、backup、archive 产物仍落在 `.runtime/local-minimal/`。

## 7. 设计约束

- `bin/` 只暴露统一入口，不复制已有编排逻辑
- Windows / PowerShell / Bash 只允许参数风格不同，不允许语义分叉
- Docker 部署继续复用既有 bootstrap 与 compose profile
- 本地进程启动继续以 `local-minimal-node` 为唯一运行目标
- 运行目录统一落在 `.runtime/`，便于后续扩展到私有化与多环境 profile
## 8. Windows Help Contract

- `bin\start-local.cmd --help` must surface the Windows GNU-style flags `--release`, `--foreground`, and `--bind-addr <host:port>`.
- `bin\status-local.cmd --help` must surface the Windows GNU-style flags `--profile <local-minimal|local-default>` and `--runtime-dir <path>`.
- This help output is part of the visible operator contract for the `.cmd` launcher and must stay aligned with the quick-start examples in this document.

```cmd
bin\start-local.cmd --help
bin\status-local.cmd --help
```
