# 停止与重启安全切换标准-2026-04-05

## 1. 目标

统一 `bin/stop-local.*` 与 `bin/restart-local.*` 的跨平台退出语义，避免出现“脚本删除 PID 并返回成功，但旧进程尚未真正退出”的竞态，从而导致：

- restart 抢占旧端口失败
- 状态脚本误判服务已停止
- 运维脚本在切换过程中产生双实例或脏 PID

## 2. 停止脚本标准

### 2.1 共性要求

- `stop-local.*` 不能在仅仅发出终止信号后就立即删除 PID 文件。
- 只有确认目标进程已经退出后，才允许删除 PID 文件并输出 `local-minimal-node stopped.`。
- 如果超时仍未退出，脚本必须返回非零退出码，并保留 PID 文件，避免把仍然存活的进程伪装成 stopped。

### 2.2 `stop-local.ps1`

- 使用 `Stop-Process` 发送停止信号后，必须调用 `Wait-Process` 等待退出。
- 建议超时时间统一为 `30` 秒。
- 若等待超时且进程仍存在，必须抛出明确错误，例如：
  - `local-minimal-node PID <pid> did not exit within 30 seconds.`

### 2.3 `stop-local.sh`

- 使用 `kill` 发送停止信号后，必须轮询 `kill -0 <pid>` 直到进程消失。
- 建议轮询上限统一为 `30` 秒。
- 若超时仍未退出，必须返回非零退出码并输出明确错误，例如：
  - `local-minimal-node PID <pid> did not exit within 30 seconds.`

## 3. 重启脚本标准

### 3.1 `restart-local.*`

- `restart-local.*` 的 stop 阶段必须继承 `stop-local.*` 的等待退出语义。
- 只有在旧进程真正退出后，才能继续执行 `start-local.*`。
- 不允许通过“删除 PID 文件后立即启动新进程”实现伪重启。

## 4. 状态一致性要求

- `stop-local.*` 成功返回后，`status-local.*` 必须能观测到：
  - `status: stopped`
  - `health status: stopped`
- 这是脚本成功语义的一部分，而不是附加信息。

## 5. 回归要求

以下行为必须长期保留在自动化验证中：

1. `stop-local.ps1` 必须包含等待退出逻辑，例如 `Wait-Process`。
2. `stop-local.sh` 必须包含超时轮询逻辑，例如 `kill -0` + `sleep`。
3. 停止脚本超时失败时不得提前删除 PID 文件。
4. Windows 真实链路上，`start-local.ps1 -> stop-local.ps1 -> status-local.ps1` 必须得到 `status: stopped` 与 `health status: stopped`。

## 6. 适用范围

- `bin/stop-local.ps1`
- `bin/stop-local.sh`
- `bin/stop-local.cmd`
- `bin/restart-local.ps1`
- `bin/restart-local.sh`
- `bin/restart-local.cmd`

后续若扩展为 systemd、launchd、Windows Service、容器 preStop 钩子，也必须继承这一套“确认退出后再切换”的标准。
