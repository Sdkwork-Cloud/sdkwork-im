# 跨平台启动与状态健康探测标准-2026-04-05

## 1. 目标

统一 Windows PowerShell、Windows CMD 包装层、Linux/macOS Bash 三套本地生命周期脚本的健康探测语义，避免出现“脚本宣告启动成功，但服务实际上未就绪”或“状态脚本声称展示健康状态，实际没有探测”的问题。

## 2. 启动脚本标准

### 2.1 `start-local.ps1`

- 后台启动完成后，必须轮询 `http://<resolved-host>:<port>/healthz`。
- 只有收到 `200 OK` 才允许返回成功。
- 探测失败或超时必须返回非零退出码。

### 2.2 `start-local.sh`

- 后台启动完成后，必须执行真实健康探测。
- 健康探测优先级：
  1. `curl`
  2. `wget`
- 如果 `curl` 与 `wget` 都不存在，脚本必须直接失败并给出明确错误，而不是把服务标记为 ready。
- 不允许因为缺少探测工具就把 `ready=1` 作为降级路径。

## 3. 状态脚本标准

### 3.1 `status-local.ps1`

- 必须输出：
  - `health: http://<resolved-host>:<port>/healthz`
  - `health status: <state>`
- 当进程存在且 `/healthz` 返回 `200` 时，`health status` 必须为 `ok`。
- 当进程不存在时，`health status` 必须为 `stopped`。
- 当进程存在但探针失败时，`health status` 必须为 `unreachable`。

### 3.2 `status-local.sh`

- 必须输出：
  - `health: http://<resolved-host>:<port>/healthz`
  - `health status: <state>`
- 状态值约束：
  - `ok`: 进程存在且探针成功
  - `stopped`: 进程不存在
  - `unreachable`: 进程存在但探针失败
  - `probe-unavailable`: 进程存在，但当前环境没有 `curl` 和 `wget`

## 4. 回归要求

以下行为必须被长期保留：

1. `start-local.sh` 必须包含 `curl` 与 `wget` 两级探测逻辑。
2. `start-local.sh` 在缺少探测工具时必须失败，不能伪造 ready 状态。
3. `status-local.ps1` 必须在真实运行链路中输出 `health:` 和 `health status: ok`。
4. `status-local.sh` 必须与 PowerShell 入口维持等价的健康状态输出约束。

## 5. 适用范围

- `bin/start-local.ps1`
- `bin/start-local.sh`
- `bin/status-local.ps1`
- `bin/status-local.sh`
- `bin/*.cmd` 包装层

后续若增加 `systemd`、`launchd`、Windows Service 安装脚本，也必须继承同一套健康探测与状态输出标准。
