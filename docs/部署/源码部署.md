# SDKWork Chat 源码部署

本文档说明如何在服务器上通过 `pnpm` 直接从源码构建和启动 Craw Chat，不再每次生成安装包或压缩包。

源码部署仍然需要构建。区别是构建产物留在源码目录中：

- 前端产物：`apps/sdkwork-chat-pc/dist`
- 后端二进制：`target/release/craw-chat-server`
- 运行配置：`/etc/sdkwork/chat/chat.toml` 和 `/etc/sdkwork/chat/server.env`

不要把 `pnpm server:dev` 用作生产启动命令。`server:dev` 是本地开发栈，会启动开发用的网关、本地业务 host、调试构建和本地默认端口。

## 服务器准备

服务器需要具备：

- Git
- Node.js 和 Corepack
- `pnpm@10.0.0`
- Rust toolchain
- PostgreSQL
- Redis

推荐源码目录：

```sh
sudo mkdir -p /opt/sdkwork/chat
sudo chown -R "$USER:$USER" /opt/sdkwork/chat
git clone <your-craw-chat-repo-url> /opt/sdkwork/chat
cd /opt/sdkwork/chat
corepack enable
corepack prepare pnpm@10.0.0 --activate
```

## 配置文件

生产配置放在服务器本机，不提交到 Git：

```text
/etc/sdkwork/chat/chat.toml
/etc/sdkwork/chat/server.env
/etc/sdkwork/chat/database.secret
/etc/sdkwork/chat/redis.secret
```

可以从 `deployments/templates/server.env.example` 复制一份到 `/etc/sdkwork/chat/server.env` 后修改。

源码部署建议至少设置这些字段：

```env
SDKWORK_CHAT_DEPLOYMENT_MODE=server
SDKWORK_CHAT_CONFIG_FILE=/etc/sdkwork/chat/chat.toml
SDKWORK_CHAT_SERVER_BIND=0.0.0.0:18080
SDKWORK_CHAT_SERVER_BASE_URL=https://chat.example.com/sdkwork/chat
SDKWORK_CHAT_SERVER_API_BASE_URL=https://chat.example.com/sdkwork/chat
SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL=wss://chat.example.com/sdkwork/chat

CRAW_CHAT_BROWSER_ORIGINS=https://chat.example.com
CRAW_CHAT_ADMIN_SITE_DIR=/opt/sdkwork/chat/apps/sdkwork-chat-pc/dist
CRAW_CHAT_PORTAL_SITE_DIR=/opt/sdkwork/chat/apps/sdkwork-chat-pc/dist
CRAW_CHAT_SERVER_BINARY_PATH=/opt/sdkwork/chat/target/release/craw-chat-server

SDKWORK_CHAT_DATABASE_ENGINE=postgresql
SDKWORK_CHAT_DATABASE_HOST=db.example.com
SDKWORK_CHAT_DATABASE_PORT=5432
SDKWORK_CHAT_DATABASE_NAME=sdkwork_chat_prod
SDKWORK_CHAT_DATABASE_SCHEMA=sdkwork_chat_prod
SDKWORK_CHAT_DATABASE_USERNAME=sdkwork_chat_prod
SDKWORK_CHAT_DATABASE_PASSWORD_FILE=/etc/sdkwork/chat/database.secret
SDKWORK_CHAT_DATABASE_SSL_MODE=require
SDKWORK_CHAT_DATABASE_MAX_CONNECTIONS=20

SDKWORK_CHAT_REDIS_ENABLED=true
SDKWORK_CHAT_REDIS_HOST=redis.example.com
SDKWORK_CHAT_REDIS_PORT=6379
SDKWORK_CHAT_REDIS_DATABASE=0
SDKWORK_CHAT_REDIS_PASSWORD_FILE=/etc/sdkwork/chat/redis.secret
SDKWORK_CHAT_REDIS_KEY_PREFIX=chat
```

`SDKWORK_CHAT_SERVER_API_BASE_URL` 和 `SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL` 是生产 base URL 的核心来源。源码构建命令会读取 `server.env`，前端构建时会把这些地址写进 Vite bundle。修改域名、路径前缀或 websocket 地址后，必须重新运行 `pnpm run server:source:build`。

## 构建

先查看计划：

```sh
cd /opt/sdkwork/chat
pnpm run server:source:plan -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat
```

执行源码构建：

```sh
cd /opt/sdkwork/chat
pnpm run server:source:build -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat
```

这条命令会复用现有生产构建流程：

```sh
pnpm run release:build:production -- --target server
```

它不会调用 `release:package`，不会生成安装包。

## 启动

前台启动，适合 systemd、Docker entrypoint 或手动验证：

```sh
cd /opt/sdkwork/chat
pnpm run server:source:start -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat
```

后台启动：

```sh
cd /opt/sdkwork/chat
pnpm run server:source:start -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat --background
```

`server:source:start` 不直接运行 `cargo run`。它会复用 `bin/start-server.sh` 或 Windows 上的 `bin/start-server.ps1`，并默认指向源码目录中的 release 二进制。

Linux 默认二进制路径：

```text
/opt/sdkwork/chat/target/release/craw-chat-server
```

Windows 默认二进制路径：

```text
target\release\craw-chat-server.exe
```

## systemd 示例

`/etc/systemd/system/craw-chat-source.service`：

```ini
[Unit]
Description=Craw Chat Source Deployment
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/sdkwork/chat
EnvironmentFile=/etc/sdkwork/chat/server.env
ExecStart=/usr/bin/pnpm run server:source:start -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat
Restart=always
RestartSec=5
User=sdkwork
Group=sdkwork

[Install]
WantedBy=multi-user.target
```

启动：

```sh
sudo systemctl daemon-reload
sudo systemctl enable craw-chat-source
sudo systemctl start craw-chat-source
```

更新源码并重启：

```sh
cd /opt/sdkwork/chat
git pull --ff-only
pnpm run server:source:build -- --env-file /etc/sdkwork/chat/server.env --config-dir /etc/sdkwork/chat
sudo systemctl restart craw-chat-source
```

## Base URL 调整规则

源码部署下，后端 base URL 需要分清两类：

- 构建期公开地址：`SDKWORK_CHAT_SERVER_API_BASE_URL`、`SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL`
- 运行期服务地址：`SDKWORK_CHAT_SERVER_BIND`、`CRAW_CHAT_WEB_GATEWAY_BIND`、`CRAW_CHAT_SERVER_API_BASE_URL`

一般单域名部署使用：

```env
SDKWORK_CHAT_SERVER_API_BASE_URL=https://chat.example.com/sdkwork/chat
SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL=wss://chat.example.com/sdkwork/chat
CRAW_CHAT_BROWSER_ORIGINS=https://chat.example.com
```

拆分 websocket 域名时使用：

```env
SDKWORK_CHAT_SERVER_API_BASE_URL=https://chat.example.com/sdkwork/chat
SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL=wss://realtime.example.com/sdkwork/chat
CRAW_CHAT_BROWSER_ORIGINS=https://chat.example.com
```

不要在生产前端中继续使用 `http://127.0.0.1:*` 作为 API base URL。只有开发命令可以使用本地地址。

## 常见问题

### 是否完全不用打包

可以不用生成 release 包，但不能跳过构建。前端需要 Vite 产物，Rust 服务需要 release 二进制。

### 只改后端代码要不要重建前端

不需要。只改 Rust 后端时，重新运行 Cargo 构建即可；用 `pnpm run server:source:build` 也可以，但会多跑一次前端构建。

### 只改域名或 websocket 地址要不要重建前端

需要。前端 bundle 会读取构建期 base URL，修改 `SDKWORK_CHAT_SERVER_API_BASE_URL` 或 `SDKWORK_CHAT_SERVER_WEBSOCKET_BASE_URL` 后必须重新构建。

### 为什么不用 `pnpm server:dev`

`pnpm server:dev` 是开发栈，会启动本地最小服务和开发网关，并使用开发端口回退逻辑。生产源码部署使用 `pnpm run server:source:build` 和 `pnpm run server:source:start`。
