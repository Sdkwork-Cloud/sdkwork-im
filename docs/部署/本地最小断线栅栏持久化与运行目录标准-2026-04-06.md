# 本地最小断线栅栏持久化与运行目录标准（2026-04-06）

## 1. 目标

`local-minimal` 私有化最小部署在不依赖外部基础设施的前提下，需要保证 `session.disconnect` 产生的 reconnect-required 栅栏可以跨进程重启恢复。

## 2. 运行目录

运行目录固定为：

```text
.runtime/local-minimal/
```

子目录包括：

- `config/`
- `logs/`
- `pids/`
- `state/`

断线栅栏文件固定为：

```text
.runtime/local-minimal/state/realtime-disconnect-fences.json
```

## 3. 配置项

`local-minimal.env` 至少包含：

- `CRAW_CHAT_BIND_ADDR`
- `CRAW_CHAT_RUNTIME_DIR`
- `CRAW_CHAT_FRIEND_REQUEST_CURSOR_HS256_SECRET`

其中：

- `CRAW_CHAT_RUNTIME_DIR` 必须指向 `.runtime/local-minimal`
- `start-local.*` 启动前必须把它注入进程环境

## 4. 启动约束

推荐启动方式：

- Windows: `bin/start-local.ps1`
- Linux/macOS: `bin/start-local.sh`

这些脚本现在不仅负责监听地址和密钥，还负责把运行目录传递给 `local-minimal-node`，从而激活本地磁盘断线栅栏持久化。

## 5. 语义约束

一旦 `session.disconnect` 成功：

1. 断线栅栏必须写入 runtime `state/` 文件
2. 进程重启后，旧会话的设备请求仍然必须被 `reconnect_required` 拒绝
3. 只有新的 `session.resume` 才能清除该栅栏

## 6. 失败语义

如果本地状态文件不可读、不可写或损坏：

- 服务必须返回受控错误
- 不能直接 panic 退出
- HTTP 对外返回 `503 Service Unavailable`
- 错误码为 `disconnect_fence_store_unavailable`

## 7. 非目标

本标准只覆盖断线栅栏持久化，不覆盖：

- realtime checkpoint 完整恢复
- route ownership epoch 恢复
- 更深层的 crash-safe route migration
